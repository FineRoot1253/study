# 스프링 시큐리티와 테스트는 어떻게 구성할까


<br><br><br>

## 스프링에서 테스트하기

&emsp;&emsp;

- 개요
	- 테스트 구성
	- 테스트 작성 진행 과정
	- 테스트 구현 방법 with Spring Security


<br><br><br>

## 테스트 구성

&emsp;&emsp;

- 설명
	- 컨트롤러 테스트 (API 인터페이스 테스트)
		- `@WebMvcTest`
		- `@AutoConfigureRestDocs`
	- 서비스 테스트 (요구사항 기반 테스트)
	- 레포지터리 테스트 (실제 레포지터리 기능 테스트)
		- `@DataJpaTest`


<br><br><br>


## 테스트 작성 진행 과정

&emsp;&emsp;

- 설명
	1. 티켓에 적힌 요구사항을 기반으로 서비스 테스트를 먼저 구성한다.
		1. 필요한 객체들을 만들어 그것들에 대한 요구사항이 존재하면 그것에 대한 테스트 또한 이때 작성해준다.
	2. 필요한 프로덕트 코드를 구현한다.
	3. 프로덕트 구현시 테스트가 필요한 레포지터리 기능을 테스트 코드로 작성한다.
	4. 레포지터리까지 테스트가 완료되면 이제 클라이언트와 연동할 컨트롤러 테스트를 작성한다.
		1. 정 급하면 컨트롤러 테스트를 빼놓을 수도 있긴하겠지만, 여기에 추가적인 요구사항이 섞여 있는 경우가 허다하고 이를 통해 문서 자동화도 구축해뒀기 때문에 협업에 필요하다.


<br><br><br>

## 테스트 구현 방법 with Spring Security

&emsp;&emsp;

- 설명
	- 서비스 테스트 구현 방법
		- 이 경우에는 Junit5 작성 방법을 찾아보면 나오는 것과 크게 다르지 않다.
	- 레포지터리 테스트 구현 방법
		- `BaseRepositoryTestConfigurer`추상 클래스를 구성한다.
		- 어노테이션 `@DataJpaTest`, `@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)`, `@Import(RepositoryTestConfig.class)`, `@Transactional`를 달아준다.
		- `BaseRepositoryTestConfigurer`를 상속 받아 테스트 클래스를 구성하여 테스트한다.
		- Auditing 관련으로 테스트가 제대로 동작하지 않는 문제가 있다.
			- 주로 null로 채워지는 것이 문제인데, SecurityContext 슬라이스를 뺀 셋업이므로 당연히 null로 채워진다.
			- 따라서 임의로 생성자에 집어 넣거나 테스트에  AuditingHandler를 주입받아 setAuditorAware에 세팅될 유저 ID를 집어 넣는 것도 방법이다.
			- 더 간단하게 쓸려면 어노테이션을 만들고, 어노테이션에 ID를 지정하고,  BeforeEachCallback, AfterEachCallback를 구현한 클래스를 만들어 어노테이션에서 UserId를 때어다가 AuditingHandler에 세팅해줘도 된다.
			- 이 방식은 쓰기는 편한데 유지보수는 복잡할거라 직접 테스트마다 주입받는 걸 추천한다.
	- 컨트롤러 테스트 구현 방법
		- 이 구성 방법은 우테코 프로젝트를 배껴서 만든게 한 90프로 이상이다.
		- `BaseControllerTestConfigurer` 추상 클래스를 구성한다.
			- `objectMapper`, `mockMvc`, `restDocumentationResultHandler`를 각각 주입받는다.
		- `@BeforeEach`애너테이션으로 mockMvc를 구성한다.
			- 깃랩 링크를 참고하자.
		- `@WebMvcTestSlice` 커스텀 어노테이션을 생성한다.
			- WebMvcTest 어노테이션을 상속받아 구현하는데 이때 필요없는 컴포넌트를 제외시켜야한다.
			- 자세한 구현은 깃랩 링크를 참고하자.
		- `@WithMockUser` 커스텀 어노테이션을 생성한다.
			- `@WithSecurityContext`애너테이션에  `WithMockCustomUserSecurityContextFactory`직접 구현한 이 클래스를 집어넣어준다.
			- 이 `WithMockCustomUserSecurityContextFactory` 클래스는 SecurityContext를 직접 구현한 클래스이며 스프링 시큐리티에서 유저 정보를 구성할때 사용되는 펙토리 클래스이다.
			- 이 펙토리 클래스가 가장 중요한데, 그 이유는 정확히 필요한 유저만 모킹하여 집어 넣기 위함이다.
				- 만약 그렇지 않으면 개발한 시큐리티 필터에 주입이 필요한 필요한 객체들을 전부 mock으로 집어넣어야 하며 이를 또 테스트하는 것이 상당히 빡시다.
				- 이렇게 구현하는게 훨씬 간편하고 쉽게 시큐리티를 의존하는 컨트롤러를 테스트하기 가장 괜찮은 방법이였다.
			- 더 자세한 구현은 깃랩 링크를 확인하자.
