# 쿼리 꿀팁: COUNT


<br><br><br>

## COUNT(expr)

&emsp;&emsp;

- 설명
	- 정의
		- 테이블 행의 수를 세는 집계 함수
	- 특징
		- `COUNT(*)`
			- 테이블 행의 총 수를 의미한다.
			- NULL인지 아닌지 검사하진 않는다.
			- 테이블의 인덱스를 전부 포함해서 행의 개수를 센다.
		- `COUNT(컬럼)` 
			- 컬럼을 읽어 들이면서 행의 수를 집계한다.
			- NULL이 아닌 값은 읽어들이지 않는다는다. 
		- `COUNT(1)`
			- MyISAM 엔진을 사용하는 레거시 최적화 방법으로 첫번째 행이 NOT NULL인 경우 카운팅하는 방식
			- https://dev.mysql.com/doc/refman/8.4/en/aggregate-functions.html#function_count
		- 속도 차이 정리
			- `COUNT(*) = COUNT(1) > COUNT(컬럼)`


<br><br><br>

## InnoDB vs MyISAM 동작 구조 차이

&emsp;&emsp;

- 설명
	- MyISAM
		- 행의 개수를 따로 저장해두기 때문에 WHERE문없이 `COUNT(*)`쿼리를 날리면 알아서 최적화된 쿼리를 날리지만 WHERE문을 붙이면 최적화가 되지 않음 이때 파일을 따로 뒤져보는 동작이 수반됨
		- 이때 첫번째 컬럼을 NOT NULL 컬럼으로 두고 COUNT(1)으로 돌리면 최적화 이득을 볼 수 있다는 이야기가 위의 링크의 이야기이다.
	- InnoDB
		- 인덱스가 데이터 파일과 동시에 저장되기 때문에 `COUNT(*)`에 인덱스 커버드 쿼리로 날리면 인덱스를 돌려서 개수를 센다.

&emsp;&emsp;

- 이걸 찾아본 이유
	- 문득 테이블 설계 도중 count가 필요한 기획이 있는데 이거 어떻게 하더라 하고 찾아보니 내가 잘못 알고 있길래 정리했다.
	- 난 `*`을 넣으면 최적화가 안되는 줄 알았는데 그게 아니였다.
		- 데이터를 읽어들여 NULL검사를 하지 않고 한번에 행만 카운팅 한다는 의미이다.
		- 참고로 모든 인덱스를 함께 찾아보는 것이 `COUNT(*)`이며 일부 인덱스 컬럼만 집어 넣는 경우 덜 조회 될 수도 있다.

&emsp;&emsp;

- 정리
	-  `COUNT(*)`로 조회하고 WHERE 조건에 인덱스 기반 조건만 잘 맞춰주기
	- 정확한 집계 개수가 필요없다면 테이블 메타데이터를 활용하는 것이 가장 깔끔하다.
	- MySQL 책에서는 특히 페이징을 구현하는 경우 집계 쿼리 자체를 사용하는 것을 지양하도록 권장한다.
