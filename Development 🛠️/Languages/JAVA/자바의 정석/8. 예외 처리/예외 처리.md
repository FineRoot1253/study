## 예외 처리 : try - catch

+ 설명
	+ 예외 처리
		+ 프로그램 실행시 발생하는 예외에 대비한 코드를 작성하는 것
		+ 프로그램의 비정상 종료를 막고 정상적인 실행상태를 유지하기 위함

&emsp;&emsp;

### 예제

``` Java
try{
	// 예외가 발생할 가능성이 있는 문장들
} catch (Exception1 ex1){
	// Exception1 예외가 발생하는 경우, 이를 처리하기 위한 문장
} catch (Exception2 ex2){
	// Exception2 예외가 발생하는 경우, 이를 처리하기 위한 문장
} catch (ExceptionN exN){
	// ExceptionN 예외가 발생하는 경우, 이를 처리하기 위한 문장
}

```

+ 설명
	+ 하나의 try 블럭에는 여러 종류의 예외를 처리할 수 있는 하나 이상의 catch 블럭이 올 수 있다.
	+ 이 중 발생한 예외의 종류와 일치하는 단 한 개의 catch 블럭만 수행된다.
	+ 발생한 예외의 종류와 일치하는 catch가 없으면 예외는 처리되지 않고 상위로 던져진다.

&emsp;&emsp;

### 예제

``` Java
package ch8;  
  
import org.junit.jupiter.api.DisplayName;  
import org.junit.jupiter.api.Test;  
  
public class ExceptionEx1Test {  
    @Test  
    @DisplayName("try catch를 해본다.")  
    void tryCatch(){  
        try {  
            try {  
                  
            }catch (Exception e){  
                  
            }  
        }catch (Exception e){  
//            try {  
//  
//            }catch (Exception e){ //            에러가 발생한다.! 변수가 중복이기 때문  
//  
//            }  
        }  
  
        try {  
        }catch (Exception e){  
  
        }  
    }  
}
```

+ 설명
	+ 이 예제는 보다시피 하는 일은 없다.
	+ 한 try 블럭 안에 중첩적으로 사용 가능하다.
	+ catch 블럭 내에서는 같은 참조 변수를 사용하면 안된다.
	+ 애초에 자바를 가지고 위의 예제에서 처럼 쓰면 안된다.
	+ try catch 블럭은 한 메서드에 하나만 넣어두는 것이 정석이고, 한 메서드는 10줄 이상을 초과하면 문제가 있는 메서드인 것이다.

&emsp;&emsp;

### 예제

``` Java

package ch8;  
  
import org.junit.jupiter.api.DisplayName;  
import org.junit.jupiter.api.Test;  
  
public class ExceptionEx2Test {  
    @Test  
    @DisplayName("divide by zero 예외를 던진다.")  
    void arithmeticEx(){  
        //given  
        int number = 100;  
        int result;  
        //when  
        for (int i = 0; i < 10; i++) {  
            result = number / (int) (Math.random() * 10);  
            System.out.println(result);  
        }  
        //then  
    }  
}

```

+ 설명
	+ 0~9 사이 정수를 나눈 결과를 출력하는 일을 10번 반복한다.
	+ 이것을 해결하면 아래와 같다.

&emsp;&emsp;

### 예제

``` Java

package ch8;  
  
import org.junit.jupiter.api.DisplayName;  
import org.junit.jupiter.api.Test;  
  
public class ExceptionEx3Test {  
    @Test  
    @DisplayName("divide by zero 예외를 던진다.")  
    void arithmeticEx(){  
        //given  
        int number = 100;  
        int result;  
        //when  
        for (int i = 0; i < 10; i++) {  
            try {  
                result = number / (int) (Math.random() * 10);  
                System.out.println(result);  
            }catch (ArithmeticException e){  
                System.out.println("0");  
            }  
        }  
        //then  
    }  
}

```

+ 설명
	+ 적절히 예외 처리한 예시이다.

## try-catch 흐름도

+ 설명
	+ try 블럭 내에 발생한 예외와 일치하는 catch 블럭이 있는지 맨 위에서부터 하나씩 찾아 내려간다.
		+ catch에 실패하면 해당 예외는 위로 던져진다.
	+ try 블럭 내에서 예외가 발생하지 않으면 catch 블럭은 무시된다.

&emsp;&emsp;

### 예제

``` Java

package ch8;  
  
import org.junit.jupiter.api.DisplayName;  
import org.junit.jupiter.api.Test;  
  
public class ExceptionEx4Test {  
  
    @Test  
    @DisplayName("catch 블럭의 흐름을 알아보자")  
     void checkCatch(){  
        System.out.println(1);  
        System.out.println(2);  
        try{  
            System.out.println(3);  
            System.out.println(4);  
        } catch (Exception e){  
            System.out.println(5);  
        }  
        System.out.println(6);  
    }  
}

```

+ 설명
	+ 예외가 발생하지 않아 5가 무시된다.

&emsp;&emsp;

### 예제

``` Java

package ch8;  
  
import org.junit.jupiter.api.DisplayName;  
import org.junit.jupiter.api.Test;  
  
public class ExceptionEx5Test {  
    @Test  
    @DisplayName("catch 블럭의 흐름을 알아보자")  
    void checkCatch(){  
        System.out.println(1);  
        System.out.println(2);  
        try{  
            System.out.println(3);  
            System.out.println(0/0);  
            System.out.println(4);  
        } catch (ArithmeticException e){  
            System.out.println(5);  
        }  
        System.out.println(6);  
    }  
}

```

+ 설명
	+ 강제로 예외를 발생시켜 5가 출력된다.

## 예외의 발생과 catch 블럭

+ 설명
	+ 예외 발생시 catch 블럭 괄호() 내에 선언된 참조변수의 종류와 생성된 예외 클래스의 인스턴스에 instanceof 연산자를 사용해 검사한다.
		+ 맨 위의 catch 블럭부터 true 결과를 만날때까지 내려간다.
	+ 참고로 Exception e를 넣으면 무슨 예외던지 다 걸리게 되어있다.
		+ 이건 대표적인 안티패턴이므로 무조건 삼가해야한다.
		+ 위에 세부적인 예외를 넣어두고 거기서 예외가 발생하면 당연히 Exception e 이렇게 걸어둔 catch 블럭은 무시된다.

&emsp;&emsp;

### printStackTrace(), getMessage()

+ 설명
	+ 예외가 발생했을 경우 위의 두 메서드로 정보를 얻을 수 있다.
	+ printStackTrace()
		+ 예외발생 당시의 호출스택에 있었던 메서드의 정보와 예외 메시지를 화면에 출력
	+ getMessage()
		+ 발생한 예외클래스의 인스턴스에 저장된 메시지를 얻을 수 있음

&emsp;&emsp;

### 예제

``` Java
package ch8;  
  
import org.junit.jupiter.api.DisplayName;  
import org.junit.jupiter.api.Test;  
  
public class ExceptionEx6Test {  
    @Test  
    @DisplayName("printStackTrace를 사용해보자")  
    void printStackTrace(){  
        System.out.println(1);  
        System.out.println(2);  
        try{  
            System.out.println(3);  
            System.out.println(0/0);  
            System.out.println(4);  
        } catch (ArithmeticException e){  
            e.printStackTrace();  
            System.out.println("예외메시지: " + e.getMessage());  
        }  
        System.out.println(6);  
    }  
}
```

+ 설명
	+ 프로그램은 catch 블럭에 의해 예외가 잘 처리되어서 정상적으로 종료된다.
	+ 참고로 printStackTrace()에 PrintStream을 파라미터로 넘기면 파일에 저장시킬 수도 있다.
		+ 허나 자바 특성상 제대로 자원회수를 시키는 것이 힘들기 때문에 이를 잘 고심해서 구현해야한다.

&emsp;&emsp;

### 멀티 catch 블럭

+ 설명
	+ catch 블럭의 ()괄호에 '|' 기호를 추가해 여러 개의 예외를 넣을 수 있다.
		+ 개수의 제한은 없다.
		+ 연결된 예외 클래스들이 상속관계라면 컴파일 에러가 발생한다.
	+ **instaceof를 사용해  어느 예외클래스인지 판단하여 해당 클래스의 메서드를 사용해야한다.**
		+ catch 블럭의 참조변수는 상수이기 때문에 값을 변경할 수 없다.
	+ 실제로는 사용할 일은 별로 없다.

## 예외 발생시키기

+ 설명
	+ 발생 방법
		1. 예외 클래스를 생성한다.
		2. throw 연산자로 발생시킨다.
	+ 일반적으로는 생성과 발생을 한줄에 넣어서 발생시킨다.
	+ 예외 던지기는 정말 자주 사용한다.
		+ 사전조건이 존재하는 모든 클래스에는 예외 던짐이 들어있다.
	+ **Exception 클래스와 그 자손들은 예외를 던질때 예외 처리를 해주지 않으면 컴파일 에러가 발생한다.**
	+ **RuntimeException 클래스와 그 자손들은 예외를 던질 때 예외 처리를 해주지 않아도 컴파일은 된다.**
		+ **하지만 예외가 발생하면 비정상적으로 종료가 된다.**
	+ **따라서 RuntimeException은 unchecked Exception, Exception 클래스들은 checked Exception이라고 부른다.**

## 메서드에 예외 선언

+ 설명
	+ 메서드 선언부에 throws 키워드를 사용하면 메서드 내에서 발생할 수 있는 예외를 여러 개 적어주어 처리할 수 있다.
	+ 장점
		+ **선언부에 예외를 선언하여 명시한 메서드는 클라이언트에서 사용할 때 쉽게 예외를 고려해서 처리해줄 수 있다.**
	+ 주의점
		+ Java API나 각종 라이브러리들의 API 문서를 보면 RuntimeException 클래스의 자손들에 대해서는 메서드에 예외 선언을 하지 않는 편이다.
			+ 그러므로 **반드시 사용할 api가 있다면 예외에 대해 전부 확인하고서 사용을 하는 것이 중요하다.**
		+ throws는 사실 상위 호출부에 예외를 전달해 예외 처리를 떠맡기는 것이다.
			+ **계속 위로 던지다가 main에서도 처리가 안되면 프로그램이 비정상 종료가 되는 것이다.**
			+ 사실상 병목현상에 대한 제약이론과 비슷한데, 처리해야할 과업은 반드시 처리되어야 한다. 따라서 처리해야할 예외는 반드시 try-catch 문으로 처리해주어야한다.
		+ 예외 처리는 호출을 하는 호출부에서나, 예외를 던지는 하위 계층의 어느 메서드에서든 처리를 할 수 있는데, 이는 어떤 방법을 택하든 상관없다.
			+ 예외의 처리 전략은 보통 사용자 요구사항에 따라 달라지는 것이 보편적이다.
				+ 대부분의 경우에 기획의 의도에 따라 달라진다는 뜻
			+ 깔끔하게 처리를 하기 위해서는 중간 레이어를 만들어 처리해 주는것도 방법이고 이를 위해 보통 프록시 클래스나, 탬플릿 메서드로 감싸주는 것도 방법이다.


## finally 블럭

+ 설명
	+ try - catch 블럭 사용시 예외의 발생 상관없이 반드시 실행해야할 코드를 포함시키는 목적으로 사용되는 블럭

&emsp;&emsp;

### 예제

``` Java
package ch8;  
  
public class FinallyTest3 {  
    public static void method() {  
        try {  
            System.out.println("method is invoked");  
            return;  
        }catch (Exception e) {  
            e.printStackTrace();  
        }finally {  
            System.out.println("method's finally block");  
        }  
    }  
}
////
package ch8;  
  
import org.junit.jupiter.api.DisplayName;  
import org.junit.jupiter.api.Test;  
  
public class FinallyTest3Test {  
    @Test  
    @DisplayName("method를 호출해 finally 호출 흐름을 파악한다.")  
    void callMethod(){  
        FinallyTest3.method();  
        System.out.println("method() 호출 이후 finally 블럭이 잘 실행되었읍니다.");  
    }  
}

```

+ 설명
	+ 반드시 finally는 return 직전, 맨 마지막에 실행된다.
	+ 주의점
		+ JVM상에서 자원 회수의 경우 finally를 사용하더라도 반드시 잘 되는 경우가 보통 없다.
			+ **그래서 try - with - resource 구문을 사용해 회수해야한다.**

## 자동 자원 반환 : try-with-resource

+ 설명
	+ jdk 1.7에 등장한 구문
	+ AutoCloseable 인터페이스을 구현한 클래스에 대해 자원 반환작업을 반드시 수행하도록 동작한다.

&emsp;&emsp;

### 예제

``` Java

package ch8;  
  
public class TryWithResourceEx implements AutoCloseable {  
  
    public void exceptionWork(boolean exception) throws WorkException {  
        System.out.println("exceptionWork("+exception+")!");  
        if (exception) {  
            throw new WorkException("Work exception!");  
        }  
    }  
  
    @Override  
    public void close() throws CloseException {  
        System.out.println("close()!");  
        throw new CloseException("Closed exception!");  
    }  
  
    class WorkException extends Exception {  
        WorkException(String message) {  
            super(message);  
        }  
    }  
  
    class CloseException extends Exception {  
        CloseException(String message) {  
            super(message);  
        }  
    }  
}

////

package ch8;  
  
import ch8.TryWithResourceEx.CloseException;  
import ch8.TryWithResourceEx.WorkException;  
import org.junit.jupiter.api.DisplayName;  
import org.junit.jupiter.api.Test;  
import org.junit.jupiter.params.ParameterizedTest;  
import org.junit.jupiter.params.provider.ValueSource;  
  
public class TryWithResourceExTest {  
    @ParameterizedTest  
    @ValueSource(booleans = {false, true})  
    @DisplayName("tryWithResource 구문 흐름에 대해 파악한다.")  
    void tryWithResource(boolean expected){  
        try(TryWithResourceEx tryWithResourceEx  = new TryWithResourceEx()){  
            tryWithResourceEx.exceptionWork(expected);  
        } catch (WorkException e){  
            e.printStackTrace();  
        } catch (CloseException e){  
            e.printStackTrace();  
        }  
        System.out.println();  
    }  
}

```

+ 설명
	+ 처음 반복의 경우
		+ close()의 예외만을 발생시킨다.
	+ 마지막 반복의 경우
		+ work와 close 예외 둘 다 발생시킨다.
		+ 이때 close()의 예외의 경우 stacktrace에 Suppressed라는 머릿말이 추가된다.
			+ WorkException이 실제로 터진 예외이고 따로 발생한 예외에 대해서는 이렇게 표현하는 것이다.
			+ 이 억제된 예외의 정보는 WorkException에 저장된다.
	+ 억제된 예외
		+ Throwable에 억제된 예외에는 void addSuppressed(Throwable exception), Throwable\[\] getSuppressed() 이렇게 두개의 메서드가 정의되어있으며 각각 억제된 예외를 추가하는 메서드와 억제된 예외 배열을 반환하는 메서드를 담당한다.
	+ 만약 기존 try - catch를 사용했다면 WorkException에 대해서는 무시된다. 
		+ 자원과 관련된 api에 대해서는 따라서 반드시 이 try-with-resource를 사용해주어야한다.

## 사용자정의 예외

+ 설명
	+ Exception 또는 RuntimeException을 상속받은 클래스를 구현해 커스텀된 클래스를 구현할 수 있다.
	+ 주의점
		+ 초기 자바는 소형 임베드 기기에 들어갈 용도로써 Exception을 상속받는 것을 권장했지만 자바가 웹으로 확장해가며 이 추세는 RuntimeException쪽으로 기울고 있다.
			+ 어느 예외를 상속받던지 반드시 처리해야하는 경우에는 Exception을 쓰고 그것이 아니면 RuntimeException을 쓰는 것이 보편적이다.
		+ 현업에서는 예외를 전부 만들어쓰기도 하고 아니면 예외 메시지 클래스만 만들고 경우에 따라 Ilegal류 RuntimeException을 사용하기도 한다.
			+ 이것은 코드 컨벤션에 맞춰서 하는 것이 좋은데 규모가 커질 수록 예외를 만들어 쓰는 것이 좋다.


<br><br>


## 예외 되던지기

+ 설명
	+ catch 블럭에서 예외클래스 참조변수를 다시 던져서 상위 호출부에서 처리하게끔 만들 수도 있다.
	+ 되던지는 이유는 예외가 발생한 메서드, 메서드를 호출한 호출부 양 쪽에서 예외를 처리해주어야하는 경우에 사용된다.
	+ 주의점
		+ **만약 try 블럭에 return이 있다면 catch 블럭에도 return을 넣어주어야한다.**
			+ 그러나 그냥 예외를 되던져 상위 호출부에서 처리하게 만들 수도 있다.
 

<br><br>

## 연결된 예외

+ 설명
	+ 예외는 연결될 수 있다.
		+ 예외 A가 예외 B를 발생시킨다면 A를 B의 원인 예외라고 부른다.
	+ Throwable에 있는 initCause() 사용해 특정 예외를 원인 예외로 등록할 수 있다.
		+ getCause()를 사용하면 원인 예외를 반환한다
	+ 사용목적
		+ 여러가지 예외를 하나의 큰 분류의 예외로 묶어서 다루기 위함이다.
			+ 상속을 이용해 예외 체계를 만들지 않는 이유
				1. 상속을 이용해 큰 분류의 예외를 만들고 이 예외를 상속받은 하위 클래스를 사용해도 실제로 큰 분류의 예외를 던지면 하위 클래스를 볼수가 없다.
				2. 상속을 이용하면 나중에 이 관계를 변경하는 것이 쉽지가 않다.
			+ 따라서 원인 예외를 제대로 표현하기 위해 이를 사용하는 것이다.
		+ checked 예외를 unchecked 예외로 되던지기 위함이다.
			+ 요즘에는 Exception으로 처리되어 예외가 던져도 호출부에서 할 수 있는게 없는 경우가 대부분인 경우가 많아졌다.
				+ 그래서 RuntimeException으로 되던져 상위 호출부를 깔끔하게 만들어줄 수 있다. (물론 처리는 해야한다.)
