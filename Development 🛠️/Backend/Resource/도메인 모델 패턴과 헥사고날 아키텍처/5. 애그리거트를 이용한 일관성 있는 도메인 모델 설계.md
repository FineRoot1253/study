## 애그리거트

&emsp;&emsp;

- 설명
	- 정의
		- DDD에 소개된 도메인 모델 구성 요소 또는 패턴 중 하나
	- 특징
		- ==**데이터 변경 목적을 위해 하나의 단위로 취급되는 연관된 객체들의 클러스터**==
			- 데이터 변경을 위해 존재한다.
		- 루트와 경계를 가진다.
		- 데이터 변경 시 하나의 단위로 취급된다.
			- 데이터 변경 일관성을 유지한다.
		- 오직 루트를 통해서 접근을 제어해야 한다.
			- 외부 객체는 루트 엔티티에만 참조를 가질 수 있다.
		- 데이터 일관성 유지
			- 경계 내의 어떠한 변경 사항이 있을 때 전체 애그리거트의 모든 불변식이 충족되어야 한다.
			- 애그리거트를 넘어서는 불변식은 이벤트나 배치 등을 통해 특정 시간 내에 해결할 수 있다.
				- 불변식은 항상 참이여야하는 조건을 의미한다. 이런 조건이 다른 애그리거트를 걸쳐서 존재한다면 이것은 이벤트나 배치 등을 사용해야한다는 것이다.
		- 검색 및 접근 방식은 항상 리포지터리를 통해서 애그리거트 루트만 직접 얻을 수 있다.
			- 내부 엔티티는 항상 루트의 연관관계로 접근해야만 한다.
		- 생명주기 관리 캡슐화
			- 팩토리와 리포지터리를 이용해 객체들을 생명주기에 걸쳐 체계적이고 의미있는 단위로 조작해야한다.
	- 경계 특징
		- 엔티티와 값 객체 하나 또는 여러 개가 존재할 수 있다.
	- 루트 특징
		- 내부에 포함된 단일 엔티티이다.
	- 최종 목표
		- 복수의 엔티티들의 일관성 유지
			- 불변식을 유지해주는 수단
		- 객체의 시작과 끝을 명확히 하여 모델을 더 쉽게 이해하게 한다.
		- 연관 관계 탐색을 제한하고 루트를 통해서만 접근하도록 해준다.
		- 생명주기 전환을 캡슐화하는 단위가 되도록 해준다.

&emsp;&emsp;

### 도메인 모델 관점으로 바라본 애그리거트

&emsp;&emsp;

- 설명
	- 예시 기준
		- 멤버를 수정하기 위해 멤버 디테일이라는 엔티티를 같이 수정되어야하는 상황
			- 멤버 애그리거트로 복수의 엔티티를 바라봐 로직을 하나로 만들어줘야한다.
			- 단, 애그리거트 루트 엔티티를 하나 선정하여 그 엔티티에서만 다른 엔티티들을 생성, 변경을 할 수 있도록 해야한다.
	- 애그리거트 적용 규칙
		- JPA의 cascading을 잘 활용해야한다.
			- 같이 영속화 시켜야하는 내부 엔티티를 제어하는데 큰 도움이 된다.
		- 리포지터리는 애그리거트 단위로만 만든다.
			- 스프링 데이터 JPA 프로젝트의 핵심 원칙
			- 리포지터리의 리턴 타입은 애그리거트 루트
		- 가능하다면 하나의 트랜잭셔넹 하나의 애그리거트만 변경되어야한다.
		- 다른 애그리거트의 참조는 애그리거트 루트에 대해서만 한다.
			- 연관관계 애그리거트 루트의 레퍼런스 대신 루트의 ID값만 저장하기도한다.
				- 내가 이렇게 하고 있다.
				- 다만 이렇게하면 토비님께선 구현이 복잡해질 수 있다는 단점을 경고하신다.
					- 이래저래 잘 트레이드 오프를 하길 권장하신다.

### 애그리거트 도입시 어려움

&emsp;&emsp;

- 설명
	1. 적절한 애그리거트 경계가 어디까지인지 선택하는 것이 매우 어렵다.
	2. 개발을 하다보면 경계 범위가 변경된다.
		1. 점점 작은 애그리거트들이 등장하게 된다.
	3. Lazy Loading을 적절히 하지 않으면 성능에 부담을 주게 된다.
	4. 조회의 경우 내부 엔티티로의 직접 접근이나 여러 애그리거트를 한번에 조회하는 기능이 필요하다
	5. 도메인 이벤트와 최종적 일관성의 사용이 요구된다.
	6. 완벽해질려고 할 수록 더 어려워진다.
		1. 모든 것을 지키려고 할 필요가 없어 가치 판단을 잘해서 도입해야한다.

&emsp;&emsp;

#### 헥사고날 아키텍처와의 궁합

&emsp;&emsp;

- 설명
	- 애그리거트 단위로 애플리케이션을 구성하는 방법이 유용하다.
	- 다른 애그리거트로의 접근을 애플리케이션 포트를 통해서 Id를 전달하는 방식으로 구현해서 좋다.
		- 애플리케이션 내부 리포지토리에서 루트 엔티티를 조회하는 방식으로 이루어지게 강제할 수 있다.
	- 다른 애그리거트의 특정 동작을 수행해 불변식을 유지해야하는경우 도메인 이벤트와 리스너를 통해 애그리거트 사이의 작업을 연결시켜 줄 수 있어 좋다.
		- 이벤트에 필요한 애그리거트 Id를 넣어줘 전달해야한다.
	- 애그리거트를 설계하고 각각을 독립적인 애플리케이션으로 분리한다.

### 애그리거트 개발

&emsp;&emsp;

- 설명
	1. 서브 엔티티의 연관관계를 잘 지정해주자.
	2. 애그리거트에 포함되는 엔티티들을 패키지를 구분지어 포함시켜주자.
	3. 서브 엔티티의 경우 애그리거트 패키지 내부에서만 접근이 가능하도록 모든 메서드에 디폴트 접근 제한자를 사용해주자.
	4. 연관관계에 캐스케이드를 `CascadeType.ALL`로 주고 `FetchType.LAZY`로 걸어주자.
		1. 나의 경우에는 연관된 애들은 엔티티로 걸지 않고 임베딩을 사용한다. 이런 경우에 잘 동작하게 되지만 1:1의 경우에는 이런 연관관계를 맺어주던가 하는게 낫다.
	5. update의 경우 dto를 그대로 엔티티 메서드 파라미터로 넣어주면 변화에 따라 자연스럽게 적은 변경으로 유지보수할 수 있다.
