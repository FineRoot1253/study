## 엔티티

&emsp;&emsp;

- 설명
	- 정의
		- 도메인 모델을 만들 때 사용하는 패턴
		- 도메인 안에 있는 대상이나 개념
	- 특징
		- 고유한 식별자를 가지고 이를 통해서 개별적으로 구분된다.
		- 생명주기를 가진다.
			- 시간의 흐름에 따라 상태가 변경된다.


<br><br><br>

## 값 객체

&emsp;&emsp;

- 설명
	- 정의
		- 도메인 모델에서 식별자가 필요하지 않고 속성/값으로만 구별되는 오브젝트
	- 특징
		- 엔티티가 너무 많은 책임을 가지고 있는 것을 방지한다.
		- 특정 속성 관련 행위를 분리해서 엔티티를 더 집중된 상태로 유지하게 해준다.
		- 불분명한 원시 타입이 아니라 객체를 정의하여 도메인 개념을 명시적으로 표현하여 모델의 명확성을 높여준다.
		- 생성 이후 상태가 변경되지 않고 변경이 필요하면 새로운 객체로 교체한다.
		- 풍부한 기능을 가진다.
		- 자체 유효성 검사도 가능하다.

## 도메인 모델 패턴

&emsp;&emsp;

- 설명
	- 정의
		- 도메인/비즈니스 로직을 구성하는 아키텍쳐 패턴의 한 가지
	- 특징
		- 도메인 모델의 속성과 행위를 모두 포함하는 도메인의 오브젝트 모델
		- object model이기 때문에 복잡한 연관 관계, 커스텀 속성, 상속 등을 사용할 수 있다.
		- TX 패턴처럼 작업 단위의 절차형 API를 만들기 어렵다.
			- 따라서 도메인 로직의 명확한 작업 단위 API를 제공하는 애플리케이션 서비스가 필요하다.
			- 한 트랜잭션으로 묶어서 처리하기 위함이다.
			- 요청을 받으면 도메인 모델들의 API활용해 사용하는 일종의 Facade이다.
	- 주의사항
		- 생성의 경우 null이 들어오면 안되는 필수 속성에 null 체크 검사는 반드시하여 null인 경우 예외를 던져야한다.
			- 예외를 던지지 않고 넘어가는 경우 그것은 논리적인 오류가 된다.
			- `Objects.requireNonNull(attribute)` 이 메서드를 사용하는 것도 방법이다.
		- 테스트를 작성할 시
			- 시간을 단축시킬 모든 수단을 동원해야한다.
			- IntelliJ에서는 Live Template을 열심히 사용해주자.
			- 템플릿의 등장 위치를 잘 지정해주자. 그렇지 않으면 시도 때도 없이 키워드만 적으면 등장해서 피곤해진다.
			- `@DisplayName` 어노테이션으로 테스트 이름을 꼼꼼히 적어줘야한다.
			- 최대한 구현하기 전에 테스트를 먼저 작성해주도록 하자.
				- 구현을 먼저 해버릇하면 테스트를 계속 안하고 넘어가게 된다.
				- 굳이 테스트가 필요 없다고 판단될 때 안해도 그만이다.
				- 강사님은 굳이 구분짓지말고 구현을 바로 할 수 있는 화면이면 구현을 하고 테스트를 짜면된다고 말씀하신다.
		- 테스트 이름 작성시 주의사항
			- 어떤 행위를 하여 어떤 결과가 나와야하는지 적어야한다.
			- 어떤 파라미터를 넘기는지, 어떤 결과를 도출하는지 상세히 적어야한다.
				- 예를 들면 "예외를 발생"보단 "예외를 던짐"이라고 표현해야한다.
				- 어떤 파라미터인지 용어 사전에 등록한 유비쿼터스 언어를 사용해 표현해야한다.
			- 강사님은 테스트를 볼때 이해가 안되는 순간에만 적어주라고 이야기하신다.
				- 내 생각에도 이게 시간낭비이긴 하지만 추후 테스트가 깨지는 순간에 바로 파악하는데 항상 도움이 되긴하기 때문에 타인을 위해서 적는다고 생각한다.
		- 예외는 직접 규정해주는 것도 좋지만 일반적으로 자바언어에 정의되어진 예외를 사용해준다.
			- 보통 예외는 지정해주는 것이 좋지만 의미 있는 비즈니스 예외의 경우에 그렇고 이걸 일일히 다 지정해주면 시간 낭비가된다.
			- 자바 언어에 정의된 런타임 예외를 활용해주던가 하자.
		- 어떤 상태에 대해서 할 수 있는 동작과 할 수 없는 동작을 규정해주는 것이 미래에 버그를 방지하는 코딩 방법이다.
			- 미리 어떤 동작에 대해 규정을 하는 것이 의미 없는 테스트가 아닌가 생각될 수도 있지만, 이렇게 규정을 미리 해주면 버그를 방지하는데 큰 도움이 된다.
			- 진정한 시간낭비는 도메인 전문가와 상의도 없이 기능을 추가하는 행위이다.
		- 구현 코드 작성시
			- 맥락이 의미있게 변경되는 부분에서는 한칸 띄어주는것이 좋다.
			- 스프링 유틸리티 클래스를 가져다 도메인 로직에 활용해도 큰 문제가 없다.
				- 기술에 의존되는 코드는 딱히 아니라고 볼 수 있다.
			- `Assert.state(상태조건, 메시지)`이 API를 활용하면 코드가 더 깔끔해진다.
			- 코틀린에서 `require, check`와 같은 기능이 자바에는 없으니 외부 라이브러리를 활용하는 것이다.
			- 구현 중에 도메인 모델에 속성, 규칙, 행위등의 추가나 수정이 필요하면 언제든 추가하면 그만이다.
				- 현업에서는 물론 도메인 전문가와의 이야기 후 진행해야한다.
			- null검사시 `@NonNull`을 파라미터마다 붙여주기 피곤하니 `package-info.java`파일을 목표 패키지에 넣어주고 여기에 `@NonNullApi`를 넣어주면 null을 허용한 파라미터만 `@Nullable`을 붙여서 사용하면 되어 매우 편리하다.
			- `@NonNull`을 붙여줘도 null을 넘기면 컴파일 에러는 나지 않으므로 그대로 배포될 수 있다. 이런 경우에는 스팟버그를 통해 그래들에서 한번씩 발견하지 못한 런타임 null 문제를 잡아주거나 소나린트를 설치해 커밋 전에 발견해주면 좋다.
			- 도메인과 관련이 있긴하지만 지나치게 기술적인 것들은 외부에서 구현해주는 것이 도메인 코드를 정해진 도메인 모델에 규정한 그대로 유지시키는데 도움이 된다.
				- 이때 인터페이스만 만들고 외부에서 구현할 것인데 이 인터페이스를 도메인 서비스(Domain Service)라고 부른다.
				- 따라서 도메인모델 문서에는 행위만 적어주면 된다.
			- 인텔리제이가 제안하는 파라미터명을 그대로 따라가면 안된다.
				- 여기서 바로 유비쿼터스 언어가 깨지기 시작한다.
			- 다 작성을 했다면 도메인 모델에 적은 규정에 맞게 잘 작성되었는지 읽어주는 시간을 잠시 가져주도록 하자.
			- 생성자의 경우 파라미터가 너무 많아지면 DTO로 리펙터링을 적극적으로 해주자.
				- 실제로 자주 일어나는 실수중 하나는 호출하는 쪽에서 파라미터 삽입 순서를 바꿔서 넣는 실수이다.
				- 이것을 방지해주는 것이 바로 DTO이다.
				- 강사님은 DTO 생성자를 그대로 생성해서 넘기지만 DTO 생성자의 빌더를 통해서 인스턴스를 만들어 넘기는게 더 낫다고 생각한다.
			- 값 객체를 적극적으로 사용해주자.
				- 다만 값객체는 JSON으로 넘길때 상당히 곤란한 부분들이 있긴한데 DTO에서 사용하기보다는 엔티티 자체에 기용하는것이 더 좋다.
				- Spring Validation에서 제공하는 `@Email` 어노테이션을 사용해 DTO에서 막아줄 수도 있다.
					- 이쪽이 좀 더 사용하기 편한데, 엔티티도 해주고 DTO도 해주고 둘 다 해주는 것이 맞다.
					- 만약 DTO를 통해서 변경이 되지 않고 다른 방식으로 변경되는 경우가 생긴다면 결국 엔티티에서도 이메일 검증을 해주는 것은 매한가지이기 때문이다.
				- 강사님은 record를 적극적으로 쓰시지만 개인적으로 롬복을 적용하기 상당히 애매해져서 나는 비추한다.
		- 롬복 사용시
			- `Getter(AccessLevel.NONE)` 을 사용하여 노출을 원하지 않은 속성은 감춰야한다.
				- 대표적으로 벨류 컬렉션은 감추거나 꼭 노출시켜야한다면 UnmodifiableCollection으로 새로 생성하여 노출해야한다.
	- Reference API 읽을때 팁
		- JSR 스펙을 찾아볼때는 `jcp.org`에서 찾아보자.
	- 트랜잭션 스크립트 방식 정의
		- 하나의 업무절차(TX)를 처리하기 위한 스크립트(메소드)를 만들고 비즈니스 로직을 순서대로 코드로 작성하는 방법
	- 패턴의 종류
		- 단순 도메인 모델 (Simple Domain Model)
			- 실 테이블과 구조가 매우 유사하게 끌고감
			- 실제 이것을 도메인 모델이라고 이야기하기 매우 힘들다.
		- 풍성한 도메인 모델 (Rich Domain Modal)
			- 상속, 전략 패턴, GoF 디자인 패턴, 연관관계
			- 복잡한 로직에 적합하지만 DB 매핑이 난해해진다.
