- 주어진 상황에서 최선의 선택과 트레이드 오프가 어떻게 이루어지는지 함께 고민하고 경험하는 것이 목표

## 도메인 주도 설계, DDD

&emsp;&emsp;

- 설명
	- 정의
		- 도메인의 복잡성이 주는 문제를 해결하기 위한 접근 방법
	- 특징
		- 도메인 모델을 개발 과정의 중심에 두는 방법
		- 개발자끼리만 해선 안되고 도메인 실무 전문가, 이해 관계자가 모두 참여해서 함께 도메인 모델을 만들고 발전시켜야 한다.
			- 한번만 참여해서 만들고 땡이 아니라 이 인원들의 지속적 참여가 필수다
			- 분석 - 설계 - 구현순으로 끊어져서 진행하게되면 각 단계에서 비즈니스의 의미가 사라지거나 엉뚱한 개념이 추가되게된다.
			- 개발을 하다가도 도메인 모델을 변경해야할거같으면 그 변경 의의가 충분히 마땅하다면 도메인 모델이 변경되어야한다.
		- 도메인 모델이 설계와 코드까지 이어져야한다.
		- 팀 안에서 도메인 모델에 기반한 단일 어휘체계를 만들고, 이를 문서, 회의, 대화 그리고 코드까지 일관되게 사용한다.
			- 이것을 유비쿼터스 언어라고 부른다.
			- 결국 잘 안되서 언어가 따로 놀게되는 경우가 많은데 이러면 DDD가 아니다.
			- 결국 참여하는 모든 인원들이 훈련이 되어야한다.
	- 주의사항
		- 복잡도가 낮은데 굳이 DDD를 위한 개발을 할 필요는 전혀없다.
			- 대신 DDD에서 사용되는 개념들을 기용하는 것은 나쁘지 않다.
		- ==**도메인에 집중하고, 코드와 모델을 일치시키고, 명확한 언어를 사용하는 것이 가장 핵심이다.**==

## 도메인 모델 만들기

&emsp;&emsp;

- 설명
	- 보편적인 케이스
		1. 도메인 전문가에게 듣고 배워야 한다.
		2. 중요한 것(개념)들 찾기 (개념 식별)
		3. 연결 고리 찾기 (관계 정의)
		4. 상세 요소들을 설명하기 (속성 및 기본 행위 명시)
		5. 그려보기 (시각화, 산출물: 컨텍스트 맵, 클래스 다이어그램)
		6. 그려본 산출물들 가지고 계속 이야기하고 다듬기 (여러번 반복 된다.)
	- 도메인 전문가가 없는 스타트업의 케이스
		1. CEO, 기획자, PO를 가상의 도메인 전문가로 활용
			- 프로젝트 참여자들이 전부 도메인 전문가가 되어야한다.
			- 경쟁 업체의 서비스의 정보구조를 분석하는 것이 좋다.
		2. 팀원들이 모여 다양한 탐색을 하고 집단적으로 학습을 해야한다.
			- 이벤트 스토밍이 대표적이다.
				- 워크숍 형태로 벽에 포스트잇을 붙여가며 이야기하는 형식
		3. 이벤트 스토밍 이후 가설 기반 점진적인 모델링을 이어나간다.
		4. 유비쿼터스 언어의 의식적인 구축과 진화
			- 이것이 가장 중요하다.
		5. 베타 테스터, 초기 사용자와의 적극적인 소통
		6. 도메인 모델의 발전과 진화, 공유를 더욱 적극적으로 해야한다.
	- 주의사항
		- 너무 자세한 디테일을 작성하면 무엇이 핵심인지 알 수 없다.
			- 간단하게 핵심만 적고 핵심 기능을 적어넣어야한다.
			- 다 적고 핵심도 추릴 수 있다면 어디까지 제공되어 런칭해야한다는 기준을 적어야한다.
		- 비즈니스 모델 또한 이 안에서 제공되어야한다.
		- 그림과 같이 문서를 작성해야하며 코드가 개발되며 대화를 하다가 변경이 필요하면 즉시 반영이 되어야한다.
			- 코드와 함께 관리되는 문서여야한다.
			- 프로젝트 내에 존재하면 자연스럽게 관리가 된다.
			- JIRA에 따로 정리하면 그거대로 정리가 안되는건 아니지만 바로바로 반영하기는 힘드니 이것은 선택사항이다.
	- 규칙
		- 주로 자주 등장하는 명사가 모델이 된다.
			- 회원, 책, 리뷰, 평점, 댓글, 팔로잉팔로우, 알림
		- 모델을 세우면 모델 간의 관계를 맺어줘야한다.
			- 카디널리티로 표현해준다. (1 - 0..\*)
		- 용어사전은 한국어, 영어, 정의(설명)으로 구분된 테이블이다.

### 예시 (Lit-Tok)

&emsp;&emsp;

- 설명
	- 릿톡은 회원이 책에 대한 리뷰를 작성하는 온라인 서비스이다.
	- 릿톡은 주기적으로 올라오는 책들에 대한 리뷰들을 바탕으로 개발되고 운영되며 발전하는 커뮤니티를 지향한다.
		- 이 생태계를 구성하는 핵심 구성 요소는 책에 대해 이야기하고 각자의 해석을 이해하며 나의 생각을 리뷰하는 회원이다.
		- 그래서 이름이 Literature + Talk = Lit-Tok이다.
		- 유사한 이름으로 Tik-Tok이란 서비스는 있지만 왓챠피디아와 Gookreads을 벤치마킹하였다.
	- Lit-Tok에서 리뷰를 읽어볼 수 있지만 최대 5개까지의 리뷰을 확인할 수 있다.
		- 다만, 회원으로 가입을 하게 되면 책에 대한 모든 리뷰를 확인할 수 있다.
		- 책에 대한 리뷰를 작성하기 위해서는 가입을 완료하고 활동 가능한 회원이 되어야 한다.
		- 가입 신청을 한 뒤 정해진 요건을 충족하면 가입이 완료된다.
	- 가입된 회원은 평점을 먼저 작성할 수 있고 이어서 리뷰를 작성할 수 있다.
		- 리뷰를 작성할 때는 태그를 넣을 수 있다.
		- 회원은 다른 회원을 팔로우 할 수 있고 팔로잉 될 수도 있다.
	- 리뷰는 책에 대해 리뷰된 글을 의미한다.
		- 리뷰를 작성하기 위해서는 가입이 필요하다.
		- 리뷰는 평점과 함께 작성되어야 한다.
		- 가입된 유저에 한해서 리뷰는 평점별로 확인이 가능하며 추천순으로도 볼 수 있어야하며 최신순 리뷰를 기본적으로 보여줘야한다.
		- (미래에는)리뷰 작성 또는 조회에 제한을 주어 구독제 서비스를 가입해야한다.
	- 리뷰에는 댓글을 작성할 수 있고 유저간의 소통이 가능해야한다.
		- 회원을 맨션할 수 있고, 댓글 작성시 회원이 맨션되면 알림이 전송된다.
	- 알림은 헤더 네비게이터에서 바로 확인이 가능한 아이콘으로 알림의 개수 만큼 뱃지 숫자가 달려있다.
		- 알림을 누르면 최근 10개의 알림에 대해 보여진다.
		- 알림 드랍다운 화면의 맨 밑을 누르면 알림만 볼 수 있는 알림 조회 화면을 볼 수 있다.
		- 댓글에서 맨션이 되거나, 댓글이 달리거나, 팔로잉을 당한다거나, 운영진이 공지사항을 전달하거나 하면 알림에 전달된다.
	- 평점은 책에 대해 1~10사이 숫자로 평가된 점수를 의미한다.
		- 평점을 작성하기 위해서는 가입이 필요하다.
	- 책은 커버이미지, 간단한 설명, 출판날짜, 글쓴이, 평균 평점등으로 구성된다.
		- 책은 모든 가입되지 않은 유저를 포함하여 검색이 가능해야한다.
		- 책은 한글 버전을 기준으로 제안되어야하며 한글 데이터가 없는 경우에는 영어 버전을 기준으로 제안되어야하며 이마저도 없는 경우에는 무시된다.
		- 메인에 여러 개의 카로셀로 통합 평점과 함께 책들이 제안되어야한다.
		- 메인에는 주간, 월간 책리스트를 아마존과 알라딘에서 주기적으로 섞어서 보여져야한다.
		- 메인에는 유저들이 자주 검색하는 책들이 랭크되어 보여져야한다.

#### 회원 (User, Entity)

&emsp;&emsp;

##### 속성

&emsp;&emsp;

- email: 이메일 - ID
- username: 유저 이름
- nickname: 닉네임
- profileImage: 프로필 이미지
- status: 회원 상태

&emsp;&emsp;

##### 행위

&emsp;&emsp;

- constructor(): 회원 생성
	- username
	- nickname
	- profileImage
	- status
- enable(): 가입 완료
- delete(): 탈퇴
- changePrivate(): 비공개
- disable(): 휴면화

&emsp;&emsp;

##### 규칙

&emsp;&emsp;

- 회원 생성 후 상태는 가입 대기
- 일정 조건을 만족하면 가입 완료가 된다.

&emsp;&emsp;

#### 회원 상태(UserStatus, Enum)

&emsp;&emsp;

##### 상수

&emsp;&emsp;

- REGISTER_PENDING: 가입 대기
- ENABLE: 가입 완료
- ENABLE_PRIVATE: 비공개 회원
- DELETED: 탈퇴 또는 강퇴 상태
- DISABLE: 휴면 상태
