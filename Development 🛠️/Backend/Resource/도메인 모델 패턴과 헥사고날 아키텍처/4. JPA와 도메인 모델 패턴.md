## JPA 모델과 도메인 모델

&emsp;&emsp;

### JPA 모델, 도메인 모델 분리 패턴

&emsp;&emsp;

- 설명
	- 정의
		- 더 확실하게 헥사곤 내부에서는 자바 코드만 유지하고 JPA와 연관된 코드들은 외부에 어뎁터를 만들어 여기에서 모든 것을 관련 기술 코드를 구현하는 방식
	- 선호하는 이유
		- 데이터 모델과 도메인 모델이 너무 다른 경우
			- 레거시 DB에 도메인 모델 설계를 진행해야하는 경우
				- DB 마이그레이션이 불가능한 경우이다.
			- 자바 스프링을 쓰는 곳은 대다수가 이런 케이스이기 때문에 이런 패턴이 선호된다.
		- 도메인 모델이 너무 복잡해 데이터 모델과 매핑이 안되는 경우
			- 이런 경우 JPA 어뎁터에서 처리를 진행해줘야한다.
		- 데이터 저장 기술이 바뀌는 경우
			- Spring Data 자체가 여러 데이터베이스를 호환되기 때문에 의미가 잘 없는 이유이긴하다.
		- 관심사가 다른 코드나 JPA 기술에 의존적이라서
			- 이건 말이 안되는게 지금 JPA 라이브러리를 다 빼고 실행해도 문제없이 동작한다. 의미가 다른 관심사의 코드가 들어있어서 보는데 방해가 된다고 생각될 수는 있겠다.
	- 실제 구현시
		1. Id를 수퍼 추상 클래스에 올려서 사용한다.
			1. 이때 JPA Buddy를 반드시 설치해주자 이 친구가 제안하는 equals() hashCode()가 jpa에서 필수적으로 구현해줘야하는 최적화된 비교구문이다.
			2. id에 `@Getter(onMethod_ = {@Nullable})`을 붙여서 명시적으로 null을 반환할 수 있음을 표현해야한다. 실제로 영속화전에 null이 할당되어있는 상태가 존재하기 때문에 이 정보는 필요하다.
			3. 사용하는 모든 엔티티에서는 `@ToString(callSuper = true)`를 해줘야한다.
		2. xml에 모든 jpa 매핑 정보를 옮겨둔다.

### 도메인 모델 패턴

&emsp;&emsp;

- 설명
	- 특징
		- 복잡한 도메인 모델은 테이블과 1:1 매핑이 안된다.
		- 제대로 최적화를 이룬 테이블이 아닌 경우 패러다임 불일치가 발생하는데 이 경우에 직면했을지라도 매핑을 어떻게 해서든 이뤄주기 위한 기술이 ORM이다.
	- 선호하는 이유
		- 데이터 모델과 도메인 모델이 다르지 않음
			- 이 자체가 보일러플레이트가 되는 경우가 대부분이다.
		- 복잡한 도메인 모델 매핑은 JPA가 충분히 지원하고 있다.
			- 하이버네이트는 스프링보다도 오래된 프로젝트로 충분히 필요한 매핑기술들이 풍부하게 존재한다.
			- 만약 너무 복잡하다면 명령, 조회로 분리해서 만들어도 잘 동작한다.
		- 모델 변환 로직으로 인한 불필요한 복잡성 증대를 방지한다.
			- 다만 대다수의 경우 유저에게 내릴 정보만 캐서 제공하기 위해 DTO는 늘 필요하지만 JPA - 도메인 분리 패턴의 경우에는 만들어야 하는 인터페이스와 클래스가 너무나 많이 증가하게 된다.
		- 자바코드 로직에 직접 침범하지 않는다

#### 대표적인 패러다임 불일치

&emsp;&emsp;

- 설명
	- 세분성(Granularity) 불일치
		- 테이블과 클래스가 1:1 매핑 되지 않는 경우
		- 만약 엔티티에서 VO를 필드로 이용한다면 이미 세분성 불일치가 발생한 것이다. 물론 JPA에서는 문제없이 수행된다.
	- 상속(Subtype) 불일치
		- RDB는 상속 개념이 없기 때문에 발생하는 경우
		- JPA는 다형성 쿼리도 지원해준다.
	- 정체성(Identity) 불일치
		- id가 다르면 다른 row이지만 자바에서 잘 지켜지지 않는 경우
		- JPA는 영속성 컨텍스트에서 이것을 정말 잘 지켜주고 있다.
	- 연관(Association) 불일치
		- rdb는 fk를 반드시 구현을 해줘야 연관이 되지만 자바에서는 필요가 없는 경우
		- JPA에서는 이 불일치 문제도 각 도메인의 특성을 유지하면서 잘 일치시켜준다.
	- 데이터 탐색(Navigation) 불일치
		- rdb는 단순히 fk를 들고온다고해서 그 정보를 전체를 알아서 조인해서 들고오진 않지만 자바는 레퍼런스를 통해서 그 데이터를 다 들고 있는게 당연한 경우
		- JPA에서는 영속상태인 엔티티가 지닌 다른 엔티티를 너무나 쉽게 불러올 수 있다.
			- 단순히 호출만해도 관련 데이터를 조인해서 들고온다.

### 스프링 데이터 프로젝트

&emsp;&emsp;

- 설명
	- 정의
		- 다양한 데이터 저장소 벤더들에 대한 데이터 접근을 단순하고 일관된 프로그래밍 모델을 제공하는 프로젝트
	- 특징
		- 보일러 플레이트 코드 감소
		- 데이터 저장소의 특성 유지
		- 확장성과 유연성
