### 반환 타입: Entity vs DTO

&emsp;&emsp;

- 설명
	- 엔티티를 리턴하면 안되는 이유
		- OSIV 때문에 대규모 트래픽을 처리하는 환경에서는 커넥션 드레인이 발생할 여지가 존재한다.
		- 도메인 로직 호출 가능성을 외부에 꺼내선 안된다.
			- 이것이 DB에 반영이 될 것이다 (물론 트랜잭셔널 내부에서만 영속성 컨텍스트가 유지되기 때문에 이상한 주장이다.)
	- DTO를 리턴하면 안되는 이유
		- DTO를 리턴하면 표현에 대한 책임이 어플리케이션과 도메인 레벨로 내려오게 된다.
			- 이게 가장 큰 문제 원인이다.
	- 개인적인 내 생각
		- 데이터를 반환처리하는 서비스만 따로 만들어서 그 서비스가 dto를 만들어 표현에 대한 책임을 지게하면 된다.
			- 이러면 osiv에 대한 걱정도 없고 괜찮아진다.
			- 거기에 쿼리프로젝션을 사용중이라면 조회에 대해서는 더 단순하게 dto를 써주면 된다.
				- 만약 쿼리DSL을 사용하지 않는다면 내가 말한대로 모든 리턴타입 dto에 대해서 생성 책임을 가지는 도메인ResponseCreateService를 만들어줘야할 것이다.
				- 이것이 파사드로써 사용되는 것이다.
		- 범균님의 경우 아에 같은 테이블을 바라보는 다른 조회용 엔티티 모델을 만든다.
			- 변경에 대해 완전히 막힌 형태로 모델을 만들어 반환하는 것이다.
			- QueryDSL을 사용하게 되면 생성자 쿼리 프로젝션 기능 덕에 위의 방식을 더 자주 사용하게되서 지금도 그렇게 쓰고 있지만 사실 이 방식을 개인적으로 더 낫다고 생각한다.

### 통합테스트

&emsp;&emsp;

- 설명
	- 통합테스트시 SpringBootTest, AutoConfigureMockMvc, Transactional 이 3 가지 어노테이션을 붙여서 모든 레이어 통합 테스트를 수행해볼 수 있다.
		- 주의점
			- 새로운 트랜잭션을 실행시키거나 하는 경우에는 롤백이 진행되지 않아 이런 경우에는 직접 롤백을 시켜줘야한다.
				- sql 어노테이션으로 롤백을 하는 쿼리를 미리 만들어두고 시작하는게 좋다.
				- 즉, 어디서부터 어디까지 테이블에 영향을 주고 어떤 결과가 나와야하는지가 명료하게 정리되어야 이것이 깔끔하게 사용가능하다.
				- 나는 지금 개발용 DB를 그대로 붙여서 따로 인메모리없이 진행하기 때문에 이런 경우 이 작업은 필수적이라고 생각한다.
	- ProblemDetail을 이용해서 오류 데이터를 제어해줘야한다.
		- 특히 public api를 만들어야하는 상황이면 이것을 안 사용해줄 이유가 없다.
		- RFC 9457을 따르며 래핑해줄 이유가 이젠 없다고 한다.
	- 테스트는 가능한 도메인에 대해서는 자세히 만들어야한다.
		- 특히 DB와 관련된 로직에 대해서는 자세하게 만들어야한다.
		- API 테스트는 시간이 없다면 간결하게 사용하는 방법만 테스트하고 특별히 예외처리되는 경우를 추가해주는 편이 좋다.
