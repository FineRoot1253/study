## 핵사고날 아키텍처

&emsp;&emsp;

### 헥사고날 아키텍처의 사실과 오해

&emsp;&emsp;

- 설명
	- 아키텍처의 특징
		- 시스템의 기본 구조를 정의
		- 시스템의 중요한 품질 속성에 큰 영향을 미친다.
		- 설계 결정의 기반이 되는 핵심적인 개념
		- 기본 구성 요소와 상호 관계, 제약 조건 원칭 등을 포함한다.
	- 레이어드 아키텍처 특징
		- 서브 시스템을 계층으로 구조화하는 아키텍처
		- 계층은 사용 관계로 연결됨
		- 사용 관계는 **단방향**이여야만 하는 핵심 제약이 존재
			- 상위에서 하위 서비스를 사용하는 하향식 흐름을 가짐
		- 각 계층이 하위 계층의 내부를 모르고 **제한된 인터페이스만 사용**해야한다.
			- 계층 격리라고 부른다.
			- 전형적인 캡슐화
			- 어떤 레이어가 다른 레이어의 컴포넌트에 영향을 주지 않도록 유지해야한다


<br><br><br>

#### 3 or 4 레이어드 아키텍처

&emsp;&emsp;

- 설명
	- 3-레이어드 아키텍처 특징
		- 가장 전통적으로 오랫동안 사용된 아키텍처
		- UI(Presentation), Domain(Business Logic), Data(Persistence, Infra로 바꾸는 것이 현대 개발 추세) 구조가 보편적
	- 4-레이어드 아키텍처 특징
		- Domain 계층은 트랜잭션 스크립트 패턴으로 구현되는 경우도 많다.
		- DDD 구조에서는 Presentation, Application Service, Domain, Infra 계층으로 나눈다.
		- 도메인 모델의 api를 조합하여 하나의 작업 단위 묶음으로 수행하기 위해 Application Service 계층이 존재한다.
			- 주로 도메인 모델끼리의 상호작용하는 방식으로 진행된다.
		- 보편적으로 Application Service 계층에서 Domain 계층을 통해서 Infra 계층을 이용하지 않고 직통으로 Infra 계층을 이용하는 경우가 더 많다.
			- Infra 계층을 생각외로 자유롭게 붙여서 호출 시킨다.
			- 이것을 완화된 계층형 아키텍처, Relaxed Layered Architecture라고 부른다.
			- 현재 Lit-Tok에 적용된 아키텍처는 이것이다.

#### 헥사고날 아키텍처

&emsp;&emsp;

- 설명
	- 특징
		- 2005년 앨리스터 코번이 제안한 아키텍처
		- 단방향 아키텍처가 아닌 대칭형 아키텍처
		- 위 아래, 좌 우로 나뉜 것이 아닌 애플리케이션의 내부와 외부 세계라는 대칭을 가진다.
			- 함수형 아키텍처와 비슷하게 코어와 외부로 나뉜다.
		-  대칭성을 지닌 구조를 그리기 쉬운 도형인 육각형으로 표현해 둔 것이다.

&emsp;&emsp;

- 특징
	- 중요한 도메인 로직을 담은 코어 애플리케이션
		- 도메인 로직을 가진 트랜잭션 스크립트
		- 애플리케이션 서비스와 도메인 모델 패턴을 유지해 만든 도메인

#### 헥사곤의 외부

&emsp;&emsp;

- 특징
	- 액터
		- 정의

		- 예시
			- 사용자, 브라우저, CLI 명령, 기계, 다른 시스템
			- 운영 환경, DB, 메시징 시스템, 메일 시스템, 원격 서비스
			- 테스트

#### 헥사고날 아키텍처의 특징과 이득

&emsp;&emsp;


- 특징
	- 앨리스터 아저씨는 온전히 테스트를 가장 고려를 많이 해서 고안한 아키텍쳐라고 이야기한다.
		- 테스트를 제대로 하지 않을 거면 도입할 이유가 없다.
	- 운영 시스템과 연결되지 않고 애플리케이션 테스트를 하는 것
	- 애플리케이션과 상호작용하는 액터가 변경되더라도 다시 빌드하지 않고 테스트한다.
	- UI의 디테일, 기술 정보가 도메인 로직 안으로 노출되지 않도록 보호해야한다.
		- 이 반대 방향도 그래야한다.
	- 컴포넌트를 각각 개발하고 연결하는 방식으로 큰 시스템 분리가 가능해진다.
	- 외부 연결을 다른 것으로 변경할 수 있다.
	- 기술 요소를 외부로 배제했기 때문에 도메인 설계에만 집중할 수 있다.
		- 스프링 프레임워크가 추구하는 이상과 굉장히 유사하다.
	- 외부에서 내부로 향하는 일종의 계층 구조로 볼 수도 있다.
	- 코드의 의존 방향은 내부로만 향한다.
		- 어뎁터 -> 애플리케이션 -> 도메인
		- **단, 사용의 흐름은 비대칭적이다.**

#### 액터와 애플리케이션의 상호작용 창구, 포트 (Port)

&emsp;&emsp;

- 설명
	- 정의
		- 애플리케이션이 외부 세계와 의도를 가지고 상호작용하는 아이디어를 캡쳐한 것
	- 특징
		- 단순 데이터 상호작용이 아닌, 명확한 목적과 방향을 가지고 외부와 연결된다.
		- 애플리케이션이 정의한 인터페이스로 만들어진다.
		- 액터는 Primary, OO으로 구분된다
	- Primary 액터
		- 정의
			- 기능 제공 인터페이스를 이용하는 엑터
	- Secondary 액터
		- 정의
			- 기능 요구 인터페이스에 기능을 제공하는 엑터


<br><br><br>

#### 인터페이스

&emsp;&emsp;

- 설명
	- Provided Interface, Lollipop
		- 정의
			- 기능 제공 인터페이스
			- 외부 엑터에게 기능을 제공하는 인터페이스
		- 특징
			- 외부 엑터가 사용할 수 있게 둥근 머리 부분을 노출시켜 그린다.
	- Required Interface, Socket
		- 정의
			- 기능 요구 인터페이스
			- 외부 엑터에게 특정 기능을 요구하는 인터페이스
		- 특징
			- 외부 엑터가 제공해줄 수 있게 레고 손 모양처럼 손 부분을 노출시켜 그린다.
			- 외부 엑터가 기능을 제공해주는 경우 이때 엑터에서 롤리팝을 그려 손부분과 연결시켜준다.

<br><br><br>

#### 어뎁터

&emsp;&emsp;

- 설명
	- 정의
		- 애플리케이션의 포트를 액터가 직접 연결할 수 없다면 인터페이스의 변환을 위한 어댑터를 도입
	- 예시
		- 애플리케이션이 브라우저를 통해 크롤링 데이터를 받아야하는 경우
			- 크롤링 정보 조회 기능 제공 인터페이스를 사용하는 웹 컨트롤러 어뎁터를 만든다.
		- 애플리케이션이 가진 크롤링 정보 저장 포트의 기능 요구 인터페이스로의 DB와 직접 연결할 수 없는 경우
			- 크롤링 정보 저장 기능 요구 인터페이스를 구현한 리포지터리 어뎁터를 만든다.
	- 특징
		- 앨리스터 콕번 아저씨는 이상하게 헥사고날 왜곡해석하는 경우가 많아 포트와 어뎁터 패턴으로 불러달라고 했지만 여전히 헥사고날로 불린다.
		- 어뎁터 또한 기능 제공 인터페이스를 사용하는 경우 Primary 어뎁터로 부른다.
			- 이를 Driving 어뎁터라고도 부르기도 한다.
		- 기능 요구 인터페이스를 제공하는 경우 Secondary 어뎁터로 부른다.
			- 이를 Driven adapter라고도 부르기도 한다.

### 헥사고날 아키텍처의 오해

&emsp;&emsp;

#### 오해 1: 애플리케이션 내부에 도메인 계층을 만들어야 한다.

&emsp;&emsp;

- 설명
	- 헥사고날 아키텍처는 애플리케이션 내부 구현에 대한 원칙이나 요구사항이 없다.
		- 스파게티로 만들어도, TX 패턴으로 만들어도 헥사고날 요구사항만 지키면 된다.
		- 도메인 계층을 포함하는 아키텍쳐는 클린 아키텍쳐이다.

&emsp;&emsp;

#### 오해 2: 헥사고날 아키텍처 패키지 구조를 따라야 한다.

&emsp;&emsp;

- 설명
	- 헥사고날은 요구하는 패키지 구조가 없다.
	- 애플리케이션과 어뎁터 패키지를 구분지어주는 것은 바람직하다.
	- 포트도 구분된 패키지를 만들어줘서 두는 것을 권장한다.

&emsp;&emsp;

#### 오해 3: 포트 이름에는 UseCase라는 접미사를 붙여줘야한다.

&emsp;&emsp;

- 설명
	- 포트의 의도가 담긴 이름을 사용하면 되지 붙여줄 이유가 따로 없다.
	- 오히려 앨리스터 아저씨가 제안한 네이밍 권장 규칙은 For + ~ing 스타일이며 이를 따를 필요도 없다.

&emsp;&emsp;

#### 오해 4: 애플리케이션에는 도메인 모델만 넣고 JPA 엔티티 등은 어댑터에 둬야한다.

&emsp;&emsp;

- 설명
	- 애플리케이션 코드와 포트 인터페이스가 외부 기술에 의존만 하지 않으면 된다.
	- 이런 규칙도 딱히 없으니 따를 이유가 없다.

&emsp;&emsp;

#### 사실: 헥사고날 아키텍처가 요구하는 규칙

&emsp;&emsp;

- 규칙
	1. 애플리케이션은 모든 외부와의 상호작용을 위해서 provided interface와 required interface를 정의한다.
		이를 어뎁터에 두는 경우가 많은데 인터페이스는 애플리케이션에 존재해야한다.
	2. 애플리케이션과 상호작용하는 액터는 런타임에 구성되어야 한다.
		애플리케이션 시작 시점에 연결되어야한다.
	3. 애플리케이션은 액터에 대한 코드 의존성을 가지면 안된다.
		액터가 변경되어도 애플리케이션에는 영향이 전혀 없어야한다.
	4. 액터는 정의된 포트를 통해서만 연결되어야 한다.
		애플리케이션 안에서 직접 외부 액터와 연결하는 어댑터 같은 것들을 new 같은 키워드로 직접 생성하면 안된다.
	5. 포트의 인터페이스 이름에는 기술 의존성을 가지지 않는다.
		웹 기술, 암호화 기술 등이 이름에 등장해선 안된다.


<br><br><br>

#### 패키지 구조

&emsp;&emsp;

- 설명
	- 도메인 패키지
		- 도메인 모델들이 모아둔 패키지
	- 어댑터 패키지
		- 컨트롤러와 같은 어댑터들을 모아둔 패키지
		- 내부 패키지 종류
			- webapi 패키지
				- 컨트롤러 어댑터들을 모아둔  패키지
			- persistence 패키지
				- DB와의 상호작용하는 어댑터들을 모아둔 패키지
			- integration 패키지
				- 외부와 상호작용하는 다양한 어댑터들을 모아둔 패키지
			- security 패키지
				- 보안과 관련된 상호작용을 하는 어댑터들을 모아둔 패키지
	- 애플리케이션 패키지
		- 애플리케이션 관련 구현 코드를 모아둔 패키지
		- 내부 패키지 종류
			- provided 패키지
				- 애플리케이션이 기능을 제공하는 인터페이스를 모아둔 패키지
			- required 패키지
				- 애플리케이션이 기능을 요구하는 인터페이스를 모아둔 패키지
	- 예시
		- domain
		- application
			- required
			- provided
		- adapter
			- webapi
			- persistence
			- integration
			- security


&emsp;&emsp;

#### 패키지 구조 팁

&emsp;&emsp;

- 설명

		- 개발가이드 md 파일을 따로 깃 레포지터리에 도메인모델.md 파일처럼 저장해주자.

			- 사진을 넣어줘도 좋다.
		- 패키지 구조 설명

## 구현

&emsp;&emsp;

### 애플리케이션 정의 

&emsp;&emsp;

- 설명
	- 기능 구현 순서
		1. 도메인 모델에 맞게 도메인을 구현해준다.
			1. 테스트까지 전부
		2. 도메인에 필요한 인터페이스를 구현해준다.
			1. 테스트까지 전부 해준다.
		3. 인터페이스를 구현하는 서비스를 구현해준다.
			1. 만약 롬복 사용시 기존 코드를 모르겠다면 Delombok기능을 이용해주자.


&emsp;&emsp;

#### 포트 정의

&emsp;&emsp;

- 설명
	1. application에 인터페이스를 만들어준다.
	2. 인터페이스 명은 어떤 기능을 제공하는지 이름을 적어주면 좋다. 보통 어뎁터에서 사용되는 빈이기에 명사로 적어주는게 좋다.
	3. 등록과 같은 경우 등록 대상 엔티티를 그대로 반환한다.
	4. 등록의 경우 리포지터리의 도움이 필요하기 때문에 required 인터페이스에 MemberRepository를 만들어주자.
		- 이때 `Repository<Entity, Identifier>`를 상속받아야 명확히 원하는 동작만 테스트하여 사용할 수 있다.
		- 이때 DataJPA 관계를 따라서 메서드를 추가해줘야 정상 동작한다.
		- DataJPA를 이용하기 때문에 어뎁터 구현 필요성이 없어져 상당히 편하다.
	5. 예시에서는 EmailSender의 send 기능도 필요하기 때문에 이또한 required 패키지에 만들어준다.
		1. 강사님은 모든 인터페이스에 어떤 행위를 하는지 한줄 정도 주석을 다는 것을 권장한다.

#### 엔티티 식별자 구현

&emsp;&emsp;

 - 설명
	 - 식별자 특징
		 - 고유성
			 - 두 개의 엔티티가 같은 식별자 값을 가지면 안된다.
		 - 불변성
			 - 한 번 값이 할당되면 엔티티 생명주기 동안 절대 변경되면 안된다.
				 - 참조 정합성을 훼손되고, 연관관계가 파괴된다.
		 - 비즈니스적으로 의미로부터 디커플링 되는 것이 낫다.
			 - 자연키 대신 인조키(대리키)를 사용하는 것이 낫다는 뜻이다.
		 - 자연키를 사용하면 키 규칙 변경에 자유롭지가 못하다.
	 - 식별자 구현 예시
		 1. 그냥 Long으로 만들어준다.
			 1. 이것을 값객체로 만들어주는 경우 고생 좀 해야한다.
			 2. 자동화를 하는 경우에는 추상 Id 객체를 만들어 BaseEntity에서 이것을 의존시켜야한다. 그렇지 않으면 리플렉션으로 직접 집어넣어야 하는데 나는 이렇게 했지만 따지고 보면 별로 좋지 않은 안티패턴이니 딱히 권장하지 않는다.
				 1. 리플렉션은 응용 레벨 개발자가 건드는 것은 왠만해서는 피해야한다.
		 2. 리포지터리에서 제대로 저장되는지 리포지터리 테스트하기
			 1. EntityManager를 따로 넣어주면 항상 테스트하기 편하다.
			 2. 이걸 BaseRepositoryTestConfigurer 같은 추상 클래스에 넣어주면 좋다.
			 3. H2 database를 쓴다면 별 추가 설정조작할거 없이 편하게 테스트할 수 있다.
		 3. 비즈니스적으로 의미가 있는 자연키와 같은 값에는 `@NaturalId`를 붙여준다.
			 1. `@NaturalIdCache`를 클래스 레벨에 붙여주면 PersistenceContext에 저장해뒀다가 제공한다.
			 2. 일반적으로 `@NaturalId`는 무조건 쿼리를 해오도록 하지만 성능상 이 캐싱을 시켜주는 것이 좋다.
			 3. 그리고 이 `@NaturalId`를 붙여주면 SQLDDL을 자동 생성할때 JPA가 알아서 유니크 어노테이션을 붙여준다.
			 4. 보통 유니크 제약조건이 붙는 순간 테이블 자체를 수정하기가 어려워져서 진짜 확실하게 결정된 사항의 경우에만 붙여주는게 좋다.

##### 테스트 구현

&emsp;&emsp;

- 설명
	1. Mock 또는 Stub을 만들어서 테스트를 구현한다.
		1. stub은 상태기반 검증으로 주로 사용되며 어떤 상태여야하는지를 검증할 때 사용된다.
		   일반적으로 고전파 테스트에 주로 사용되며 왠만해서는 테스트가 쉽게 깨지지 않는다.
		2. 목은 주로 상호작용 기반 검증에 사용되며 어떤 상호작용이 일어났는가를 검증할 때 사용된다.
		   일반적으로 런던파 테스트에 주로 사용되며 테스트가 쉽게 깨지곤 한다.
		3. 강의에서는 mock으로 테스트를 진행하는데 테스트를 작성하는 속도는 mock이 압도적으로 빠르다. 다만 테스트가 잘 깨지기 때문에 작성은 빠르지만 변화에 대해서는 그리 빠르게 대응되긴 힘들어진다. 일반적으로 서비스가 어느정도 빌드업된 상황에서는 단위테스트를 스텁으로 세밀하게 짜는 것을 더 선호한다.
		4. 토비님도 기능이 크게 바뀐게 없는데 내부 구조를 리팩터링했더니 테스트도 같이 리펙터링되어야했다면 테스트대상(이하 SUT)설정이 문제가 있는거 아닌가 하고 되돌아봐야한다 라고 이야기하신다.
		5. 목은 주로 실수로 구현에 변경을 줬다거나 하는 경우에 가장 확실한 이득을 가져다준다. 토비님도 그것을 이메일 전송을 기준으로 보여주신다.
			1. 목은 이런식으로 외부 시스템에 의존적인 기능을 테스트할때 가장 효과가 좋다.
	2. 구현한 서비스를 대상으로 테스트를 작성한다.
		1. 토비님은 인터페이스를 기준으로 서비스 이름도 바뀌는게 맞긴하다고 말씀하시지만 강의에서는 일단 간단하게 진행한다.
	3. SUT에 쓰일 스텁과 목을 동시에 작성한다.
		1. 목은 주로 외부 시스템에 의존적이라면 구현해주는 것이 좋고 토비님은 일부러라도 강요해서라도 스텁으로 일단 작성하는 것을 권장하신다.
		2. 나도 스텁으로 테스트하는 것을 가장 잘 사용하곤하는데 실제 프로젝트에서는 그러지 못해서 반성하고 잘 작성해야겠다.
		3. 모키토를 사용하는 경우 any()와 eq() 조합을 잘 사용해서 적용해야한다.
			1. 검증을 원하는 값은 eq()로 검증하고 아닌 값은 any()로 통과시키면 된다. 둘다 matcher 스태틱 메서드다.
			2. 라이브템플릿에 다양한 테스트 패턴들을 등록해두고 빠르게 짜서 넘기는 과정들을 해둬야한다고 생각한다. 토비님도 테스트를 빠르게 만드는 방법을 고안하는게 중요하다고 강조하셨다.
		4. 참고로 모키토를 사용한다고 해서 목으로만 개발을 할 수 있는건 아니다. 모키토를 사용하더라도 단순히 지정된 값을 반환하게 만들면 스텁처럼 개발을 할 수 있는 것이다.
	4. required 인터페이스로 만들어진 특히 레포지터리의 경우에는 메모리 DB를 이용하는 것이 가장 좋다고 말씀하신다.
		1. 나는 왠만해서 실제 DB를 그대로 쓸 수만 있다면 가져다 쓰고 환경만 잘 분리를 해두면 된다는 마인드이다.
		2. 물론 테스트를 할때는 실제 데이터를 넣어두고 사용하거나 그런다면 외부 의존성이 추가된 것이라 애매하겠지만 이건 테스트를 구성하기 나름이다. BeforeEach로 미리 데이터를 저장한다던가 given 절에서 미리 환경을 구성해준다던가 하면 왠만해서는 거의 대다수 로직 테스트는 가능하다.
	5. 서비스를 통합 테스트하실때 SpringBootTest 어노테이션을 붙여서 테스트 하신다.
		1. 사실상 통합테스트이고, 각각의 유닛테스트가 되어있다면 진행하는 테스트이다.
		2. 이때 필요한 빈은 실제 빈을 사용하면 안되고 `@TestConfiguration`으로 테스트용 빈을 넣어서 원하는 데이터를 반환하게끔 만드는 방식이다.
			1. 따라서 의존성이 분리가 안된건 아니고 빈으로서 필요한 로직을 사용한다고 보면 된다.
			2. 이 컨피그 클래스는 외부에 만들면 `@Import`를 해줘야한다.
			3. `@TestConfiguration`이 우선이고 없으면 프로덕션쪽에서 빈을 찾는다.
			4. 애초에 컴퍼넌트 스캔 범위를 빼두는 것이 좋다.
	6. 도메인에서 사용되는 예외는 전부 도메인 모델 문서에 정리해줘야한다.
	7. public 메서드에는 간결하게 볼 수 있도록 디테일한 로직은 private 메서드로 분리를 해주자
	8. 항상 트랜잭셔널을 넣어주자.
	9. `@Validated`를 이용해 유효성 검사를 해주자.
	10. 일반적으로 ID를 찾을려했지만 못찾겠다는 예외는 사용자에게 알려주면 안된다. 이런 경우는 orElseThrow 정도로만 걸러주고 사용자에게는 서버에 문제가 생겼다고만 알려줘야한다.
	11. 상태를 저장할 때는 save()를 호출해줘야한다. 이게 공식이다.
		1. 그래야 다른 DB를 사용해도 손댈 이유가 없어진다.
		2. 도메인 이벤트 발행(Domain Event Publication)이 발생하지 않기 때문에 이것을 회피하기 위한 이유가 가장 크다.
		3. auditing을 위해서도 반드시 필요하다.
	12. 리포지터리 테스트시 given 절에서 어느 케이스를 구축 했다면 이후 flush, clear까지 해줘야 제대로 given절이 구현된 것이다.
	13. cqrs는 미리 구현을 해줘야 테스트 짤때부터 훨씬 편하다.
		1. 쿼리 자체를 구현을 해두고 예외처리까지 해주면 서비스에서 쿼리용 포트를 주입받아 사용할때 훨씬 깔끔하게 사용가능해진다.
	14. 기능을 얼추 구현하면, 코드 폴리싱 작업을 해주면 좋다.
		1. 보통 이걸 할 시간은 현업에서는 잘 없다.
		2. 테스트 커버리지를 최대한 높이시는 것을 토비님은 좋아하시는 것 같지만 실전에서는 70프로만 넘기면 되는 경우가 대다수이다.
		3. 보통 무의미한 메서드명을 수정해서 의미를 부여해주기, 안쓰는 import 제거하기 이정도다.
			1. organize import 이 버튼을 사용하면 한번에 지워준다.
		4. 소나린트를 쓰면 매 커밋마다 얼추 해결된다.
	15. 더미로 붙여둔 스텁 컴포넌트의 경우, 보통 `@Primary`나 `@Qurifier`를 붙여서 실제 구현 빈을 주입 받겠지만 토비님의 경우에는 그냥 이런 스텁 컴포넌트에 `@FallBack` 어노테이션을 붙여 구현 빈이 아니니 실제 구현 빈이 있는 경우 무시하도록 만들었다.
	16. Application 클래스도 mockito로 테스트도 가능하다.
		1. 뭐 굳이 필요한가 싶긴하다.
		2. mockStatic()을 써야 static 메서드에 대해 테스트를 할 수 있다.
		3. 참고로 try-with-resource 구문으로 해줘야한다. 안그럼 테스트가 간헐적으로 실패한다.
	17. 모키토는 최신 버전을 사용하기 위해서는 미리 모키토 에이전트를 따로 주입시켜줘야한다.
