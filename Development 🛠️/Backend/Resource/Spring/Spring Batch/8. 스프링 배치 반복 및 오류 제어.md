# 배치 반복 및 오류 제어

&emsp;&emsp;

## 배치 반복

&emsp;&emsp;

### Repeat

&emsp;&emsp;

- 정의
	- 얼마나 작업을 반복할지 알려주는 기능

&emsp;&emsp;

- 특징
	- 특정 조건의 만족할 때까지( 또는 특정 조건이 아직 만족되지 않을 때까지) Job 또는 Step을 반복해서 배치 애플리케이션을 구성할 수 있다.
	- 스프링 배치에서는 Step과 반복 Chunk 반복을 RepeatOperation을 사용해서 처리하고 있다.
	- 기본 구현체는 RepeatTemplate을 제공한다.
		- **Step과 Chunk 반복 구간마다 사용**되며 이 **구간들마다 주요 도메인 객체를 통해 제어**할 수 있다.

&emsp;&emsp;

- Repeat 기본 동작 구조
	1. Job 내부에 있는 Step이 실행된다.
	2. Step 내부에 있는 반복 제어 구현체인 RepeatTemplate를 통해 Tasklet을 실행한다.
	3. Tasklet 내부에 있는 반복 제어 구현체인 RepeatTemplate를 통해 청크 블록을 실행한다.

&emsp;&emsp;

- 주요 도메인 객체
	- `RepeatStatus`
		- 스프링 배치 처리 결과를 정의한 enum
		- `CONTINUABLE`
			- 작업이 더 남아있음을 뜻함
			- 반복 추가 수행
		- `FINISHED`
			- 작업이 더 남아있지 않음을 뜻함
			- 반복 중단 수행
	- `CompletionPolicy`
		- RepeatTemplate iterate 메소드 안에서 반복을 중단할 지 결정
		- 실행 횟수 또는 완료 시기, 오류 발생시 수행할 작업에 대한 반복 여부 결정
		- 정상 종료를 알리는데 사용
		- 여러개를 set 했다면 마지막에 적용한 CompletionPolicy만 적용된다.
			- `CompositeCompletionPolicy`에 여러개를 적용하고 이 `CompositeCompletionPolicy`를 적용해야 여러 종료 조건을 전부 검증하며 Or 조건으로 검증된다. 둘 중 하나라도 참이면 참이다.
		- API 종류
			- `boolean isComplete(RepeatContext, RepeatStatus)`
				- 콜백의 최종 결과를 참조해 배치가 완료되어있는지 검증
			- `boolean isComplete(RepeatContext)`
				- 콜백이 완료 될 때까지 기다리지 않고 바로 완료되어있는지 검증
		- 구현체 종류
			- `SimpleCompletionPolicy`
				- 현재 반복 횟수가 청크 개수보다 크면 종료
			- `CountingCompletionPolicy`
				- 일정한 카운트를 계산 및 집계하여 카운트 제한 조건이 만족하면 반복 종료
			- `TimeoutCompletionPolicy`
				- 반복 시점부터 현재 시점까지 소요된 시간이 설정된 시간보다 크면 반복 종료
			- `CompositeCompletionPolicy`
				- 여러 종료 정책을 뭉쳐둔 정책들을 하나씩 검증 하여 반복 종료
	- `ExceptionHandler`
		- RepeatCallback 안에서 예외가 발생하면 RepeatTemplate가 ExceptionHandler를 참조해서 예외를 다시 던질지 여부 결정
		- 예외를 받아서 다시 던지게 되면 반복 종료
		- 비정상 종료를 알리는데 사용된다.
		- API 종류
			- `void handleException(RepeatContext, Throwable) throws Throwable`
		- 구현체 종류
			- `LogOrRethrowExceptionHandler`
				- 예외를 로그로 기록할지, 아니면 다시 던질지 결정
				- 로그로 기록되면 반복문은 다시 수행된다.
				- 다시 던지면 반복은 종료된다.
			- `RethrowOnThresholdExceptionHandler`
				- 지정된 유형의 예외가 임계 값에 도달하였다면 이때 다시 예외를 던짐
				- 아직 임계 값 도달 전이라면 반복 문은 다시 수행된다.
				- 다시 던지면 반복은 종료된다.
			- `SimpleLimitExceptionHandler`
				- 예외 타입 중 하나가 발견되면 정수 카운터가 증가하고 한계를 초과하였는지 여부를 검증하고 초과하였다면 Throwable을 다시 던짐
				- 정수 카운터가 한계를 초과하지 않았다면 반복은 다시 수행된다.
				- 다시 던지면 반복은 종료된다.
			- `CompositeExceptionHandler`
				- 여러 예외 핸들러를 뭉쳐서 하나씩 검증하여 던질지 결정
				- 다시 던지면 반복은 종료된다.
			- `DefaultExceptionHandler`
				- 기본 예외 핸들러

&emsp;&emsp;

- RepeatOperations 동작 구조
	1. `RepeatOperations`의 `RepeatStatus iterate(RepeatCallback)`을 통해 `RepeatCallback`을 반복 호출한다.
	2. `RepeatTemplate`를 통해 반복을 종료할 때까지 `RepeatCallback`을 반복해서 콜백을 호출한다.
	3. `RepeatCallback`의 `RepeatStatus doInteration(RepeatContext)`를 통해 구현된 비즈니스 로직을 실행하한다.
		1. 수행되는 로직은 tasklet에 구현한 비즈니스 로직이다.
			1. 만약 수행 도중 예외가 발생된다면, 그리고 `ExceptionHandler`를 적용했다면 이 예외 정책에 따라 반복 여부 결정이 가능하다.
		2. 수행 도중 일시적으로 사용할 필요가 있는 데이터는 `RepeatContext`에 저장하고 반복 종료와 함께 데이터는 삭제된다.
		3. 수행이 종료되면, 그리고 `CompletionPolicy`를 설정했다면 이 종료 조건 정책에 따라 반환되는 `RepeatStatus`에 따라 반복 여부 결정이 가능하다.

### FaultTolerant

&emsp;&emsp;

- 정의
	- 스프링 배치가 Job 실행 중 오류 발생이나 장애를 처리하기 위한 기능으로 복원력을 향상시키 위한 내결함 기능이다.

&emsp;&emsp;

- 특징
	- 오류가 발생해도 Step이 즉시 종료되지 않고 , Retry 혹은 Skip 기능을 활성화하여 내결함성 서비스가 가능하도록 한다.
	- 프로그램 내결함성을 위해 Skip과 Retry 기능을 제공한다.
	- Skip은 ItemReader, ItemProcessor, ItemWriter에 적용 가능하다.
	- Retry는 ItemProcessor, ItemWriter에만 적용 가능하다.
	- FaultTolerant 구조는 청크 기반 프로세스 기반위에 Skip과 Retry 기능이 추가되어 재정의 되어있다.
	- `stepBuilder` API를 통한 `FaultTolerantStepBuilder` API 종류
		- `<I, O>chunk(int)`
			- 반드시 먼저 호출되어야하는 메서드
		- `reader(ItemReader)`
		- `writer(ItemWriter)`
		- `faultTolerant()`
			- 이 메서드를 호출해야 `FaultTolerantStepBuilder`가 반환되어 `FaultTolerantStepBuilder` API 이용이 가능하다.
		- `skip(Class<? extends Throwable>)`
			- 예외 발생시 Skip 해야할 예외
		- `skipLimit(int)`
			- skip 제한 횟수
		- `skipPolicy(SkipPolicy)`
			- skip을 어떤 조건과 기준으로 적용할 지 검증 조건 설정
		- `noSkip(Class<? extends Throwable>)`
			- 예외 발생시 Skip 하지 않을 예외 타입 설정
		- `retry(Class<? extends Throwable>)`
			- 예외 발생시 retry 할 예외 타입 설정
		- `retryLimit(int)`
			- retry 제한 횟수 설정
		- `retryPolicy(RetryPolicy)`
			- retry를 어떤 조건과 기준으로 적용할 것인지 정책 설정
		- `backOffPolicy(BackOffPolicy)`
			- retry 하기 까지의 지연시간 설정
		- `noRetry(Class<? extends Throwable>)`
			- 예외 발생 시 retry 하지 않을 예외 타입 설정
		- `noRollback(Class<? extends Throwable>)`
			- 예외 발생 시 rollback 하지 않을 예외 타입 설정

&emsp;&emsp;

- 기본 구조
	- `SimpleStepBuilder` 인터페이스
		- `SimpleChunkProcessor`
		- `SimpleChunkProvider`
	- `FaultTolerantStepBuilder`, `SimpleStepBuilder` 재정의 구현체
		- `FaultTolerantChunkProcessor`, `SimpleChunkProcessor` 재정의 구현체
		- `FaultTolerantChunkProvider`, `SimpleChunkProvider` 재정의 구현체

&emsp;&emsp;

- 기본 동작 구조
	1. `ChunkOrientedTasklet`이 `ChunkProvider`의 메소드 `provide()`를 호출해 `Chunk` 데이터 읽기 프로세스를 수행한다.
	2. `ChunkProvider`가 `ItemReader`의 `read()`를 호출해 청크를 읽어 들인다.
	3. 이 과정은 chunkSize 만큼 반복된다.
	4. 이 와중 예외 발생시 이 던져진 예외가 `Skip` 설정에 걸어둔 예외라고 하면 반복하지 않고 skipCount 만큼 예외를 건너 뛴다.
	5. 읽기 처리가 끝나면 `ChunkOrientedTasklet`이 `ChunkProcessor`의 메소드 `process(inputs)`를 호출해 `Chunk`의 데이터 가공 프로세스를 수행한다.
	6. `ChunkProcessor`는 `RetryTemplate`의 `execute()` 메서드를 호출해 Chunk 사이즈 만큼 읽기를 시도해 읽어들인 item 개수 만큼 처리를 반복 수행 시도한다.
	7. `RetryTemplate`은 `ItemProcessor`의 메서드 `process()`메서드를 호출해 데이터 가공 처리를 한다.
	8. 이 과정은 itemReader가 읽어 들인 갯수 만큼 수행한다.
	9. 이 와중 예외 발생시  이 던져진 예외가`Retry` 설정에 걸어둔 예외라고 하면 retry를 retryCount 만큼 시도한다.
		1. 또는 skip설정에 걸어둔 예외라고 하면 skipCount만큼 시도를 건너 뛴다.
	10. process() 과정 중 예외 발생 후 retry, skip 처리 이후 마지막에는 RecoveryCallback이 호출된다.
		1. skip 검증 후 retry 시도가 더 이상 불가능한지 검증하고 이곳에서 `RetryException`을 던진다.

### Skip

&emsp;&emsp;

- 정의
	- 데이터 처리 도중 예외가 던져지는 경우 해당 데이터 처리를 건너뛰는 기능

&emsp;&emsp;

- 특징
	- 데이터의 사소한 오류에 대해 Step 실패처리 대신 Skip을 함으로써, 배치 수행의 빈번한 실패를 줄여준다.
	- 오류 발생시 스킵 설정에 의해 예외를 던진 Item을 제외하고 그 다음 Item 부터 수행한다.
	- ItemReader는 예외 발생시 해당 아이템만 스킵하고 계속 읽기 처리를 진행한다.
	- ItemProcessor와 ItemWriter는 예외 발생시 Chunk의 처음으로 돌아가 예외를 던졌던 아이템을 제외한 나머지 아이템들을 가지고 처리하여 스킵하게 된다.
		- ==**한번 ItemWriter에서 예외가 발생하면 Writer의 경우에만 개별 처리 프로세스로 넘어간다.**==
		- 당연히 이 과정이 매우 비효율적이기 때문에 왠만하면 대부분 검증은 processor에서 쓰기 수행이 가능한 Item인지 검증 해야한다.
	- 내부적으로 SkipPolicy를 통해 구현되어있다.
	- ==**skipLimit는 ItemReader, ItemProcessor, ItemWriter 횟수 합**==이다.
		- 임계치로써 설정되는 값이며 이 이상 실패하면 step이 실패하므로 job 자체가 실패한다.
	- Skip 여부 판별 방식
		- 스킵 대상에 포함된 예외인지 검증
		- 스킵 카운터를 초과한 상태인지 검증
		- 둘 다 검증 실패시 Step이 실패하며 당연히 Job은 실패로 끝난다.
		- AND 조건으로 모두 true 인 경우에만 스킵 기능이 동작된다.
	- SkipPolicy API
		- `SkipPolicy` 인터페이스
			- `boolean shouldSkip(Throwable t, int skipCount)`
		- `SubclassClassifier<Throwable, Boolean>`
			- `Map<Class, boolean> classified`
				- 예외 분류 클래스 맵
				- `noSkip()`를 통해 등록된 예외 클래스는 여기에 `false`로 저장된다.
	- SkipPolicy 구현체 종류
		- `LimitCheckingItemSkipPolicy`
			- 설명
				- Skip 카운터 및 예외 분류 맵에 따라 skip 여부 결정
				- 기본 SkipPolicy이다.
			- `int skipLimit`
				- 스킵 횟수
			- `Classifier<Throwable, Boolean>`
				- 예외 분류 객체
		- `AlwaysSkipItemSkipPolicy`
			- 항상 Skip 한다.
		- `ExceptionClassifierSkipPolicy`
			- 예외 대상을 분류해 Skip 여부 결정
		- `CompositeSkipPolicy`
			- 여러 SkipPolicy를 탐색해 검증하여 skip 여부 결정
			- 각 SkipPolicy 중 하나라도 Skip 판정이 되면 Skip 된다.
		- `NeverSkipItemSkipPolicy`
			- 항상 skip을 하지 않는다.

### Retry

&emsp;&emsp;

- 정의
	- ItemProcess, ItemWriter 과정 중 던져진 예외가 미리 지정해둔 재시도 예외인 경우 재시도를 하는 기능

&emsp;&emsp;

- 특징
	- Skip과 매한가지로 배치 수행의 빈번한 실패 방지 기능
	- 예외 던짐 발생시 재시도 설정에 의해 chunk 단계 처음부터 시도한다.
		- 아이템은 ItemReader에서 캐시된 아이템 리스트를 사용하기 때문에 읽기 처리는 건너뛴다.
		- 단, `RepeatTemplate`이 아닌 `RetryTemplate`을 통해 재시도가 수행된다.
			- 따라서 동작상 chunk 처음단계인 것 같지만 skip과 달리 그게 아니라 Step의 `RepeatTemplate`이 호출됬던 단계까지 되돌아간다.
	- skip과 달리 inputs에 변동은 없고 단순 재시도만 한다.
	- Retry 기능은 내부적으로 RetryPolicy를 통해서 구현되어있다.
		- BackOffPolicy 구현체도 사용해야한다.
		- 재시도 하기 까지 지연시간을 설정한다.
		- 처리시간이 길다면 재시도 간격 조절이 가능하다.
	- Retry 여부 판별 방식
		- 재시도 대상에 포함된 예외인지 검증
		- 재시도 카운터를 초과했는지 검증
		- Skip과 동일하게 둘 다 AND 조건으로 true여야 재시도가 동작된다.
	- ==**무조건 맨 처음 RetryTemplate을 통해 재시도 정책을 검사한 후 청크를 처리한다.**==
		- 특이하게 skip이 발생하면 repeat 내부에서 검증하던 것과 다른 아키텍쳐이다.
		- 항상 RetryTemplate 내부에서 청크 처리가 이뤄지기 때문에 재시도가 가능한 것이다.
		- 청크 처리 도중 예외가 발생하면 `RetryPolicy`를 통해 검증한다.
			- 검증 이후 `BackOffPolicy`를 통해 Step 반복문 처음부터 다시 시작한다.
				- 보통 여기서 시간 지연을 두는 경우가 대부분이다.
	- RetryContext가 Item 마다 생성되어 추적되며 이 정보를 바탕으로 재시도를 한 item인지 카운팅하거나 할 수 있다.
		- ==**따라서 Skip과 달리 Retry의 Count는 Item 별로 개별 합산 방식이다.**==
	- Retry와 Skip을 동시에 설정했다면, RecoveryCallback에서 Skip 여부를 검증한다.
		- 이때 skip이 가능하면 해당 아이템이 제거된 체 다시 반복한다.
			- 다시 청크 프로세스부터 시작한다.
	- 만약 직접 커스텀하여 RetryTemplate을 세팅한다고 가정하면 이때 RetryCallback과 RecoveryCallback을 따로 세팅하고 거기에 DefaultRetryState를 세팅해서 넘겨줘야한다.
		- 만약 **DefaultRetryState를 세팅하지 않게되면 null로 인정되어 shouldRethrow() 과정중 null 검사에서 false로 판별되어 Retry가 발생하지 않는다.**
	- `RetryOperations` 인터페이스 API
		- `T execute(RetryCallback<T, E>)`
			- `RetryCallback`을 재시도 설정만큼만 호출
		- `T execute(RetryCallback<T, E>, RecoveryCallback<T>)`
			- `RetryCallback`을 재시도 설정만큼만 호출 한 뒤 `RecoveryCallback`을 한 번 호출
	- `RetryTemplate` 인터페이스
	- `RetryCallback`, `RetryTemplate` 인터페이스 구현체
		- `T doWithRetry(RetryContext)`
	- `RecoveryCallback`, `RetryTemplate` 인터페이스 구현체
		- `T recover(RetryContext)`
	- `RetryPolicy` 인터페이스 API
		- `boolean canRetry(RetryContext)`
		- `RetryContext open(RetryContext)`
		- `void close(RetryContext)`
		- `void registerThrowable(RetryContext, Throwable)`
	- `RetryPolicy` 구현체 종류
		- `AlwaysRetryPolicy`
			- 항상 재시도를 허용
		- `ExceptionClassifierRetryPolicy`
			- 예외 분류 검증 이후 재시도 검증
		- `CompositeRetryPolicy`
			- 여러 RetryPolicy를 탐색하면서 재시도 검증
		- `SimpleRetryPolicy`
			- 재시도 횟수 및 등록된 예외 분류 맵에 따라 재시도 검증
			- 기본으로 설정된다.
			- `Classifier<Throwable, Boolean>`
			- `int retryLimit`
		- `MaxAttemptsRetryPolicy`
			- 재시도 횟수에 따라 재시도 여부 결정
		- `TimeoutRetryPolicy`
			- 주어진 시간에 따라 재시도 허용
		- `NeverRetryPolicy`
			- 최초에 한 번만 허용하고 그 이후로는 허용하지 않음

### Skip & Retry 정리

&emsp;&emsp;

#### ItemReader Skip

&emsp;&emsp;

- 동작 구조
	1. Step에서 `RepeatTemplate`을 통해 청크 기반 테스클릿을 실행
	2. `ChunkOrientedTasklet`에서는 `SimpleChunkProvider`를 상속 받은 `FaultTolerantChunkProvider`를 통해 다시 `RepeatTemplate`을 수행한다.
	3. `ItemReader`가 `read()`를 수행하는 도중 `Exception`이 발생하는 경우 만약 Skip 예외 인지 검증한다.
		1. Skip 예외인 경우 그냥 무시하고 다시 `FaultTolerantChunkProvider`가 제공하고 있는 `RepeatTemplate`문을 진행한다.
	4. Skip 예외가 아닌 경우 전체 `RepeatTemplate` 반복문이 종료된다.


&emsp;&emsp;

#### ItemProcessor Skip & Retry

&emsp;&emsp;

- 동작 구조
	1. Step에서 `RepeatTemplate`을 통해 청크 기반 테스크릿을 실행
	2. `ChunkOrientedTasklet`에서는 `SimpleChunkProvider`를 상속 받은 `FaultTolerantChunkProvider`를 통해 다시 `RepeatTemplate`이 아닌 `ChunkIterator`가 제공하는 `RetryTemplate`을 수행한다.
	3. RetryCallback을 통해 Retry를 해야하는 예외 상황인지 검증한다.
		1. 예외 상황이지만 Retry가 불가능한 경우 `RecoveryCallback`을 호출한다.
			1. 리커버리를 수행하여 스킵이 가능하다면 아이템을 제거하고 `RetryTemplate` 처음으로 돌아가 다음 아이템을 시도한다.
			2. Skip이 불가능한 경우 Skip 초과 예외를 던져 Step이 실패 종료된다.
	4. 예외 상황이 아니라면 `ItemProcessor`가 `process()`를 수행
	5. `process()` 수행 도중 예외 발생 후 RetryLimit이 초과한 경우 Step을 전체를 재시도한다.
		1. `process()` 수행 도중 예외 발생 후 RetryLimit이 초과하지 않은 경우`RetryTemplate` 처음으로 돌아가 다음 아이템을 시도한다.


&emsp;&emsp;

#### ItemWriter Skip & Retry

&emsp;&emsp;

- 동작 구조
	1. Step에서 `RepeatTemplate`을 통해 청크 기반 테스크릿을 실행
	2. `ChunkOrientedTasklet`에서는 `SimpleChunkProvider`를 상속 받은 `FaultTolerantChunkProvider`를 통해 다시 `RepeatTemplate`이 아닌 `ChunkIterator`가 제공하는 `RetryTemplate`을 수행한다.
	3. RetryCallback을 통해 Retry를 해야하는 예외 상황인지 검증한다.
		1. 예외 상황이지만 Retry가 불가능한 경우(RetryLimit 초과 등) `RecoveryCallback`을 호출한다.
			1. 리커버리를 수행하여 스킵이 가능하다면 `doScan()` 으로 복구작업을 한다.
			2. 스킵이 불가능한 경우 탈진 재시도 예외를 던져 Step이 실패 종료된다.
	4. 예외 상황이 아니라면 ItemWriter의 `write()`를 수행한다.
	5. `write()`를 수행 도중 예외가 발생하는 경우 Step을 재시도한다
		1. `write()`를 수행 도중 예외가 발생하지 않는다면 Step 성공 종료한다.
