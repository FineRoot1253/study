# 스프링 배치 실행 - Flow

## Job and Flow

&emsp;&emsp;

### FlowJob

&emsp;&emsp;

#### FlowJob 개념

&emsp;&emsp;

- 정의
	- Step을 순차적으로 구성하는 것이 아닌 특정 상태에서 흐름이 전환되어야하는 경우에 사용되는 Job

&emsp;&emsp;

- 특징
	- FlowJobBuilder에 의해 생성
	- 사용 예시
		- Step이 실패해도 Job은 실패하지 않아야하는 경우
		- Step이 성공했을 때 다음에 실행할 Step을 구분해서 실행해야 하는 경우
		- 특정 Step은 전혀 실행되지 않게 구성해야 하는 경우
	- Flow와 Job의 흐름을 구성하는데만 관여하고 실제 비즈니스 로직은 Step에서 이루어진다.
	- 내부적으로 SimpleFlow 객체를 포함하고 있으며 Job 실행시 호출한다.
	- all or Nothing이 아닌 항상 일부 흐름으로는 완료시켜야하는 경우에 유용하다.
	- FlowJob의 생성구조가 SimpleJob보다 굉장히 복잡하고 다양한 API를 제공한다.

&emsp;&emsp;

- SimplJob vs FlowJob
	- SimpleJob
		- 순차적 흐름
			- a, b, c 스탭 중 a가 실패하면 b, c는 수행하지 않음
	- FlowJob
		- 조건적 흐름
			- a, b, c 스탭중 a가 실패하거나 성공하는 여부에 따라 b 또는 c 수행

&emsp;&emsp;

#### JobFlowBuilder API

&emsp;&emsp;

- 종류
	- start(Flow or Step)
		- Flow를 시작하는 Step 설정
		- 처음에 실행할 Flow를 설정한다.
		- 이곳에 Flow를 집어넣으면 Flow api를 바로 사용할 수 있는 JobFlowBuilder를 반환하고 step을 집어넣으면 SimpleJobBuilder를 이용한다.
		- 단순히 on 없이 flow()를 넣어 구분을 지으면 flow 별로 테스트를 할 수 있어서 많이들 사용한다.
		- 보통 Job은 하나의 작업 단위 이지만 그 내부의 연관성이 있는 논리 흐름들을 묶어주는 역할은 Flow()가 한다.
		- **flow의 흐름을 정의하는 역할**
	- next(Step or Flow or JobExecutionDecider)
		- 다음으로 이동할 Step을 지정한다.
		- **flow의 흐름을 정의하는 역할**
	- from(Step)
		- 이전 단계에서 정의한 Step의 Flow를 추가적으로 정의한다.
		- **flow의 흐름을 정의하는 역할**
	- on(String pattern)
		- Step의 실행 결과를 돌려받는 종료 상태를 캐치하여 매칭하는 패턴
		- TransitionBuilder를 반환한다.
		- **조건에 따라 흐름을 전환시키는 역할**
	- to(Step)
		- 다음으로 이동할 Step을 지정한다.
		- **조건에 따라 흐름을 전환시키는 역할**
	- stop()
		- Flow를 중지한다.
		- **조건에 따라 흐름을 전환시키는 역할**
	- fail()
		- Flow를 실패처리한다.
		- **조건에 따라 흐름을 전환시키는 역할**
	- end()
		- Flow를 종료한다.
		- build()보다 앞에서 사용하면 FlowBuilder를 닫고 SimpleFlow 객체를 생성한다.
		- **조건에 따라 흐름을 전환시키는 역할**
	- stopAndRestart()
		- Flow를 중지하고 재시작한다.

&emsp;&emsp;

	- next(Step or Flow or JobExecutionDecider)
	- from(Step or Flow or JobExecutionDecider)
	- on(String pattern)

### Transition 

&emsp;&emsp;

#### BatchStatus 개념

&emsp;&emsp;

- 정의
	- JobExecution과 StepExecution의 속성으로 Job과 Step의 종료 후 최종 결과 상태가 무엇인지 정의

&emsp;&emsp;

- Job 별 특징
	- SimpleJob
		- 마지막 Step의 BatchStatus 값을 Job의 최종 BatchStatus 값으로 반영
		- Step이 실패할 경우 해당 Step이 마지막 Step이 된다.
	- FlowJob
		- Flow 내 Step의 ExitStatus 값을 FlowExecutionStatus 값으로 저장
		- 마지막 Flow의 FlowExectionStatus 값을 Job의 최종 BatchStatus 값으로 반영한다.

&emsp;&emsp;

- 상태 종류
	- COMPLETED
	- STARTING
	- STARTED
	- STOPPING
	- STOPPED
	- FAILED
	- ABANDONED
		- 실패했지만 재시작시 건너뛰는 단계
	- UNKNOWN

&emsp;&emsp;

#### ExitStatus 개념

&emsp;&emsp;

- 정의
	- JobExecution과 StepExecution의 속성으로 Job과 Step의 실행 후 어떤 상태로 종료되었는지 정의
	- 일반적으로 ExitStatus와 BatchStatus는 동일한 값으로 설정된다.

&emsp;&emsp;

- Job 별 특징
	- SimpleJob
		- 마지막 Step의 ExitStatus와 값을 Job의 최종 ExitStatus와 값으로 반영
	- FlowJob
		- Flow 내 Step의 ExitStatus 값을 FlowExecutionStatus 값으로 저장
		- 마지막 Flow의 FlowExectionStatus 값을 Job의 최종 BatchStatus 값으로 반영한다.

&emsp;&emsp;

- 상태 종류
	- EXECUTING
	- COMPLETED
	- NOOP
	- FAILED
	- STOPPED

&emsp;&emsp;

- 구현
	- step의 tasklet 구현시 contribution에서 setExitStatus()를 사용해 상태를 지정해줄 수 있다.
		- 이때 step exitStatus가 job에도 반영된다.
		- 다만, BatchStatus는 따로 지정해주지 않으면 별개로 설정된다.
	- flowExection의 경우에도 결과 값이 FlowJob에 반영된다.

&emsp;&emsp;

#### Transition 개념

&emsp;&emsp;

- 정의
	- Flow내 Step의 조건부 전환(전이)를 정의

&emsp;&emsp;

- 특징
	- Job의 API 설정에서 on(String pattern) 메소드를 호출하면 TransitionBuilder가 반환되어 Transition Flow를 구성할 수 있음
	- Step의 종료상태(ExitStatus)가 어떤 Pattern과도 매칭되지 않으면 스프링 배치에서 예외를 발생하고 Job은 실패한다.
	- transition은 구체적인 것부터 그렇지 않은 순서로 적용된다.
	- **FlowExectionStatus은 Job의 BatchStatus와 ExitStatus에만 영향을 끼치며 Step에는 영향을 끼치지 않는다.**

&emsp;&emsp;

#### Transition API

&emsp;&emsp;

- 종류
	- 조건 정의 API 종류
		- on(String pattern)
			- TransitionBuilder를 반환한다.
			- 일종의 if 역할
			- 조건부 api
			- Step의 실행 결과인 ==**ExitStatus와 매칭**==하는 패턴 스키마
				- ==**BatchStatus와 매칭하는 것이 아니다!!**==
			- 특수 문자는 두 가지만 허용한다.
				- "*"
					- 0 개 이상의 문자와 매칭, 모든 ExitStatus와 매칭된다.
				- "?"
					- 정확히 1개의 문자와 매칭
				- 예시
					- `c*t`: cat, count 등 매칭 가능
					- `c?t`: cat만 매칭 가능
		- to(Step or Flow or JobExecutionDecider)
			- 조건부 api인 on(String pattern)이 참인 경우 동작하는 Step or Flow or JobExecutionDecider
			- 다음으로 실행할 단계를 지정
		- from()
			- 이전 단계에서 정의한 transition을 새롭게 추가해서 정의함
			- 일종의 else 역할
	- 조건에 따른 행동 정의 API 종류
		- stop()
			- 조건부 api인 on이 거짓인 경우 동작
			- FlowExecutionStatus가 STOPPED 상태로 종료되는 transition
			- Job의 BatchStatus, ExitStatus도 STOPPED로 종료된다.
		- fail()
			- 조건부 api인 on이 거짓인 경우 동작
			- FlowExecutionStatus가 FAILED 상태로 종료되는 transition
			- Job의 BatchStatus, ExitStatus도 FAILED로 종료된다.
		- end()
			- 조건부 api인 on이 거짓인 경우 동작
			- FlowExecutionStatus가 COMPLETED 상태로 종료되는 transition
			- Job의 BatchStatus, ExitStatus도 COMPLETED로 종료된다.
			- **Step의 ExitStatus가 FAILED이더라도 Job의 BatchStatus가 COMPLETED로 종료하게 만든다.**
				- ==**따라서 이 경우 Job 재시작은 불가능하다.**==
		- stopAndRestart(Step or Flow or JobExecutionDecider)
			- 조건부 api인 on이 거짓인 경우 동작
			- stop과 기본 흐름은 동일하다. 
				- Step or Flow or JobExecutionDecider 중 하나를 재시작한다.
			- 특정 Step에서의 작업을 중단하도록 설정하면 중단 이전의 Step만 COMPLETED로 저장되고 이후의 Step은 실행되지 않고 STOPPED 상태로 Job이 종료된다.
			- Job이 다시 실행됬을때 실행해야 할 Step을 restart 파라미터로 넘기면 이전에 COMPLETED로 저장된 step은 건너뛰고 중단 이후 step 부터 시작한다.

### 사용자 정의 ExitStatus

&emsp;&emsp;

#### 사용자 정의 ExitStatus 개념

&emsp;&emsp;

- 정의
	- ExitStatus에 존재하지 않는 exitCode를 새롭게 정의해서 설정

&emsp;&emsp;

- 특징
	- 기본 동작 방법은 StepExecutionListener의 afterStep() 메서드에서 커스텀 exitCode 생성 후 새로운 ExitStatus 반환
	- Step 실행 후 완료 시점에서 현재 exitCode를 사용자 정의 exitCode로 수정할 수 있음
	- 만약 이렇게 on절에서 실패한 경우나 성공한 경우의 step이 발생하지 않으면 자동으로 Job은 실패한다.

### JobExecutionDecider

&emsp;&emsp;

#### JobExecutionDecider 개념

&emsp;&emsp;

- 정의
	- ExitStatus 조작, StepExecutionListener 등록 없이 Transition처리를 하는 클래스
&emsp;&emsp;

- 특징
	- Step과 Transition 역할을 명확히 분리할 수 있음
	- Step ExitStatus가 아닌 JobExecutionDecider의 FlowExecutionStatus 상태 값을 새롭게 설정해서 반환한다.
	- `FlowExecutionStatus decide(JobExecution jobExecution, StepExecution stepExecution)`

&emsp;&emsp;

#### FlowJob 아키텍처

&emsp;&emsp;

- 실행 흐름
	1.  JobLauncher가 JobParameters, JobInstance를 통해 JobExecution을 생성하여 ExecutionContext에 저장
		1. 이때 배치 상태는 STARTED
	2. FlowJob 생성
	3. JobListener beforeJob() 호출
	4. FlowExecutor 동작
	5. FlowExecution 객체 생성
	6. SimpleFlow 생성
	7. SimpleFlow의 State 속성에 들어있는 Step, Flow, Decider를 통해 Flow를 처리
	8. FlowExecutionStatus 결과 값을 반환
	9. FlowExecution에 반영
	10. JobListener afterJob() 호출
	11. JobExecution의 BatchStatus, ExitStatus를 FlowExecutionStatus에 맞춰서 업데이트

### SimpleFlow

&emsp;&emsp;

#### SimpleFlow 개념

&emsp;&emsp;

- 정의
	- 스프링 배치에서 제공하는 Flow의 구현체
	- 각 요소(Step, Flow, JobExecutionDecider)들을 담고 있는 State를 실행시키는 도메인 객체

&emsp;&emsp;

- 특징
	- FlowBuilder를 사용해서 생성하며 Transition과 조합하여 ==**여러 개의 Flow 및 중첩 Flow**==를 만들어 Job 구성이 가능하다.

&emsp;&emsp;

#### SimpleFlow API

&emsp;&emsp;

- 종류
	- Flow
		- `String getName()`
			- Flow 이름 조회
		- `State getState(String stateName)`
			- State 명으로 State 조회
		- `FlowExecution start(FlowExecutor executor)`
			- Flow를 실행
		- `FlowExecution resume(String stateName, FlowExecutor executor)`
			- 다음에 실행할 State를 구해서 FlowExecutor에게 위임
		- `Collection<State> getStates()`
			- Flow가 가지고 있는 모든 State를 Collection 타입으로 반환
	- SimpleFlow
		- `String name`
			- Flow 이름
		- `State startState`
			- 가장 처음 실행될 첫 State
		- `Map<String, Set<StateTransition>> transitionMap`
			- State 이름으로 매핑된 `Set<StateTransition>`
		- `Map<String, State> stateMap`
			- State 이름으로 매핑된 `State`
		- `List<StateTransition> stateTransitions`
			- State와 Transiton 정보를 가지고 있는 StateTransition 리스트
		- `Comparator<StateTransition> stateTransitionComparator`

&emsp;&emsp;

#### SimpleFlow 아키텍처

&emsp;&emsp;

- StateTransition 특징
	1.  FlowBuilder에서 start(), next(), from()에 각각 다른 값 지정에 값을 지정한다.
		1. step을 지정했다면 StepState라는 객체를 만들어 Step을 저장한다.
		2. flow를 지정했다면 FlowState를 객체를 만들어 Flow를 저장한다.
		3. decider를 지정했다면 DecisionState를 만들어 JobExecutionDecider를 저장한다.
		4. 그 외 split()의 경우에는 SplitState를 만들어 복수 개의 Flow를 저장한다.
	2. StateTransition의 속성으로 1단계에서 만든 속성들을 저장시키며 on()에서 지정한 transition과 다음 state를 들고 있다.
		1. 현재 state, 전이 조건, 다음 state를 들고 있다.
	3. SimpleFlow에는 이 StateTransition과 State들을 맵으로 가지고 있으며 특히 흐름 관련은 추가로 StateTransition List 형태로 가지고 있다.

&emsp;&emsp;

#### SimpleFlow 실행 흐름

- StepState의 경우
	1. FlowJob에서 .start()를 통해 실행
	2. SimpleFlow가 실행되며 state.handle(executor)를 호출하여 실행
		1. 이때 루프를 돌며 resume() 수행하며 state의 개수 만큼 수행한다.
		2. resume은 현재 state 실행이 완료되면 nextState()를 호출해 다음 state를 실행한다.
			1. 실행순서는 순차적 개념이 아닌 설정 및 조건에 따라 다르다.
		3. state가 null이거나 실행 불가능 상태면 loop는 종료된다.
	3. StepState에서 executor.executeStep() 호출
	4. jobFlowExecutor가 stepHandler.handlerStep(step)을 호출
	5. step에서 tasklet 실행
	6. 최종적으로는 FlowExecutionStatus를 반환

&emsp;&emsp;

- FlowState의 경우
	1. FlowJob에서 .start()를 통해 실행
	2. SimpleFlow가 실행되며 state.handle(executor)를 호출하여 실행
		1. 이때 루프를 돌며 resume() 수행하며 state의 개수 만큼 수행한다.
		2. resume은 현재 state 실행이 완료되면 nextState()를 호출해 다음 state를 실행한다.
			1. 실행순서는 순차적 개념이 아닌 설정 및 조건에 따라 다르다.
		3. state가 null이거나 실행 불가능 상태면 loop는 종료된다.
	3. FlowState의 flow.start() 호출
	4. SimpleFlow에서 다시 State들 중 하나를 실행
	5. 재귀적으로 2~4단계 반복 호출
	6. 최종적으로는 FlowExecutionStatus를 반환

&emsp;&emsp;

- DecisionState의 경우
	1. FlowJob에서 .start()를 통해 실행
	2. SimpleFlow가 실행되며 state.handle(executor)를 호출하여 실행
		1. 이때 루프를 돌며 resume() 수행하며 state의 개수 만큼 수행한다.
		2. resume은 현재 state 실행이 완료되면 nextState()를 호출해 다음 state를 실행한다.
			1. 실행순서는 순차적 개념이 아닌 설정 및 조건에 따라 다르다.
		3. state가 null이거나 실행 불가능 상태면 loop는 종료된다.
	3. DecisionState에서 decider.decide(jobExecution, step execution) 호출
	4. JobExecutionDecider가 결과 반환
	5. 최종적으로는 FlowExecutionStatus를 반환

&emsp;&emsp;

- SplitState의 경우
	1. FlowJob에서 .start()를 통해 실행
	2. SimpleFlow가 실행되며 state.handle(executor)를 호출하여 실행
		1. 이때 루프를 돌며 resume() 수행하며 state의 개수 만큼 수행한다.
		2. resume은 현재 state 실행이 완료되면 nextState()를 호출해 다음 state를 실행한다.
			1. 실행순서는 순차적 개념이 아닌 설정 및 조건에 따라 다르다.
		3. state가 null이거나 실행 불가능 상태면 loop는 종료된다.
	3. SplitState에서 SimpleFlow 두 가지를 각각 실행
	4. 최종적으로는 FlowExecutionStatus를 반환

&emsp;&emsp;

#### State 인터페이스

&emsp;&emsp;

- 특징
	- 실제 실행 기준 주체
	- Step, Flow, JobExecutionDecider의 각 요소들을 저장함
	- Flow를 구성하면 내부적으로 생성되어 Transition과 연동
	- handle() 메소드를 실행하여 FlowExectionStatus를 반환한다.
		- FlowExectionStatus는 해당 Flow의 Exit결과와 매핑되며 이 결과로 흐름을 제어한다.

## @JobScope / @StepScope


&emsp;&emsp;

### Scope

&emsp;&emsp;

#### Scope 개념

&emsp;&emsp;

- 정의
	- Spring 컨테이너에서 Bean이 관리되는 범위

&emsp;&emsp;

- 특징
	- singleton, prototype, request, session, application이 있고 기본은 싱글턴이다.

&emsp;&emsp;

#### Spring Batch용 Scope

&emsp;&emsp;

- 정의
	- Job과 Step의 빈 생성과 실행에 관여하는 스코프

- 특징
	- 프록시 모드가 기본인 스코프
		- `@Scope(value = "job", proxyMode = ScopedProxyMode.TARGET_CLASS)`
	- 해당 스코프 선언시 빈이 앱 실행시점이 아닌 빈 실행 시점에서 생성된다.
		- `@Values`를 주입해서 빈의 실행 시점에 값을 참조할 수 있으며 일종의 Lazy Binding이 가능해진다.
	- `@values`를 사용하는 경우 빈 선언문에 `@JobScope`, `@StepScope`를 정의하지 않으면 오류라 발생한다.
	- 프록시 모드로 빈이 선언되기때문에 앱 구동 시점에는 빈의 프록시 객체만 생성되고 실행 시점에 실제 빈을 호출한다.
	- 병렬처리 시 각 스레드마다 생성된 스코프 빈이 할당되어 스레드에 안전하게 실행가능하다.

&emsp;&emsp;

- 예시
	- `@Value("#{jobParameters[파라미터명]}"),@Value("#{jobExecutionContext[파라미터명]}"),@Value("#{stepExecutionContext[파라미터명]}")`

&emsp;&emsp;

#### `@JobScope`

&emsp;&emsp;

- 특징
	- `Step` 선언문에 정의한다.
	- `@Value`에 할당 가능한 변수
		- jobParameter
		- jobExecutionContext
	- ==**`@value` 사용시에는 이 스코프가 무조건 필요하다.**==


#### `@StepScope`

&emsp;&emsp;

- 특징
	- `Tasklet`, `ItemReader`, `ItemWriter`, `ItemProcessor` 선언문에 정의한다.
	- `@Value`에 할당 가능한 변수
		- jobParameter
		- jobExecutionContext
		- stepExecutionContext


<br><br><br>

### @JobScope / @StepScope 아키텍처

&emsp;&emsp;

#### Proxy 객체

&emsp;&emsp;

- 설명
	- JobScope, StepScope
		- Proxy 객체의 실제 대상이 되는 Bean을 등록, 해제
		- 실제 빈을 저장하고 있는 JobContext, StepContext를 가지고 있다.
	- `@JobScope`
		- `@Scope(value = "job", proxyMode = ScopedProxyMode.TARGET_CLASS)`
	- `@StepScope`
		- `@Scope(value = "step", proxyMode = ScopedProxyMode.TARGET_CLASS)`
	- JobContext, StepContext
		- 스프링 컨테이너에서 생성된 빈을 저장하는 컨텍스트 역할
		- Job의 실행 시점에서 프록시 객체가 실제 빈을 참조할 때 사용된다.

&emsp;&emsp;

#### Scope의 Proxy 아키텍처

&emsp;&emsp;

- `@JobScope` 초기화 실행 흐름
	1. `@JobScope`가 존재하면 Step의 Proxy를 생성 
	2. JobLauncher에서 Job이 Step의 Proxy 객체를 저장 
	3. 실제 빈 객체를 참조할 때 Step 빈을 생성한다.
		1. Step 빈 생성은 JobScope가 JobContext에서 빈을 꺼내어 준다.
		2. 빈이 꺼내서 사용될 때(프록시가 Step의 메서드를 실행시키는 순간), `@Value`의 데이터도 바인딩되어 넘겨진다.
