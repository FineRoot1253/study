# 스프링 배치 프로세스 1

## Chunk

&emsp;&emsp;

### Chunk

&emsp;&emsp;

#### Chunk 개념

&emsp;&emsp;

- 정의
	- 여러 개의 아이템을 묶은 하나의 덩어리, 블록

&emsp;&emsp;

- 특징
	- 한번에 하나씩 아이템을 입력 받아 Chunk 단위의 덩어리로 만든 후 Chunk 단위로 트랜잭션을 처리
		- ==**즉, Chunk 단위의 Commit과 Rollback이 이루어짐**==
	- 일반적으로 대용량 데이터는 한번에 처리하는 것이 아니고 청크 단위로 쪼개어서 더 이상 처리할 데이터가 없을 때까지 반복해서 입출력하는데 사용된다.
	- Chunk는 내부 클래스 구현체인 ChunkIterator를 가지고 있는데  Iterable를 구현하고 있다.

&emsp;&emsp;

- 기본 동작 개념
	- 입력기가 아이템을 하나씩 읽어들여 청크의 사이즈가 정해진 특정 사이즈가 되면 출력기에 보내 처리한다.
	- 원본 → ItemReader → `Chunk<I>` → ItemProcessor → `Chunk<O>` → ItemWriter
	- 이 일련의 하나의 동작이 트랜잭션으로 보호된다.

&emsp;&emsp;

- 동작 흐름
	1. 원본으로부터 ItemReader가 Item을 읽어들여 `Chunk<I>`에 있는 items에 추가
	2. 만약 추가 후 사이즈가 ItemProcessor가 읽어들일 사이즈가 되었으면ItemProcessor에 전달한다.
		1. 사이즈가 아직 읽어들일 사이즈가 아니라면 1번 흐름부터 반복한다.
	3. ItemProcessor는 Iterator 패턴으로 순회하여 Item들을 처리하고 `Chunk<O>`의 items에 하나씩 추가한다.
	4. 만약 추가 후 사이즈가 ItemWriter가 읽어들일 사이즈가 되었으면 ItemWriter에 전달한다.
		1. 사이즈가 아직 읽어들일 사이즈가 아니라면 3번 흐름부터 반복한다.
	5. ItemWriter가 `Chunk<O>`를 파일이나 DB에 작성한다.

&emsp;&emsp;

- Chunk 종류
	- `Chunk<I>`
		- 입력용 청크
		- ItemReader로 읽은 하나의 아이템을 Chunk에서 정한 개수만큼 반복해서 저장
	- `Chunk<O>`
		- 출력용 청크
		- ItemReader로부터 전달받은 `Chunk<I>`를 참조해서 ItemProcessor에서 적절히 가공, 필터링하여 다음 ItemWriter에 전달하는 타입

### ChunkOrientedTasklet

&emsp;&emsp;

#### ChunkOrientedTasklet 개념

&emsp;&emsp;

- 정의
	- Chunk 지향 프로세싱을 담당하는 도메인 객체로 스프링 배치에서 제공하는 Tasklet 구현체

&emsp;&emsp;

- 특징
	- ItemReader, ItemWriter, ItemProcessor를 사용해 Chunk 기반의 데이터 입출력 처리를 담당
	- TaskletStep에 의해 반복적으로 실행되며 ChunkOrientedTasklet이 실행 될 때마다 매번 새로운 트랜잭션이 생성되고 이 안에서 처리된다.
	- exception 발생시 해당 Chunk는 롤백되며 이전에 커밋한 Chunk는 완료된 상태가 유지
	- 내부적으로 ItemReader를 핸들링하는 ChunkProvider와 ItemProcessor, ItemWriter를 핸들링하는 ChunkProcessor 타입의 구현체를 가지고 있다.

&emsp;&emsp;

- 기본 동작 개념
	- chunkContext
		- ItemReader가 Item을 읽어들이면 이 Item들이 이 버퍼에 저장된다.
		- Chunk 처리 중 예외 발생으로 인해 재 시도하는 경우 ItemReader로 다시 읽어들이는 것이 아닌 여기에 있는 데이터에서 다시 들고와 처리한다.
			- 이렇게 해야 IO 시간을 줄일 수 있다.
			- 이 버퍼링 옵션을 꺼두면 그냥 다시 읽어들이게끔도 가능 하다.
		- Chunk 단위 입출력이 끝나면 버퍼에 Chunk 데이터를 없앤다.
	- RepeatStatus.continueIf()
		- 읽을 Item이 더 존재하는지 체크하여 존재하면 Chunk 프로세스를 반복한다.
		- 읽을 Item이 없으면 RepeatStatus.FINISHED를 반환하고 Chunk 프로세스를 종료한다.

&emsp;&emsp;

- 기본 동작 흐름
	1. Tasklet에서 execute() 호출
		1. ChunkOrientedTasklet에서 provide() 호출
		2. ChunkProvider에서 read() 호출
			1. ItemReader가 반복적으로 read()를 호출해 목표 청크 사이즈 만큼 Item을 읽어들임
		3. ChunkOrientedTasklet가 process(inputs)을 호출
		4. ChunkProcessor가 process()를 호출
			1. ItemReader가 읽어들여 전달한 아이템 개수 만큼 반복한다.
		5. ChunkProcessor가 write(items)를 호출
		6. ItemWriter가 item 개수만큼 반복하여 파일 또는 DB등에 저장
		7. 이 일련의 과정들이 읽을 Item이 없을 때까지 Chunk 단위로 반복된다.
			1. 이 과정은 Transaction 경계 내에서 이루어진다.

&emsp;&emsp;

- StepBuilder Chunk API
	- `<I, O>chunk(10)`
		- 청크 사이즈 설정
		- commit interval도 의미한다.
		- input, output 제네릭 타입도 이곳에서 설정한다.
		- 반환
			- SimpleStepBuilder
	- `<I, O>chunk(CoimpletionPolicy)`
		- Chunk 프로세스를 완료하기 위한 정책 설정 클래스 지정
		- 숫자만 넣어 지정을 해도 되지만 이렇게 설정 클래스를 넘길 수도 있다는 뜻
		- 반환
			- SimpleStepBuilder
	- `reader(itemReader())`
		- 소스로부터 Item을 읽거나 가져오는 ItemReader 구현체 지정
	- `writer(itemWriter())`
		- Item을 목적지에 쓰거나 보내기 위한 ItemWriter 구현체 지정
	- `processor(itemProcessor())`
		- Item을 변형, 가공, 필터링 하기 위한 ItemProcessor 구현체 지정
	- `stream(itemStream())`
		- 재시작 데이터를 관리하는 콜백 스트림 등록
	- `readerIsTransactionalQueue()`
			- Item이 JMS, Message Queue Server와 같은 외부 트랜잭션에 의해 읽혀지고 캐싱할 지 여부 설정
			- default: false
	- `listener(ChunkListener)`
		- Chunk 프로세스가 진행되는 특정 시점에 콜백을 제공받을 수 있는 수단인 ChunkListener 설정

### ChunkProvider

&emsp;&emsp;

#### ChunkProvider 개념

&emsp;&emsp;

- 정의
	- ItemReader를 사용해서 소스로부터 아이템을 Chunk size 만큼 읽어서 Chunk 단위로 만들어 제공한느 도메인 객체

&emsp;&emsp;

- 특징
	- `Chunk<I>`를 만들고 내부에서 반복문으로 ItemReader.read()를 계속 호출하여 item을 chunk에 쌓는다.
	- 외부로부터 **ChunkProvider가 호출될 때마다 항상 새로운 Chunk가 생성**된다.
	- 반복문 종료 조건 
		- Chunk size만큼 item을 읽으면 반복문이 종료 
			- ChunkProcessor로 흐름이 넘어간다.
		- ItemReader가 읽은 item이 null일 경우 반복문 종료 
			- 해당 Step 반복문까지 종료된다.
	- 기본 구현체로서 SimpleChunkProvider, FaultTolerantChunkProvider가 있다.
	- `ChunkProvider<I>`는 `Chunk<T> provide(StepContribution)`을 가지는 인터페이스이다.
		- `SimpleChunkProvider<I>`는 `ItemReader<I>`, `RepeatOperations`, `I read(StepContribution, Chunk<T>)`를 구현하고 있는 `ChunkProvider<I>`의 구현체이다.


<br><br><br>

### ChunkProcessor

&emsp;&emsp;

#### ChunkProcessor 개념

&emsp;&emsp;

- 정의
	- ItemProcessor를 사용하여 Item을 변형, 가공, 필터링하고 ItemWriter를 사용해서 Chunk 데이터를 저장, 출력하는 도메인 객체

&emsp;&emsp;

- 특징
	- `Chunk<O>`를 만들고 앞에서 넘어온 `Chunk<I>`의 item을 한 건씩 처리한 후 `Chunk<O>`에 저장
	- 외부로부터 **ChunkProcessor가 호출될 때마다 항상 새로운 Chunk가 생성**된다.
	- ItemProcessor는 설정 시 선택사항으로 객체가 존재하지 않으면 ItemReader에서 읽은 Item이 그대로 `Chunk<O>`에 저장한다.
	- ItemProcessor 처리가 완료되면 `Chunk<O>`에 있는 items를 ItemWriter에 전달
	- ItemWriter 처리가 완료되면 Chunk 트랜잭션이 종료하게 되고 Step 반복문에서 ChunkOrientedTasklet이 새로 생성된다.
	- ItemWriter는 Chunk size 만큼 데이터를 commit하기 때문에 이것이 커밋 주기가 된다.
	- 기본 구현체로 SimpleChunkProcessor, FaultTolerantChunkProcessor가 있다.
	-  `ChunkProcessor<I>`는 `void process(StepContribution, Chunk<I>)`을 가지는 인터페이스이다.
		- `SimpleChunkProcessor<I, O>`는 `ItemProcessor<I, O>`, `ItemWriter<O>`, `Chunk<O> transform(StepContribution, Chunk<I>)`, `void write(StepContribution, Chunk<I>, Chunk<O>)`
		- transform 내부에서 doProcess로 실제 프로세스 과정이 이뤄진다.
			- write 호출도 이곳에서 한다.

### ItemReader

&emsp;&emsp;

#### ItemReader 개념

&emsp;&emsp;

- 정의
	- 다양한 입력으로부터 데이터를 읽어 제공하는 인터페이스

&emsp;&emsp;

- 지원 입력 종류
	- Flat 파일
		- csv, txt
	- XML, JSON
	- Database
	- JMS, RabbitMQ, Apache Kafka 와 같은 Messaging Queuing 서비스
		- AmqpItemReader와 KafkaItemReader(ItemStreamSupport 구현체) 구현체가 대표적으로 사용된다.
	- Custom Reader
		- 이런 직접 구현하는 경우에는 Threadsafe하게 구현해야한다.

&emsp;&emsp;

- 특징
	- **ChunkOrientedTasklet 실행 시 필수**적 요소로 설정해야 한다.
	- `ItemReader<T>` 인터페이스에는 `T read() throws Exception, UnexpectedInputException, ParseException, NonTrasientResourceException`가 있다.
		- read() 특징
			- 입력 데이터를 읽고 다음 데이터로 이동
			- 아이템 하나를 리턴하고 더 이상 아이템이 없으면 null 리턴
			- 아이템 하나는 파일의 한줄, DB의 한 row 혹은 XML 파일에서 하나의 엘리먼트가 될 수 있다.
			- 더 이상 처리해야 할 Item이 없어도 예외가 발생하지 않고 ItemProcessor와 같은 다음 단계로 넘어간다.
	- 다수의 구현체들이 ItemReader와 ItemStream 인터페스를 동시에 구현한다.
		- 파일의 스트림을 열거나 DB 커넥션을 열거나, 종료, 입력 장치 초기화 등의 작업을 위함
		- ExecutionContext에 read와 관련된 여러가지 상태 정보를 저장해서 재시작 시 다시 참조 하도록 지원
	- **일부를 제외하고 하위 클래스들 대부분 Thread-Safe 하진 않아서 병렬 처리를 만들 예정이라면 데이터 정합성을 위한 동기화 처리가 필요함**

&emsp;&emsp;

- 대표적 구현체 종류
	- File 관련
		- FlatFileItemReader
			- csv, txt
		- StaxEventItemReader
			- XML
		- JsonItemReader
			- Json
		- MultiResourceItemReader
			- 여러 개의 파일 조합
	- DB 관련
		- JdbcCursorItemReader
			- JDBC Cursor Type
		- JpaCursorItemReader
			- JPA Cursor Type
		- JdbcPagingItemReader
			- JDBC Paging Type
		- JpaPagingItemReader
			- JPA Paging Type
		- ItemReaderAdapter
			- Reuse Exist Service
	- 커스텀 관련
		- SynchronizedItemStreamReader
			- Convert to Thread-safe ItemReader
		- CustomItemReader
			- Customzing ItemReader

### ItemWriter

&emsp;&emsp;

#### ItemWriter 개념

&emsp;&emsp;

- 정의
	- Chunk 단위로 데이터를 받아 일괄 출력 작업을 위한 인터페이스

&emsp;&emsp;

- 지원 출력 종류
	- Flat File
		- csv, txt
	- XML, Json
	- Database
	- JMS, RabbitMQ와 같은 Messaging Queuing 서비스
	- Mail Service
	- Custom Writer

&emsp;&emsp;

- 특징
	- 아이템 하나가 아닌 아이템 리스트를 전달받는다.
	- **ChunkOrientedTasklet 실행 시 필수**적 요소로 설정해야 한다.
	- `ItemWriter<T>` 인터페이스에는 `void write(List<? extends T> items) throws Exception`가 있다.
		- write 특징
			- 출력 데이터를 아이템 리스트로 받아 처리
			- 출력이 완료되고 트랜잭션이 종료되면 새로운 Chunk 단위 프로세스로 이동
	- 다수의 구현체들이 ItemWriter와 ItemStream을 동시에 구현하고 있다.
	- 대부분 ItemReader 구현체와 1:1 대응관계 ItemWriter 구현체들이 구성되어 있다.

&emsp;&emsp;

- 대표적 구현체 종류
	- File 관련
		- FlatFileItemWriter
			- csv, txt
		- StaxEventItemWriter
			- XML
		- JsonFileItemWriter
			- Json
		- MultiResourceItemWriter
			- 여러 개의 파일 조합
	- DB 관련
		- JdbcBatchItemWriter
			- JDBC Type
		- JpaItemWriter
			- JPA Type
		- ItemWriterAdapter
			- Reuse Exist Service
	- 커스텀 관련
		- CustomItemWriter
			- Customzing ItemWriter

### ItemProcessor

&emsp;&emsp;

#### ItemProcessor 개념

&emsp;&emsp;

- 정의
	- 데이터 출력 전 데이터를 가공, 변형, 필터링하는 역할

&emsp;&emsp;

- 특징
	- ItemReader 및 ItemWriter와 분리되어 비즈니스 로직을 구현할 수 있다.
	- ItemReader로부터 받은 아이템을 특정 타입으로 변환해서 ItemWriter에 넘겨줄 수 있다.
	- ItemReader로부터 받은 아이템들 중 필터과정을 거쳐 원하는 아이템들만 ItemWriter에게 넘겨줄 수 있다.
		- ==**ItemProcessor에서 process() 실행결과 null을 반환시 `Chunk<O>`에 저장되지 않기 때문에 ItemWriter는 전달되지 않는다.**==
	- ChunkOrientedTasklet 실행 시 선택적 요소이기 때문에 청크 기반 프로세싱에서 ItemProcessor 단계가 반드시 필요한 것은 아니다.
		- ==**다만 대부분의 경우에 여기에 비즈니스 로직을 구현하기 때문에 결국 이게 핵심이다.**==
	- `ItemProcessor<I, O>` 인터페이스에는 `O read(@NonNull I item) throws Exception`가 있다.
		- process 특징
			- `<I>` 제네릭은 ItemReader에서 받을 데이터 타입
			- `<O>` 제네릭은 ItemWriter에서 받을 데이터 타입
			- 아이템 하나씩 가공처리하며 null 리턴시 해당 아이템은 `Chunk<O>`에 저장하지 않는다.
	- ==**ItemStream을 구현하지 않는다.**==
	- ==**대부분 커스텀해서 쓰기 때문에 기본적으로 제공되는 구현체가 현저히 적다**==

&emsp;&emsp;

- 대표적 구현체
	- ClassifierCompositeItemProcessor
		- 분류에 따른 선택적 처리
	- CompositeItemProcessor
		- ItemProcessor들을 연결해서 처리
	- CustomItemProcessor
		- 커스터마이징 ItemProcessor

### ItemStream

&emsp;&emsp;

#### ItemStream 개념

&emsp;&emsp;

- 정의
	- ItemReader와 ItemWriter의 처리 과정 중 상태를 저장하고 오류가 발생하면 해당 상태를 참조하여 실패한 곳에서 재 시작하도록 지원하는 도메인 객체

&emsp;&emsp;

- 특징
	- 리소스를 열고 닫아야하며 입출력 장치 초기화를 해야하는 경우 사용된다.
	- ExecutionContext를 매개변수로 하여 상태 정보를 업데이트 한다.
	- ItemReader 및 ItemWriter는 ItemStream을 구현해야 한다.
	- `ItemStream` 인터페이스
		- `void open(ExecutionContext) throws ItemStreamException`
			- read, write 호출 전에 파일이나 커넥션이 필요한 리소스에 접근하도록 초기화 작업
		- `void update(ExecutionContext) throws ItemStreamException`
			- 현재까지 진행된 모든 상태를 저장
		- `void close() throws ItemStreamException`
			- 열려 있는 모든 리소스를 안전하게 해제하고 닫음

&emsp;&emsp;

- 기본 동작 흐름
	1. ItemStream은 ItemReader 사용하기 전 open() 메소드를 이용해 리소스를 연다.
		1. 최초 한 번만 수행한다.
		2. ItemWriter 또한 똑같이 수행된다.
	2. ItemReader가 read()를 호출한다.
		1. 호출될 때마다 ItemStream의 update()또한 호출되어 상태정보가 저장된다.
		2. 이 주기는 Chunk Size 만큼 반복된다.
		3. ItemWriter 또한 write()를 수행될때 똑같이 수행된다.

### Chunk Process Architecture

&emsp;&emsp;

- 실행 흐름
	1.  Job에서 Step이 Tasklet을 수행
	2. Tasklet에서 RepeatTemplate을 수행
	3. ChunkOrientedTasklet이 SimpleChunkProvider와 SimpleChunkProcessor를 통해 청크 단위 작업을 수행
		1. 청크 단위 뿐 아니라 이 작업의 시작 또한 트랜잭션에 경계에 존재한다.
	4. SimpleChunkProvider에서 RepeatTemplate을 수행해 청크 사이즈 만큼 read()
		1. 이것이 트랜잭션이 시작하는 시점이다.
	5. SimpleChunkProvider에서 null을 읽어들인 경우 모든 반복문 및 Chunk 프로세스 종료
		1. null이 아닌 경우 `Chunk<I>`에 아이템을 저장하고 청크 사이즈를 확인한다.
			1. 초과하지 않은 경우 read()를 반복한다.
			2. 초과한 경우 items를 SimpleChunkProcessor에 전달한다.
	6. SimpleChunkProcessor에서는 process()를 반복하여 `Chunk<O>`에 저장한다.
	7. 모두 저장했으면 write()를 호출해 ItemWriter에게 전달한다.
		1. 이것이 트랜잭션이 종료되는 시점이며 종료 후 바로 다시 시작된다.
		2. chunk 단위로 커밋이 진행되며 트랜잭션의 범위는 read() ~ write() 단계라고 보면 된다.
	8. 어떤 read(), process(), write() 경우에 예외가 발생한다면 롤백 처리된다.
		1. 재시작을 하는 경우 ItemReader가 읽어들였던 부분부터 다시 시작한다.
