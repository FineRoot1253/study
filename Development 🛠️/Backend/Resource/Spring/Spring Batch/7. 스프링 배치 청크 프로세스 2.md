# 스프링 배치 프로세스 2

## ItemReader

&emsp;&emsp;

### FlatFileItemReader

&emsp;&emsp;

#### FlatFileItemReader 개념

&emsp;&emsp;

- 정의
	- 2차원 데이터(표)로 표현된 유형의 파일을 처리하는 ItemReader

&emsp;&emsp;

- 특징
	- 일반적으로 고정 위치로 정의된 데이터 필드나 특수 문자에 의해 구별된 데이터 행을 읽음
	- Resource, LineMapper 두 가지 요소를 필요로 함

&emsp;&emsp;

- `FlatFileItemReader` 구성
	- `String encoding = DEFAULT_CHARSET`
	- `int linesToSkip`
		- 툴마다 csv의 시작하는 줄에 이것 저것 해더 정보를 넣는 편이라 보통 이게 있어야 정상 작동한다.
	- `String[] comments`
		- 이 기호가 있는 라인은 건너뛴다.
	- `Resource resource`
	- `LineMapper<T> lineMapper`
		- 파일 라인 한줄을 Object로 변환해 FlatFileItemReader로 변환하는 핵심 클래스
		- 단순 문자열을 받아 문자열을 토큰화하여 객체로 매핑하는 과정이 필요하다.
	- `LineCallbackHandler skippedLinesCallback`
		- 건너뛴 라인마다 발생시키는 콜백

&emsp;&emsp;

- 핵심 구현체
	- Resource
		- FileSystemResource
			- 실제 파일 시스템 기반
		- ClassPathResource
			- 클래스 경로 기반
	- LineMapper
		- FieldSet
			- 라인을 필드로 구분해 만든 배열 토큰을 전달하면 토큰 필드를 참조할 수 있다.
			- ResultSet과 비슷하다.
		- LineTokenizer
			- 입력받은 라인을 FieldSet으로 변환해서 리턴한다.
			- 파일마다 형식이 다르기 때문에 문자열을 FieldSet으로 변환하는 작업을 추강화시켜야한다.
		- FieldSetMapper
			- FieldSet 객체를 받아서 원하는 객체로 매핑해서 리턴
			- RowMapper와 동일한 패턴을 사용한다.

&emsp;&emsp;

- LineMapper 구조
	- `LineMapper<T>` 인터페이스
		- `T mapLine(String line, int lineNumber)`
			- 현재 라인과 라인 넘버를 인자로 받고 객체로 반환
	- `DefaultLineMapper`, `LineMapper<T>` 인터페이스의 구현체
		- `LineTokenizer`
			- `FieldSet tokenize(@NullableString line)`
		- `FieldSetMapper<T>`
			- `T mapFieldSet(FieldSet fieldSet)`
	- `LineTokenizer`의 구현체
		- `DelimitedLineTokenizer`
			- 구분자를 이용해 필드로 구분하는 파일에 사용
		- `FixedLengthTokenizer`
			- 각 필드를 고정된 길이로 정의하는 파일에 사용
			- 
	- `FieldSetMapper<T>`의 구현체
		- `BeanWrapperFieldSetMapper`
			- 객체 필드명과 일치하는 FieldSet의 필드를 자동으로 매핑

&emsp;&emsp;

- Item 읽기 흐름
	1. Step에서 read() 호출
	2. FlatFileItemReader의 readLine()을 호출해 라인을 한 줄 읽음
	3. LineMapper의 mapLine(line)을 호출해 객체로 변환 시도
	4. FieldSetMapper의 mapFieldSet()을 통해 FieldSet을 찾음
	5. LineTokenizer의 tokenize()를 통해 배열 토큰으로 변환해 FieldSet에 전달
	6. FieldSet은 FieldSetMapper에 전달되어 다시 객체로 변환한다.
	7. 변환된 객체는 read()의 최종 반환 값으로 반환된다.

&emsp;&emsp;

- Builder API
	- `name(String)`
		- 이름 설정
		- ExecutionContext에서 key로 세팅됨
	- `resource(Resource)`
		- 읽어야 할 리소스
	- `delimited().delimiter(String)`
		- 구분자
		- 이렇게하면 구분자 기반으로 읽어들인다.
	- `fixedLength()`
		- 고정 길이
		- 이렇게하면 길이 단위로
	- `addColumns(Range...)`
		- 고정 길이 범위를 설정
	- `names(String[] fieldNames)`
		- LineTokenizer로 구분된 라인의 항목을 객체의 필드명과 매핑하도록 설정
	- `targetType(Class class)`
		- 라인의 각 항목과 매핑할 객체 타입 설정
	- `strict(boolean)`
		- 라인을 읽거나 토큰화할 때 Parsing 예외가 발생하지 않도록 검증 생략 설정
	- `encoding(String)`
		- 파일 인코딩
	- `linesToSkip(int)`
		- 파일 상단 라인 무시 할 라인 수
	- `saveState()`
		- 상태 정보를 저장할 것인지 설정
	- `setLineMapper(LineMapper)`
	- `setFieldSetMapper(FieldSetMapper)`
	- `setLineTokenizer(LineTokenizer)`

&emsp;&emsp;

##### DelimitedLineTokenizer

&emsp;&emsp;

- 정의
	- 한 개의 라인을 구분자 기준으로 나누어 토큰화하는 방식

&emsp;&emsp;

- 구조
	- `AbstractLineTokenizer` 인터페이스
		- `String[] names`
		- `boolean strict`
		- `FieldSetFactory fieldSetFactory`
	- `DelimitedLineTokenizer`, `AbstractLineTokenizer` 구현체
		- `String delimiter`
		- `FieldSet tokenize(@Nullable String line)`

&emsp;&emsp;

- 구현시 특징
	- `BeanWrapperFieldSetMapper`로 세팅을 하고 targetType인 목표 객체를 집어넣어주면 **FieldSetMapper를 구현해줄 필요가 전혀없다.**

&emsp;&emsp;

##### FixedLengthTokenizer

&emsp;&emsp;

- 정의
	- 한 개의 라인의 String을 사용자가 설정한 고정길이 기준으로 나누어 토큰화 하는 방식

&emsp;&emsp;

- 특징
	- 범위는 문자열 형식으로 설정 가능
		- 예시
			- 1-4
			- 1-3, 4-6, 7
			- 1-3, 4-6, 7-10
		- **마지막 범위가 열려 있는 경우 나머지 행이 해당 열로 읽어들여짐**


&emsp;&emsp;

- 구조
	- `FixedLengthTokenizer`
		- `Range[] ranges`
		- `int maxRange = 0`
		- `Boolean open = false`

&emsp;&emsp;

- 동작 구조
	1. FlatFileItemReader가 `readLine()`으로 한 줄 씩 읽어들인다.
	2. lineNumber을 매겨 mapper에 넣는다.
	3. 데이터는 tokenizer에 의해 토큰화 된다.
		1. `setNames()`와 `setColumns()`에 맞춰 그 글자 수 대로 데이터를 쪼갠다.
		2. 쪼갠 이름과 데이터는 맵에 세팅된다.

&emsp;&emsp;

##### Exception Handling

&emsp;&emsp;

- 정의
	- 라인을 읽거나 토큰화 하는 동안 발생하는 Parsing 예외를 처리할 수 있는 계층

&emsp;&emsp;

- 특징
	- 토큰화 검증을 엄격하게 적용하지 않게 설정하면 예외 던짐이 발생하지 않게 만들 수 있다.
		- `tokenizer.setStrict(false);`
		- 예외 던짐이 발생하지 않으며 컬럼 개수와 맞지 않는 컬럼은 빈 토큰으로 채워진다.
	- `ItemReaderException` 예외
	- `FlatFileParseException`, `ItemReaderException` 구현체
		- ItemReader에서 파일을 읽어들이는 동안 발생하는 예외
		- 구체적이진 않다.
	- `FlatFileFormatException` 예외
	- `IncorrectTokenCountException`, `FlatFileFormatException` 구현체
		- `DelimitedLineTokenizer`로 토큰화 할 때 컬럼 개수와 실제 토큰화 한 컬럼 수와 다를 때 발생하는 예외
	- `IncorrectLineLengthException`, `FlatFileFormatException` 구현체
		- `FixedLengthLineTokenizer`로 토큰화 할 때 라인 전체 길이와 컬럼 길이의 총합과 일치하지 않을 때 발생

### StaxEventItemReader

&emsp;&emsp;

#### JAVA XML API 개념

&emsp;&emsp;

- 정의
	- DOM 방식
		- 문서 전체를 메모리에 로드 후 tree 형태로 만들어 데이터를 처리하는 방식
		- pull 방식
		- 엘리멘트 제어에 유용하며 문서 크키가 클 수록 메모리 사용이 많고 느리다.
	- SAX 방식
		- 문서 항목을 읽을 때마다 이벤트가 발생하여 데이터를 처리하는 방식
		- push 방식
		- 메모리 비용이 적고 속도가 빠르지만 엘리멘트 제어가 어렵다.
	- StAX 방식
		- DOM과 SAX의 장점과 단점을 보완한 API 모델
		- pull, push 방식 동시 제공
		- XML 문서를 읽고 쓸 수 있는 양방향 파서기 지원
		- XML 파일의 항목에서 항목으로 직접 이동하면서 Stax 파서기를 통해 구문 분석

&emsp;&emsp;

- StAX특징
	- Iterator API
		- `XMLEventReader`의 `nextEvent()`를 호출해 이벤트 객체를 가지고 옴
		- 이벤트 객체
			- 요소, 택스트, 주석 등에 대한 XML 태그 유형 정보 제공
	- Cursor API
		- JDBC ResultSet 처럼 작동하는 API
		- `XMLStreamReader`는 xml 문서의 다음 요소로 커서를 이동
		- 커서에서 직접 메서드를 호출해 이벤트에 대한 자세한 정보를 얻음

&emsp;&emsp;

#### StaxEventItemReader API

&emsp;&emsp;

- 정의

 - 특징

		- 이 루트 조각 단위가 객체와 매핑되는 기준이 된다.

		- Unmarshaller 객체 설정
	- saveState(boolean)

		- 기본값

### DB Accessing ItemReader

&emsp;&emsp;

#### Cursor & Paging

&emsp;&emsp;

- 정의
	- 실시간 처리가 힘든 대용량 데이터 처리를 위해 제공하는 두 가지 해결 방안인 Cursor Based 프로세싱, Paging Based 프로세싱이 존재한다.

&emsp;&emsp;

- 특징
	- Cursor Based 프로세싱
		- JDBC Result Set 메커니즘을 따름
		- 현재 행에 커서를 유지하며 다음 데이터 호출시 다음 행으로 이동하여 데이터 반환이 이뤄지는 Streaming 방식 I/O
		- ResultSet이 open 될 때마다 next() 메소드가 호출되어 DB의 데이터가 반환되고 객체와 매핑을 이룬다.
		- DB Connection이 연결되면 배치 처리가 완료될 때까지 데이터를 읽어오기 때문에 DB와 SocketTimeout을 충분히 큰 값으로 설정할 필요가 있다.
		- 모든 결과를 메모리에 할당하기 때문에 충분한 메모리가 요구된다.
		- Connection 연결 유지 시간과 메모리 공간이 충분하면, 대량 데이터 처리에 적합한 방식이다. (Fetch Size 조절)
			- 스트리밍 방식이여도 페치 사이즈는 조절 가능하다.
			- 한번 페칭이 시작되면 커넥션은 계속 유지된다.
	- Paging Based 프로세싱
		- 페이지 단위로 데이터를 조회하는 방식으로 Page Size만큼 한번에 메모리로 가지고 온 다음 한 개씩 읽는다.
		- 한 페이지를 읽을 때마다 Connection을 맺고 끊기 때문에 대량의 데이터를 처리하더라도 SocketTimeout 예외가 거의 일어나지 않는다.
		- 시작 행 번호를 지정하고 페이지에 반환시키고자 하는 행의 수를 지정한 후 사용한다.
			- Offset과 Limit 이야기이다.
		- 페이징 당위의 결과만 메모리에 할당하므로 메모리 공간 요구량이 적다.
		- Connection 연결 유지 시간이 길지 않고 메모리 공간을 효율적으로 사용해야하는 경우에 적합하다.
			- 한번 페칭이 시작되면 페이징 단위로만 커넥션이 유지된다.

&emsp;&emsp;

#### JdbcCursorItemReader

&emsp;&emsp;

- 정의
	- Cursor 기반 JDBC 구현체, ResultSet과 함께 사용됨

&emsp;&emsp;

- 특징
	- DB에서 Connection을 얻어와서 SQL 실행
	- Thread 안전하지 않아 멀티스레드 환경에서는 동시성 이슈가 존재해 동기화 처리가 필수적이다.
	- `JdbcCursorItemReaderBuidler` API
		- `name(String)`
			- 이름 지정
		- `fetchSize(int)`
			- 한번에 메모리에 할당할 크기 지정
		- `datasource(DataSource)`
			- DataSource 지정
		- `rowMapper(RowMapper)`
			- 쿼리 결과를 Object에 매핑하는 매퍼 설정
		- `beanRowMapper(Class<T>)`
			- 별도의 RowMapper를 설정하지 않는 경우 클래스 타입을 지정하면 자동으로 객체와 매핑
		- `sql(String)`
			- ItemReader가 조회시 사용할 쿼리 문장
		- `queryArguments(Object...)`
			- 쿼리 파라미터 설정
		- `maxItemCount(int)`
			- 조회 할 최대 Item 수
		- `currentItemCount(int)`
			- 조회 Item 시작 지점
		- `maxRows(int)`
			- ResultSet 오브젝트가 포함할 수 있는 최대 행 수

&emsp;&emsp;

- 동작 방식

### ItemProcessor

&emsp;&emsp;

#### CompositeItemProcessor

&emsp;&emsp;

- 정의
	- ItemProcessor 들을 연결해서 위임하면 각 ItemProcessor를 실행시킨다.
		- 즉 각 처리 로직을 체이닝해서 순차적으로 Item 들을 가공처리하는 로직을 구성하는 것이다.

&emsp;&emsp;

- 특징
	- 이전 ItemProcessor 반환 값은 다음 ItemProcessor 값으로 연결된다.
	- `CompositeItemProcessorBuilder<>`
		- `delegates(ItemProcessor<?, ?>...)`
			- 체이닝 할 ItemProcessor

&emsp;&emsp;

#### ClassifierItemProcessor

&emsp;&emsp;

- 정의
	- Classifier로 라우팅 패턴을 구현하여 ItemProcessor 구현체 중에서 하나를 호출하는 역할을 한다.

&emsp;&emsp;

- 특징
	- `ClassifierItemProcessorBuilder<>`
		- `classifier(Classifier)`
			- 분류자 설정
	- `Classifier<C, T>` 인터페이스 API
		- `T classify(C classifiable)`
	- Classifier 구현체를 통해 원하는 프로세서를 반환하는 방식이다.
		- 개인적으로 DB에 연동이 가능하지 않나 싶다.

&emsp;&emsp;

- 동작 구조
	1. `ItemReader`가 `ClassifierCompositeItemProcessor`를 통해 읽어낸 Item을 `process(Item)`를 통해 넘긴다.
	2. `ClassifierCompositeItemProcessor`는 분류 조건에 따라 맞는 `ItemProcessor`를 선택한다.
	3. 내가 구현해둔 `ItemProcessor` 구현체들중 하나를 반환한다.
