# 배치 도메인의 이해

&emsp;&emsp;

## Job

&emsp;&emsp;

### Job

&emsp;&emsp;

#### Job 개념

&emsp;&emsp;

- 정의
	- 배치 계층 구조에서 가장 상위에 있는 개념
		- 하나의 배치 작업을 의미
		- 예시
			- API 서버에 있는 로그 데이터를 통계 서버로 이동
			- 원천 데이터 스크래핑 결과 파일을 메인 서버로 이동

&emsp;&emsp;

- 특징
	- Job Configuration을 통해 생성되는 객체 단위로서 배치 작업을 어떻게 구성하고 실행할 것인지 전체적으로 설정하고 명세해 놓은 객체
	- 배치 Job을 구성하기 위한 최상위 인터페이스이며 스프링 배치가 기본 구현체를 제공한다.
	- 여러 Step을 포함하고 있는 컨테이너로 **반드시 한 개 이상의 Step으로 구성**해야한다.
	- 내부에 AbstractJob이 JobRepository를 지니고 있는데, 이는 JobBuilder를 통해 Job을 생성해줄때 매번 지정해줘야하는 메타데이터 저장소이다.

&emsp;&emsp;

#### Job 구현체

&emsp;&emsp;

- SimpleJob
	- **순차적으로 Step을 실행**시키는 Job
	- 모든 Job에서 유용하게 사용할 수 있는 표준 기능을 갖고 있다.

&emsp;&emsp;

- FlowJob
	- **특정한 조건과 흐름에 따라 Step을 구성**하여 실행시키는 Job
	- Flow 객체를 실행시켜서 작업을 진행한다.

&emsp;&emsp;

##### AbstractJob

&emsp;&emsp;

- 구성
	- name
		- Job 이름
	- restartable
		- 재시작 여부
		- 디폴트는 true

	- JobExecutionListener
		- Job 이벤트 리스너

		- JobParameter 증가기
	- JobParametersValidator

	- SimpleStepHander
		- Step 실행 핸들러

### JobInstance

&emsp;&emsp;

#### JobInstance 개념

&emsp;&emsp;

- 정의
	- Job이 실행될 때 생성되는 Job의 논리적 실행 단위 객체
	- 고유하게 식별이 가능한 작업 실행

&emsp;&emsp;

- 특징
	- Job의 설정과 구성은 동일하지만 실행되는 시점과 내용은 늘 다르기 때문에 Job의 실행은 하나의 단위로써 구분이 되어야 하므로 그 단위로써 사용된다.
	- 생성시 처음 시작하는 Job + JobParameter 인 경우 JobInstance가 생성된다.
	- 생성시 이전과 동일한 Job + JobParameter인 경우 내부적으로 JobRepository를 탐색해 JobInstance를 리턴한다.
		- 내부적으로 JobName + JobKey(jobParameter의 해시값)을 가지고 JobInstance 객체를 얻는다.
	- Job과 1:N 관계를 가진다.
	- Job은 Service, JobInstance는 Entity와 같은 관계라고 보면된다.

&emsp;&emsp;

- BATCH_JOB_INSTANCE 테이블과 매핑
	- JOB_NAME(job)과 JOB_KEY(JobParameter 해시값)가 동일한 데이터는 중복 저장이 불가능하다.
	- 탐색을 위한 제약 조건이다.

### JobExecution

&emsp;&emsp;

#### JobExecution 개념

&emsp;&emsp;

- 정의
	- JobInstance에 대한 한번의 시도를 표현하는 단위 객체
	- Job 실행 중 발생한 정보들을 저장하고 있는 객체

&emsp;&emsp;

- 특징
	- 속성
		- 시작시간, 종료시간, 상태, 종료 상태
	- JobInstance와의 관계 규칙 (도메인 규칙, 불변식)
		- JobExecution은 FAILED 또는 COMPLETED 등의 실행 결과 상태를 가진다.
		- **JobExecution의 실행 상태 결과가 COMPLETED**면 JobInstance 실행이 완료된 것으로 간주해서 **재 실행이 불가**
		- JobExecution의 실행 상태 결과가 FAILED 이면 JobInstance 실행이 완료되지 않은 것으로 간주하여 재 실행이 가능하다.
			- JobParameter가 동일한 값으로 Job을 실행할 지라도 JobInstance를 계속 실행할 수 있음
		- JobExecution의 실행 상태 결과가 COMPLETED 될 때까지 하나의 JobInstance 내에서 여러 번의 시도가 생길 수 있음

&emsp;&emsp;

- BATCH_JOB_EXECUTION 테이블과 매핑
	- JobInstance와 JobExecution은  1:N 관계
	- JobInstance에 대한 성공/실패 내역이다.

## Step

&emsp;&emsp;

### Step

&emsp;&emsp;

#### Step 개념

&emsp;&emsp;

- 정의
	- Batch Job을 구성하는 독립적인 하나의 단계
	- 실제 배치 처리를 정의하고 컨트롤하는 데 필요한 모든 정보를 가지고 있는 도메인 객체

&emsp;&emsp;

- 특징
	- 단순한 단일 태스크 뿐 아니라 입력과 처리, 출력과 관련된 복잡한 비즈니스 로직을 포함하는 모든 설정들을 담고 있다.
	- 배치 작업을 어떻게 구성하고 실행할 것인지 Job의 세부 작업을 Task 기반으로 설정하고 명세해 놓은 객체
	- 모든 Job은 반드시 하나 이상의 Step을 소유하고 있다.
	- 호출하는 API에 따라 생성되는 Step 구현체가 달라진다.
		- tasklet()
			- TaskletStep
		- chunk().reader().writer()..
			- ChunkOrientedTasklet
		- job().launcher().parameterExtractor()...
			- JobStep
	- Tasklet 인터페이스를 직접 구현해서 등록을 해도 가능하다.

&emsp;&emsp;

#### Step 구현체

&emsp;&emsp;

- 종류
	- TaskletStep
		- 가장 기본이 되는 클래스
		- Tasklet 타입의 구현체들을 제어한다.
	- PartitionStep
		- 멀티 스레드 방식으로 Step을 여러 개로 분리해서 실행
	- JobStep
		- Step 내부에서 Job을 실행
	- FlowStep
		- Step 내부에서 Flow를 실행

&emsp;&emsp;

##### AbstractStep 구성

&emsp;&emsp;

- 구성
	- name
		- 이름
	- startLimit
		- 실행 제한 횟수
	- allowStartIfComplete
		- 실행 이후 재실행 여부
	- stepExecutionListener
		- Step 이벤트 리스너
	- jobRepository
		- Step 메타데이터 저장소

### StepExecution

&emsp;&emsp;

#### StepExecution 개념

&emsp;&emsp;

- 정의
	- Step에 대한 한번의 시도를 의미하는 객체
	- Step 실행 중 발생한 정보들을 저장하고 있는 객체

&emsp;&emsp;

- 특징
	- Step이 매번 시도될 때마다 생성되며 각 Step 별로 생성된다.
	- Job이 재시작하더라도 이미 성공적으로 완료된 Step은 재실행되지 않고 실패한 Step만 실행된다.
	- 이전 단계에서 실패해서 현재 단계가 실행되지 않았다면 StepExecution을 생성하지 않는다.
		- **실제로 시작될 때만 생성**한다.
	- JobExecution 과의 관계
		- Step의 StepExecution이 모두 정상적으로 완료되어야 JobExecution이 정상적으로 완료
		- Step의 StepExecution이 하나라도 실패하면 JobExecution은 실패
		- All or Nothing 성질을 지닌다.

&emsp;&emsp;

- BATCH_STEP_EXECUTION 테이블
	- JobExection과 StepExection은 1:M 관계
	- 하나의 Job에 여러 Step을 구성한 경우 각 StepExecution은 하나의 JobExection 부모를 가진다.
	- Step과 StepExecution은 1:1 관계이다.
		- Job과 JobExecution은 1:N 관계라서 헷갈릴 수도 있다.
		- 한 Step당 tasklet은 하나만 적용 가능하므로 헷갈리면 안된다.

&emsp;&emsp;

- 구성
	- 시작시간
	- 종료시간
	- 상태
	- 커밋 횟수
	- 롤백 횟수

&emsp;&emsp;

### StepContribution

&emsp;&emsp;

#### StepContribution 개념

&emsp;&emsp;

- 정의
	- 청크 프로세스의 변경 사항을 버퍼링하여 StepExecution에 업데이트 하는 도메인 객체
	- StepExecution 업데이트 용 DTO라고 보면 편하다.

&emsp;&emsp;

- 특징
	- 청크 커밋 직전에 StepExection의 apply 메서드를 호출하여 상태를 업데이트한다.(Hook)
	- ExitStatus의 기본 종료 코드외 사용자 정의 종료 코드를 생성해서 적용할 수 있다.

&emsp;&emsp;

- 구조
	- 읽기 횟수
	- 쓰기 횟수
	- 필터 횟수
	- 부모 스킵 횟수
	- 읽기 스킵 횟수
	- 쓰기 스킵 횟수
	- 프로세스 스킵 횟수
	- 종료 상태
	- StepExection 레퍼런스

&emsp;&emsp;

#### StepContribution 동작 구조

1. TaskletStep에서 StepExecution 생성
2. StepExecution에서 StepContribution 생성
3. TaskletStep에서 `execute(contribution, chunkContext)` 수행
	1. ItemReader가 청크 아이템을 읽어오거나 스킵한다.
	2. ItemProcess가 청크 아이템을 필터링하거나 수행한다.
	3. ItemWriter가 청크 아이템을 쓰거나 쓰기 작업을 스킵한다.
4. StepContribution에 3단계의 결과를 각 `ItemReader, ItemProcessor, ItemWriter`에서 값을 들고와서 청크 프로세스의 변경 사항을 버퍼링한다.
5. StepContribution이 `apply(contribution)`에 의해 StepExection의 상태를 업데이트한다.

## ExcutionContext

&emsp;&emsp;

### ExecutionContext

&emsp;&emsp;

#### ExectionContext 개념

&emsp;&emsp;

- 정의
	- 프레임워크에서 유지 및 관리하는 키/값으로 된 컬렉션으로 StepExection 또는 JobExection 객체의 상태를 저장하는 공유 객체
	- Job 생성시 생성되는 해당하는 Job에서만 공유되는 임시 공유 데이터 공간으로 생각하면 된다.
	- SecurityContext 또한 한 세션에서만 살아있는 공유 데이터 공간이였다. 그것과 비슷하지만 이 ExecutionContext는 실제로 공유 데이터를 DB에 넣어 사용된다. 
		- 이래야 재실행을 해도 동일하게 동작하기 때문이다.

&emsp;&emsp;

- 특징
	- DB 직렬화한 값으로 저장된다.
		- key-value json 형태라는 뜻
	- Job 재시작 시 이미 처리한 Row 데이터는 건너띄고 이후로 수행하도록 할 때 상태 정보를 활용한다.

&emsp;&emsp;

- JobExectionContext는 Job 내부 Step끼리 데이터 공유으로 필요하지만 StepExecutionContext도 필요한 이유
	- 재 실행하는 경우에 앞 Step은 건너 띄고 다시 중간 Step만 실행해야하는 경우에 유용하기 때문에 반드시 필요하다.

&emsp;&emsp;

- 공유 범위
	- Step 범위
		- 각 Step의 StepExection에 저장되며 Step 간 서로 공유가 되지 않는다.
	- Job 범위
		- 각 Job의 JobExecution에 저장되며 Job간에 공유는 안되지만 한 Job 내부의 Step 들 사이에서는 공유가 가능하다.

&emsp;&emsp;

- `execute(contribution, chunkContext)` 구현시 접근 방법
	- JobExecutionContext
		- `contribution.getStepExecution().getJobExecution().getExecutionContext()`
	- StepExecutionContext
		- `contribution.getStepExecution().getExecutionContext()`
	- 메시지 체이닝이 역겹기때문에 보통 `@StepScope `or `@JobScope` + `@Value("#{jobParameters['fileName']}")` 조합으로 사용하며 스프링 팀에서도 이것을 권장한다.

## JobRepository

&emsp;&emsp;

### JobRepository

&emsp;&emsp;

#### JobRepository 개념

&emsp;&emsp;

- 정의
	- 배치 작업 중의 정보를 저장하는 저장소 역할

&emsp;&emsp;

- 특징
	- Job의 수행 시간, 종료 시간, 수행 횟수, 실행에 대한 결과 등의 배치 작업의 수행과 관련된 모든 metadata를 저장
		- JobLauncher, Job, Step 구현체 내부에서 CRUD 기능을 처리한다.
	- 등록시 Proxy 객체로 생성된다.
		- Advice와 같은 AOP 기술 적용을 위함이다.


&emsp;&emsp;

- 설명
	- `@EnableBatchProcessing` 어노테이션  방식 
		- 이 어노테이션 선언시 자동으로 빈으로 등록
	- JDBC 방식으로 결정
		- JobRepositoryFactoryBean
			- 내부적으로 AOP 기술을 통해 트랜잭션 처리
			- **트랜잭션 isolation의 기본값은 SERIALIZEBLE, 즉 최고 수준이기 때문에 쓰기 작업 도중 읽기 작업 자체가 불가능하다.**
			- 그 이유는 다른 Job의 실행을 방지하기 위함이다.
				- 다른 레벨도 지정은 가능하다.
			- 메타테이블 Table Prefix를 변경할 수 있다.
				- 기본은 `BATCH_`이다.
	- In Memory 방식으로 결정
		- MapJobRepositoryFactoryBean
			- 성능 등의 이유로 도메인 오브젝트를 굳이 데이터베이스에 저장하고 싶지 않을 경우
			- Test나 프로토타입의 빠른 개발이 필요할 때

## JobLauncher

&emsp;&emsp;

### JobLauncher

&emsp;&emsp;

#### JobLauncher 개념

&emsp;&emsp;

- 정의
	- 배치 Job을 실행시키는 러너

&emsp;&emsp;

 - 특징
	 - Job과 Job Parameters를 인자로 받으며 요청된 배치 작업을 수행한 후 최종 client에게 JobExecution을 반환
	 - 스프링 부트 배치가 구동이 되도 자동 생성 되지 않아 따로 빈 등록을 해줘야한다.
	 - 수동으로 TaskExecutor를 설정해줘야 동기/비동기 여부를 선택할 수 있다.

&emsp;&emsp;

- 실행 과정
	- JobLauncher.run(job, jobParameters)
	- 스프링 부트 배치에서는 JobLauncherApplicationRunner가 기존에는 자동으로 실행시켜줬지만 지금은 따로 등록을 해야한다.

&emsp;&emsp;

- 동기별 실행 종류
	- 동기
		- taskExecutor를 SyncTaskExecutor로 설정하는 경우
		- JobExecution을 획득하고 배치 처리를 최종 완료한 이후 Client에게 JobExecution을 반환
		- 스케줄러에 의한 배치처리에 적합함
			- 새벽에 주로 실행되서 길어도 상관이 없는 애들
			- 원천 데이터 스크래핑 등
		- 반환될 때  ExitStatus
			- FINISHED
			- FAILED
	- 비동기
		- taskExecutor를 SimpleAyncTaskExecutor로 설정하는 경우
		- JobExecution을 획득하고 Client에게 JobExecution을 반환하고 배치 처리를 최종 완료한다.
		- Http 요청에 의한 배치처리에 적합하다.
			- 응답이 늦어지면 안되는 경우
			- 즉시 바로 요청 때려넣고 나중에 확인할려는 경우
		- 반환될 때의 ExitStatus
			- UNKNOWN
