&emsp;&emsp;

- 설명

### 시작하기

&emsp;&emsp;

#### 배치 활성화

&emsp;&emsp;

- 설명
	- `@EnableBatchProcessing` 
		- Configuration이나 스프링부트 애플리케이션 main 쪽 클래스 래벨에 추가한다.
		- 총 4개의 설정 클래스를 실행시키며 스프링 배치의 모든 초기화 및 실행 구성을 한다.
		- 자동 설정 클래스가 실행되며 빈으로 등록된 모든 Job을 검색하여 초기화와 동시에 Job을 수행하도록 구성된다.

&emsp;&emsp;

#### 배치 설정 클래스 종류

&emsp;&emsp;

- 설명
	- BatchAutoConfiguration
		- 스프링 배치가 초기화될 때 자동 실행되는 설정 클래스
		- Job을 수행하는 JobLauncherApplicationRunner 빈을 생성
	- SimpleBatchConfiguration
		- JobBuilderFactory와 StepBuilderFactory 생성
		- 스프링 배치의 주요 구성 요소 생성
			- 프록시 객체로 생성된다.
	- BatchConfigurerConfiguration
		- BasicBatchConfigurer
			- SimpleBatchConfiguration에서 생성한 프록시 객체의 실제 대상 객체를 생성하는 설정 클래스
			- 빈으로 의존성을 주입받아서 주요 객체들을 참조해서 사용할 수 있다.
		- JpaBatchConfigurer
			- JPA 관련 객체를 생성하는 설정클래스
			- BasicBatchConfigurer를 상속한 클래스
	- 사용자 정의 BatchConfigurer 인터페이스
		- BasicBatchConfigurer, JpaBatchConfigurer 둘 다 이것을 구현 중
		- 이것을 직접 사용자 정의로 구현하여 사용도 가능하다.

### 간단하게 Hello Batch! 출력 돌려보기

&emsp;&emsp;

- 구현 설명
	1. `@Configuration` 선언
	2. JobRepository를 생성자 주입
	3. PlatformTransactionManager를 생성자 주입
	4. JobBuilder로 Job 생성
		1. step 지정은 start(step) 이렇게 해야한다.
	5. StepBuilder로 Step 생성
	6. helloJob, helloStep 이름으로 Job, Step 하나씩 생성
	7. Step 안에서 Hello Batch!를 출력하는 tasklet 로직 구현
		1. tasklet은 단일 테스크로 수행되는 로직이다.

&emsp;&emsp;

- 추가 설명
	- 위의 구성은 사실 보안과 명시성 강화를 위해 5버전부터 실행이 되진 않는다.
	- 거기에 더불어 배치용 메타데이터 테이블이 없으면 실행되지도 않는다.
		- https://docs.spring.io/spring-batch/reference/schema-appendix.html
		- 이 스키마는 내부적으로 제공을 해주며 DDL을 복사 붙여넣기해서 사용하면 된다.
		- 늘 Flyway같은 마이그레이션 라이브러리를 사용하여 추가하는 것을 권장한다.
	- 테이블 추가 이후 실행 방법 3 가지
		- `JobLauncher`를 빈으로 추가 등록해 등록한 잡을 레포지터리에서 찾아서 실행
		- 아니면 ApplicationRunner 인터페이스 구현체를 만들어 여기에 JobLauncher와 Job 주입받고 이를 토대로 실행
		- 혹은, application.yaml에 enabled를 true로 설정한다.
			- 4버전에서는 true가 디폴트인데 5버전부터 점진적으로 변경, 삭제시켜주기 위한 일종의 플래그 값으로 설정하는 레거시 호환 설정이라고 보면 된다.
	- 정확히 명확하게 원하는 잡만 찾아서 수행시켜주기 위해 변경되었다.
	- 그 전에는 그냥 잡 빈을 등록만하면 실행이 되었지만 바뀌었다.

&emsp;&emsp;

- 메타데이터 테이블 ERD

![Spring Batch Metadata ERD](https://docs.spring.io/spring-batch/reference/_images/meta-data-erd.png)

&emsp;&emsp;

- 메타데이터 스키마 구성
	- Job 관련
		- BATCH_JOB_INSTANCE
			- Job 실행시 JobInstance 정보
			- job_name + job_key 복합키를 사용한다.
				- 따라서 같은 잡과 같은 키는 존재할 수 없다.
			- job_key는 job_name과 job_params의 값을 합쳐서 해시값으로 만든 값이다.
		- BATCH_JOB_EXECUTION
			- Job의 실행 정보
				- 생성 시간
				- 종료 시간
				- 실행 상태
				- 메시지
		- BATCH_JOB_EXECUTION_PARAMS
			- Job 실행시 에 넘겨진 파라미터 정보
		- BATCH_JOB_EXECUTION_CONTEXT
			- Job의 실행동안 여러가지 상태 정보, 공유 데이터
			- Json 형식으로 직렬화되어 저장된다.
			- **Step간 데이터 공유는 이 데이터를 사용한다.**
	- Step 관련
		- BATCH_STEP_EXECUTION
			- Step의 실행 정보
				- 생성 시간
				- 종료 시간
				- 실행 상태
				- 메시지
			- 실제 상세한 스텝 실행시 절차들의 Count 정보들도 추가로 들어간다.
				- 커밋 횟수
				- 쓰기, 읽기, 필터 횟수
				- 읽기, 쓰기, 필터 스킵 횟수
				- 프로세스 스킵 횟수
				- 롤백 횟수
		- BATCH_STEP_EXECUTION_CONTEXT
			- Step의 실행동안 여러가지 상태 정보, 공유 데이터
			- Json 형식으로 직렬화되어 저장된다.
			- **Step간 이 데이터는 공유될 수 없다.**

&emsp;&emsp;

- initialize-schema
	- always
		- 항상 생성을 시도한다.
	- embeded
		- 기본 값
		- yml 설정에 추가된 DataSource 환경에 따라서 결정된다.
	- never 
		- 운영에서는 명시적으로 꺼주는 것이 좋다.

&emsp;&emsp;

- 동작 설명
	- 전체 흐름
		- Job을 구동하고 Step을 실행하며 Step은 tasklet을 구동한다.
	- Job
		- 역할
			- 일, 일감 
			- 작업의 단위이다.
	- Step
		- 역할
			- 일의 항목, 단계
			- 작업의 구성요소
	- Tasklet
		- 역할
			- 실제 작업 로직
			- 비즈니스 로직 구현은 여기에 존재한다.
		- 특징
			- execute()를 가지고 있는 인터페이스이다.
				- 상태를 null로 반환하면 한번 반복한다.
				- 기본적으로 무한 반복을 하며 상태를 반환해 완료를 정의해줘야한다.
