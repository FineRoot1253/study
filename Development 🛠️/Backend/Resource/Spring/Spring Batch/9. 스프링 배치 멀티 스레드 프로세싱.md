# 스프링 배치 멀티 스레드 프로세싱

&emsp;&emsp;

## 기본 개념

&emsp;&emsp;

### 단일 스레드 vs 멀티 스레드

&emsp;&emsp;

- 특징
	- 멀티스레드는 보통 복잡한 처리, 대용량 데이터를 다루는 작업일 경우 전체 소요 시간 및 성능상의 이점을 가져오기 위해 멀티 스레드 방식을 선택
	- **멀티 스레드는 데이터 동기화 이슈가 존재**하므로 최대한 고려해서 결정해야한다.

&emsp;&emsp;

#### 멀티 스레드 프로세싱

&emsp;&emsp;

- 동작 구조
	1. Step에서 `RepeatTemplate`의 구현체인 `TaskExecutorRepeatTemplate`을 사용해 수행한다.
	2. `TaskExecutor`의 `execute()`를 호출해 정해진 스레드 수 대로 동작을 수행함
		1. 스레드들은 동작 이후 소멸되는 것이 아닌 스레드 풀에서 꺼내져서 사용되고 반환된다.
	3. `Thread`들은 `ExecutingRunnable` 구현체로 `run()`을 호출해 동작한다.
	4. `ExecutingRunnable`이 소유한 `RepeatCallback` 을 통해 `ChunkOrientedTasklet`을 통해 청크 기반 프로세스가 수행된다.
	5. 저장이 다되면 `BlockingQueue` 안에 `put(Runnable)` 메소드를 통해 자기 자신을 저장한다.
		1. `BlockingQueue`는 스레드 안전성을 보장한다.
	6. `TaskExecutorRepeatTemplate`에서 `BlockingQueue`에서 `Runnable take()` 메소드를 호출해 결과를 꺼내 반복할지 말지 결정한다.

&emsp;&emsp;

- 특징
	- `ExecutingRunnable` 인터페이스 API
		- `RepeatCallback callback`
		- `ResultQueue<ResultHolder> queue`
		- `RepeatStatus result`
	- 스프링 배치 스레드 모델 특징
		- 기본 동작은 스프링 배치는 단일 스레드 기반으로 동작한다.
		- 성능 향상과 대규모 데이터 작업을 위한 비동기 처리 및 Scale out 기능을 제공한다.
		- Local과 Remote 처리를 지원한다.
	- 스프링 배치 스레드 모델 종류
		- `AsyncItemProcessor` & `AsyncItemWriter`
			- `ItemProcessor`에게 별도의 스레드가 할당되어 작업을 처리하는 방식
		- `Multithreaded Step`
			- Step 내 Chunk 구조인 ItemReader, ItemProcessor, ItemWriter마다 여러 스레드가 할당되어 실행하는 방법
		- `Remote Chunking`
			- 분산환경 처럼 Step 처리가 여러 프로세스로 분할되어 외부의 다른 서버로 전송되어 처리하는 방식
		- `Parallel Steps`
			- Step마다 스레드가 할당되어 여러개의 Step을 병렬로 실행하는 방법
		- `Partitioning`
			- Master/Slave 방식으로서 Master가 데이터를 파티셔닝한 다음 각 파티션에게 스레드를 할당하여 Slave가 독립적으로 작동하는 방식

## AsyncItemProcessor & AsyncItemWriter

&emsp;&emsp;

- 정의
	- Step 안에서 ItemProcessor가 비동기적으로 동작하는 스레드 모델

&emsp;&emsp;

- 특징
	- `AsyncItemProcessor` & `AsyncItemWriter`가 한 몸처럼 구성되어야한다.
	- `AsyncItemProcessor` 로부터 `AsyncItemWriter`가 받는 결과물은 `List<Future<T>>` 타입이며 비동기 실행이 완료될 때까지 대기한다.
	- `spring-batch-integration` 의존성이 필요하다.
	- `AsyncItemProcessor` 인터페이스 API
		- `ItemProcessor<I, O> delegate`
			- 실제 프로세스를 수행하는 `ItemProcessor`
		- `TaskExecutor taskExecutor = new SyncTaskExecutor();`
			- Thread를 생성하고 Task에 할당하는 분배기
		- `FutureTask<O> task = new FutureTask<>(Callable<V>)`
			- Thread가 수행하는 Task 로서 Callable를 실행시키고 결과를 `Future<V>`에 담아 반환한다.
			- `Runnable`이 아닌 `Callable`인 이유는 결과 값을 반환 받아야하기 때문이다.
	- 빌더 설정 방법
		- 스텝 빌더의 processor와 writer 설정시, 각각 `AsyncItemProcessor` & `AsyncItemWriter`를 세팅해주면 된다.
		- 각자 클래스 내부에 `setDelegate()` 메소드가 있는데 여기에 구현해둔 기존 `ItemProcessor`, `ItemWriter`를 세팅하면된다.
		- `AsyncTaskExecuter`는 `setTaskExecutor()`에다가 세팅하면된다.
			- 예제에서는 `SimpleAsyncTaskExecutor`를 사용하셨다.

&emsp;&emsp;

- 기본 동작 구조
	1. `Job`안에서 `Step`이 실행되면 `ItemReader`로 읽는다.
	2. `AsyncItemProcessor`는 내부적으로 `ItemProcessor`가 위임하여 프로세스를 처리하는데 이때 `TaskExecutor`를 사용해 비동기로 수행한다.
	3. `AsyncItemWriter`가 `List<Future>` 타입의 결과물을 기다리고 있다가 마찬가지로 내부적으로 `ItemWriter`에게 위임하여 작성 처리를 진행한다.

&emsp;&emsp;

- 상세 동작 구조
	1. `Job`이 `TaskletStep`을 통해 작업 실행
	2. `ChunkOrientedTasklet`을 통해 `ItemReader`, `AsyncItemProcessor`, `AsyncItemWriter`를 각각 수행한다.
	3. `AsyncItemProcessor`는 `TaskExecutor`에게, 그리고 `TaskExecutor`는 WorkThread에 `ItemProcessor` 작업을 내포하는 `FutureTask`를 담아 수행한다.
	4. `AsyncItemWriter`는  `List<Future<T>>`에 있는 작업을 돌면서 처리하는데 이때 실행 값들이 아직 받아오지 않은 값이 있다면 그 자리에서 대기한다.
		1. 실제 내부에서는 delegate에게 넘길 아이템을 리스트에 하나씩 담아서 delegate에 넘기게되고 Future.get()을 하는 경우에 이때 대기를 하기 때문에 실제 대기는 write 단계에서 하기 때문에 이 단계에서 분산되어 대기하는 구조이다.

## Multi-threaded Step

&emsp;&emsp;

- 정의
	- Step내에서 멀티 스레드를 통해 Chunk 기반 프로세스가 이뤄지는 모델

&emsp;&emsp;

- 특징
	- `TaskExecutorRepeatTemplate`이 반복자로 사용되며 설정한 개수 만큼 스레드를 생성하여 수행
	- `TaskExecutorRepeatTemplate` API
		- `int throttleLimit = DEFAULT_THROTTLE_LIMIT;`
			- 조절 제한 개수
		- `TaskExecutor taskExecutor = new SyncTaskExecutor();`
			- Thread를 조절 제한 수 만큼 생성하고 Task를 할당
	- 스텝 빌더 세팅시 주의사항
		- `ItemReader`를 스텝 빌더에 등록할 때, `JdbcPagingItemReader`나 `JpaPagingItemReader`를 해두면 스레드 안정성을 보장해준다.
		- 스프링에서 만든 `ThreadPoolTaskExecutor`를 써주면 편하다. `taskExecutor()`에 세팅만하면 바로 `Multi-threaded Step`으로 동작하게 된다.
	- ==**파일에서는 쓸 수 없는 방식이다.**== 읽기 처리가 스레드 안정성이 보장되지 않기 때문이다.

&emsp;&emsp;

- 기본 동작 구조
	1. Job에서 Step 수행
	2. Step에서 `TaskExecutorRepeatTemplate`를 수행
	3. `TaskExecutorRepeatTemplate`에서 Runnable을 통해 `RepeatCallback`을 수행
	4. `RepeatCallback`에는 지정해뒀던 `ChunkOrientedTasklet`을 통해 작업을 수행
	5. 각 `ItemReader`, `ItemProcessor`, `ItemWriter`에 각각 워커 스레드가 붙어서 동작하게 된다.
		1. ==**이때 `ItemReader`는 스레드가 안전하게 보장되어야한다.**==

&emsp;&emsp;

- 상세 동작 구조
	1. `ItemReader`는 `Thread-safe`인지 반드시 확인해야한다.
		1. 데이터를 소스로부터 읽어오는 역할이기 때문에 스레드마다 중복해서 데이터를 읽어오지 않도록 동기화가 보장되어야한다.
		2. 그 이유는 읽어올때 같은 데이터를 중복해서 읽어오면 안되기 때문
	2. 스레드마다 새로운 Chunk가 할당되어 데이터 동기화가 보장된다.
		1. 스레드끼리 Chunk를 공유하지 않는다.
			1. 즉, 한 스레드에서 이뤄진 작업은 한 스레드에서 마무리된다.
		2. 매 `ItemReader`, `ItemProcessor`에 작업 단계에서 Chunk는 새로 생성된다.  
			1. 스레드 격리를 보장하는 것

## Parallel Steps

&emsp;&emsp;

- 정의
	- `SplitState`를 사용해서 여러 개의 Flow 들을 병렬적으로 실행하는 구조

&emsp;&emsp;

- 특징
	- 실행이 다 완료된 후 FlowExecutionStatus 결과들을 취합해서 다음 단계 결정
	- `SplitState` 필드 및 메서드
		- `Collection<Flow> flows`
			- 병렬로 수행할 `Flow`들이 담긴 컬렉션
		- `TaskExecutor taskExecutor`
			- Thread를 생성하고 Task를 할당하는 실행기
		- `FlowExecutionAggregator aggregator`
			- 병렬로 수행 후 하나의 종료 상태로 집계처리하는 클래스
		- `FlowExecutionStatus handle(FlowExecutor)`
			- 실행 메서드
	- `JobBuilder` API 사용법
		- `start(Flow)`
			- 기반 실행 Flow를 생성
		- `split(TaskExecutor).add(Flow...)`
			- 이 Flow들은 `TaskExecutor`가 개수 만큼 스레드를 생성해 Flow를 실행시킨다.
			- 이때 앞서 `start()`에 세팅한 Flow도 함께 합쳐져서 병렬 수행된다.
		- `next(Flow)`
			- 이 플로우는 split 처리가 완료된 후 실행이 된다.

&emsp;&emsp;

- 기본 동작 구조
	1. `Job`에서 `Flow` 호출
	2. `Flow`에서 `SplitState`를 호출
	3. `TaskExecutor`에서 각 워커들마다 `SimpleFlow`를 들고 있는 `FutureTask`에 작업을 수행
		1. `SplitState`에 복수 개의 `flows`들을 들고 있어 이렇게 수행이 가능하다.
	4. `result`들 (`Collection<FlowExecution>`)을 `FlowExecutionAggregator`가 집계 후 최종 결과인 `FlowExecutionStatus`로 반환한다.
		1. `COMPLETED`, `STOPPED`, `FAILED`, `UNKNOWN`으로 최종 실행 결과의 상태 값을 반환하여 다음 스텝을 결정한다.

## Partitioning

&emsp;&emsp;

- 정의
	- `MasterStep`이 `SlaveStep`을 실행시키는 구조

&emsp;&emsp;

- 특징
	- `SlaveStep`은 각 스레드에 의해 독립적으로 실행이 됨
	- `SlaveStep`은 독립적인 `StepExecution` 파라미터 환경을 구성한다.
		- 따라서 스레드 안전한 환경이 구성된다.
	- `SlaveStep`은 `ItemReader`, `ItemProcessor`, `ItemWriter` 등을 가지고 동작하고 작업을 독립적으로 병렬 처리한다.
	- `MasterStep`은 `PartitionStep`이며 `SlaveStep`은 `TaskletStep`, `FlowStep` 등이 올 수 있다.
	- 실제로는 `SlaveStep`이 여러 개가 생성해서 사용되는 것이 아니라, 하나의 `SlaveStep`에서 각자 독립적인 `StepExecution` 이 `execute()`에 넣어져 수행되어 여러 개의 `SlaveStep`이 수행된 것으로 보이게 된다.
	- 구성 구현체 종류
		- `PartitionStep`
			- 특징
				- 파티셔닝을 수행하는 Step
				- 이 `Step`이 Master 역할을 한다.
				- 파티셔닝 수행 후 `StepExecutionAggregator`를 사용해 `StepExecution`의 정보를 최종 집계한다.
			- 구성
				- `PartitionHandler partitionHandler`
				- `StepExecutionSpliter stepExecutionSpliter`
				- `StepExecutionAggregator stepExecutionAggregator`
		- `PartitionHandler`
			- 특징
				- `PartitionStep`에 의해 호출되며 스레드를 생성해 `WorkStep`을 병렬 수행하는 수행기
				- `WorkStep`에서 사용할 `StepExecution` 생성은 `StepExecutionSplitter`와 `Partitioner`에게 위임한다.
				- `WorkStep`을 병렬로 실행 후 최종 결과를 담은 `StepExecution`을 `PartitionStep`에 반환한다.
			- 구성
				- `Collection<StepExecution> handle(StepExecutionSpliter stepSpliter, StepExecution masterStepExecution)`
		- `StepExecutionSpliter`
			- 특징
				- `WorkStep`에서 사용할 `StepExecution`을 `gridSize`만큼 사용
				- `Partitioner`를 통해 `ExecutionContext`를 얻어서 `StepExecution`에 매핑하는 `StepExecution` 분배기
			- 구성
				- `Set<StepExecution> split(StepExecution masterStepExecution, int gridSize)`
		- `Partitioner`
			- 특징
				- `StepExcution`에 매핑할 `ExecutionContext`를 `gridSize` 만큼 생성하는 `ExecutionContext` 생성기
				- 각 `ExecutionContext`에 저장된 정보는 `WorkStep`을 실행하는 스레드마다 독립적으로 참조 및 활용 가능
			- 구성
				- `Map<String, ExecutionContext> partition(int gridSize)`
	- `StepBuilder` API를 통한 구현 방법
		- `partitioner("Slave Step Name", new ColumnRangePartitioner())`
			- `PatitionStep` 생성을 위한 `PartitionStepBuilder` 를 반환하는 빌더 메서드
			- 여기에서 `Partitioner`를 설정한다.
		- `step(slaveStep())`
			- SlaveStep을 설정
			- TaskletStep, FlowStep 등이 사용 가능
		- `gridSize(4)`
			- 파티션 구분을 위한 값 설정
			- 몇 개의 파티션을 사용할 것인지를 세팅한다.
		- `taskExecutor(taskExecutor())`
			- PartitionStep 생성
			- 최종적으로 `MasterStep`이 생성된다.

&emsp;&emsp;

- 기본 동작 구조
	1. `Job`에 `Step`이 호출된다.
	2. `Step` 들 사이에 `MasterStep`이 `SlaveStep`들을 각자 병렬 수행한다.
		1. `MasterStep`은 `PartitionHander`의 `handle()`을 통해 수행한다.
		2. `PartitionHandler`는 `StepExecutionSpliter`를 통해 `gridSize`만큼 `StepExecution`과 `ExecutionContext`를 세팅한 Set을 만든다.
		3. `TaskExecutor`는 `StepExecutionSpliter`가 세팅해준 Set을 통해 `SlaveStep`들을 독립적인 `StepExecution`와 스레드간 공유되는 `ExecutionContext`를 통하여 병렬 실행시킨다.
			1. 이 뜻은 각 `ItemReader`, `ItemProcessor`, `ItemWriter`의 한 스레드끼리는 같은 데이터를 보존하고 하여 이 데이터만 공유하지 다른 데이터를 공유한다는 뜻이 아니다.
			2. 기본적으로 `ItemReader`, `ItemProcessor`, `ItemWriter`각각 프록시 빈이 생성되어 각 스레드 별로 생성해서 사용된다.
			3. 따라서 반드시 각자 `@StepScope`를 사용해줘야한다.
		4. `StepExecutionAggregator`가 실행 결과를 취합하여 최종 결과를 반환한다.

## SynchronizedItemStreamReader

&emsp;&emsp;

- 정의
	- 스레드 안전 하지 않은 `ItemReader`를 스레드 안전하게 처리해주는 스트림 리더

&emsp;&emsp;

- 특징
	- 4.0 버전부터 지원중이다.
	- 각 워커들이 읽기 시도를 하면 wait()을 하게 된다.
	- `SynchronizedItemStreamReaderBuilder` API
		- `delegate(ItemReader)`
			- 이 메서드를 통해 사용할 스레드 안전하지 않은 `ItemReader`를 위임받아 사용한다.
