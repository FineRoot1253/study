## 배치 초기화 설정

&emsp;&emsp;

### 초기화 설정

&emsp;&emsp;

#### JobLauncherApplicationRunner

&emsp;&emsp;

- 정의
	- Spring Batch 작업을 시작하는 ApplicationRunner
	- BatchAutoConfiguration에서 생성

&emsp;&emsp;

- 특징
	- 곧 Deprecated 예정이다.
	- 빈으로 등록된 모든 Job을 실행시킨다.
		- 이 점이 Deprecated 대상인 이유이다.
		- 의도치 않게 등록된 Job 빈이 실행되서 운영DB에 오염이 되는 경우가 심심치 않게 많다고 한다.
	- 최신 버전에서는 `implements CommandLineRunner`를 하여 러너를 구현해서 컴포넌트로 등록해야한다.

&emsp;&emsp;

#### BatchProperties

&emsp;&emsp;

- 정의
	- 스프링 배치 환경 설정 값 클래스

&emsp;&emsp;

- 특징
	- Job 이름, 스키마 초기화 설정, 테이블 Prefix등을 지님
	- application.yml에 설정하는 구조

&emsp;&emsp;

- 예시
	- batch
		- job
			- names: `${job.name:NONE}`
		- initialize-schema: NEVER
		- tablePrefix: Lit-Tok

&emsp;&emsp;

#### Job 실행 옵션

&emsp;&emsp;

- 설정
	- 지정한 Batch Job만 실행하는 옵션
	- spring.batch.job.names

&emsp;&emsp;

##### 실 예시

```java
@Component 
public class FlexibleBatchRunner implements CommandLineRunner { 

	@Value("${job.name:NONE}") 
	private String jobName; 
	
	@Override 
	public void run(String... args) throws Exception { 
		if (!"NONE".equals(jobName)) { 
			Job job = applicationContext.getBean(jobName, Job.class);
			jobLauncher.run(job, createJobParameters()); 
			} 
	} 
}
```

&emsp;&emsp;

- 설명
	- 실제로는 Deprecated되었기 때문에 이렇게 사용해야 레거시방식을 재활용할 수 있다.

## Job and Step

&emsp;&emsp;

### JobBuilder

&emsp;&emsp;

#### JobBuilder 개념

&emsp;&emsp;

- 정의
	- Job을 구성하는 설정 조건에 따라 구체 빌더 클래스를 생성하고 실제 Job 생성을 위임한다.

&emsp;&emsp;

- 구체 클래스
	- SimpleJobBuilder
		- 특징
			- SimpleJob을 생성하는 Builder 클래스
			- Job 실행과 관련된 여러 설정 API 제공
		- API
			- `start(step)`
			- `next(step)`
			- `on(regexPattern)`
				- TransitionBuilder 반환
			- `start(decider)`
				- FlowBuilder 반환
			- `next(decider)`
				- FlowBuilder 반환
			- `split(taskExecutor)`
				- SplitBuilder 반환
	- FlowJobBuilder
		- 특징
			- FlowJob을 생성하는 Builder 클래스
			- 내부적으로 FlowBuilder를 반환해 Flow 실행과 관련된 여러 설정 API를 제공
			- JobFlowBuilder는 이 빌더의 구현체이다.
		- `start(flow)`
			- FlowBuilder 생성
		- `start(step)`
			- JobFlowBuilder 생성
	- JobFlowBuilder
		- 특징
			- FlowJobBuilder를 구현하고 있는 구현체
		- API
			- `start(step or flow)`
			- `next(step or flow)`
			- `on(regexPattern)`
				- TransitionBuilder 반환
			- `start(decider)`
				- FlowBuilder 반환
			- `next(decider)`
				- FlowBuilder 반환
			- `from(decider)`
				- FlowBuilder 반환

&emsp;&emsp;

- 특징
	- `start()`에 파라미터로 step을 넣으면 SimpleJobBuilder, flow를 넣으면 FlowJobBuilder가 생성된다.
		- `flow(step)`을 해도 FlowJobBuilder를 반환한다.

&emsp;&emsp;

### SimpleJob

&emsp;&emsp;

#### SimpleJob 개념

&emsp;&emsp;

- 정의
	- Step을 실행시키는 기본 구현체

&emsp;&emsp;

- 특징
	- SimpleJobBuilder가 생성한다.
	- 여러 단계의 Step으로 구성할 수 있으며 Step을 순차적으로 실행시킨다.
	- 모든 step의 실행이 성공해야 Job이 성공적으로 완료처리된다.
		- 그래서 맨 마지막에 실행한 Step의 BatchStatus가 Job의 최종 BatchStatus가 된다.


&emsp;&emsp;

#### SimpleJobBuilder API

&emsp;&emsp;

- 종류
	- `start(step)`
		- 처음 실행할 step 설정
		- 최초에 한번만 설정된다.
		- SimpleJobBuilder 생성 후 반환
	- `next(step)`
		- 다음에 실행될 Step들을 순차적으로 연결하도록 설정
		- 여러 번 설정이 가능하며 모든 next()의 Step이 종료되면 Job이 종료된다.
	- `incrementer(jobParametersIncrementer)`
		- JobParameter의 값을 자동 증가시켜주는 jobParametersIncrementer 설정
		- 기존 JobParameter 변경 없이 Job을 여러 번 시작하고 싶을 때 사용된다.
			- 기본 Job의 도메인 규칙은 같은 Job으로 JobParameter의 변경없이 시도하면 시도가 안되기 때문
		- RunIdIncrement 구현체를 지원하며 JobParametersIncrementer를 구현해서 사용해도 된다.
	- `preventRestart(true)`
		- job의 재시작 여부의 설정
		- true가 디폴트
		- 기본 Job의 도메인 규칙은 실패하면 재시도 가능이지만 이것을 false로 놓으면 재시작이 불가능하다.
			- JobRestartException이 던져진다.
		- 생성 초기에는 JobExecution이 초기화되기 전이기 때문에 검증하지 않는다.
	- `validator(jobParameterValidator)`
		- jobParameter를 검증하는 jobParameterValidator 설정
		- DefaultJobParameterValidator 구현체를 지원하고 있다.
			- requiredKeys, optionalKeys 이 두가지로 왠만한 검증은 다 가능하다.
		- 더 복잡한 제약조건은 직접 JobParameterValidator 인터페이스를 구현해야한다.
			- 필수적인 비즈니스 로직, 특히 불변식은 여기에 구현해야한다.
		- ==**이곳에서 던진 예외는 DB에 아예 남질 않는다.**==
	- `listener(jobExecutionListener)`
		- Job 라이브 사이클 시점에 콜백을 제공받도록 
		- jobExecutionListener 설정

&emsp;&emsp;

#### SimpleJob 아키텍쳐

&emsp;&emsp;

- run 시 동작 흐름
	1. JobLauncher가 JobParameters와 Job을 가지고 실행시킴
	2. JobInstance가 생성된다.
	3. JobExecution이 있는지 찾아보고 생성한다.
		1. 이때 status를 STARTED로 둔다.
		2. JobExecutionContext를 생성한다.
	4. JobExecutionListener beforeJob이 호출된다.
	5. for 루프를 돌면서 step들을 순회하며 수행함
		1. 매 step 마다 StepExecution이 생성된다.
		2. 작업 수행시 StepExecutionContext을 생성한다
	6. JobExecutionListener afterJob이 호출된다.
	7. 작업 상태를 마지막 step의 상태를 가져와 동일하게 업데이트한다.

### StepBuilder

&emsp;&emsp;

#### StepBuilder 개념

&emsp;&emsp;

- 정의
	- Step을 구성하는 설정 조건에 따라 다섯 개의 하위 빌더 클래스를 생성하고 실제 Step 생성을 위임한다.

&emsp;&emsp;

- 구현체 종류
	- TaskletBuilder
		- TaskletStep을 생성하는 기본 빌더 클래스
	- SimpleStepBuilder
		- TaskletStep을 생성하며 내부적으로 청크 기반 작업을 처리하는 ChunkOrientedTasklet 클래스를 생성
	- PartitionStepBuilder
		- PartitionStep을 생성하며 멀티 스레드 방식으로 Job을 실행
	- JobStepBuilder
		- JobStep을 생성하여 Step 안에서 Job을 실행
	- FlowStepBuilder
		- FlowStep을 생성하여 Step 안에서 Flow를 실행

&emsp;&emsp;

- 생성 과정
	1. StepBuilder에 이름과 jobRepository를 넘겨 빌더를 생성한다.
	2. 이후 호출된 api에 따라 다른 빌더들을 반환한다.
		- TaskletStepBuilder
			- tasklet(tasklet())
		- SimpleStepBuilder
			- chunk(chunkSize)
			- chunk(completionPolicy)
		- PartitionStepBuilder
			- partioner(stepName, partioner)
			- partioner(step)
		- JobStepBuilder
			- job(job)
		- FlowStepBuilder
			- flow(flow)

### TaskletStep

&emsp;&emsp;

#### TaskletStep 개념

&emsp;&emsp;

- 정의
	- 스프링 배치에서 제공하는 Step 구현체로 Tasklet을 실행시키는 도메인 객체

&emsp;&emsp;

- 특징
	- RepeatTemplate을 사용해 Tasklet 구문을 트랜잭션 경계 내에서 반복해서 실행한다.
	- Task 기반과 Chunk 기반으로 나누어 Tasklet을 실행한다.

&emsp;&emsp;

- Task vs Chunk
	- chunk
		- 하나의 큰 덩어리를 n개씩 나누어 실행한다
		- 대용량 처리에 효과적인다.
		- ItemReader, ItemProcessor, ItemWriter를 사용하며 이를 기반한 구현체인 ChunkOrientedTasklet 구현체가 사용된다.
	- Task
		- 단일 작업 기반으로 처리되는 것이 효율적인 경우 사용된다.
		- 주로 Tasklet 구현체를 만들어 사용한다.
		- 대량 처리를 하는 경우 chunk 기반에 의해 더 복잡한 구현을 필요로 한다. 
			- 따라서 간단한 경우를 제외하면 chunk 기반을 기용한다.
			- 애초에 스프링 배치는 CPU bound용 라이브러리라는 것을 잊지 말아야한다.

&emsp;&emsp;

#### TaskletStepBuilder API

&emsp;&emsp;

- 종류
	- tasklet(Tasklet)
		- Taskley 클래스 설정
		- TaskletStepBuilder를 반환
		- 익명 클래스나 구현 클래스로 만들어 사용한다.
		- 하나의 Tasklet만 설정이 가능하며 두개 이상을 설정하는 경우 마지막에 설정한 객체가 실행된다.
		- 단일 트랜잭션으로 보장해야하는 단일 작업에만 사용해야한다.
	- startLimit(Integer)
		- Step의 실행 횟수 설정
		- 설정한 값 만큼 실행되고 초과시 오류(StartLimitExceededException) 발생
		- 기본값
			- INTEGER.MAX_VALUE
	- allowStartIfComplete(Boolean)
		- 성공 실패 상관없이 항상 Step을 실행하기 위한 설정
		- 재시작 가능한 Job에서 Step 이전 성공 여부와 상관없이 다시 실행하기 위한 설정
		- 기본값
			- false
		- 필요한 경우
			- 실행마다 유효성을 검증해야하는 Step이 전처리로 필요함
			- 실행마다 전처리 작업이 꼭 필요함
	- listener(StepExecutionListener)
		- Step 라이프사이클 Hook
		- StepExecutionListener 설정

##### Tasklet 개념

&emsp;&emsp;

- 정의
	- Step 내에서 구성되고 실행되는 도메인 객체로서 주로 단일 태스크를 수행하기 위한 것

&emsp;&emsp;

- 특징
	- TaskletStep에 의해 반복적으로 수행되며 반환 값에 따라 계속 수행 혹은 종료
	- RepeatStatus - Tasklet 반복 여부 상태 값
		- FINISHED
			- Tasklet 종료, null을 반환하는 경우 FINISHED로 간주된다.
		- CONTINUABLE
			- Tasklet 반복
			- FINISHED가 리턴되거나 실패 예외가 던져지기 전까진 while문 내부에서 무한 루핑을 돈다.

&emsp;&emsp;

- TaskletStep 동작 순서
	1. Job에서 StepExecution을 ExecutionContext에 BatchStatus를 STARTED로 두고 ExitStatus를 EXECUTING으로 둔다.
	2. Step 실행 전 StepExecutionListener에서 훅을 호출한다.
		1. `CompositeStepExecutionListener.beforeStep()`
	3. Step의 RepeatTemplate에서 Tasklet을 기반으로 반복문을 돌린다.
		1. 이때 RepeatStatus는 CONTINUABLE이다.
		2. 비즈니스 로직을 수행한다.
			1. 만약 예외가 발생하면 Step을 종료한다.
			2. 이 경우 RepeatStatus는 CONTINUED를 반환한다.
		3. 성공한 경우 RepeatStatus를 FINISHED로 반환한다.
	4. Step 실행 후 StepExecutionListener에서 훅을 호출한다.
	5. ExitStatus와 BatchStatus를 RepeatStatus에 따라 업데이트한다.
		1. FINISHED면 COMPLETED로 둔다.
		2. 아니면 FAILED로 둔다.

### JobStep

&emsp;&emsp;

#### JobStep 개념

&emsp;&emsp;

- 정의
	- Job에 속하는 Step 중 외부의 Job을 포함하고 있는 Step

&emsp;&emsp;

- 특징
	- 외부의 Job이 실패하면 해당 Step이 실패하므로 최종 기본 Job도 실패한다.
	- 모든 메타데이터는 기본 Job과 외부 Job을 별개로 저장한다.
	- 커다란 시스템을 작은 모듈로 쪼개고 Job의 흐름을 관리하고자 할때 주로 사용한다.
	- 주로 각 단계가 다른 Job에 영향을 주지만 막상 각 Job들이 각자 다른 인프라를 이용해야하는 경우에 유용하다.
	- **부모가 일부 스탭을 실패해도 자식 job은 실패하지 않았다면 이건 JobExecution에 COMPLETED로 기록된다.**
	- **자식이 스탭을 실패하면 부모 자식 모두가 FAILED로 기록된다.**

&emsp;&emsp;

- API
	- job(job)
		- 실행할 Job 설정
	- launcher(jobLauncher)
		- Job을 실행할 런처
	- parameterExtractor(jobParameterExtractor)
		- ExecutionContext를 Job이 실행되는데 필요한 JobParameter로 변환해주는 변환기
		- 만약 Listener에서 미리 beforeStep()에서 StepExecution에 값을 미리 넣어주면, 이것을 다음 job에서 jobParameters에 추가한다.
