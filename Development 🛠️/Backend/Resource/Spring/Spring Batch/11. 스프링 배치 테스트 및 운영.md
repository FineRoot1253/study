# 스프링 배치 테스트 및 운영

&emsp;&emsp;

## Spring Batch Test

&emsp;&emsp;

- 특징
	- 스프링 배치 4.1.x 이상 버전 (부트 2.1) 기준
	- 테스트 코드 작성시 필요한 어노테이션
		- Job, Step 테스트
			- `@SpringBatchTest`
				- 자동으로 ApplicationContext에 테스트에 필요한 여러 유틸 Bean을 등록해주는 어노테이션
			- `@SpringBootTest`
				- 자동으로 Spring Boot 컨텍스트를 등록하는 어노테이션
				- 이걸 통해서 Job 설정 클래스, 통합 테스트를 위한 여러 의존성 빈들을 주입받기 위해 필요하다.
		- 설정 클래스
		- `@Configuration`
		- `@EnableAutoConfiguation`
		- `@EnableBatchProcessing`
			- 테스트 시 배치 환경 및 설정 초기화를 자동 구동하기 위한 어노테이션
			- 테스트 클래스마다 선언하지 않고 공통으로 사용하기 위함
			- `@EnableJpaTest`와 비슷하다.
	- `JobLauncherTestUtils`
		- `launchJob()`, `launchStep()`과 같은 스프링 배치 테스트에 필요한 유틸성 메소드 지원
		- API 종류
			- `@Autowired void setJob(Job)`
				- 실행할 Job을 자동으로 주입받는다.
				- 한 개의 Job 만을 받을 수 있다.
					- 따라서 Job 설정 클래스를 한 개만 지정해서 테스트하는 방식이다.
			- `JobExecution launchJob(JobParameters)`
				- Job을 실행시키고 JobExecution을 반환
			- `JobExecution launchStep(String stepName)`
				- Step을 실행시키고 JobExecution을 반환
	- `JobRepositoryTestUtils`
		- `JobRepository`를 통해 `JobExecution` 생성, 삭제 기능 메소드 지원
		- API 종류
			- `List<JobExecution> createJobExecutions(String jobName, String[] stepNames, int count)`
				- JobExecution 생성
			- `void removeJobExecutions(Collection<JobExecution> list)`
				- JobExecution 삭제
	- `JobScopeTestExecutionListener`
		- `@JobScope` 컨텍스트를 생성
		- `@JobScope` 컨텍스트를 통해 `JobParameter`등을 단위 테스트에서 DI 받을 수 있음
	- `StepScopeTestExecutionListener`
		- `@StepScope`
		- `@StepScope` 컨텍스트를 통해 `JobParameter`등을 단위 테스트에서 DI 받을 수 있음

&emsp;&emsp;

- 작성 방법
	1. `@SpringBootTest`에 테스트할 Job을 설정해둔 클래스와 테스트용 설정 클래스를 어노테이션의 맴버인 `classes`로 넣어줘야한다.
	2. 필요한 유틸 클래스를 `@Autowired`로 주입받자.
		1. `JobLauncherTestUtils`, `JobRepositoryTestUtils` 등이 있다.
	3. 테스트를 작성시
		1. mock으로 JobParameter를 임의로 만들어준다.
		2. `JobLauncherTestUtils`의 `launchJob(JobParameter)`로 Job을 실행한다.
		3. `launchJob(JobParameter)` 결과인 `JobExecution`의 결과 상태를 검증하거나 리스너 유틸등을 통해 상태를 검증하자.

&emsp;&emsp;

- 주의사항
	- Rollback이 안된다. 따라서 Job 수행 결과는 알아서 롤백 해줘야한다.
	- 항상 청크 사이즈의 + 1번 커밋을 하게된다.
		- 마지막에 더 남은 데이터가 있는지 검증하기 때문이다.

## JobExplorer, JobRegistry, JobOperator

&emsp;&emsp;

### JobExplorer

&emsp;&emsp;

- 정의
	- JobRepository의 readOnly 버전

&emsp;&emsp;

- 특징
	- 실행 중인 Job의 실행 정보인 JobExecution 또는 Step의 실행 정보인 StepExecution을 조회할 수 있다.

&emsp;&emsp;

- API
	- `List<JobInstance> getJobInstances(String jobName, int start, int count)`
		- start 부터 count 만큼 jobInstances를 가져온다.
	- `JobExecution getJobExecution(Long executionId)`
		- `JobExecutionId`를 이용해 `JobExecution`을 가져온다.
	- `StepExecution getStepExecution(Long jobExecutionId, Long stepExecutionId)`
		- jobExecutionId과 stepExecutionId을 이용해 StepExecution을 가져온다.
	- `JobInstance getJobInstance(Long instanceId)`
		- `JobInstanceId`를 이용해 `JobInstance`을 가져온다.
	- `List<JobExecution> getJobExecutions(JobInstance jobInstance)`
		- `JobInstance`을 이용해 `JobExecution` 리스트를 가져온다.
	- `Set<JobExecution> findRunningJobExecutions(String jobName)`
		- jobName을 이용해 실행중인 Job의 JobExecution 들을 조회한다.
	- `List<String> getJobNames()`
		- 실행 가능한 Job들의 이름을 가져온다.

&emsp;&emsp;

### JobRegistry

&emsp;&emsp;

- 정의
	- 생성된 Job을 자동으로 등록, 추적 및 관리하며 여러 곳에서 Job을 생성한 경우 ApplicationContext에서 Job을 수집해서 사용할 수 있다.

&emsp;&emsp;

- 특징
	- 기본 구현체로 map 기반 MapJobRegistry 클래스를 제공
		- jobName이 Key, job이 Value인 엔트리로 매핑된다.
	- 초기 Job 등록 단계
		-  `JobRegistryBeanPostProcessor`
			- `BeanPostProcessor` 단계에서 bean 초기화 시 자동으로 JobRegistry에 Job을 등록시켜준다.
			- 이것을 따로 bean으로 등록해야 JobRegistry를 통해서 등록된 job에 접근하거나 따로 추가 등록등을 할 수 있다.

&emsp;&emsp;

- API
	- `void register(JobFactory)`
		- JobFactory에 Job 등록
	- `void unregister(String jobName)`
		- jobName으로 JobFactory에서 Job 삭제
	- `Job getJob(String jobName)`
		- jobName으로 Job을 가져온다.
	- `Set<String> getJobNames()`
		- jobName 들을 얻는다.

&emsp;&emsp;

### JobOperator

&emsp;&emsp;

- 정의
	- `JobExplorer`, `JobRepository`, `JobRegistry`, `JobLauncher`를 포함하며 배치 중단, 재시작, Job의 요약등의 모니터링을 제공하는 클래스

&emsp;&emsp;

- 특징
	- 기본 구현체로 SimpleJobOperator 클래스를 제공한다.

&emsp;&emsp;

- API
	- `Set<String> getJobNames()`
		- 실행 가능한 JobName들을 가져온다.
	- `int getJobInstanceCount(String jobName)`
		- JobInstance 개수 조회
	- `List<JobInstance> getJobInstances(String jobName, int start, int count)`
		- start부터 count만큼의 JobInstance 들의 Id를 조회
	- `List<Long> getRunningExecutions(String jobName)`
		- jobName을 통해 실행중인 Job의 Execution 들의 Id 들을 조회.
	- `Properties getParameters(long executionId)`
		- Job의 ExecutionId를 통해 Job의 Parameters를 조회
	- `void start(String jobName, Properties jobParameters)`
		- JobName, Job의 Parameters를 통해 Job 시작
	- `void restart(long executionId, Properties restartJobParameters)`
		- JobExecutionId를 이용해 정지되거나 이미 종료된 Job 중 재실행이 가능한 Job을 재시작
	- `Long startNextInstance(String jobName)`
		- JobName을 통해 항상 새로운 Job을 실행한다.
		- 이 메소드는 Job에 문제가 있어 처음부터 재시작할 때 주로 호출된다.
	- `void stop(long executionId)`
		- JobExecutionId를 이용하여 실행 중인 Job을 정지
		- Stop은 graceful하게 동작
			- 즉시 Stop이 이뤄지는 것이 아니라 현재 실행 중이던 Step은 다 실행하고 Job이 Stop된다는 뜻이다.
	- `JobInstance getJobInstance(long executionId)`
		- JobExecutionId를 통해 JobInstance 조회
	- `List<JobExecution> getJobExecutions(JobInstance instance)`
		- JobInstance를 통해 JobExecution 조회
	- `JobExecution getJobExecution(long executionId)`
		- JobExecutionId를 통해 JobExecution 조회
	- `List<StepExecution> getStepExecutions(long jobExecutionId)`
		- JobExecutionId를 통해 StepExecution을 조회
