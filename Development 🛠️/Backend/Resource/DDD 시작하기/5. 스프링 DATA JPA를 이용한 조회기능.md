## 개요

- 설명
	- CQRS
		- 정의
			- 명령 모델과 조회 모델을 분리하는 패턴
		- 특징
			- 명령(Command)과 조회(Query)를 분리하는 패턴이다.
			- DDD에서는 명령에 쓰이는 엔티티와 조회에 쓰이는 엔티티가 다르다.

## 검색을 위한 스펙

``` java
public interface Specification<T>{
	public boolean isSatisfiedBy(T agg);
}
```


``` java
public class OrdererSpec implements Specification<Order> {
	private String ordererId;

	public OrdererSpec(String ordererId){
		this.ordererId = ordererId;
	}

	public boolean isSatisfiedBy(Order agg){
		return agg.getOrdererId().getMemberId().getId().equals(ordererId);
	}
}
```

- 설명
	- 검색조건이 고정되어 있는 경우 특정 조건으로 조회하는 기능을 만들면 된다.
	- 목록 조회의 경우 다양한 검색조건을 필요로 한다.
		- 이때 스펙을 사용하면된다.
			- `isSatisfiedBy(T agg);` agg는 검사 대상이 되는 객체이다.
				- 검사 대상이 조건을 충족하는지 안하는지를 구현한다.
			- 위의 OrdererSpec이 예시 구현체이다.
	- 실무에서는 이렇게 하진 않는다.
		- 이런 구조가 추상화된 어떤 기술을 사용하는 것이 보편적이며 책에서는 크리테리아를 사용하지만 난 QueryDsl를 사용할 예정이다.

## 하이버네이트 @Subselect

- 설명
	- @Subselect 정의
		- 쿼리의 결과를 엔티티로 사용할 수 있게 만들어준다.
	- 사용방법
		1. 쿼리 결과 엔티티 클래스 레벨에 `@Immutable`, `@Subselect`를 붙여준다.
			1. 조회 쿼리를 값으로 가질 수 있게 된다.
			2. 당연하지만 ==**엔티티이기 때문에 변경을 하는 순간 update 쿼리가 나가게된다. 그런데 문제는 매핑 한 테이블이 따로 없으니(조회 쿼리 결과니까) 에러가 발생한다.**== 그래서 Immutable 어노테이션을 붙여 불변하게 만들어준다.
		2. 쿼리를 사용할 엔티티 클래스 레벨에 `@Synchronize({"변경감지테이블1","변경감지테이블2"})`를 붙여준다.
			1. 하이버네이트는 엔티티 로딩 전 지정된 테이블과 관련된 변경이 발생하면 플러시를 한다. 그런데 만약 엔티티 로딩 이후 관련 데이터를 변경하고 트랜잭션 내에서 바로 조회를 하면 당연히 제대로 반영된 엔티티 조회 결과를 불러오지 못한다. 
			2. 싱크로나이즈는 이점을 고려해 변경을 감지하여 최신화된 테이블을 불러올 수 있도록 목표 테이블을 걸어줄 수 있다.
				1. 자신의 버전보다 더 높은 tx에 의한 변경이 일어난 상태면, 이미 엔티티가 로딩이 되어 1차 캐시에 로딩이 된 상황이여도 한번 더 목표 엔티티를 로딩한다.
	- 개인 의견
		- 프로젝션DTO를 거는게 더 일반적이니 이런게 있다는 거 정도만 봐주자.
		- 여기서 싱크로나이즈는 확실히 쓸만해보인다.
