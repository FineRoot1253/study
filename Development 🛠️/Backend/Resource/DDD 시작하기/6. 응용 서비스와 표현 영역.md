## 표현 영역과 응용 영역

- 설명
	- 도메인 영역 만큼 중요한 것은 응용 영역과 표현 영역이다.

&emsp;&emsp;

### 표현 영역

- 설명
	- 사용자가 웹 브라우저에 던진 요청을 처리한다.
	- 각종 쿠키처리, 헤더처리등을 이곳에서 해야한다.

&emsp;&emsp;

### 응용 영역

- 설명
	- 실제 사용자가 원하는 기능을 제공하는 응용 영역에 위치한 서비스
	- ==**반환되는 데이터의 형식은 여기서 신경쓸 이유가 없다**==. 여기에서는 원하는 기능의 결과만 던져주고 그 ==**결과의 데이터를 다듬는 것은 표현영역에서 해야한다**==.


&emsp;&emsp;

## 응용 서비스의 역할

- 설명
	- 서비스의 주요 패턴
		1. 리포지터리에서 애그리거트 구하기
			1. 필요하면 다른 애그리거트에서도 구해야한다.
		2. 애그리거트의 도메인 기능을 실행하여 원하는 결과를 만든다.
		3. 결과를 반환한다.
	- 서비스의 애그리거트 생성 패턴
		1. 데이터가 괜찮은지 검사한다. (이건 인풋값 검사는 Validator로 처리해야한다.)
			1. 데이터 중복 검사등도 여기서한다.
		2. 애그리거트를 생성한다.
		3. 리포지터리에 애그리거트를 저장한다.
			1. 내 패턴의 경우, 필요한 연관 엔티티들도 여기서 다 생성해줘서 save를 여러번해줘야한다.
				1. 책에 적혀있는 것처럼 벨류 엔티티를 적극적으로 사용할 경우 애그리거트 루트만 save 한번 해주면 되지만, 데이터를 세밀하게 다뤄야하는 현업에서는 아쉬운 소리이다.
		4. 결과를 리턴한다.
	 - 주의 사항
		 - ==**도메인 로직을 넣지 않아야한다.**==
			 - 예를 들면, 패스워드 변경의 경우 계정 애그리거트가 제공해야하는 기능이다. 이런걸 서비스에 둬선 안된다.
			 - 패스워드 변경의 사전 검증으로 패스워드 검증 로직의 경우 괜찮은 패스워드인지 아닌지 if 로직자체를 서비스에 남겨서는 안된다.
			 - 오로지 패스워드를 입력 받았으면 그 검증은 changePassword() 내부에서 처리하고 여기에서 예외를 던지는 식으로 만들어야한다.
			 - 참고로 이 검증 로직을 도메인에 넣지 않으면 검증 로직이 여기저기에 파편화가 될 것이다.
				 - 그래서 이런 검증 자체를 WebUtils 같은 곳에 빼고 치우는 곳이 허다하다
		 - 만약 위의 패턴에 벗어난 행동을 한다면 도메인 로직의 행동을 서비스에서 대신 처리하는 안티 패턴일 확률이 매우 높다.
		 - 응용서비스는 트랜잭션의 역할도 수행하며, 상태를 변경했는데 DB에 반영하는 도중에 문제가 발생하면 수행하는 롤백의 책임을 여기서 져야한다.
			 - 서비스의 주요 역할은 트랜잭션, 접근 제어, 이벤트 처리 총 3가지이다.

&emsp;&emsp;

## 응용 서비스의 구현

- 설명
	- 특징
		- 응용서비스는 파사드 역할을 한다.
			- 내가 생각한 패턴인 오케스트레이션 패턴도 파사드 패턴의 일환이다.
	- 응용서비스의 구현 방식 2가지
		- 한 응용서비스에 모든 도메인 기능 제공하기
		- 구분되는 기능별로 응용서비스 클래스 따로 구현하기
			- 이 방식이 테스트 코드 편하게 짜기 엄청 수월하고 보기도 편하다.
			- 책의 필자도 이 방식으로 만들고 공통 로직은 도메인에 헬퍼 클래스를 만들어 빼는걸 추천한다.
	- 주의사항
		- 인터페이스는 그닥 필요하지 않는다.
			- 테스트 더블을 만들어 대응하기 위해서 좋게 쓰이겠지만 mokito가 있는데 굳이 사용할 이유가 없다.
			- 관련 버전을 여러개 만들어 런타임에 대응하기 위함이라면 요즘은 아무도 이런식으로 블루 그린 배포를 하진 않는다.
		- 주로 글을 작성하면, 상세 결과를 알기 위해서 결과 데이터로 식별자를 리턴하는 것이 좋다.
			- 댓글의 경우에는 이후 댓글 목록을 리턴하거나 하는 것인데, 이건 댓글 목록 재 요청을 하는게 낫다.
			- 글 작성의 경우에는 식별자를 리턴해주는 것이 좋다.
		- 애그리거트 전체를 리턴하는 것은 금물이다.
			- OSIV 관점에서보면 커넥션 드레인을 만들어내는 원인이다.
			- 객체지향 관점에서 보면 로직을 분산시키는 원인이며 애그리거트에 대한 변경권 자체를 여기 저기에 흩뿌리는 위험한 행동이다.
		- 표현 영역에 해당하는 객체를 응용서비스에 파라미터로 던지면 안된다.

&emsp;&emsp;

## 표현 영역

- 설명
	- 특징
		- 사용자가 시스템을 사용할 수 있는 흐름을 제공하고 제어
		- 사용자의 요청을 알맞는 응용 서비스에 전달하고 결과를 사용자에게 제공
		- 사용자의 세션을 관리
	- 주의사항
		- 핵심은 세션의 관리도 여기서 처리해야하지만 더 고려해야할 것이 많다.
			- 인증, 권한처리도 서비스로 나눠서 처리하는 것도 방법이다. 요즘은 인증, 세션용 서버도 따로 두는게 정석이다.
			- 멀티 클러스터를 염두로 개발하는 게 일상이기 때문에 세션까지 여기서 관리할 생각은 접는게 아키텍처 설계상 건강하다.

&emsp;&emsp;

## 값 검증

- 설명
	- 특징
		- 원칙적으로 응용서비스에서 모든 값에 대한 검증을 하는 것이 좋다.
	- 주의사항
		- 스프링 MVC의 경우 BindingResult와 같은 사용하려면 표현 영역에서 처리해야한다.
			- 내 경우 @Valid를 컨트롤러, 서비스에 둘다 넣어준다.
				- 컨트롤러와 서비스 둘다 올라오는 예외가 다르게 올라온다.
				- 예외 핸들러에서 각자 처리해주면 된다.
			- 주로 컨트롤러에서는 값의 형식의 예외를, 응용서비스에서는 데이터의 존재의 유무, 특정 도메인 로직에 의한 논리적 오류를 검증하면 된다.

## 권한 검사

- 설명
	- 특징
		- 스프링에서는 구현 방법이 여러가지이다.
			- Spring Security 사용하기
				- 스프링 필터 기반으로 동작하며 결국 커스텀 필터를 끼워 넣어서 처리하는 것은 매한가지이긴 하지만 여러 CORS, CSRF등의 정책 기반 필터가 미리 만들어져 있어 이를 활용하기 좋아 많이들 사용하는 편이다.
			- `MethodArgumentsHander`에서 검증하기
				- 이는 인터셉터의 기능을 활용하는 것과 비슷하지만 알아두면 여러모로 써먹을 구석이 많다.
			- 커스텀 필터를 만들어서 검증하기
			- 인터셉터를 활용해서 검증하기
				- 인터셉터를 활용하는 경우 MVC에 대한 컨텍스트를 벗어나는 것은 아니기 때문에 보안상 사용하는 것은 권장하지 않는다.
	- 주의사항
		- 커스텀 필터나 스프링 시큐리티의 경우, Springboot MVC 컨텍스트를 벗어나는 영역이다.
			- 즉, 여기에서 등록된 빈을 활용하는 것은 괜찮지만 MVC 관련 기능을 활용할 수는 없다.
			- 특히! 트랜잭션과 같은 기능은 사용불가능하다! 필터 테스트하는 경우 당황해서는 안된다.
		- 책에서는 여러 방식을 예시에 넣어뒀는데 나의 경우에는 시큐리니나 커스텀 필터를 쓰는 것을 권장한다.
			- 이곳이 여러 공격을 미연에 방지하기가 가장 수월한 편이다.

## 조회 전용 기능과 응용 서비스

- 설명
	- 책에서 필자는 단순 조회의 경우 조회 서비스를 따로 만들지 않고 그냥 DAO를 만들어 붙여버린다.
		- 따지고 보면 리포지터리를 바로 가져다 붙이는 것인데 더 대단한 기능이 딱히 없으면 이렇게 구성해도 상관은 없다.
		- 심지어 트랜잭션도 필요없으니 이렇게 구성하는 것이다.
