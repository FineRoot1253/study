## JPA를 이용한 리포지터리 구현


### 모듈 위치

- 설명
	- DIP에 따라 리포지터리의 구현 클래스는 인프라에 위치시킨다.
		- 주로 QueryDSL을 구현해야하는 경우 이런식으로 사용하는 경우가 많다.

&emsp;&emsp;
### 리포지터리 기본 기능 구현

- 설명
	- 요구사항
		- ID로 애그리거트 조회
		- 애그리거트 저장
	- 구현 방법
		- 반환 타입은 Optional로 해도되고 안해도 된다.
			- NPE를 회피하기위한 수단으로 사용하는 것이 보편적인데 사용해도 무방하다고 본다.
		- 책에서는 변경 이후 굳이 persist나 flush를 호출하진 않는데 flush를 호출해 바로 저장시켜주는 것이 일반적이긴하다.
			- 어차피 호출 이후 더티해진 상태의 모델을 알아서 flush를 하긴하게된다. 우린 이걸 더티체킹이라고 부른다.
			- 더티체킹은 일반적으로 트랜잭션이 종료가 되어야 적용이 되는데, 만약 상태를 변경하고 적flush()를 하지 않고, 다른 애그리거트에서 변경된 데이터를 다시 조회하여 사용하려는 경우, 이런경우 제대로 적용이 안된 상태를 가져온다는 것이다.
				- 따라서 보편적으로 변경시 곧바로 flush()를 호출하는 것이 일반적이다.
				- 물론 이걸 없애야 코드는 깔끔해지긴하다.
		- ID가 아닌 조건으로 조회시 By뒤에 조건을 넣어주자
			- DataJpa에 의존적으로 만들면 자동으로 구현을 다해준다.
			- 위빙으로 다 구현해준다.
			- 만약 페치 사이즈를 따로 조정해줘야하는 복수 개의 값이 기대된다면, 그땐 QueryDSL로 따로 구현하는 편이 좋다.
		- 삭제 기능 구현시 delete로 만들어주자
			- Data JPA에도 구현되어있다.
				- 물론 물리적인 삭제는 대부분 팀 표준에서는 금지되어있다.
					- 그러지 않은 회사도 있지만 이런 경우에는 힘내길 바란다...
				- 책에서도 직접적으로 삭제하지 않는 경우가 대다수라고 지적한다.
					- 그래서 대강 알려주고 넘어간다.

## 매핑 구현

- 설명
	- 기본 규칙
		- 애그리거트 루트는 `@Entity`로 매핑한다.
		- 벨류는 `@Embeddable`로 매핑한다.
		- 벨류 타입 프로퍼티는 `@Embedded`로 매핑 설정한다.
		- 식별자는 `@Embedded`와 함께 `@AttributeOverrides(@AttributeOverride(name = "매핑 엔티티의 Id 객체 프로퍼티명", column = @Column(name="fk 컬럼명")))` 이렇게 둬야한다.
		- 만약 여러가지면 `@AttributeOverrides({@AttributeOverride(name = "매핑 엔티티의 객체 프로퍼티명1", column = @Column(name="실 테이블 컬럼명1")),@AttributeOverride(name = "매핑 엔티티의 객체 프로퍼티명2", column = @Column(name="실 테이블 컬럼명2")), ...})`
			- 이렇게 넣어주면 된다.
			- Id가 아니여도 괜찮다. 해당 값이 실제 DB에서는 어떻게 되어있냐를 제대로 적어두면 된다.
			- 책의 예제에서는 복수 벨류 값을 이렇게 표현하고 있다.

### 기본 생성자

- 설명
	- 기본 생성자는 보통 필요 없지만, JPA에는 기본생성자를 통해 객체를 생성하고 setter를 위빙으로 집어넣어 이 setter를 이용해 데이터를 초기화하는 구조를 가지고 있다.
		- ==**따라서 귀찮아도 반드시 `protected`나 `private` 생성자를 넣어주어야한다.**==
		- `@NoArgsConstructor(access = AccessLevel.PROTECTED)`롬복을 쓴다면 이거 한줄 클래스에 넣어주면 된다.
		- `@Entity`와 `@Embeddable`로 선언된 엔티티, 벨류 클래스에는 전부 넣어주자

&emsp;&emsp;

### 필드 접근 방식 사용

- 설명
	- @Access(AccessType.FIELD)
		- 하이버네이트가 필드 접근을 필드를 통해 접근한다.
	- @Access(AccessType.PROPERTY)
		- 하이버네이트가 필드 접근을 getter/setter를 통해 접근한다.
		- 대부분의 경우 DTO가 아니고서야 setter를 구현하지 않기 때문에 이는 사용되진 않는다.
	- 명시적으로 표시 하지 않는 경우
		- @ID나 @EmbeddedId의 위치를 보고 판단한다.
			- 필드인 경우 필드 접근, 메서드인 경우 프로퍼티 접근을 한다.

### AttributeConverter를 이용한 벨류 매핑 처리

``` java
public interface AttributeConverter<X,Y> {  
	public Y convertToDatabaseColumn (X attribute);  
	public X convertToEntityAttribute (Y dbData);  
}
```

``` java
@Converter(autoApply = true)
public class MoneyConverter implements AttributeConverter<Money, Integer> {  
  
    @Override  
    public Integer convertToDatabaseColumn(Money money) {  
        return money == null ? null : money.getValue();  
    }  
  
    @Override  
    public Money convertToEntityAttribute(Integer value) {  
        return value == null ? null : new Money(value);  
    }  
}
```

- 설명
	- 만약 Length 벨류에 값과 단위를 적어야한다고 가정한다.
		- 그리고 DB에는 width라는 `varchar(20)` 컬럼에서 이 값을 사용한다고 가정해보자.
		- 이 변환을 처리하기 위한 방법은 AttributeConverter가 진행한다.
			- 위의 인터페이스를 상속받아 컨버터를 구현하면 된다.
		- 책에서는 MoneyConverter를 구현하고 있다.
	- 클래스 레벨에 @Converter를 지정해주지않으면 직접 해당하는 필드마다 컨버터를 직접 지정해줘야한다.

### 벨류 컬렉션: 별도 테이블 매핑

``` java
@ElementCollection(fetch = FetchType.LAZY)
@CollectionTable(name = "order_line", joinColumns = @JoinColumn(name = "order_number"))  
@OrderColumn(name = "line_idx")
```

- 설명
	- `@ElementCollection(fetch = FetchType.LAZY)`는 컬렉션 테이블 매칭에 필요한 애너테이션이다.
		- Lazy 설정은 여기서 할 수 있다.
	- `@OrderColumn(name = "line_idx")`이 설정은 순서가 상관이 없는 컬렉션이라면 굳이 넣어주지 않아도 된다.
		- 순서가 중요한 리스트 데이터라면 컬렉션용 테이블마다 line_idx와 같은 인덱스 필드를 반드시 넣어줘야한다.
	- 이 구현이 생각보다 중요한 이유는 1:N 관계로 벨류 관계를 가지고 있는 경우에는 이렇게 만드는 것이 DDD 관점에서 중요하게 생각하기 때문이다.
		- 보통 이런 경우에는 엔티티를 따로만들고 N:1관계로 OrderLine에 OrderNO를 달아두는 편이 일반적이지만 책에서는 이렇게 하지 않았다.
	- 불편해도, 성능에 문제가 생길것 같아도, 이렇게 구현한 이유는 이 테이블의 생체 주기가 Order 애그리먼트에 의존적이기 때문이다.
		- Order가 생성될때 같이 생성되고 같이 삭제되는 OrderLine은 벨류로써의 가치가 있지 독단적인 생체주기를 지니진 않았기 때문에 이렇게 사용되는 것이다.
		- Littok 기준 자원-권한 테이블은 예를들면, 독단적인 생체주기를 가지고 있다. 이런 경우에는 벨류 컬렉션으로 자원이나 권한쪽 애그리먼트에 넣어줄게 아니라, 그냥 독단적인 관계테이블을 하나 엔티티로 만들어두는게 낫다.
		- ==**이 벨류냐 엔티티로 만드느냐, 벨류중에 다른 테이블에 있는 벨류라 복수개라서 이걸 엔티티로 봐야하냐 벨류 컬렉션으로 봐야하느냐 이런 기준은 생체주기에 달려있다.**==

### 별도 테이블에 저장하는 벨류 매핑

- 설명
	- 기본 매핑 규칙
		- 애그리거트에서 ==**루트 엔티티를 뺀 나머지 구성요소들은 대부분 벨류**==이다.
			- 루트 엔티티 외 ==**또 다른 엔티티가 존재한다면 진짜 엔티티는 맞는지 존재를 의심**==해야한다.
			- 별도에 **테이블에 데이터를 저장한다고 해서 엔티티인 것은 아니다.**
				- OrderLine이 대표적인 사례이다.
		- 벨류가 아니라 ==**엔티티가 확실하다면(본인만의 생존 사이클을 지니는 데이터라면) 다른 애그리거트인지 확인해야한다.**==
			- 특히 책에서는 독자적인 라이프사이클을 지닌다면 다른 애그리거트일 확률이 높다고 이야기한다.
			- 댓글과 게시글의 관계가 가장 헷갈리는 관계인데, 댓글과 게시글은 작성 주체가 다르고, 생성 시점도 다르다. 따라서 다른 애그리거트로 보는 것이 마땅하다.
		- 엔티티의 식별자와 DB 테이블의 식별자와 동등하게 바라봐선 안된다.
			- User의 경우, User가 자주 들고와야하는 정보(예를 들면 타인의 리스트나, 자주 프론트에서 보여줘야하는 개인 프로필 기초BIO, 프로필 사진, 닉네임, 이메일등이 대표적이다.)
				- 이런식으로 1:1 테이블을 분할 해놓는 경우, 이들은 벨류지 엔티티가 아니다.
				- 이런경우 `@SecondaryTable(name="user_profile", pkJoinColumns= @PrimaryKeyJoinColumn(name="id"))`를 사용하고 `UserProfile`에는 `@Embedded` 어노테이션을 붙여준다. 이러면 user를 조회만하면 알아서 해당 테이블에서 데이터를 조인해서 User 데이터 전체를 조회한다.
				- 만약 유저 목록용 데이터 조회를 위해 조인이 필요하지 않아 Lazy로 두고싶다면, ==**Lazy 설정을 넣지말고, 조회용 엔티티를 따로 만들어서 DAO를 따로 만드는 것이 낫다.**==

### 벨류 컬렉션을 @Entity로 매핑하기

- 설명
	- 개념적으로 벨류이지만, 구현 기술의 한계나 팀 표준을 위해 엔티티로 만들어야하는 경우도 많다.
		- 대다수의 경우 Entity로 만드는 것이 대부분이다.
		- 컬렉션을 따로 만들어서 매핑하는 것은 성능에 관련 문제로 상당히 머리가 아파진다.
	- 책에서는 상속 매핑을 Entity로 구현해야하는 경우를 보여준다.
		- `@Embeddable`이 아닌 `@Entity`를 사용해야 상속 매핑이 가능하다.
		- 추상 클래스에 `@Inheritance(stategy = InheritanceType.SINGLE_TABLE)` , `@Discriminator(name="구분자 타입 컬럼 이름")` 이것을 엔티티 애너테이션과 같이 넣어준다.
		- 상속을 받는 클래스에는 구분자 매핑을 해준다.
			- 엔티티 에너테이션과 함께`@DiscriminatorValue("구분자 이름")`를 넣어준다.
		- 컬렉션으로 구현하는 측에서는 `@OneToMany(cascade = {CascadeType.PERSIST, CascadeType.REMOVE}, orphanRemoval = true)`를 넣어준다.
			- 라이프 사이클을 같이 묶어주는 것이다.
			- 상속 관련은 대부분 현대 객체지향에서는 지양하는 구현이라 난 이걸 굳이 알필요는 없는 구현이라고 생각한다.
			- 다만 엔티티를 컬렉션으로써 사용한다면 이렇게 사용해야한다. 그래야 라이프사이클이 이곳에 묶이게 되며 굳이 이렇게 구현해야하나 싶어도 이렇게 안하면 DDD 관점에서 벗어난 구현을 만들게 된다.
	- ==**하이버네이트는 @Embeddable 타입에 대한 컬렉션의 clear() 메서드를 호출하면 실제로 삭제를 시켜버린다.**==
		- 현업 개발에서 Hard Delete는 금물이다. 코드는 깔끔하게 만들수는 있지만 굳이 이런 구현은 왜 만들었는지는 모르겠다. 오히려 사고가 날 확률이 높은 구현이라고 생각한다.

### 매핑 구현에 대한 정리

- 설명
	- ==**벨류 컬렉션은 대부분 실무에서는 금지하고 있는 기술**==이다.
		- CascadeType.REMOVE, orphanRemoval = true 이 설정이 기본으로 동작하게 된다.
			- 놀랍게도 컬렉션에 변화가 생기거나 컬렉션이 삭제가되면 같이 삭제가 물리적으로 되버리는데 이는 매우 위험하다.
			- 물론 한 트랜잭션에서 직접 같이 삭제를 시키는 것(Soft Delete의 의미로써)이 맞지만, 하드 딜리트를 해버리는 문제가 발생한다.
		- ==**`@Entity`와 1:N 매핑을 사용하자.**==
			- 물론 이것도 성능관련으로 문제가 발생할 가능성이 있어서 이 또한 금지하는 팀도 존재한다.
			- ==**개인적으로는 모든 테이블을 엔티티로 만들기로 작정했다면 그냥 N:1 매핑으로 해결하는 것을 권장**==한다.
			- 물론 의미론상으로 **객에 대한 리포지터리에서 무언가를 조회해서 주에 가져와야한다던지하는 상황**들이 벌어지겠지만 (**주객전도**된 상황) 괜찮다 **이게 더 안전한 구현 방식**이다.
			- 이것을 해결하고 싶다면 더 나은 프레임워크를 직접 만드는 수 밖에 없지만 이것도 상식밖의 행동이다. 결국 의미론적으로 주가 객에 대한 정보를 알아서 들고 있어야하는 것이 맞지만 **(애그리거트를 조회(생성)하면 온전한 데이터를 다 들고 있어야 DDD 관점에 맞는 설계이다.)**, 그렇게 만들면 성능에 문제가 발생하는 것이 당연하기 때문
			- 결국은 트레이드 오프이기때문에 잘 생각해서 표준을 만들고 그에 맞게 구현해야한다.

&emsp;&emsp;

## 애그리거트 로딩 전략

- 설명
	- 애그리거트의 모든 데이터는 생성 즉시 온전하게 로딩이 되어야한다.
		- 원한다면 즉시 로딩 상태로 만들어두면 된다.
			- 다만 이런경우 대부분 원치않게 N + 1 에러가 발생한다.
			- 보통은 원하는 데이터가 존재하면, 이 경우에만 join fetch를 따로 써주는것이 사고 예방책으로 더 낫다.
	- 현실적으로 애그리거트 루트를 생성한다고 해서 루트 엔티티에 대한 모든 데이터를 로딩할수는 없다.
		- 애그리거트가 처음 로딩시 완전해야하는 이유
			- 상태를 변경하는 기능 실행시 애그리거트의 상태는 전부 불러와야 온전하게 관련 하위 애그리거트까지 일관적으로 데이터를 변경할 수 있다.
			- 표현 영역에서 애그리거트 상태 정보를 보여줄 때 필요하다.
				- 이건 그냥 관련 엔티티를 따로 만들어두는게 낫다.
		- 결론
			- 지연로딩을 사용해도 성능에 큰 문제를 주는 경우는 별로 없다.
				- 관련 컬렉션을 수정해도 실제 컬렉션에 접근할때 별도로 따로 쿼리를 날려서 엔티티의 데이터를 채우기 때문에 괜찮다.
			- ==**일단 지연로딩을 선택하고, 성능에 문제가 발생하는 조회에는 join fetch를 붙여줄 준비를 하자.**==

## 애그리거트 영속성 전파

- 설명
	- 애그리거트를 저장할 때에도 하나로 처리해야한다.
		- 저장 메서드는 애그리거트에 속한 모든 객체를 저장해야한다.
		- 삭제 메서드는 애그리거트에 속한 모든 객체를 삭제해야한다.(물론 이것은 소프트딜리트로 대체되어야하는 경우가 대다수이기 때문에 대부분 Embeddable은 사용할 수 없다.)
	- 책에서는 Cascade 설정을 추가해 저장과 삭제를 자동으로 되게끔 1:N 매핑을 처리하는데, 현업방식과 맞지 않는다.
		- 원치않게 변경이 되는 것을 방지할 방법도 없고, 원치않게 삭제까지 되는 것을 방지할 방법또한 없다.
		- 현업에서는 모든 데이터는 전부 영속화하는 것이 보안, 운영상 항상 맞는쪽에 속한다.
	- **애그리거트 루트 내에서 1:N 연관관계를 사용하는 경우:**
	    - **장점:** 애그리거트 루트가 컬렉션의 생명주기를 일관되게 관리할 수 있으므로, 애그리거트 루트에서 변경을 관리할 수 있음
	    - **단점:** 컬렉션에서 요소가 제거되면 자동으로 삭제가 진행되어, 의도치 않은 삭제가 발생할 위험이 있습니다. 
		    - 이 단점이 너무 커서 1:N orphanRemoval 자체를 꺼두거나 손보지 않는다.
	- **N:1 매핑으로 독립 엔티티로 관리하는 경우:**
	    - **장점:** 각 자식 엔티티가 독립적인 생명주기를 가지므로, 삭제 로직(예: 소프트 딜리트)을 명시적으로 제어할 수 있습니다.
	    - **단점:** 삭제 기능을 애그리거트 루트가 아니라 서비스 레이어나 별도의 도메인 로직에서 구현해야 하므로, 도메인 모델 내에서의 응집성이 떨어질 수 있습니다.
		    - 보통 도메인 모델 내의 응집성이 떨어지더라도, 변경과 삭제를 한 도메인 모델 내에서 처리하지 않고 서비스 레벨에서 처리하게 만든다.
			- 따라서 변경, 삭제에 대한 메서드를 모든 엔티티에 구현해두고 서비스 레벨에서 이를 처리한다.
	- 정리
		- 영속성 전파의 영역은 언제나 같은 애그리거트에 있는 다른 벨류, 또는 하위 애그리거트를 의미하며 이를 위해서 서비스영역에서 충분히 한 트랜잭션에서 영속성 전파 관리가 가능하다.
			- 지금 영속성 설명은 한 애그리거트에서 타 애그리거트의 데이터를 변환하는 것에 대한 설명이 아니다.
		- 한 애그리거트 루트에서 다른 애그리거트 루트의 상태를 변환해야한다면 이벤트를 이용해야한다.
			- 이것은 사가패턴을 추가로 학습해야한다. 롤백 구현을 직접해야한다.
