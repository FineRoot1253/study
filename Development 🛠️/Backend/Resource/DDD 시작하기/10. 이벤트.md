## 시스템 간 강결합 문제

- 설명
	- 책의 예시
		- 주문 애그리거트의 환불 기능은 환불 서비스를 파라미터로 받는다.
			- 이 환불 기능은 외부 결제 시스템이 제공하는 환불 서비스를 호출한다.
			- 문제 1
				- 외부 서비스가 정상이 아닌 경우 트랜잭션 처리를 어떻게 해야하는가?
				- 환불 실패시 트랜잭션을 롤백 하거나 주문을 취소 상태로 만들고 나중에 다시 시도 가능하게 만드는 방식으로 만들 수 있다.
			- 문제 2
				- 외부 시스템의 응답이 길어지면 대기시간이 길어지는데 외부 시스템의 의존성을 어떻게 줄이는가?
		- 애그리거트 기능 메서드의 파라미터로 서비스 전달시 생기는 문제
			- 문제 1 
				- 도매인 객체에 로직이 뒤섞인다.
			- 문제 2
				- 기능 추가시 또 다른 로직이 다시 뒤섞이게 된다.
	- 문제의 원인
		- 바운디드 컨텍스트 간의 강결합이 원인이다.
			- 예시에서는 주문과 결제가 강하게 결합되어 주문 바운디드 컨텍스트가 결제 바운디드 컨텍스트에 영향을 받는다.
		- 책에서는 이벤트를 권장한다.

## 이벤트 개요

- 설명
	- 구성요소
		- 이벤트 생성 주체
		- 이벤트
		- 이벤트 디스패쳐(이벤트 퍼블리셔)
		- 이벤트 핸들러(이벤트 구독자)
	- 이벤트 구성
		- 이벤트 종류
		- 이벤트 발생시간
		- 추가 데이터
	- 예시
		- 주문 애그리거트의 배송지 변경 기능에 이벤트를 발생시킨다.
		- 배송지 변경 핸들러에서는 이벤트를 구독하고 있다가 추가로 필요한 데이터는 리포지터리에서 조회하여 처리한다.
	- 이벤트의 목적
		- 트리거
			- 주로 도메인 상태 변경의 후킹 작업을 추가하기 위해 사용된다.
		- 데이터 동기화
			- 배송지 변경시 외부 배송 서비스에 바뀐 배송지 정보를 전송해야한다면 주문 도메인은 이벤트를 발생시키고 이벤트 핸들러가 외부 배송 서비스와 배송지 정보를 동기화하는 것이다.
	- 이벤트의 장점
		- 서로 다른 로직이 뒤섞이는 것을 방지한다.
			- 예시에서는 구매 취소 로직이 발생하는 것을 그저 이벤트를 발행하여 외부에 핸들러에서 처리하게 만들었다.
			- 심지어 추가 로직들은 핸들러에 추가하면 된다.

## 비동기 이벤트 처리

- 설명
	- 예시
		- 이메일 전송
	- 구현 방법
		- 로컬 핸들러를 비동기로 실행
		- 메시지 큐 사용하기
		- 이벤트 저장소와 이벤트 포워더 사용하기
		- 이벤트 저장소와 이벤트 제공 API 사용하기
	- 로컬 핸들러 비동기실행
		- `@EnableAsync`를 설정 클래스에 붙여준다.
		- 비동기로 실행시킬 클래스에 `@Async`를 붙여준다.
	- 메시징 시스템을 이용한 비동기 구현
		- 주의 사항
			- 앱의 트랜잭션의 범위는 이벤트 메시지 전송으로 제한되고 나머지는 메시징 시스템이 담당한다.
			- 카프카는 글로벌 트랜잭션을 지원하지 않고 래빗MQ는 글로벌 트랜잭션을 지원한다.
	- 이벤트 저장소를 이용한 비동기 처리
		- 포워더 방식
			- 로컬 핸들러가 DB에 이벤트를 저장하고 포워더가 주기적으로 이벤트를 DB에서 읽어 이벤트 핸들러에 전달한다.
			- 로컬 핸들러가 DB에 이벤트를 저장하고 외부 핸들러가 직접 API를 통해 추적한다.
		- 구현 예시
			- API 방식
			- 포워더 방식
				- `@Scheduled`어노테이션으로 등록을 해두고 주기적으로 저장소에서 읽게 만든다.
	- 주의 사항
		- 자동 증가 컬럼의 경우 insert를 해도 커밋을 해야 반영이 되므로 insert를 했더라도 다른 포워더나 핸들러가 조회를 해도 조회가 되지 않는다. 
			- DB에 반영되는 시점이 달라진다는 것이다.
			- 만약 트랜잭션 1과 2 둘다 insert 쿼리를 차례 날린 상태에서 2가 먼저 커밋을 한다고 가정하면, 트랜잭션 1의 ID를 2가 사용해 상태를 저장하게된다.
				- ==**즉, ID와 상태가 매칭되지 않는 상태에서 저장이 된다.**==
			- ==**ID를 기준으로 지연조회 하는 방식을 사용해야한다.**==

## 이벤트 적용시 추가 고려 사항

- 설명
	- 이벤트 객체에 이벤트 발생 주체 추가 여부
		- 어떤 애그리거트가 발생시킨 이벤트만 조회를 한다고 가정하면 이벤트 발생 주체가 필요하다.
	- 포워더에서 전송 실패를 얼마나 허용할 것인지 여부
		- 포워더의 전송 실패시 실패한 이벤트부터 읽어와 전송을 시도도 하는데 계속 실패할 경우 어떻게 해야할지 지정해야한다.
		- 주로 재시도 횟수를 제한하고 그 이후는 실패처리하고 넘어간다는 식의 도메인 정책 합의가 필요하다.
			- 실패 처리도 DB에 남기는 식으로 만들어야 추후 문제 해결이나 롤백 처리에 유용하니 필수적이다.
	- 이벤트 손실에 대한 것
		- 이벤트를 비동기로 처리하면 이벤트 처리 실패시 이벤트를 유실하게 된다.
	- 이벤트 순서에 대한 것
		- 이벤트 발생 순서대로 외부 시스템에 전달해야할 경우 이젠트 저장소를 사용하는 것이 좋다.
			- 이벤트 발생 순서대로 DB에 저장하고 그 순서대로 이벤트 목록을 제공하기 때문이다.
			- 메시징 시스템은 벤더사의 구현 기술에 따라 순서가 다를 수 있다.
	- 메시지 재처리에 대한 것
		- 동일한 이벤트를 재처리를 해야하는 경우를 정해야한다.
			- 마지막으로 처리한 이벤트의 순번을 기억해뒀다가 이미 처리한 순번의 이벤트 도착시 해당 이벤트를 처리하지 않고 무시하는 것이 가장 쉬운 구현 방법이다.
		- 예를 들면, 회원 가입 이메일 발송의 경우 재전송을 여러번 하여도 멱등하게 처리되어야한다.

### 이벤트 처리와 DB 트랜잭션 고려

- 설명
	- 만약 이미 이벤트는 전송했는데 트랜잭션이 실패해서 다같이 롤백이 되어야한다면 어떻게 해야할까
		- `@TransctionalEventListener`를 사용한다.
			- phase 속성값으로 트랜잭션 커밋이 실행되는 경우(성공인 경우)에만 동작하게끔 만드는 것이다.

&emsp;&emsp;

### 주의 사항

- 이 책에서는 비동기 로직이 실패하여 다같이 롤백해야하는 경우는 고려하지 않았다.
- 보통 비동기 로직이 실패하여 다같이 롤백해야하는 경우에는 트랜잭션을 세밀하게 나누어 설계하고 제어해야한다.
	- 만약 주문-결제의 경우, 주문을 성공하고 이벤트를 전송한 후 결제 서비스에서 이를 받는다고 가정하면, 결제 서비스에서 실패를 하는 경우, 실패 이벤트를 날리게 된다.
	- 그리고 주문쪽에도 실패한 경우의 주문 실패 기능을 만들어 주문 시도를 롤백시키게 만든다.
	- 주문 생성 후 결재 시도와 주문 실패 처리 기능을 합쳐서 하나의 트랜잭션으로 볼 수있는데 이런 트랜잭션을 보상 트랜잭션이라고 부른다.
	- 주문 - 결제에서 결제 시도의 경우 피봇 트랜잭션이라고 부른다.
	- 결제 이후 시도하는 여러 조회(예를 들면 주문번호)들을 재시도 가능 트랜잭션이라고 부른다.
	- 이런 패턴을 우리는 사가 패턴이라고 부르며, 주로 MSA에서 자주 쓰이는 패턴이다.
