## 네 개의 영역

- 설명
	- 표현
		- 컨트롤러
	- 응용
		- 서비스
			- 책에서는 사용자에게 제공해야할 기능을 큰 개념으로 클래스 별로 만들어서 제공한다.
			- 이렇게 할 필요는 없긴하지만 이러면 테스트를 구현할때는 주입할 의존객체가 적어 매우 편해진다.
				- 책에서는 주문 취소 서비스를 예시로 드는데 처음 예시에는 취소에 대한 책임을 주문에게 주고 있다.
				- 나중에는 취소에 대한 정책을 따로 클래스로 만들어 이를 서비스에서 주입받고 이에 따라 주문을 취소할 지 말지를 정하고 있다.
	- 도메인
		- 도메인 영역을 구현하는 곳이다.
		- 주문의 경우 배송지 변경, 결제 완료, 주문 총액 계산등의 핵심 로직을 구현하고 있다.
		- littok의 account의 경우 비밀번호 검증, 리뷰의 경우 Rating 총합 계산 등이 있다.
			- 아직 기능이 적긴하다.
	- 인프라
		- DB 연동, 메시징 큐에 메시지 전송과 수신, Redis와 데이터 연동, SMTP 메일 발송, HTTP 클라이언트를 이용한 API 호출 등
		- 개념 보단 실제 구현을 다루는 곳이다.
		- 인프라를 제외한 영역은 인프라에서 제공하는 기능을 사용해 필요한 기능을 개발한다.

## 계층 구조 아키텍쳐 (레이어드 아키텍쳐)

- 설명
	- 상위에서 하위까지 계층은 상위의 계층을 의존하지 않는 것이 특징이다.
	- 응용 계층에 서비스를 엮고, 엔티티는 도메인에 두고, 서비스는 엔티티와 DB 모듈을 의존한다.
		- 책에서는 Drools라는 룰 엔진을 인프라로 받고 이를 서비스에서 활용한다.
			- 다만 이 예제에서는 Drools에 종속된 서비스 코드를 보여줘 이렇게 개발하면 안된다는 것을 보여준다.
			- 대게 큰 회사에서의 큰 프로젝트에는 이런 룰엔진을 도메인 전문가들이 동적으로 사용하여 앱을 관리하려고 하기 때문에 이런식으로 개발하는 회사도 많다.
				- 금융쪽이나 보험쪽 같은 규칙이 정말 많은 곳에서 사용된다.

## DIP

- 설명
	- 책에서는 위의 Drools 엔진 예제를 든다.
		- 기존에는 Drools 인프라 클래스를 주입받아 사용했는데 새로운 코드에서는 인터페이스만을 만들고 그 인터페이스를 사용한다.
		- 이러면 변경도 쉽고 테스트도 쉬워진다.
	- 다만 어차피 스프링은 cglib 기반 프록시로 빈을 띄우기 때문에, 이러한 Drools 엔진 클래스를 컴포넌트로 등록시키면 쉽게 변경도 가능하고 테스트도 쉽다.
		- DIP를 추상화하여 사용하기 쉽게 고도화하다보니 나온 기술이 동적 프록시이다.
		- mockito를 이용하면 쉽게 테스트도 가능하고 그렇다.
	- 주의 사항
		- 예시에서는 RuleEngine 인터페이스를 인프라에 두는데 이는 고수준 계층인 도메인에 두는 것이 맞다.
			- 책에서는 이를 통해서 고수준에서 저수준 모듈에 의존하지 않도록 만들었다고 하는데, 사실 이렇다 해도 대신 인프라가 고수준을 의존하는 구조이긴하다.
			- 온전히 서로에게 의존하지 않게 만들려면 중간 계층을 하나 더 추가해서 그곳에서 인프라와 도메인을 섞어서 사용해야한다.
			- 가장 대표적인 방법이 펙토리이다.
		- 인터페이스는 사용하면 할수록 결합도는 낮아지고 코드베이스의 크기는 급증한다.
			- 꼭 필요한 곳에서 사용해야하는 것이 마땅한데 이건 개발을 많이 진행해봐야 깨닫는 부분이다.
			- 항상 적용 범위를 고려해보고 적용해야하는 기술인 것이다.
	- DIP와 아키텍쳐
		- 인프라는 모든 계층에 걸쳐 의존하는 계층이다.

## 도메인 영역의 주요 구성 요소

- 설명
	- 엔티티
		- 고유 식별자를 갖는 객체
		- 자신의 생체주기를 가짐
		- 주문, 회원, 상품과 같은 도메인의 고유한 개념을 가진다.
		- 도메인 모델의 데이터를 포함한다.
			- 해당 데이터와 관련된 기능도 같이 제공해야한다.
	- 벨류
		- 고유한 식별자를 가지지 않는 객체
		- 개념적으로 값을 표현하는데 사용된다.
		- 주소나 금액이 대표적이다.
			- 엔티티 속성으로 사용할 뿐 아니라 벨류 타입의 속성으로도 사용 가능하다.
	- 애그리거트
		- 연관된 엔티티와 벨류 객체를 개념적으로 하나로 묶은 것이다.
		- Order 엔티티, OrderLine 벨류, Orderer 벨류 등이 주문 에그리거트로 묶을 수 있다.
	- 리포지터리
		- 도메인 모델의 영속화를 담당한다.
	- 도메인 서비스
		- 특정 엔티티에 속하지 않은 도메인 로직을 제공한다.
		- 할인 금액 계산은 상품, 쿠폰, 회원 등급, 구매 금액 등 다양한 조건을 이용해서 구현하게 되는데, 이렇게 도메인 로직이 여러 엔티티와 벨류를 필요로 하면 도메인 서비스에서 로직을 구현한다.
		- 주로 다른 도메인 영역의 모델과 함께 처리가 필요해지는 경우, 서비스가 필요해진다.
			- 대표적으로 할인 금액 계산과 마찬가지로, 주문 취소의 경우, 어떤 맴버의 요청인지 알아야하므로, 회원이 함께 사용되어야하기 때문에 서비스에서 이뤄진다.

&emsp;&emsp;

### 엔티티와 벨류

- 설명
	- 엔티티
		- 엔티티는 데이터만 담는 구조체가 아니라 항상 기능을 같이 제공해야 의미가 존재한다.
		- 단순히 DB 테이블와 다르게, 여러 정보들이 하나의 정보를 나타내는 경우에는 묶어서 벨류 타입으로 만들 수도 있다.
			- 책에서는 Orderer가 대표적이며 특히 회원 ID를 벨류로 만들어두고 이를 다시 벨류 객체에서 재사용하는 것이 상당히 인상적이다.

&emsp;&emsp;

### 에그리거트

- 설명
	- 정의
		- 관련 도메인 객체를 하나로 묶은 군집이다.
	- 특징
		- 도메인이 커질수록 개발할 도메인 모델이 커지며 더 많은 엔티티와 벨류가 출현한다.
			- 엔티티와 벨류 개수가 많아지면 모델은 점점 더 복잡해진다.
			- 도메인 모델이 복잡해지면 개발자가 전체 구조가 아닌 한 개의 엔티티와 밸류에만 집중하는 상황이 발생한다.
			- 상위 수준에서 모델을 관리하지 않고 개별 요소에만 초점을 맞추다 보면 큰 수준에서 모델을 이해하지 못해 큰 틀에서 모델을 관리할 수 없는 상황에 빠질 수 있다.
				- 즉, 모델끼리도 상위 모델, 하위 모델을 만들어야 전체 구조에서 관망하기 편한 것이다.
				- 보통 애그리거트 루트와 애그리거트로 부르긴한다.
		- 예시에서 주문이 대표적이며 주문, 배송지 정보, 주문자, 주문 목록, 총 결재 금액의 하위 모델로 구성되어 하나의 주문이라는 상위 개념으로 표현할 수 있다.
		- 루트 엔티티가 애그리거트 루트로 불린다.
			- 루트 엔티티는 애그리거트에 속해 있는 엔티티와 벨류 객체를 이용해 에그리거트가 제공해야할 기능을 구현하고 있어야 한다.
			- 애그리거트를 사용하는 코드는 애그리거트 루트가 제공하는 기능만을 실행하고 애그리거트 루트를 통해서 간접적으로 애그리거트 내의 다른 엔티티나 벨류 객체에 접근한다.
			- 이것은 애그리거트의 내부 구현을 숩겨 애그리거트 단위로 구현을 캡슐화할 수 있도록 돕는다.
		- 애그리거트 루트만을 통해서 외부 코드들이 애그리거트에 접근해야만 한다는 것이다.
	- 장점
		- 개별 객체가 아닌 관련 객체를 묶어서 객체 군집 단위로 모델을 바라볼 수 있게 된다.
			- 개별 객체간 관계가 아닌 애그리거트간의 관계로 도메인 모델을 이해하고 구현할 수 있게 되며 큰 틀에서 도메인 모델을 관리할 수 있다.
	- 예시
		- 주문 에그리거트
			- 구성
				- 주문 엔티티 (루트)
					- 주문 항목 벨류
					- 주문 요청자
					- 배송지 정보 엔티티
						- 주소 벨류
						- 착신자 벨류
			- 특징
				- 주문 애그리거트에서는 Order 엔티티를 거치지 않고 배송지를 변경할 방법을 제공하지 않는다.
				- 애그리거트 구현시 고려할 것이 많은데, 트랜잭션 범위가 달라지기도한다.
					- 보통 트랜잭션의 범위는 애그리거트 내부로 제한하는 경우가 대다수이다.

&emsp;&emsp;

### 리포지터리

- 설명
	- 정의
		- 물리적인 저장소로 도메인 객체를 영속화하기 위한 도메인 모델
	- 특징
		- 개념적으로 요구사항에서 도출되는 도메인 모델이 아닌 구현을 위한 도메인 모델이다.
		- 예시에서 주문 취소의 경우 리포지터리를 통해 도메인 객체인 주문 객체를 얻은 뒤 도메인 객체의 기능을 실행한다.
		- 리포지터리의 인터페이스 자체는 고수준 모듈에 속한다.
			- 이를 구현한 클래스는 저수준 모듈로 인프라영역에 속한다.
			- 설명은 이렇지만 실제로는 DAO같은 경우에는 이 패키지를 따로 만들어두기만 한다.
		- 응용 서비스는 DIP와 같은 방식으로 실제 구현에 접근한다.
		- 응용 서비스는 리포지터리와 밀접한 연관을 가진다.
				- 필요한 도메인 객체를 구하거나 저장할 때 리포지터리를 사용한다.
				- 응용 서비스는 트랜잭션을 관리하는데 트랜잭션 처리는 리포지터리 구현 기술의 영향을 받는다.
		- 리포지터리를 사용하는 주체는 응용 서비스이기 때문에 리포지터리는 응용서비스가 필요로 하는 메서드를 제공한다.
			- 애그리거트를 저장하는 메서드
			- 애그리거트 루트 식별자로 애그리거트를 조회하는 메서드

## 요청 처리 흐름

- 설명
	- 표현 영역
		- 사용자의 요청을 가공해 데이터를 응용 서비스 영역으로 넘기며 응용서비스에 기능 실행을 위임한다.
	- 응용 서비스
		- 도메인 모델을 이용해 기능을 구현한다.
		- 도메인의 상태를 변경하는 경우 트랜잭션을 잘 관리해야한다.

## 인프라스트럭쳐

- 설명
	- 정의
		- 표현 영역, 응용 영역, 도메인 영역을 지원하는 저수준 모듈을 모은 영역
	- 특징
		- 다른 영역에서 필요로 하는 프레임워크, 구현 기술, 보조 기능을 지원한다.
		- 이 응용, 도메인 두 영역에서 정의한 인터페이스를 사용하는 것이 더 유연하게 사용이 가능하지만 항상 이에 대한 의존을 제거할 필요는 없다.
			- 예를 들면 응용서비스는 기본적으로 트랜잭셔널 애너테이션을 이용하는 것이 간편하다.
				- XML 보단 편안한 설정이라는 뜻이다.
			- 물론 MSA를 사용하는 경우 상태를 전역 트랜잭션 이벤트 기반으로 관리해야하므로 이땐 카프카나 Rabbitmq와 같은 메시징 큐가 필요해진다.
			- DIP가 주는 장점 만큼 편리함이 존재하는 것이 분명하기 때문에 장점을 해치치 않는 선에서 응용, 도메인 영역에서 구현 기술에 대한 의존을 가져가는 것이 나쁘지는 않다.
				- MSA기반으로 구현을 해야하는 경우, 트랜잭션의 구현방식을 3가지로 구분하여 구현해야하므로 의존성은 줄어도 오히려 구현은 복잡해진다. (사가 패턴)
		- 표현영역은 인프라영역와 쌍을 이루는데 MVC를 이용해 웹 요청을 처리한다고 했으면 스프링이 제공하는 MVC 프레임워크에 맞게 표현 영역을 구현해야한다.
			- 프레임워크에 의존적으로 구현되는 영역이라는 것이다.

## 모듈 구성

- 설명
	- 책에서의 모듈 구성의 특징
		- 책에서는 애그리거트별 패키지 내부 구성을 ui, application, domain, infra 4가지로 두고 있으며 이런 구조를 지닌 각 애그리거트 별 패키지로 전체 프로젝트를 구성하고 있다.
		- 도메인이 점점 커지면 같은 구성으로 애그리거트를 분리한다.
		- 책 예시의 경우 카탈로그 애그리거트의 경우 하위 애그리거트로 상품 애그리거트와 카테고리 애그리커트가 카탈로그 애그리거트의 도메인영역이 구성되어있다.
		- 애그리거트, 모델, 리포지터리는 같은 패키지에 위치시킨다.
			- 개인적으로는 이들 또한 패키지로 구분 지어 두는 것을 좋아하는 편이다. 이러는 편이 파일 찾기 수월하다.
		- 서비스와 DTO는 응용 영역에 배치한다.
			- 서비스의 인터페이스는 도메인에 배치한다.
		- 책에서는 한 패키지에 가능하면 10~15개의 개수의 타입 개수를 유지하려고 하며 이를 넘어가면 분리한다고 한다.
			- 개인적으로는 기능적으로 책임이 커진 도메인이 점점 커진다면 이를 분리해서 빼내는 편이다.
	- 모듈 구성의 장점
		- MSA로 전환하는 것이 한층 수월해진다.
			- 물론 트랜잭션을 쪼개는 수고로움을 견뎌야하는 것은 여전하긴하다.
		- 기능을 추가하고 제어하는 것에 대한 시간적 소모가 점증적으로 천천히 증가한다.
			- 단순 레이어드 패턴은 기능을 추가함에 따라 증대되는 복잡성이 도메인패턴 기준 애그리거트가 5개 이상 추가되는 경우 제어하기 힘든 수준으로 올라가는데 도메인 모델은 확실히 감당 가능한 수준으로 천천히 증대된다.
