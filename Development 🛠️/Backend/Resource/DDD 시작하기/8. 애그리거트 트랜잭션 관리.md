## 애그리거트와 트랜잭션

- 설명
	- 책의 예시
		- 운영자가 배송상태를 변경하는 도중 고객 스레드가 배송지를 변경하면 어떻게 되는가?
	- 해결 방법
		- 운영자가 배송 상태 조회 후 상태 변경 도중 고객은 애그리거트를 수정하지 못하게 막는다.
		- 운영자가 배송지 정보를 조회한 후 고객이 정보 변경시 운영자가 애그리거트를 다시 조회하여 수정하도록 만든다.
		- 책에서는 선점 잠금 (비관적 락), 비선점 잠금(낙관적 락) 해결 방법 두가지를 보여준다.
		- 이 경우는 레거시들에는 비관적 락이 많고 요즘 신규 시스템에는 낙관적 락이 많다.
		- 비관적락은 디비 수준에서 행단위 락을 거는 것이고(select for update가 대표적) 낙관적 락은 tx 버전에 따라 우선순위를 매겨 변경하려는 tx의 버전이 최신이 아닌 경우 변경을 하지 못하게 막는 것이다.
			- 낙관적락이나 비관적 락 둘다 앱단에서 잘 지원은 되지만 주로 비관적락은 sql 스크립트 단위로 사용되는 편이다.

## 선점 잠금 (비관적 락)

- 설명
	- 정의
		- 애그리거트 조회시 행단위 락을 걸어 다른 관련 tx들이 접근하지 못하게 막는다.
	- 특징
		- 일반적으로 RDB에서 재공하는 행단위 잠금으로 구현한다.
			- select for update가 대표적이다.
		- LockMode.PESSIMISTIC_WRITE로 entityManager.find()메서드에 파라미터로 선점 잠금을 할 수있다.
		- Data JPA에서는 쿼리 메서드에 `@Lock(LockMode.PESSIMISTIC_WRITE)`을 붙여주면 된다.
	- 주의사항
		- 선점 잠금에는 데드락의 위험이 도사리고 있다.
			- 서로에 대한 애그리거트를 동시에 선점 잠금으로 구하면 해당 행이 잠기게 된다.
			- `@QueryHint(name="javax.persistance.lock.timeout", value = "2000")`
				- 이런식으로 타임아웃을 정하면 잠금을 구하지 못하면 익셉션을 발생시킨다.

## 비선점 잠금

- 설명
	- 사실 선점 잠금으로도 해결 못하는 문제도 있다.
		- 만약 먼저 운영자가 선점 잠금으로 애그리거트를 얻고 커밋만 한 다음, 사용자가 선점잠금으로 애그리거트를 얻고 배송지를 변경한 다음 커밋까지 하고, 그 다음 그 전 정보를 얻은 상태인 운영자가 배송 정보를 변경하려고 하면 이때의 배송정보는 옛날 정보여서 예전 정보로 배송지가 다시 변경되어 배송되게 된다.
		- 이는 선점 잠금으로도 해결을 못했다는 것에 대한 반증이다.
	- 비선점 잠금을 구현하고 싶으면 버전 컬럼을 하나 추가해 줘야한다.
		- `@Version` 어노테이션을 붙인 long 타입을 하나 붙여주면 된다.
		- JPA는 엔티티가 변경되어 UPDATE 쿼리 실행시 비선점 잠금 쿼리를 실행한다.
	- 버전 충돌 예외 던짐시 이에 대한 반환 메시지등도 만들어 주자

&emsp;&emsp;

### 강제 버전 증가

- 설명
	- 비선점 잠금의 또다른 문제점
		- 애그리거트 외 다른 엔티티도 가져와 기능 실행중인데 이 다른 엔티티의 값이 변경되면 애그리거트 루트의 버전 변경에는 변함이 없다는 문제가 발생한다.
			- 문제는 일반적인 데이터 변경이지만 루트 엔티티가 아닌 다른 엔티티의 값인 경우에도 버전은 변경이 필요한데 그렇지 않은 것이다.
			- 하위 엔티티나 애그리거트에도 버전 변경에 맞춰 변경이 필요하다는 의미이다.
	- 해결방법
		- `@Lock(LockModeType.OPTIMISTIC_FORCE_INCREMENT)`
			- ==**애그리거트 루트의 데이터가 아닌 다른 엔티티의 데이터 변경이더라도 이것을 이용해 애그리거트 루트를 쿼리하면 트랜잭션 마지막에 버전을 올리는 update 쿼리를 실행하고 커밋한다.**==

## 오프라인 선점 잠금

- 설명
	- 선점 잠금 방식과 비선점 잠금 방식의 문제점
		- 오프라인 선점 잠금 문제를 해결 하지 못한다.
	- 선점 잠금 방식과 비선점 잠금 방식의 대표 문제
		- 운영자가 강제로 사용자의 게시글을 수정 중인데, 사용자가 게시글을 동시에 수정하려고 한다고 생각해보자.
			- 운영자나 사용자가 원하는대로 데이터를 넣지 못한 상태로 저장이 될 수도 있다.
			- 머지 충돌을 생각해보면 간단하다.
	- 대표 문제의 경우
		- 특정 자원에 대해 다른 사용자가 동시에 수정하는 문제가 분명한 경우
	- 오프라인 선점 잠금
		- 수정의 경우, 2 개의 트랜잭션을 수행한다.
			1. 사용자가 수정 폼을 요청하고 현재 데이터를 반영해 수정 폼을 리턴한다.
				1. 이때 오프라인 잠금을 얻는다.
				2. 다른 사용자가 얻으려고 하면 에러를 반환해야한다.
			2. 사용자는 수정완료 폼을 전송하여 변경한다.
				1. 시스템은 수정을 한다.
				2. 오프라인 잠금을 해제한다.
				3. 시스템은 결과를 응답한다.
			3. 사용자는 수정 처리 결과를 응답받는다.
	- 오프라인 선점 잠금 필요 기능
		- 선점 시도
			- 타입과 id를 파라미터로 받고 락ID를 리턴한다.
			- 이미 락이 걸린 경우에는 예외를 던진다.
		- 잠금 확인
			- 락ID를 파라미터로 받는다.
			- 이미 락이 걸린 경우에는 예외를 던진다.
		- 잠금 해제
			- 락ID를 파라미터로 받는다.
			- 락이 안걸린 경우에는 예외를 던진다.
		- 잠금 유효시간 연장
			- 락ID와 시간을 파라미터로 받는다.
			- 락이 안걸린 경우에는 예외를 던진다.
	- 오프라인 잠금 기능 구현 방법
		- 서비스에는 락 매니저를 사용해 락과 함께 데이터를 넘겨받아야한다.
		- DB를 이용해 Lock 테이블을 만들어 애그리거트에 대한 락을 넣는다.
