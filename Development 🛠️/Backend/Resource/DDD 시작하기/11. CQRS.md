## 단일 모델의 단점

- 설명
	- 문제점
		- 식별자를 이용해 애그리거트를 참조하는 방식은 즉시 로딩과 같은 방식의 JPA 쿼리 최적화 기능을 사용할 수 없다.
			- join Fetch와 같은 기능을 사용할 수가 없다는 뜻이다.
		- 조회 기능 구현시 JPA 네이티브 쿼리기능을 사용해야 할 수도 있다.
	- 해결 방법
		- 상태 변경용 모델과 조회용 모델을 따로 만들어 구분지어 구현하는 것
			- 이것을 CQRS라고 부른다.

&emsp;&emsp;

## CQRS

- 설명
	- 특징
		- 복잡한 도메인에 적합하다.
		- 보통 DAO는 마이바티스나 DATA JDBC, JDBC 템플릿등을 적용하는 편이다.
		- 보통 캐싱용 DB를 따로두고 여기에서 조회하도록 만들 수 있는데 이때 주로 변경에 대한 후킹으로 이벤트 핸들러에서 캐시DB에 결과를 저장한다.
			- 특정 시간 안에 동기화 해도 된다면 비동기로 데이터를 전송하면 된다.
			- 통계는 수초, 수분, 1시간 단위로 데이터를 반영해도 보통 문제가 없다.

&emsp;&emsp;

### 장단점

- 설명
	- 장점
		- 로직 복잡도가 낮아져서 유지보수가 수월해진다.
		- 조회 성능을 향상시키기 유리하다.
	- 단점
		- 구현해야할 코드와 관련 기술들이 더 늘어난다.
