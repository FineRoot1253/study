## 도메인 전문가와 개발자 간 지식 공유

- 설명
	- 개발자는 도메인 전문가들이 원하는 요구사항을 분석하여 설계하고 코드를 작성해 테스트하고 배포해야한다.
		- 개발에는 요구사항 분석이 첫 단추를 끼우는 것과 같으며 코드는 단추가 아니라서 잘못 끼우면 손실이 어마무시하다.
	- 개발은 요구사항을 올바르게 이해하는 것이 가장 중요하다.
		- 아직까지도 요구사항을 잘못 이해하여 다시 제품을 만드느라 납기 기간이 길어지는 불상사가 일어나고 있다.
	- 요구사항은 개발자와 전문가가 직접 대화해야 올바르게 이해가 가능하다.
		- 중간에 전달자로 기획자가 어설프게 끼어들어가면 정보가 왜곡되고 시간 손실이 발생할 가능성이 매우 커진다.
		- 적어도 회의에는 기능 개발 담당자들이 항상 참여해야 시간 손실 발생 가능성이 적어진다.
	- 도메인 전문가 만큼은 아니지만 이해관계자(기획자나 임원)와 개발자는 도메인 지식을 갖춰야한다.
		- 제품 개발과 관련된 도메인 전문가, 관계자, 개발자가 같은 지식을 공유하고 직접 소통할 수록 도메인 전문가가 원하는 제품을 만들 가능성이 높아진다.
	- 도메인 전문가가 항상 올바른 요구사항을 주지도 않는다.
		- 기존에 만들어진 소프트웨어를 기준으로 요구사항을 제공하기도 하는데 이런 기능들이 항상 좋은 기능들이 아닐 수도 있으며 본인들이 실제 원하는게 잘 표현하지 못하거나 잘 알지도 못하는 경우가 상당히 많다.
	- 개발자는 요구사항을 들을 때 왜 이런 기능을 요구하는 지 배경을 제대로 이해해야 전문가가 원하는 진짜 기능을 구현해낼 수 있다.

## 도메인 모델

- 설명
	- 객체 모델로 주문 모델을 구성하면 다음과 같다.
		- 주문
			- 주문 번호 
			- 총 주문 금액
			- 주문 상태
			- 요청 항목 (컬렉션)
				- 항목
					- 가격
					- 개수
					- 행동
						- 총 개수 가격
					- 상품
						- 이름
						- 가격
						- 상세 정보
			- 제품 정보
			- 배송정보
				- 배송지 주소
				- 주문자
				- 수신자
			- 주문 요청자
				- 주문 요청자 이름
			- 배송상태
			- 행동
				- 배송지 변경
				- 주문 취소
	- 상태별 주문 상태 모델
		- 주문 전
			- 주문하고 결제 전 경우 결제 대기중으로 변경
			- 주문을 결제 후인 경우 상품 준비 중으로 변경
		- 결제 대기중
			- 주문 취소시 주문 취소됨으로 변경
			- 결제 시 상품 준비 중으로 변경
		- 상품 준비 중
			- 택배사에서 상품 수령하는 경우 출고 완료됨으로 변경
		- 출고 완료됨
			- 택배사에서 배송을 시작하는 경우 배송 중으로 변경
		- 배송 중
			- 고객이 상품을 수령하는 경우 배송 완료됨으로 변경
	- 도메인 모델
		- 도메인 자체를 이해하기 위한 모델이며 개념 모델을 코드로 바로 작성하는 것은 아니기 때문에 구현 기술에 맞는 구현 모델이 필요하다.
		- 개념 모델과 구현 모델은 서로 다른 것이며 구현 모델이 최대한 개념모델에 가깝게 구현할 수 있다.
			- 객체 지향 언어인 자바나 코틀린으로 객체 모델에 맞게 구현할 수 있다.
	- 각 하위 도메인 마다 모델을 만들어야한다.
		- 카탈로그와 배송 도메인을 하나의 다이어그램으로 묶는다면 상품은 카탈로그의 상품이거나 배송의 상품을 의미하여 카텔로그 도메인을 이해하는데 방해가 된다.
		- ==**각 모델의 구성요소는 특정 도메인으로 한정지어야 의미가 완전해진다.**==
			- 즉, 카탈로그의 하위 도메인 모델과 배송 하위 도메인 모델을 만들어야 상품 도메인이 제대로 의미가 완전해진다.
		- 비슷한 예로 거진 모든 데이터를 객체로 만들려는 노력을 하라는 객체지향 개발자 선배의 조언이 있었다.
			- 이런식으로 개발을 하게되면 나중에 데이터가 가지는 의미가 커지는 순간, 나중가서 분해하려고 애쓰는 것보단 미리 객체로 정의되어 있으니 도메인 모델 내부에서 해당 데이터에 엮여있는 관련 로직을 하나하나 생선 살 바르듯이 분해하기는 것보다 더 쉽고 간단하게 해결이 된다.
				- 해당 데이터의 클래스 파일을 다른 도메인 패키지로 이동만 시키면 되기 때문이다.

## 도메인 모델 패턴

- 설명
	- 도메인 아키텍처
		- 표현(컨트롤러), 응용(서비스), 도메인(서비스에 사용되는 각 객체), 인프라(서비스나 DB와 통신하는 연결 객체), 레포지터리(데이터베이스, 영속화를 위해 사용되는 객체) 레이어를 나눠 영역을 구성한다.
		- 표현
			- 사용자의 요청을 처리하거나 외부 시스템과의 통신을 다루기도한다.
			- 컨트롤러
		- 응용
			- 사용자가 요청한 기능을 처리하며 업무 로직을 직접 구현하지 않고 도메인 계층을 조합해서 기능을 실행하는 역할을 한다.
			- 서비스
		- 도메인
			- 시스템이 제공할 도메인 규칙을 구현한다.
			- 사용되는 객체들
		- 인프라스트럭처
			- 데이터베이스나 메시징 시스템과 같은 외부 시스템과의 연동을 처리한다.
			- 레포지터리 혹은 카프카와 같은 전역 트랜잭션 메시징큐 등등
	- 레이어드 아키텍처
		- 서비스, 컨트롤러, 레포지터리 3계층으로 나눠 문제를 해결하는 패턴
		- 장점
			- 초기에 서비스의 기능을 빠르게 구현하는 데에 용이하다.
		- 단점
			- 서비스간 의존성 해결이 상당히 난해하여 데이터 일관성을 해치기 쉬운 구조이다.
				- 특히 JPA를 사용하는 경우 더욱 도메인 개념을 분리하기 난해해진다.

``` java
@Entity  
@Table(name = "purchase_order")  
@Access(AccessType.FIELD)  
public class Order {  
  
    @Embedded  
    private ShippingInfo shippingInfo;  
  
    @Column(name = "state")  
    @Enumerated(EnumType.STRING)  
    private OrderState state;  
  
    protected Order() {  
    }  
  
    public Order(OrderNo number, Orderer orderer, List<OrderLine> orderLines,  
                 ShippingInfo shippingInfo, OrderState state) {  
        setNumber(number);  
        setOrderer(orderer);  
        setOrderLines(orderLines);  
        setShippingInfo(shippingInfo);  
        this.state = state;  
        this.orderDate = LocalDateTime.now();  
        Events.raise(new OrderPlacedEvent(number.getNumber(), orderer, orderLines, orderDate));  
    }  
    public ShippingInfo getShippingInfo() {  
        return shippingInfo;  
    }  
  
    public OrderState getState() {  
        return state;  
    }  
  
    public void changeShippingInfo(ShippingInfo newShippingInfo) {  
        verifyNotYetShipped();  
        setShippingInfo(newShippingInfo);  
        Events.raise(new ShippingInfoChangedEvent(number, newShippingInfo));  
    }  
  
    public void cancel() {  
        verifyNotYetShipped();  
        this.state = OrderState.CANCELED;  
        Events.raise(new OrderCanceledEvent(number.getNumber()));  
    }  
}
```

``` java
public enum OrderState {  
    PAYMENT_WAITING{  
        public boolean isShippingChangeable(){  
            return true;  
        }  
    }, PREPARING{  
        public boolean isShippingChangeable(){  
            return false;  
        }  
    }, SHIPPED, DELIVERING, DELIVERY_COMPLETED, CANCELED;  
    public boolean isShippingChangeable(){  
        return false;  
    }  
}
```

- 설명
	- 주문 상태에 따라 주문이 변경이 가능한 상태인지에 대한 코드를 구현한 것이다.
		- 나중엔 이 로직을 enum이 아닌 도메인에 포함시킨다.
			- 주문 상태 판단을 주문에게 맡기는 것이다.
		- enum에 배송지 변경이 가능한지 아닌지에 대한 규칙은 상태값 만으로 하는 것이 아니기 때문에 이것을 넘긴 것이다.
		- 주로 enum으로 분기처리를 하는 경우가 상당히 많은데 의존관계가 문제가 생긴다면 해당 분기 처리의 책임을 생각해서 다시 옮길 필요가 있을 것이다.
			- 예를 들어서 지금의 케이스가 그러하다.
	- 처음부터 모든 것을 고려해 만든 구현 모델이 아니다.
		- 개념모델을 만들 당시 성능등을 고려해서 만드는 것은 불가능하다.
			- 미래를 예측하려고 하는 것과 동일하기 때문이다.
		- 통찰력은 개념 모델을 토대로 구현 모델을 만들다가 새로운 지식이 쌓이면서 생기는 것이기 때문에 전반적인 개요을 알 수 있는 수준으로 개념 모델을 작성해야한다.
	- 프로젝트 초기에는 개요 수준의 개념모델로 도메인에 대한 윤곽을 잡고, 구현하는 과정에서 개념 모델을 구현 모델로 점진적으로 발전시켜 나가야 한다.

## 도메인 모델 도출

- 설명
	- 주문 도메인 모델에 대한 요구사항
		- 최소 한 종류 이상의 상품을 주문
		- 한 상품을 한 개 이상 주문
			- 책에서는 이 검사를 Order에서 구현한다.
				- 주문 객체 생성시 초기화 단계에서 한다.
			- OrderLines라는 컬렉션에서 이와 같은 검사를 해도 무리는 아니지만 책임의 주체를 Order로 넘긴 것이다.
		- 총 주문 금액은 각 상품들의 구매 가격의 합
		- 각 상품의 구매 가격 합은 상품 가격 합을 모두 더한 금액
		- 주문할 때 배송지 정보를 반드시 지정해야한다.
			- 책에서 이 검사를 Order에서 구현한다.
				- 주문 객체 생성시 초기화 단계에서 한다.
			- 이 과정은 OrderLines의 상태를 알아야하기 때문에 Order에서 처리하는 것이다.
		- 배송지 정보는 받는 사람의 이름, 전화번호, 주소로 구성
		- 출고시 배송지변경이 불가능하다.
		- 출고전 주문 취소가 가능하다.
		- 고객이 결제 완료하기 전까진 상품을 준비하지 않는다.
	- 주문 도메인의 요구사항에 따라 필요한 행동
		- 배송으로 변경
		- 배송지 정보 변경
		- 주문 취소
		- 결제 완료
	- 위의 요구사항에 대한 검사는 테스트코드로 만들어두는 것이 매우 좋다.
		- 필수 요소라고 봐도 무방하다.
			- 정말 시간이 너무 촉박하다면 어쩔 수 없이 넘어가겠지만 말이다.
	- 문서화를 해야한다.
		- 책에서는 코드 자체를 문서처럼 여길 수 있도록 잘 작성하는 것이 중요하다고 강조한다.
		- 팀에서의 룰에 따라 가면 된다.
			- 주석처리도 주석에 따라 다르다.
				- 주석처리를 해도 코드를 바꾸면 주석도 함께 바꿔야하니 작성된 코드에 비해 해야할 일이 2배로 증가된다.
			- 주석처리를 꼭해야한다면 왠만하면 퇴사를 해야하는 순간이 아니면 하지 않는 것을 권장한다.
				- 그리고 퇴사를 해야하는 순간이라면 문서화를 주석으로 때우지말고 그냥 레퍼런스 문서를 만들어서 제공하여 빌미를 없애는게 낫다.
		- 특히 프론트에 제공할 문서화가 필요하다면, RESTDoc과 같은 컨트롤러 테스트에 결합된 자동화 도구를 이용하면 상당히 괜찮다.
			- 요구사항 변경시 테스트가 자동으로 실패하게 되어 수정이 필요한 위치를 알게 된다.
			- 물론 이 또한 시간 소모이므로 잘 고려해서 적용해보도록하자.

## 엔티티와 벨류

- 설명
	- 모델의 종류
		- 엔티티
			- 식별자를 가지는 객체
				- 엔티티 객체 별로 서로 다른 식별자를 가진다.
				- 책에서는 hashCode와 isEquals와 같은 동등성 비교 메서드를 만들어 구현하지만 롬복을 쓰면 편하게 처리 가능하다.
				- 벨류 객체 또한 동등성 비교 메서드가 필요하지만 들어있는 데이터가 동등하면 이땐 동등해야하지만 엔티티는 동등해도 식별자만 다르면 상관 없다.
		- 벨류

### 엔티티의 식별자 생성

- 설명
	- 엔티티의 식별자 생성 방식
		- 특정 규칙에 따라 생성
		- snowflake, uuid나 nano id와 같은 고유 식별자 생성기 사용
		- 값을 직접 입력
			- 회원 아이디, 이메일 등이 있다.
		- 일련번호 사용 (시퀀스, auto increment 등)
			- 이 방식도 멀티 노드 환경에서는 제대로 동작하지 않는 경우가 많다.
			- 특히 DB 까지 여러 노드로 둔다고 가정하면 auto increment는 성능에 문제가 생겨 쓸 수 없는 수준이 된다.
	- 가장 자주 사용되는 규칙
		- 현재 시간 + 랜덤값 붙이기
		- 스노우플레이크 구조
			- 현재 타임스탬프
			- 노드 ID
			- 순차적으로 증가하는 숫자값
		- 요즘 MSA 구조에서는 아예 식별자 생성을 담당하는 서버를 하나 두기도 한다.

### 벨류 타입

- 설명
	- 특징
		- 개념적으로 완전한 하나를 표현할 때 사용한다.
			- 예를 들면 착신자는 이름과 전화번호, 주소는 시, 구(군), 동(읍)까지 적어야 제대로 된 의미를 가진다.
		- 단순한 하나의 데이터라도 어떤 개념을 의미할 때도 이를 따로 개념으로 표현하는 것이 맞다.
			- 예를 들면, 돈은 단순한 데이터가 아니다. Balance나 Money라는 객체로 따로 표현하는 것이 객체를 제대로 바라보며 개발 할 수 있다.
		- 불변해야한다.
			- 값을 변경한다면, 새로운 값으로 대체하여야한다.
			- 불변한다면 멀티 스레드 환경에서 안전하다.
			- 다른 도메인에서 함부로 값을 변경할 수 없게 된다.
				- 예를 들면, 가격을 Money 객체로 표현하고 setter를 구현하지 않는다면, 파라미터로 r가격객체의 참조를 전달하더라도 안전하게 사용할 수 있다.


&emsp;&emsp;

### 엔티티 식별자와 벨류 타입

- 설명
	- 책에서는 식별자 값또한 벨류 타입으로 만든다.
		- 이것이 DDD의 첫 걸음이긴하다.
		- 이러면 JPA 주인관계를 풀어야하는 불편함이 있지만 결합도를 낮춰 더 많은 기능들을 확장하기 편해진다.
		- JPA로 개발하다 보면 복잡해지는 것이 바로 주인관계인데, 이는 정규화를 잘못나눴거나, 덜 나눴거나, 억지로 한방쿼리를 짤려고하거나하면 이렇게 된다.
			- 이러한 문제를 해결하기 위해서는 queryDSL과 SQL과 거의 흡사한 문법 구조를 지닌 도구를 사용하는 것이 좋은데 문제는 이렇다고 해서 더 많은 관계가 늘어나면 늘어날 수록 더 많은 의존관계가 꼬이기 시작하고 DB로는 관계를 표현하지만 객체로는 제대로 의존관계를 관리하기 힘들어지는 순간들이 오기 시작한다.

&emsp;&emsp;

### 도메인 모델에 setter 넣지 않기

- 설명
	- 책에서는 getter는 public, setter는 private으로 하기를 권장한다.
		- setter는 존재하는건 나쁘진 않다. 다만 내부에서 값을 초기화하고 초기화시 검증하는 로직을 배치하기 위해서 존재하는게 나쁘지 않다는거지 외부에 여는 순간 데이터 유지 일관성이 깨져 지옥문이 열린다.
	- getter와 setter는 특히 톰캣에서 사용되는 빈 모델에서의 필수 구현 메서드였다보니 잘못 알려진 사례가 많다고 한다.
		- 이러한 getter setter도 JSP와 같은 표현 계층에서는 상관 없겠지만 시간이 없으니 이를 그대로 복사 붙여넣기하여 개발하는 습관들을 잘못 가르쳐진 사례가 상당히 많다고 한다.
	- 책에서는 생성자 시점에 모든 필요한 데이터를 전부 전달받아야한다고 가르친다.
		- 나도 이게 맞다고 생각한다.
	- 롬복을 잘 이용하면 DTO도 불변객체로 만들 수 있다.
		- 이는 표현 계층과 도메인 계층간 소통을 위해 사용된 구조체이지만 결합도를 낮추기 위해서 어디서든 쓰일 수 있는 객체다.
			- 한 계층에서 다른 계층에 전달한다고 하면 DTO를 써주는 것이 좋다.
			- OVI 개념에서도 좋다.

### 도메인 용어와 유비쿼터스 언어

-  설명
	- OrderState와 같이 상태를 숫자로만 표현한다면, 도메인 규칙이 드러나지 않는다.
		- 따라서 위에 올린 예시와 같이 코드로 표현한다면 이를 해석하는데 들어가는 시간이 줄어든다.
	- 도메인 용어를 사용해 도메인 규칙을 코드로 작성해야 버그도 줄어든다.
		- 이를 유비쿼터스 언어라고 에릭 에반스는 표현한다.
	- 책에서는 영어사전을 사용해서라도 알맞는 용어를 사용하라고 가르치는데 요즘은 지피티를 사용하면 기깔나게 알려준다.
