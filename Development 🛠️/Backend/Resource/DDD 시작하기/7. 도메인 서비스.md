## 여러 애그리거트가 필요한 기능

- 설명
	- 책에서의 결제 금액 계산 로직의 요구사항과 각 애그리거트가 대응하는 기능
		- 상품 애그리거트
			- 구매하는 상품의 가격이 필요하다.
				- 상품에 따라 배송비가 추가되기도 한다.
		- 주문 애그리거트
			- 상품별 구매 개수가 필요하다.
		- 할인 쿠폰 애그리거트
			- 쿠폰별로 지정한 할인 금액이나 비율에 따라 주문 총 금액을 할인한다.
				- 쿠폰 사용에 대한 여러 정책들이 붙기 시작하면 할인 계산이 복잡해진다.
				- 이걸 트랜잭션 스크립트로 해결한 레거시들이 정말 많다.
		- 회원 애그리거트
			- 회원 등급에 따라 추가 할인이 가능하다.
	- 책에서의 핵심 내용
		- 위에 적은 각 애그리거트에서 제공하는 모든 기능을 사용해야 계산 추론이 가능하다.
		- 책에서는 처음엔 주문 애그리거트에서 기능을 구현했지만, 이론상 가능은 하지만 주문에 집어넣게 되면 그 책임을 벗어난다고 이야기한다.
		- 책에서는 도메인 서비스로 이를 해결한다.

&emsp;&emsp;

## 도메인 서비스

- 설명
	- 도메인 서비스가 필요한 경우
		- 여러 애그리거트의 협력이 필요한 경우
			- 회원가입, 게시글 작성, 변경, 댓글 작성, 변경 등등
		- 한 애그리거트에 넣기에는 다소 복잡한 계산 로직
	- 외부 시스템 연동이 필요한 도메인 로직
		- 구현하기 위해 타 시스템을 사용해야하는 도메인 로직
			- 외부 모듈 API 연동, 이메일 전송, 알림 전송, 이벤트 전송등이 있다.

&emsp;&emsp;

### 계산 로직과 도메인 서비스

- 설명
	- 책에서는 할인 계산 서비스를 만들고, 이 서비스를 다시 주문 애그리거트에 총액 계산 메서드의 파라미터로 넣는다.
	- 계좌 이체의 경우 계좌 애그리거트를 이체 서비스 메서드의 파라미터로 넘긴다.
	- 도메인 서비스는 도메인 로직을 수행할 뿐 응용 로직을 수행하지 않는다.
		- 트랜잭션 처리와 같은 로직은 응용 로직이므로 도메인 서비스가 아닌 응용 서비스에서 처리해야한다.
	- 주의 사항
		- 응용 서비스, 도메인 서비스 기준점은 애그리거트 상태의 변경이거나 상태를 계산하거나 둘 중 하나를 봐야한다.
			- 계좌 이채나 도메인 서비스는 둘다 한 애그리거트에 넣기 애매하므로 도메인 서비스로 구현한다.
		- 정확한 것은 트랜잭션에 의존하는 서비스를 밖으로 뺀다는 것이 아니다.
			- 애그리거트끼리 계산을 해야해서 특정 애그리거트의 값을 변경할 때 사용되는 것이다.

&emsp;&emsp;

### 외부 시스템 연동과 도메인 서비스

- 설명
	- 이 경우 도메인에 인터페이스를 넣고 인프라쪽에 구현 코드를 만들어 연동 로직을 구현한다.


&emsp;&emsp;

### 도메인 서비스의 패키지 위치

- 설명
	- 도메인 서비스는 도메인 영역에 있어야 한다.
	- 도메인 개수가 많고 엔티티나 벨류와 같은 다른 구성요소와 구분하고 싶어지면 domain.model, domain.service, domain.repository 와 같이 하위 패키지를 구분하여 위치시켜도 된다.

&emsp;&emsp;

### 도메인 서비스의 인터페이스와 클래스

- 설명
	- 별도의 룰 엔진 하에 계산하여야하는 경우에는 도메인 서비스 자체를 인터페이스로 만들고 인프라에서 서비스를 구현하면 된다.
