## 도메인 모델과 경계

- 주의사항
	- 도메인 모델 설계시 ==**도메인을 완벽하게 표현하는 단일 모델을 만들려고 노력하므로써 문제가 발생**==한다.
		- 한 도메인은 여러 도메인으로 구분되기 때문에 처음부터 도메인을 모두 표현하는 것은 불가능하다.
			- 상품의 경우, 주문에서의 상품, 카탈로그에서의 상품, 배송에서의 상품은 전부 다르게 여러 개 존재한다.
		- 논리적으로 같은 존재처럼 보이지만 하위 도메인에 따라 다른 도메인 용어를 사용하는 경우도 많다.
			- 회원 도메인에서의 회원은 주문 도메인에서는 주문자 또는 배송 받는 사람, 배송 도메인에서는 배송 보내는 사람으로 불린다.

&emsp;&emsp;

- 결론
	- 하위 도메인마다 사용하는 용어가 달라 ==**하위 도메인마다 모델을 만들어야한다**==.
	- 모델은 컨텍스트 하에 완전 다른 의미를 가지므로 같은 제품이라도 카탈로그 컨텍스트와 재고 컨텍스트는 의미론상 완전히 다르다.
		- 구분이 되는 경계를 갖는 컨텍스트를 바운디드 컨텍스트라고 부른다.

&emsp;&emsp;
## 바운디드 컨텍스트

- 설명
	- 정의
		- 모델의 경계를 결정하는 문맥
	- 특징
		- 한 개의 바운디드 컨텍스트는 논리적으로 한 개의 모델을 갖는다.
		- 바운디드 컨텍스트는 용어를 기준으로 구분된다.
			- 카탈로그와 재고는 서로 다른 용어를 사용해 상품을 바라보므로 이 용어를 기준으로 컨텍스트를 분리할 수 있다.
		- 바운디드 컨텍스트는 실제로 사용자에게 기능을 제공하는 물리적 시스템이다.
		- 도메인 모델은 바운디드 컨텍스트 안에서 도메인을 구현한다.
		- 바운디드 컨텍스트는 팀 표준에 따라 결정된다.
			- 주문 처리 팀과 결제 처리 팀이 따로 존재하는 팀 구조라면 주문 하위 도메인에 주문 바운디드 컨텍스트, 결제 바운디드 컨텍스트가 존재하게 된다.
			- 카탈로그 하위 도메인과 재고 하위 도메인에서 하나의 바운디드 컨텍스트에서 구현하기도 한다.
		- 규모가 작은 기업은 한 팀에서 전체 시스템을 구현한다.
			- 온라인 쇼핑의 경우 하나의 시스템에서 회원, 카탈로그, 재고, 구매, 결제와 관련된 모든 기능을 제공한다.
				- 여러 하위 도메인을 한개의 바운디드 컨텍스트에서 재공하는 구조이다.
				- 이런 구조에서는 하위 도메인 모델이 섞이지 않게만 만들어도 감지덕지하다.
					- ==**전체 하위 도메인을 위한 단일 모델을 만들고 싶은 유혹에 넘어가면 안된다.**==
				- 각 도메인 모델을 만들어야 모델이 뒤섞이지 않고 하위 도메인마다 바운디드 컨텍스트를 갖는 효과를 누릴 수 있다.
		- 바운디드 컨텍스트는 구현하는 하위 도메인에 알맞는 모델을 포함한다.
			- 예를 들면 주문과 회원 바운디드 컨텍스트가 갖는 회원 모델이 다르다.
				- Member는 애그리거트 루트이지만 주문 Orderer의 벨류가 된다.
				- 카탈로그의 Product(애그리거트 루트)는 상품이 속할 Category와 연관을 갖지만 재고의 Product(애그리거트 루트)는 카탈로그의 Category와 연관을 맺지 않는다.

## 바운디드 컨텍스트 구현

- 설명
	- 구현시 주의사항
		- ==**바운디드 컨텍스트는 도메인 모델만 포함하지 않고 모든 영역을 포함한다.**==
			- 예를 들면 단순 리뷰에는 CRUD 방식으로 개발해도 큰 문제는 없다.
			- 이런 경우 도메인 기능이 서비스에 흩어지지만 기능이 단순하면 유지보수에 큰 문제가 없다.
			- CQRS의 경우 두 방식을 혼합해서 사용한다.
				- 다같이 바운디드 컨텍스트에 포함이 되는 것이다.
			- 책에서는 바운디드 컨텍스트별로 패키지를 분할하고 있다.

&emsp;&emsp;

## 바운디드 컨텍스트 간 통합

- 설명
	- 예시
		- 카텔로그 하위 도메인에 개인화 추천 기능을 추가하고 팀을 추가로 꾸린다고 가정하면 카탈로그 하위 도메인에 카탈로그 바운디드 컨텍스트와 추천 바운디드 컨텍스트가 생기게 된다.
			- 이때 통합이 필요한 기능이 무조건 발생한다.
			- 예시로 사용자가 제품 상세를 볼때 보고있는 상품과 유사한 상품 목록을 하단에 보여준다.
				- 사용자가 카탈로그 바운디드 컨텍스트에 추천 제품 목록을 요청하면 카탈로그 바운디드 컨텍스트는 추천 바운디드 컨텍스트로부터 추천 정보를 읽어와 추천 제품 목록을 제공한다.
					- 이때 카탈로그 컨텍스트와 추천 컨텍스트의 도메인 모델은 다르다.
					- 카탈로그는 제품을 중심으로 도메인 모델을 구현하지만 추천은 추천 연산을 위한 모델을 구현한다.
				- 추천 시스템은 상품의 상세 정보를 포함하지 앟고 상품 번호 대신 아이템 ID라는 용어를 사용해 식별자를 표현하고 추천 순위와 같은 데이터를 담게 된다.
				- 카탈로그 시스템은 추천 시스템으로부터 추천 데이터를 받아오지만 카탈로그 시스템에서는 추천의 도메인 모델을 사용하지 않고 카탈로그 도메인 모델을 사용해 추천 상품을 표현해야만 한다.
					- 책에서는 외부 추천 시스템을 인프라로 구현하고 인터페이스를 카탈로그 도메인 패키지에 구현하고 있다.
					- 외부 모델과 상품간 변환 과정이 복잡하여 변환기를 두고 있다.
					- 메시지 큐를 이용한 방법도 알려준다.
						- 추천 바운디드 컨텍스트는 큐에서 사용자 이력 메시지를 읽고 추천을 계산하고 사용자 이력 메시지는 카탈로그 컨텍스트에서 메시지 큐에 등록한다.
						- 각 팀에서는 주고 받을 데이터 형식을 협의해야한다.
						- 카탈로그는 뷰 로그, 주문 로그, 주문 상품 로그를 등록한다.
						- 추천 도메인에서는 활동 로그로 바라본다.
							- 그래서 실제 구현에서는 클라이언트에서 모든 로그를 활동 로그 형태로 변환하여 메시지 큐에 저장한다.

## 바운디드 컨텍스트 간 관계

- 설명
	- 관계 종류
		- Pub/Sub
			- 특징
				- 한쪽에서는 정보를 제공하고 한쪽에서는 정보를 소비만한다.
				- 제공하는 쪽을 상류, 소비 하는 쪽은 하류로 구분한다.
					- 예시에서의 카탈로그는 하류, 추천 시스템은 상류이다.
					- 추천 아이템을 카탈로그에 던져주기 때문
				- 각 팀에서 개발한다면 상호 협력이 매우 중요하다.
					- 추천 시스템에서는 gRPC와 같은 프로토콜을 통해 제공할 수도 있다.
					- 만약 하류 팀이 여러 개라면, 이를 서비스 형태로 공개해서 서비스의 일관성을 유지할 수 있다.
					- 이것을 공개 호스트 서비스(Open Host Service, 이하 OHS)라고 부른다.
						- 검색 시스템이 가장 대표적인 공개 호스트 서비스이다.
						- 블로그, 카페, 게시판과 같은 서비스를 제공하는 포털은 각 서비스별로 검색 기능을 구현하지 않고 검색을 위한 전용 시스템을 구축하고 검색 시스템과 각 서비스를 통합한다.
						- 검색 시스템은 상류이며 블로그, 카페, 게시판은 하류가 된다.
						- 검색 시스템은 각 하류 팀에서 요구사항을 수용하는 단일 API를 만들어 공개하고 공개된 팀은 API를 통해 검색 기능을 구현한다.
				- 상류 컴포넌트 서비스는 상류 바운디드 컨텍스트의 도메인 모델을 따른다.
					- 따라서 하류 컴포넌트는 상류 서비스의 모델이 자신의 도메인 모델에 영향을 받지 않도록 완충지대(DIP 구조, 도메인 인터페이스 - 인프라 구현체)가 필요하다.
					- 이것을 우리는 Anti-Corruption Layer, 이하 ACL로 부른다.
		- 공유 커널
			- 특징
				- 두 바운디드 컨텍스트가 같은 모델을 공유하는 경우
				- 운영자를 위한 주문 관리를 만드는 팀과 고객을 위한 주문 서비스를 만드는 팀이 다를 경우 주문을 표현하는 모델을 공유하여 주문과 관련된 중복 설계를 막을 수 있다.
				- 중복을 줄여주는 장점이 있다.
				- 두 팀이 밀접하게 소통하지 않으면 함부로 모델을 변경이나 기능 추가를 하지 못해 오히려 개발 속도가 더 더뎌지는 단점이 있다.
		- 독립 방식
			- 특징
				- 서로 통합하지 않는 방식
				- 서로 연동을 하지 않으므로 사용자가 직접 각 도메인 서비스를 이용해야한다.
				- 수동 방식에는 한계가 있으므로 두 바운디드 컨텍스트를 통합하는 별도의 시스템을 만들어야할 수도 있다.

## 컨텍스트 맵

- 설명
	- 주의사항
		- 개별 바운디드 컨텍스트에 매몰되면 전체를 조망하지 못하게 된다.
			- 전체를 조망할 지도인 컨텍스트 맵은 항상 필요하다.
	- 특징
		- 각 바운디드 컨텍스트의 경계가 명확히 드러나고 서로 어떤 관계를 맺고 있는지 알 수 있다.
		- 주로 그릴때 정보를 사용하는 계층은 ACL, 정보를 제공하는 쪽은 OHS로 표현한다.
