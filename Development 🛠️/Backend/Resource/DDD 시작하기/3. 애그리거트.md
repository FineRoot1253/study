![[littok_aggregate.png]]


- 설명
	- 책 예제
		- 주문은 회원과 결제, 상품에 의존적이라는 것을 보여준다.
		- 상품리뷰는 상품에 의존적이라는 것을 보여준다.
	- Littok
		- 리뷰는 책에 의존적이다.
		- 코멘트, 평점은 리뷰에 의존적이다.
		- 팔로잉, 계정은 유저에 의존적이다.
		- 권한은 계정에 의존적이면 안된다. 다중계정을 허용할 예정이기 때문에 권한은 유저에 의존적이여야한다.
		- 자원은 권한에 의존적이다. (그림이 잘못됬다고 생각한다)
	- 특징
		- 백 개 이상의 ERD를 모두 표시하면 개별 테이블 간의 관계 파악에 큰 틀에서 데이터 구조 파악에 어려움을 겪게 된다.
			- 요즘 ERD 툴들은 애그리거트 구분을 잘 짓도록 테이블끼리의 영역 표시를 할 수 있도록 잘 나와있다.
			- 주요 도메인 요소 간의 관계를 파악하기 어렵다는 것은 코드를 변경하고 확장하는 것이 어려워 진다는 것을 의미한다.
			- 상위 수준에서 어떻게 엮여 있는지 알아야 전체 모델을 망가뜨리지 않으면서 추가 요구사항을 모델에 반영할 수 있는데, 세부 모델만 이해하면 코드 수정이 꺼려지기 때문에 코드 변경을 최소화하는 방향으로만 협의를 하게 된다.
				- 꼼수를 부려 당장 돌아가는 코드를 추가할 수 있지만 장기적으로 코드를 더 수정하기 힘들게 만든다.
		- 복잡한 도메인을 이해하려면 상위 수준에서 모델을 조망할 수 있어야한다는 것이다.
		- 모델을 이해하는데 도움을 주며, 일관성을 관리하는 기준이 된다.
			- 모델을 보다 잘 이해할 수 있고 애그리거트 단위로 일관성을 관리하기 때문에 애그리거트는 복잡한 도메인을 단순한 구조로 만들어준다.
			- 복잡도가 낮아지는 만큼 도메인의 기능을 확장하고 변경하는 데 필요한 노력도 줄여준다.
		- 애그리거트는 관련된 모델을 한데 모으기 때문에 한 애그리거트에 속한 객체는 유사하거나 비슷한 라이프 사이클을 가진다.
			- 주문 애그리거트에 속한 Order, Orderer, OrderLine과 같은 객체들은 함께 살고 함께 죽어야한다.
				- Order는 생성했지만 ShippingInfo는 만들지 않거나 ShippingInfo를 생성하면서 Orderer를 생성하지 않는 경우는 없다.
				- 도메인 규칙에 따라 최소 주문 시점에 일부 객체를 만들 필요가 없는 경우도 있지만 애그리거트가 속한 구성요소는 대부분 함께 생성하고 함께 제거한다.
		- 애그리거트는 경계를 가지며 다른 애그리거트에 동시에 속하지 않는다.
		- 애그리거트는 독립된 객체 군이며 각 애그리거트는 자기 자신을 관리할 뿐 다른 애그리거트를 관리하지 않는다.
			- 주문 애그리거트는 배송지를 변경하거나 다른 애그리거트를 관리하지 않는다.
				- 배송지 변경, 주문 상품 개수 변경등은 자신이 하지만 주문 애그리거트에서 회원의 비밀번호를 변경하거나 상품의 가격을 변경하지 않는다.
			- litok의 경우, 회원가입 인증 코드나 비밀번호 재설정 인증 코드는 Auth 영역에서 처리하는게 맞지 유저가 하는게 아니다.
		- 경계 설정의 기본 규칙은 **도메인 규칙**과 **요구사항**이다.
			- 도메인 규칙에 따라 추가되는 요구사항은 주로 해당 애그리거트에 속할 가능성이 높다.
			- 주문할 상품의 개수, 배송지 정보, 주문자 정보는 주문 시점에 함께 생성되므로 이들은 한 애그리거트에 속한다.
				- OrderLine변경시 총 주문 금액을 새로 계산해야 한다.
				- 사용자 요구사항에 따라 주문 상품 개수와 배송지를 함께 변경할 수도 있다.
			- ==**함께 변경되는 빈도가 높은 객체들은 한 애그리거트에 속할 가능성이 높다.**==
		- A가 B를 갖는다는 요구사항이 존재해도 라이프사이클은 다를 수 있다.
			- 이는 속하는 애그리거트가 다를 수 있다는 의미이다.
			- 게시글과 댓글이 가장 좋은 예이다.
				- 게시글과 댓글은 동시에 생성되서 동시에 죽는 존재는 아니다.
					- 물론 대다수는 게시글이 죽으면 댓글이 같이 사라지게끔 하는 것이 일반적이긴하다.
					- 다만 생성되는 순간을 같이 하진 않으니 댓글이 게시글 애그리거트에 속한다고 볼수는 없다.
				- littok의 경우에는 책은 littok 시스템이 올리고 리뷰는 유저가 올리는 것이니 올리는 주체도 다르다.
		- 도메인 모델 설계시 처음은 거대해보이는 애그리거트로 보일수 있지만 경험이 쌓이고 도메인 규칙을 제대로 이해할수록 애그리거트의 실제 크기는 줄어든다.
			- ==**다수의 애그리거트는 한 개의 엔티티 객체만 가지는 경우가 대다수이며 두 개 이상의 엔티티로 구성되는 애그리거트는 드물다.**==

## 애그리거트 루트

- 설명
	- 예시
		- 주문 애그리거트의 항목
			- totalAmounts를 가지는 Order 엔티티
			- quantity와 price를 갖고 있는 OrderLine 벨류
		- 상품의 개수를 변경하면 OrderLine의 quantity를 변경하고 더불어 order의 totalAmounts도 변경해야한다.
		- ==**애그리거트는 여러 객체로 구성되기 때문에 한 객체만 상태가 정상이면 안된다.**==
	- 정의
		- 애그리거트에 속한 모든 객체가 일관된 상태를 유지하게 만드는 것이 중요하며 이를 주관하는 객체가 애그리거트의 루트 엔티티이며 이것이 애그리거트의 대표 엔티티이다.
	- 특징
		- 애그리거트에 속한 객체는 애그리거트 루트 엔티티에 직접 또는 간접적으로 속하게 된다.
			- 주문 애그리거트에서 루트 역할은 Order이다.
				- OrderLine, ShippingInfo, Orderer등 주문 애그리거트에 속한 모델은 Order에 직 간접적으로 속한다.
			- littok 기준 유저 애그리거트의 루트 역할은 User, 책 애그리거트의 루트 역할은 Book이다.

### 도메인 규칙과 일관성

- 설명
	- 애그리거트 루트가 제공하는 메서드는 도메인 규칙에 따라 애그리거트에 속한 객체의 일관성이 깨지지 않도록 구현해야한다.
		- 배송이 시작되기 전까지만 배송지 정보를 변경할 수 있다는 규칙이 있다면, changeShippingInfo()메서드는 이 규칙에 따라 사전조건으로 배송 시작 여부를 확인하고 규칙에 허용되는 경우 배송지 정보를 변경해야한다.
	- 책에서는 changeShippingInfo 구현시, Order에서 직접 수정하는 것이 아닌, 새로운 shippingInfo를 외부에서 받아서 새로운 레퍼런스값으로 교체하는 것을 필수요소로 여기고 있다.
		- 즉, Order에서는 ShippingInfo를 내부에 지니고는 있지만 변경 권한을 쥐어주진 않고 새로운 초기화 권한만 주어진 것이다.
			- 거기에 초기화 전에 검증 로직만 넣어주면 된다.
		- 개인적으로는 ShippingInfo 자체에 검증 로직을 넣어줄 수 있으면 넣어주는 것을 권장하는데, 이 요구사항에서는 ShippingInfo 애그리거트 외부 상태 값인 배송 출발 여부를 확인해야하므로 이에 대한 의존성이 생기면 애그리거트를 벗어나 결합도가 굉장히 커진 생성자를 가지게 된다.
			- 따라서 지금같은 구현이 옳게된 구현이라고 보면 좋다.
			- 보통 생성자에 사전조건이 필요한 경우 값자체에 생성조건이 존재하는 경우에 넣어주는 것이 좋다.
				- 예를들면 나이, 신장과 같은 경우 공백이거나, 음수일수는 없다. 이런 경우는 생성자에서 사전조건 검증로직을 집어넣어주는 것이 좋다.
					- Spring Validator를 이용하는 것도 방법이긴 하지만 내부에서 데이터 변경시 문제되는 경우를 막아준다.
					- 예를 들면 나이의 계산 공식이 잘못되어 1살을 올린다는 것이 -1살을 올린다는 극단적인 사례를 들어보자, 이런 경우에도 적어도 데이터 생성 단계에서 검증이되어 예외를 잡아낼 수 있고 스테이징 단계에서 APM이나 테스트에서 확인해서 고치면 된다.

### 애그리거트 루트의 기능 구현

- 설명
	- 책에서는 Order의 OrderLine 계산 (totalAmounts 집계 산출이다.), Member의 password 변경 로직(이런건 보통 Account(계정)와 Member(회원, 또는 User)는 분리해서 관리하는 경우에는 Account(계정)에 존재해야하는 로직이다.), 또한 OrderLine의 변경로직등을 예시로 들고 있다.
	- 벨류를 만들었으면 여러 기능들은 이쪽에 분산시켜주는 것이 마땅하다. 물론 테스트 클래스는 보기 싫게 늘어나긴하지만 어차피 만들때만 시간이 들지 나중에 수정할때는 도움이 되는 녀석이니 분리를 했다면 기능도 분리해주자.

### 트랜잭션 범위

- 설명
	- 트랜잭션 범위는 작을 수록 좋다.
		- 한 개의 트랜잭션이 한 개의 테이블을 수정하는 것보다 3개의 테이블을 수정하는 것이 훨씬 성능이 떨어진다. (차례대로 Predict(where 결과 범위)에 따라 행단위 범위지정 락을 걸기 때문이다.)
	- ==**한 트랜잭션에서는 한 개의 애그리거트만 수정해야한다.**==
		- 두 개 이상의 애그리거트를 수정해야하는 경우 응용 영역의 서비스에서 두 개의 애그리거트를 수정하면 된다.
			- 한방쿼리를 사랑하는 어떤 사람들은 매우 별로라고 생각 할 수도 있다.
			- 한방쿼리는 말 그래도 select에만 사용하는게 낫고 동시에 여러 테이블을 한번에 수정하는 것은 정말 삼가해야한다.
				- 데드락을 야기할 수도 있는 방식이다.
					- 굳이 하고 싶다면 수축-확장을 거친 2 phase 락을 걸어야한다.
		- 도메인 이벤트를 사용하면 한 개의 애그리거트를 수정하면서도 동기나 비동기로 다른 애그리거트의 상태를 변경하는 코드를 작성할 수 있다.
	- 한 트랜잭션에서 복수의 애그리거트를 수정 여부의 기준
		- 팀 표준
			- 팀이나 조직의 표준에 따라 사용자 유스케이스와 관련된 응용 서비스의 기능을 한 트랜잭션으로 실행해야하는 경우가 있다.
		- 기술 제약
			- 기술적으로 이벤트 방식을 도입할 수 없는 경우 한 트랜잭션에서 복수의 애그리거트를 수정해서 일관성을 처리해야한다.
		- UI 구현의 편리
			- 운영자의 편리함을 위해 주문 목록 화면에서 여러 주문의 상태를 한 번에 변경하고 싶을 것이다.
				- 이 경우 한 트랜잭션에서 여러 주문의 애그리거트의 상태를 변경해야한다.

## 리포지터리와 애그리거트

- 설명
	- 객체의 영속성을 처리하는 리포지터리는 애그리거트 단위로 존재한다.
		- ==**만약 OrderLine을 위한 DB 테이블이 존재하더라도 이를 위한 리포지터리를 만들지 않아야한다.**==
		- Littok의 경우, 권한-자원 관계 테이블이 존재하더라도, 이에 해당하는 리포지터리는 굳이 만들지 않는 다는 것이다.
	- 애그리거트를 새로 만들면 저장소에 애그리거트를 영속화하고 애그리거트를 사용하려면 저장소에서 애그리거트를 읽어야 하므로, 리포지터리는 보통 다음의 두 메서드를 기본으로 제공한다.
		- save, findById
		- 두 메서드 외에도 다양한 조건으로 애그리거트를 검색하는 메서드나 애그리거트를 삭제하는 메서드를 추가할 수 있다.
	- 책에서는 팀 표준에 따라서 엔티티를 설계하라고 나온다.
		- DB 테이블을 그대로 모방하는 구조로 만들거면 그것에 따라가야한다고 이야기한다.
		- 문제는 이러면 책에서 이야기하는 애그리거트당 엔티티 1개는 좀 힘들 수 있다.
	- 애그리거트는 리포지터리에서 불러오는 경우 온전한 상태여야한다
		- 리포지터리에서 불러온 뒤 다시 다른 리포지터리에서 데이터를 더해야 온전히 기능이 수행가능한 형태여서는 안된다는 뜻이다.
			- 이는 NPE를 야기한다.
	- 애그리거트에서 변경된 모든 사항은 리포지터리에 반영되어야한다.

## ID를 이용한 애그리거트 참조

- 설명
	- 스프링 JPA를 사용해보면, 한 엔티티에서 다른 엔티티를 N:1 관계를 가질때, 1에 해당하는 객체를 M에 해당하는 객체가 지니게 된다.
		- ManyToOne 매핑이며 대부분의 레이어드 구조로 구성된 프로젝트에는 이렇게 되어있다.
			- ==**문제는 이렇게 구성하면 의존성이 분리가 되지 않는다.**==
			- ==**같은 애그리거트에 있는 경우에는 상관이 없지만 다른 애그리거트인 경우에는 문제가 발생한다.**==
		- 책에서는 매핑의 주인은 누구인지 지정하는 매핑 어노테이션은 완벽히 제거해야한다고 제안한다.
	- 한 객체가 다른 객체를 참조하는 것처럼 애그리거트도 다른 애그리거트를 참조한다.
		- ==**애그리거트 관리 주체는 애그리거트 루트이므로 애그리거트에서 다른 애그리거트를 참조하는 것은 다른 애그리거트의 루트를 참조하는 것과 같다.**==
	- 연관 애너테이션의 단점
		- 편한 탐색의 오용
			- 난 편한 탐색이 주는 오용문제보단 편리성이 더 혜택이 크다고 생각한다.
			- 책에서 알려주는 오용은 다른 애그리거트의 데이터를 쉽게 변경이 가능해, 이러한 경우에 데이터 일관성이 무너질 가능성의 존재해 이것에 대해 지적한다.
		- 성능에 대한 고민
			- 성능은 여러가지 대응책이 있어서 괜찮다.
		- 확장의 어려움
			- 이것이 가장 큰 문제라고 할 수 있다.
			- 여러 엔티티에 n:1로 매핑이 묶이기 시작하면 기능 추가를 하려고 하면 점차 답이 없어진다.
			- 테스트만해도 한 클래스의 10가지 동작만 정의한다고 가정하면, 이런 클래스가 50개만 있어도 필요한 테스트는 1000개에 해당한다. 최소를 따지면 그렇다. 예외 테스트를 전부 따지면 2배까지도 늘어난다.
			- 문제는 엔티티에 마구잡이로 연관을 섞는 순간, 테스트의 의존성 문제가 큰 문제로 다가오게된다.
				- 서비스를 여러개로 잘 쪼개고 오케스트레이션 서비스등으로 잘 나눈다고 해도 근본적인 문제해결이 되진 않는다.
	- 연관 애너테이션의 문제를 해소할 방법은 ID(식별자) 객체를 만드는 것이다.
		- DB 테이블에서 외래키를 참조하는 것과 동일한 방식으로 사용하면 된다. 간단하다.
	- 책에서는 주문자 객체에 Member 객체가 아닌, MemberId를 객체를 참조하게 만들고 연관 애너테이션을 제거했다.
	- ID 객체의 장점
		- 구현 복잡도가 낮아지고 응집도가 올라간다.
		- 즉시 로딩, 지연 로딩 고민이 없어진다.
			- 이건 당연히 항상 지연로딩으로 세팅을 하고 한방 쿼리가 필요한 경우에만 QueryDSL을 날리는 것이 보편적이다.
		- 한 애그리거트에서 다른 애그리거트를 수정하는 문제를 미연에 방지한다.

### ID를 이용한 참조와 조회 성능

- 설명
	- ID를 참조하는 경우 여러 애그리거트를 읽는 경우에 조회 성능이 떨어진다.
		- N+1 문제가 발생하게 된다.
	- 책에서는 ID를 사용하여 관련 객체를 함께 가져오는 일종의 조인이 필요한 쿼리를 사용해야하는 경우 조회 전용 쿼리를 사용하는 것을 권장한다.
		- 조회를 위한 별도의 DAO를 만들고 조회 메서드에 조인을 이용해 한 번의 쿼리로 필요한 데이터를 로딩한다.
		- QueryDSL이나 DataJPA를 이용해도 쉽게 구현이 가능하다.
			- 책에서는 JPQL을 직접 사용한 DAO를 만들었는데, 그냥 DATA JPA나 QueryDSL을 활용해주는 편이 좋다.
	- 애그리거트마다 다른 저장소를 사용한다면 (인증쪽은 Redis, 유저 쪽은 Rdb를 쓰는 경우는 흔하다.) 한 번의 쿼리로 관련 애그리거트를 조회할 수 없다.
		- 이런 경우에는 조회 성능을 높이기 위해 캐시를 사용하거나 조회 전용 저장소를 따로 구성한다.
		- 코드가 복잡해지긴 하지만 시스템 처리량은 높일 수 있다.
		- 한 대의 DB 장비로 대응 불가능한 수준의 트래픽 발생시 캐시나 조회 전용 저장소는 필수이다.
			- Redis, Memory DB Table 등을 의미한다.

## 애그리거트 간 집합 연관

- 설명
	- 1:N, M:N 연관은 컬렉션이 필요하다.
		- JPA을 정석으로 사용한다면 이런 연관 자체를 걸진 않고 N:1 단방향을 거는 것이 일반적이다.
	- 1:N 관계
		- Set 컬렉션으로 표현 가능하다.
			- 보통 항목에 대한 컬렉션인 경우 항목은 주로 페이징을 사용해 처리해야한다.
				- 이런 것이 골치아프니 보통 반대로 N:1 연관을 거는 것이 일반적이다.
			- 책에서도 성능 문제로 보통은 사용하지 않는다고 알려준다.
				- N:1 연관을 애용하라고 알려준다. 이게 당연히 보편적이다. 직접 개발을 해보면 깨닫게 된다.
	- M:N 관계
		- 양쪽 애그리거트에 Set 컬렉션 연관을 만들어 구현 가능하다.
			- 다만 1:N 연관처럼 성능 문제를 가지고 있다. DB상에서는 애초에 관계 테이블을 따로 만들어진 구조이다.
		- 보통은 단방향으로 정보를 원하는 것이 일반적이라 한쪽에만 Set 컬렉션으로 구현해둔다.
			- JPQL의 `member of` 연산자를 통해서 특정 항목을 걸러서 조회하는 것도 가능해진다.
		- 이런 N:M 관계는 왠만하면 피해야한다. 성능문제를 항상 야기한다.
			- 어느 단방향으로 거는 상대 참조의 항목 가지수가 적으면 괜찮겠지만 이게 아니면 일반적으로 조회성능이 무너질 가능성이 높다.
			- 단순히 조회를 한다면 N+1 문제가 무조건 발생한다.
				- JPQL에 Join fetch를 반드시 넣어줘야한다.

## 애그리거트를 팩토리로 사용하기

- 설명
	- 책에서는 애그리거트를 생성할 데이터로 다른 애그리거트를 생성해야하는 경우 펙토리 메서드를 사용하는 것을 권장한다.
		- 예제에서는 상품을 생성하는데 Store의 Id가 필요한데, 이 경우에는 Store 애그리거트에 상품 생성 펙토리 메서드를 만들어 상품 생성시 Store의 상품 생성 펙토리 메서드를 통해서 생성한다.
		- 실세계의 실제 요구사항을 그대로 반영하여 기능을 만들어낸 모습이다.
	- 만약 더 많은 데이터를 생성시 요구한다면, 해당 파라미터는 DTO로 만들고, 그 DTO와 Store의 정보를 파라미터로 받는 상품 생성 펙토리 클래스를 따로 만들어서 거기서 생성하게끔도 만들 수 있다.
		- 이러면 Store에서 상품을 생성하는 기능 자체는 그대로 유지할 수 있는 장점이 있다.
	-
