## 리눅스 기반 소켓 구조

>[!info]
> 소켓은 전화기와 흡사하다!

&emsp;&emsp;

- 설명
	- 전화기에 비유한 소켓 동작
		1. 전화기를 구매한다. (invoke socket())
		2. 구매 이후 동작으로 전화기의 역할이 달라진다.
			1. 전화기에 전화번호를 부여하는 경우 이 전화기는 서버 역할을 한다. (invoke bind())
				1. 전화기를 연결 대기 상대로 둔다. 전화가 오면 수화기를 들어 전화를 받을 준비만 한다. (invoke listen())
				2. 전화가 오는 경우 전화를 받는다. (invoke accept())
			2. 전화기를 다른 전화기에 바로 연결하는 경우 이 전화기는 클라이언트 역할을 한다.(invoke connect())
		3. 전화가 끝나면 전화기 전원을 끈다. (invoke close())
	- API 종류
		- `int socket(int domain, int type, int protocol);`
			- 역할
				- 소켓 생성
			- 특징
				- 성공시 fd, 실패시 -1 반환
		- `int bind(int socketfd, struct sockaddr *address, socklen_t addrlen);`
			- 역할
				- IP주소와 Port 번호 할당
			- 특징
				- 성공 시 0, 실패 시 -1 반환
		- `int listen(int socketfd, int backlog);`
			- 역할
				- 소켓을 연결요청 가능 상태로 변경
			- 특징
				- 성공시 0, 실패 시 -1 반환
				- 연결요청 대기상태에 두고자 하는 FD값을 넘긴다.
					- 해당 소켓은 리스닝 소켓(서버 소켓)이 된다.
				- backlog는 연결요청 대기 큐이며 대기 시킬 총 클라이언트의 수를 넣는다.
					- 이 수가 다 채워지면 나머지 요청은 거부된다.
				- ==**웹 서버의 경우 backlog는 15이상을 전달하는 것이 현명하다.**==
		- `int accept(int socktd, struct sockaddr *clientAddress, socklen_t addrlen);`
			- 역할
				- 연결요청 수락
			- 특징
				- 성공시 fd, 실패 시 -1 반환
					- 성공시 FD 값이므로 연결 요청이 수락되면 새로운 소켓(파일)이 생성된다는 의미이다.
					- **==새로 생성된 소켓은 연결 요청을 한 클라이언트 소켓과 자동으로 연결되는 소켓==**이다.
				- 첫 번째 인자는 서버의 FD 값이다.
				- 두 번째 인자는 연결요청한 클라이언트의 주소정보를 담을 변수의 주소 값이다.
					- 함수 호출 완료시 이 주소의 변수에 클라이언트 주소 정보가 담긴다.
				- 마지막인자는 클라이언트의 변수 크기이다.
					- 크기 정보를 변수에 저장한 다음 변수의 주소값을 전달한다.
					- 함수 호출 완료시 크기 정보로 채워진 변수에는 클라이언트 주소정보 길이가 바이트 단위로 계산되어 채워진다.
		- `int connect(int socktd, struct sockaddr *serverAddress, socklen_t addrlen);`
			- 역할
				- 연결 요청
			- 특징
				- 클라이언트에 의해서 함수 호출시 둘 중 한가지 상황이 되어야 함수가 반환된다.
					- 서버에 의해 연결 요청이 접수된 경우
						- 서버 연결요청 대기큐에 있다가 접속되는 경우를 의미함
					- 네트워크 단절 등의 오류 상황이 발생해서 연결 요청이 중단된 경우

## 리눅스 기반 파일 조작 구조

&emsp;&emsp;

>[!info]
> 리눅스는 모든 것이 파일이다.
> 따라서 소켓도 파일이며, 파일을 조작하는 방법을 이해해야만 소켓을 다루는 구조와 방식 이유를 알게 된다.

&emsp;&emsp;

- 설명
	- 리눅스 소켓 통신
		- 특징
			- 소켓 파일에 파일 스트림을 연결해 파일에 데이터를 읽고 작성하여 통신한다.
	- 저수준 파일 입출력
		- 특징
			- 표준에 상관없이 운영체제가 독립적으로 제공하는 파일 입출력
			- 보통 커널이 담당한다.
				- 예외적인 경우는 DBMS로 관리시스템이 알아서 파일 입출력 과정을 효율적으로 관리한다.
				- 추상적인 page 단위를 구성해 입출력한다.
			- ==**ANSI 표준에서 정의한 함수가 아니며 리눅스가 제공하는 함수이다.**==
	- 파일 디스크립터
		- 역할
			- 시스템으로부터 할당 받은 파일 또는 소켓에 부여된 정수
		- 특징
			- ==**운영체제는 파일 또는 소켓을 생성할 때마다 해당하는 파일 또는 소켓에 부여된 숫자를 하나 건네준다.**==
			- ==**그동안 입출력 대상으로 여겼던 표준 입출력 및 표준 에러 또한 파일 디스크립터를 할당한다.**==
			- ==**파일과 소켓은 생성의 과정을 거쳐야 파일 디스크립터가 할당된다.**==
			- 기초 할당 규칙에 적어둔 ==**표준 입출력 및 에러**==는 ==**따로 생성하지 않아도 프로그램이 실행되면 자동으로 할당되는 파일 디스크립터**==이다.
		- 기초 할당 규칙
			- 0
				- 표준 입력
			- 1
				- 표준 출력
			- 2
				- 표준 에러
		- 리눅스 API
			- `int open(const char *path, int flag);`
				- 역할
					- 파일 열기
				- 특징
					- 성공시 fd, 실패시 -1
					- flag에는 파일의 오픈 모드 정보를 전달한다.
				- flag 종류
					- O_CREAT
						- 역할
							- 필요하면 파일 생성
					- O_TRUNC
						- 역할
							- 기존 데이터 전부 삭제
					- O_APPEND
						- 역할
							- 기존 데이터 보존, 뒤에 이어서 저장
					- O_RDONLY
						- 역할
							- 읽기 전용으로 파일 오픈
					- O_WRONLY
						- 역할
							- 쓰기 전용으로 파일 오픈
					- O_RDWR
						- 역할
							- 읽기, 쓰기 겸용으로 파일 오픈
			- `int close(int fd);`
				- 역할
					- 파일 닫기
				- 특징
					- 성공시 0, 실패시 -1
			- `ssize_t write(int fd, const void *buf, size_t nbytes);`
				- 역할
					- 파일에 데이터 작성
				- 특징
					- 성공시 전달한 바이트 수, 실패시 -1 반환
			- ``ssize_t read(int fd, const void *buf, size_t nbytes)`
				- 역할
					- 파일 읽기
				- 특징
					- 성공시 수신한 바이트 수, 실패시 -1 반환

## Windows 기반 소켓 구조

&emsp;&emsp;

>[!info]
> TCP 클라이언트의 경우 대다수의 경우는 윈도우 환경이 대다수이다.
> 그러므로 윈도우의 API 정도는 알아두고 개발할 필요가 있다.

&emsp;&emsp;

- 설명
	- 특징
		- 소스에 winsock2 헤더를 include 해야한다.
		- ws2_32.lib 라이브러리를 링크 시켜야한다.
			- 보통 리눅스에서는 라이브러리 링크 과정을 cmake로 손수해야한다.
		- 리눅스는 소켓도 파일로 취급하지만 윈도우는 핸들을 반환하는 건 파일과 같지만, 명백히 구분하이 취급한다.
		- SOCKET이란 새로 정의된 (typedef) 자료형을 사용하며 정수형이다.
		- 리눅스와 윈도우, 두 운영체제 간의 커널 구현 방식의 차이 때문에 사용하는 구현 함수도 약간 다르다.
	- 윈속(winsock)
		- 특징
			- 프로그래밍시 WSAStartup 함수 호출로 초기화를 진행해야한다.
	- Windows API
		- `int WSAStartup(WORD wVersionRequested, LPWSDATA lpWSAData);`
			- 역할
				- 프로그램에서 요구하는 윈도우 소켓 버전을 알리고 해당 버전을 지원하는 라이브러리 초기화
			- 특징
				- wVersionRequested는 요구 윈속의 버전정보
					- 윈도우 소켓에는 여러 버전이 존재해 사용할 소켓의 버전 정보를 WORD 형으로 구성해서 전달해야함
					- 상위 8비트는 부버전, 하위 8비트는 주 버전 정보를 표시해서 전달한다.
						- 1.2라면 0x0201을 전달한다.
					- MAKEWORD() 함수를 사용하면 편하게 조립이 가능하다.
				- lpWSAData는 WSADATA라는 구조체 변수의 주소 값
					- LPWSADATA는 WSADATA의 포인터 형이다.
		- `int WSACleanup(void)`
			- 역할
				- 초기화된 윈속 라이브러리를 해제
			- 특징
				- 할당된 윈속 라이브러리가 윈도우 운영체제에 반환되며 윈속 관련 함수 호출이 불가능해진다.
				- 보통 프로그램 정상 종료시 호출된다.
					- 윈속 라이브러리가 더 이상 필요 없는 경우에 해제하면 되나 보통 오늘날의 모든 프로그램은 늘 TCP가 시작부터 연결되어 정상 종료할 때까지 살아있어야한다.
		- `SOCKET socket(int af, int type, int protocol)`
			- 역할
				- 리눅스와 동일
			- 특징
				- 성공시 소켓 핸들, 실패시 INVALID_SOCKET 반환
		- `int bind(SOCKET s, const struct sockaddr *name, int namelen);`
			- 역할
				- 리눅스와 동일
			- 특징
				- 성공 시 0, 실패 시 SOCKET_ERROR 반환
		- `int listen(SOCKET s, int backlog);`
			- 역할
				- 리눅스와 동일
			- 특징
				- 성공 시 0, 실패 시 SOCKET_ERROR 반환
		- `SOCKET accept(SOCKET s, struct sockaddr *addr, int *addrlen);`
			- 역할
				- 리눅스와 동일
			- 특징
				- 성공시 소켓 핸들, 실패시 INVALID_SOCKET 반환
		- `int connect(SOCKET s, const struct sockaddr *name, int namelen);`
			- 역할
				- 리눅스와 동일
			- 특징
				- 성공 시 0, 실패 시 SOCKET_ERROR 반환
		- `int closesocket(SOCKET s);`
			- 역할
				- 소켓을 닫는다.
			- 특징
				- 성공 시 0, 실패 시 SOCKET_ERROR 반환
				- 윈도우와 리눅스의 윈속 라이브러리 초기화 및 해제를 제외하고 또다른 차이점이다.

## 윈도우 기반 입출력 구조

&emsp;&emsp;

- 설명
	- 특징
		- 윈도우와 리눅스의 가장 큰 특징이다.
			- 리눅스는 파일 입출력 함수를 소켓과 파일 구분 없이 둘 다 사용한다.
			- 윈도우는 소켓 전용 입출력 함수를 사용한다.
	- API
		- `int send(SOCKET s, const char *buf, int len, int flags)`
			- 역할
				- 데이터를 전송
			- 특징
				- 성공 시 전송된 바이트 수, 실패 시 SOCKET_ERROR 반환
		- `int recv(SOCKET s, const char *buf, int len, int flags)`
			- 역할
				- 데이터를 수신
			- 특징
				- 성공 시 수신한 바이트 수(단, EOF 전송시 0), 실패 시 SOCKET_ERROR 반환

## 소켓의 타입

&emsp;&emsp;

- 설명
	- 특징
		- 대표적인 데이터 전송 방식은 스트림과 데이터 블록방식이 있다.
	- 종류
		- 연결지향형 소켓(SOCK_STREAM)
			- 특징
				- 중간에 데이터 소멸 없이 목적지로 전송
				- 전송 순서대로 데이터 수신
				- 전송되는 데이터의 경계가 없음
					- 이는 쓰기 횟수와 읽기 횟수는 큰 의미가 없다는 의미이다.
					- ==**전송 측에서 전송되는 데이터가 얼마이건 받는건 받는 측 마음대로 진행된다는 뜻**==
				- 소켓에 들어있는 버퍼에 수신된 데이터는 읽혀진 만큼 비워지고 놓여진다.
					- 즉, 덜 읽혀진 데이터가 버퍼가 꽉채워저 더이상 수신이 못할 지언정 ==**읽혀질 때까진 데이터가 그대로 존재한다.**==
					- 따라서 ==**연결 지향형 소켓은 수신 소켓의 상태를 봐가며 데이터를 전송한다는 의미이다.**==
					- 데이터 손실은 특별한 경우가 아니면 일어나진 않는다.
				- 반드시 1:1 연결이여야 한다.
				- 소켓 생성시 두 번째 인자로 SOCK_STREAM, 마지막 인자로 IPROTO_TCP를 넣어준다.
		- 비 연결지향형 소켓(SOCK_DGRAM)
			- 특징
				- 전송된 순서에 상관없이 가장 빠른 전송을 지향함
				- 데이터 손실 우려가 있고 파손 우려도 있다.
				- 전송되는 데이터의 경계가 존재한다.
					- 전송되는 데이터가 2개면 받는 측도 2개를 받아야는 특성을 의미함
				- 한번에 전송가능한 데이터의 크기가 제한된다.
				- 소켓 생성시 두 번째 인자로 SOCK_DGRAM, 마지막 인자로 IPROTO_UDP를 넣어준다.

## 윈도우 socket 함수

&emsp;&emsp;

- 설명
	- 특징
		- 윈도우에서 지정한 자료형들은 지켜주는 것이 좋다.

## 네트워크 바이트 순서와 인터넷 주소 변환

&emsp;&emsp;

- 설명
	- 특징
		- CPU에 따라 정수를 저장하는 방식은 (바이트로 저장하는 방식) 달라진다.
			- 특히 바이트를 저장하는 순서가 다르다.
		- 만약 데이터 메모리 저장 방식이 다른 pc가 데이터를 전달하게 된다면, 서로 거꾸로 바이트를 저장해 이해하게된다!!
			- 따라서 ==**CPU에 맞게 IP주소와 PORT 번호를 저장하기 위해 통일한 바이트 순서를 네트워크 바이트 순서**==라고 부른다.
			- ==**이 순서는 빅 엔디안 방식에 통일되어 있다.**==
			- ==**따라서 대부분 상용 컴퓨터에서 개발하는 우리들은 바이트 순서를 변환해야한다.**==
			- 이 순서 변환은 ==**sockaddr_in 구조체를 초기화할 때 빼고 나머지는 자동으로 변환**==하여 수신하기 때문에 ==**sockaddr_in 구조체를 초기화할 때만 신경써서 변환**==해주면 된다.
	- 데이터 메모리 저장 방식
		- 빅 엔디안
			- 특징
				- 상위 바이트 값을 작은 번지수에 저장하는 방식
				- 0x123456을 저장 할때, 0x20은 0x12, 0x21은 0x34, 0x22는 0x56 이렇게 저장되는 형식이다.
				- 최상위가 0x12, 최하위는 0x56이다.
				- 주로 메인프레임이나 일부 전자기기용 cpu에나 들어가는 방식이며 만질일은 거의 없다.
		- 리틀 엔디안
			- 특징 상위 바이트 값을 큰 번지수에 저장하는 방식
				- 56, 34, 12 순으로 저장되는 방식
			- 상용 CPU들이 가장 많이 사용하는 방식이다.
	- 바이트 순서 변환 API
		- `unsigned short htons(unsigned short)`
			- 역할
				- short 형 호스트(h)의 순서를 네트워크 바이트(n) 순서로 변환한다
			- 특징
				- PORT 번호 변환에 사용된다.
		- `unsigned short ntohs(unsigned short)`
			- 역할
				- short 형 네트워크 바이트(n)의 순서를 호스트(h) 순서로 변환한다
			- 특징
				- PORT 번호 변환에 사용된다.
		- `unsigned long htonl(unsigned long)`
			- 역할
				- long 형 호스트(h)의 순서를 네트워크 바이트(n) 순서로 변환한다
			- 특징
				- IP 주소 변환에 사용된다.
		- `unsigned long ntohl(unsigned long)`
			- 역할
				- long 형 네트워크 바이트(n)의 순서를 호스트(h) 순서로 변환한다
			- 특징
				- IP 주소 변환에 사용된다.

## 인터넷 주소의 초기화와 할당

&emsp;&emsp;

- 설명
	- IP주소 초기화 API
		- `in_addr_t inet_addr(const char *string);`
			- 역할
				- IP 주소 문자열을 네트워크 바이트 순서에 맞게 변환하여 반환
			- 특징
				- 성공 시 빅 엔디안으로 변환된 32비트 정수 값, 실패 시 INADDR_NONE 반환
				- 유효하지 않은 IP 주소 검출도 진행한다.
					- 따라서 굳이 검출 로직을 짤 필요는 없다.
		- `int inet_aton(const char* string, struct in_addr *addr);`
			- 역할
				- inet_addr과 기능은 동일하지만 in_addr를 이용한다
			- 특징
				- 성공 시 1, 실패 시 0을 반환한다.
				- 마지막 파라미터를 이용해 이 포인터에 변환된 데이터를 담는 방식이다.
				- 당연하지만 성능과 범용성은 이게 더 낫기 때문에 inet_addr보다 inet_aton보다 더 많이 사용된다. (생성+복사와 참조+대입 차이)
				- ==**윈도우에는 이 함수가 없다.**==
		- `char *inet_ntoa(struct in_addr addr);`
			- 역할
				- in_addr를 기반으로 문자열로 변환한다.
			- 특징
				- 성공 시 변환된 문자열의 주소 값, 실패 시 -1을 반환
				- 참고로 ==**이 문자열은 곧 바로 호출한 스택 프레임에서 다른 변수에 저장하지 않는 이상 구문을 벗어나는 순간 문자열은 메모리에서 바로 삭제되므로 주의해야한다.**==
	- 초기화 특징
		- memset으로 sockaddr_in을 0으로 초기화 해야한다.
			- 위에서 설명했듯 마지막 데이터 공간인 sin_zero를 0으로 채워넣기 위해 필요한 작업이다.
			- ==**이게 bind()의 필수요구사항이다.**==
		- IP주소와 포트번호는 프로그램의 외부에서 주입가능하게 만드는 것이 현명하다.
			- 서버 프로그램의 경우 INADDR_ANY를 보통 사용하는 것이 보편적이다.
				- ==**NIC가 두 개 이상인 컴퓨터**==의 경우에도 ==**PORT 번호만 일치하면 받아들일 수 있는 방식**==이라 가장 보편적이다.
					- ==**참고로 이것이 bind때마다 굳이 매번 주소를 넣어 초기화하는 이유**==이다.
		- 윈도우의 경우 WSAStringToAddress와 WSAAddressToString이 있다
			- 장점
				- 다양한 프로토콜에 적용이 가능하다.
			- 단점
				- 이식성이 떨어진다.
					- 이 문제점이 커서 잘 사용안한다.

## Iterative 기반 서버, 클라이언트 구현

&emsp;&emsp;

- 설명
	- 특징
		- 지금 구현한 서버와는 달리 더 listen 이후 accept를 무한 반복하는 구조로 만든 서버이다.
			- 그래서 반복적인 구조의 서버, 클라이언트라는 의미이다.
		- 이런 구조여도 현재 서버는 한 번에 한 클라이언트만 서비스 제공이 가능하다.
			- 프로세스와 스레드를 배워야 동시 요청 처리가 가능한 서버를 구현할 수 있다.
	- 예시 단순 에코 클라이언트 
		- 문제점
			- write, read는 1024만큼의 양만 한번씩 입출력을 진행한다.
				- 즉, 이 분량의 문자열만 입출력이 진행되는 구조이다.
					- write를 하고 쪼개진 fragments가 전부 read 되지 못하는 문제도 있을 수 있다.
						- 허나 이건 재전송과정을 거치기 때문에 괜찮다.
			- 단순 에코 서버 연관된 문제점은 클라이언트는 read를 한번만 하기 때문에 문자열 데이터를 한방에 전송받기를 원하고 있는 것이 문제이다.
				- 물론 시간이 좀 지나서 read를 호출하면 한번에 받을 수는 있지만 그러지 못할 확률을 생각해야한다.
		- 해결 방법
			- read를 수신할 데이터의 크기 만큼 반복해서 호출하여 버퍼에 쌓으면 된다.
				- 단, 대다수의 경우 이 수신할 데이터의 크기를 알 수가 없다.

## TCP 소켓 입출력 버퍼와 Flow Control

&emsp;&emsp;

- 설명
	- 특징
		- TCP는 입력 버퍼와 출력 버퍼가 존재해 상황에 맞게 적절히 버퍼를 이용해 데이터를 전달한다.
			- write를 하면 출력 버퍼에 데이터가 들어간다.
				- 이것이 상대방의 입력버퍼로 전송된다.
				- ==**이 함수가 반환되는 시점은 출력버퍼가 수신측 입력버퍼로 이동이 완료되는 시점이다.**==
			- read를 하면 입력 버퍼에서 데이터를 읽어온다.
		- 입출력 버퍼는 TCP 소켓 각각에 별도로 존재한다.
		- 입출력 버퍼는 소켓 자동 생성시 생성된다.
		- 소켓을 닫아도 출력버퍼에 남아있는 데이터는 계속해서 전송을 한다.
		- 소켓을 닫으면 입력버퍼에 있는 데이터는 소멸된다.
		- 슬라이딩 윈도우라는 프로토콜이 존재해 입력버퍼가 작아도 이 이상의 데이터는 전송되지 않는다.
	- 동작 원리
		- 소켓 연결
			- 특징
				- 흔히, 이 동작 과정을 3 way handshake라고 부른다.
				- SYN
					- 연결 요청
				- ACK
					- 수신 확인
					- ==**상대방이 어디서부터 메시지를 전달해주길 바라는지 순서번호**==를 넣는다.
					- 만약 수신 버퍼가 작아서 덜 받았다면 어디서부터 이어서 넘겨주길 원하는지에 대한 순서번호를 넘기면 된다.
						- SEQ가 1000이고 100을 받아야하는데 입력버퍼가 작아서 20만 받았다면
							- ACK에 1081을 담아 응답한다.
					- 연결시에는 수신을 확인 했는지 데이터를 담아서 보내 연결 요청 상호 확인의사를 전달하는 용도로 사용된다.
				- SEQ
					- 데이터 동기화 메시지
					- 수신 확인 이후 상대방에게 ==**이 수신 이후에는 데이터를 어디서부터 이어서 보낼 것인지 에 대한 순서번호**==를 넣는다.
			- 과정
				1. 연결을 원하는 호스트 A가 호스트 B에게 SYN 요청을 보낸다.
					1. SEQ에는 호스트 A가 B에게 지금 보내는 요청 메시지를 잘 받았다면, ACK에 그 다음 번호 + 1을 담아서 보내길 바라며 순서번호를 부여한다.
				2. 호스트 B측은 SYN + ACK을 함께 보낸다.
					1. SEQ에는 호스트 B가 지금 보내는 메시지를 잘 받았다면 다음 메시지의 ACK에 그 다음 번호 + 1을 담아서 보내길 바라며 순서번호를 부여한다.
					2. ACK에는 호스트 A의 SYN 요청을 잘 받았으니 수신 데이터의 SEQ + 1 값을 넣어 수신 확인의 의사를 담는다.
				3. 호스트 A측은 ACK를 보낸다.
					1. SEQ에는 호스트 B가 보낸 ACK 1001을 담는다.
					2. ACK에는 호스트의 SYN + ACK 메시지를 잘 받았다는 의사 표현으로 수신 데이터의 SEQ + 1 값을 넣어 수신확인의 의사를 담는다.
				4. 이제 호스트 A와 B의 데이터 송수신을 위한 연결 작업은 마무리되었다.
		- 소켓과 데이터 송수신
			- 특징
				- 데이터를 단순히 주고 받는 과정
			- 과정
				1. 데이터를 전송하는 A와 데이터를 수신 받는 B가 있다.
				2. A는 SEQ에 보낼 데이터의 패킷 번호를 부여하고 데이터를 담아서 전송한다.
				3. B는 수신 확인 메시지를 위해 ACK와 SEQ값 + 전송된 바이트 크기 + 1을 담아서 전달한다.
					1. 이 데이터는 어디까지 수신되었는지, 수신측의 입력 버퍼를 확인하고 조절하여 보내기 위해 반드시 필요한 메시지이다.
				4. 만약 수신 오류가 발생해 time out이 발생된 경우
					1. 송신 측은 수신 확인 메시지를 받지 못했기 때문에 다시 메시지를 전송한다.
		- 소켓 연결 해제
			- 특징
				- 흔히, 이 동작 과정을 4 way handshake라고 부른다.
				- 연결 해제 수신 측(서버)이 메시지를 연결 해제 요청 확인 메시지을 연결 해제 요청을 두 번 나눠서 보낸다.
					- 두 번 나눠서 보내는 이유
						- half-close 지원을 위함인데 만약 ==**클라이언트가 연결 해지를 요청해서 더이상 메시지를 서버에 전달하지 않는다고 해도 서버에서 더 보낼 메시지가 존재한다면 이를 전달하고 나서 이어서 연결 해제 요청을 이어서 보내기 위함**==이다.
			- 과정
				1. 연결 해제를 요청하는 A(클라이언트)가 B(서버)에게 FIN 메시지를 보낸다.
					1. SEQ에는 확인의사 표현을 받기 위한 용도로 순서번호를 담아서 보낸다.
				2. B는 ACK 메시지를 보내며 ACK에 SEQ + 1를 담아 수신확인 요청을 보낸다.
					1. SEQ에는 확인 의사 표현을 받기 위한 용도로 순서번호를 담아서 보낸다.
				3. B는 다시 한번 FIN 메시지를 추가로 보낸다. 
					1. ACK는 2번 과정의 데이터와 동일한 데이터를 담는다.
						1. 이는 A의 연결 해제 요청 메시지를 이어서 수신확인차 보내는 것을 의미한다.
					2. 이때 SEQ에는 본인이 보낸 SEQ + 1을 담는다.
				4. A는 ACK 메시지를 보내며 SEQ에는 B의 ACK 데이터를 담는다.
					1. ACK는 B의 SEQ + 1을 담아 보낸다.

## UDP 소켓

&emsp;&emsp;

- 설명
	- 특징
		- Flow Control 과정이 존재하지 않는다.
		- 신뢰성이 현저히 떨어진다.
		- 호스트로 수신된 패킷을 PORT 정보를 참조해 UDP 소켓까지 빠르게 전달하는 것이 핵심 역할이다.
		- 주로 실시간 비디오, 동영상 데이터 전송과 같은 상대적으로 빠르게 전달되어야하지만 데이터의 유실을 굳이 걱정할 필요없는 위주의 프로토콜에서 자주 사용된다.
		- 데이터 전송마다 목적지의 주소정보를 별도로 추가하여 전달해야한다.
			- 우편물에 주소정보를 써 넣는 것과 비슷하다.
	- API
		- `ssize_t sendto(int sockFd, void *buff, size_t nbytes, int flags, struct sockaddr *to, socklen_t addrlen)`
			- 역할
				- 바이트를 전달함
			- 특징
				- 성공 시 전송된 바이트의 수, 실패 시 -1 반환
				- buff는 전송할 데이터의 주소, nbytes는 전송될 데이터의 크기이다.
				- to에 목적지 주소 정보가 담긴 구조체 변수의 주소값을 넣는다.
				- sendto 사용전 bind를 호출해 직접 ip와 port를 지정해줄 수도 있고 bind를 호출하지 않았다면 자동으로 할당된다.
					- 따라서 ==**UDP 클라이언트 입장에서는 따로 주소정보를 할당하는 과정이 필요하지 않다.**==
				- ==**입력함수와 출력함수의 횟수가 완벽하게 일치해야 송신된 데이터를 전부 수신할 수 있다.**==
					- TCP는 출력을 여러번 해도 입력 함수 호출 한번으로 수신받을 수 있지만 UDP는 그렇지 않다.
				- 소켓의 종류는 connected 소켓, unconnected 소켓으로 나뉜다.
					- connected 소켓
						- 목적지 정보가 등록되어있는 소켓
						- 하나의 호스트와 오랜기간 통신해야하는 경우 이 소켓을 유지하는 것이 효율적이다.
							- 전송 과정중 2가지 과정을 줄이면 전송과정의 1/3을 줄인 것과 비슷한 효과를 볼 수 있다.
						- ==**`sockaddr`을 초기화 한 다음 connect를 호출하면 세팅된다.**==
						- ==**등록을 하면 read, write 함수로도 데이터 송수신이 가능해진다.**==
					- unconnected 소켓
						- 목적지 정보가 등록되어있지 않은 소켓
			- 전송 과정
				1. UDP 소켓에 목적지의 IP와 PORT 번호 등록 (connected)
				2. 데이터 전송
				3. UDP 소켓에 등록된 목적지 정보 삭제 (unconnected)
		- `ssize_t recvfrom(int sockFd, void *buff, size_t nbytes, int flags, struct sockaddr *to, socklen_t addrlen)`
			- 역할
				- 바이트를 수신함
			- 특징
				- 성공시 수신한 바이트의 수, 실패 시 -1 반환
				- 모든 파라미터는 sendto와 정반대의 파라미터들이라고 보면 된다.

## Half-Close: Graceful Shutdown

&emsp;&emsp;

>[!info] Definition
> 소켓을 반만 닫아 연결을 종료하는 방식


&emsp;&emsp;

- 설명
	- 기존 종료의 문제점
		- 한쪽에서 일방적인 종료를 진행하면 전송 도중 도착해야할 중요한 데이터가 유실될 수 있다.
		- 전송만 가능하던지, 수신만 가능하던지 둘 중 하나의 상황을 만들면 우아하게 종료시킬 수 있다.
		- 클라이언트 측에서 파일의 끝을 알 수 없으므로 수신만 하다가 블로킹 상태에 빠질 수도 있다.
			- 기존 close()을 호출하면 EOF는 전송되지만 상대방이 전송하는 데이터를 더 이상 수신할 수 없는 상태가 된다.
	- Half-Close 특징
		- 소켓을 통해 ==**상호간 데이터 송수신이 가능한 상태**==가 되는데 이것을 ==**스트림이 형성된 상태**==라고 한다.
		- 스트림은 ==**양방향을 위해 소켓 하나당 입력과 출력, 총 2개의 스트림이 형성**==된다.
		- ==**Half-Close는 2개의 스트림중 하나의 스트림을 닫는 것**==을 의미한다.
		- 서버측에서 출력 스트림을 Half-Close를 하여 더 이상 보낼 데이터가 없음을 표현하면, 클라이언트 측에서는 마지막으로 EOF를 확인했다는 의미의 메시지를 보내 종료하면 된다.
			- 이러면 종료를 해도 ==**제대로 종료를 인지했는지**==, 혹은 더 ==**클라이언트 측에서 보낼 데이터가 남아 있는지 등을 확인**==할 수 있다.
	- API
		- `int shutdown(int sock, int howto);`
			- 역할
				- 스트림의 일부를 종료
			- 특징
				- 성공 시0, 실패 시 -1 반환
				- Fd와 종료 방법에 대한 정보를 파라미터로 전달함
			- 두번째 인자의 종류
				- SHUT_RD
					- 입력 스트림 종료
					- 데이터 수신 불가 상태로 전환
				- SHUT_WD
					- 출력 스트림 종료
					- 데이터 송신 불가 상태로 전환
				- SHUT_RDWR
					- 입력 출력 스트림 종료
					- 데이터 송수신 불가 상태로 전환

## DNS

&emsp;&emsp;

>[!info]
> DNS는 UDP를 쓰는 대표적인 시스템이다.

&emsp;&emsp;

- 설명
	- DNS 구조
		1. 로컬 영역에서 기본 DNS 서버에 쿼리를 함
		2. 만약 찾아봐도 계속 없으면 Root DNS 서버까지 쿼리를 타고 올라감
		3. Root DNS는 해당 도메인의 IP를 아는 Server를 알고 있으므로 이를 통해서 목적지 IP를 알아낸다.
	- 특징
		- `ping www.google.com` 이 명령어를 cmd에 입력하면 어디에서나 ip를 찾을 수 있다.
			- ip는 자주 변동된다.
		- `nslookup` 이 명령어를 cmd에 입력하면 기본 DNS 서버의 주소를 알 수 있다.
			- 보통 ISP의 DNS 서버, 혹은 구글 DNS 서버가 가장 대표적인 기본 DNS 서버이다.
			- 리눅스의 경우에는 로컬에 있는 host 파일을 먼저 살펴보고 없으면 기본 DNS 서버에 물어본다.
		- hostent 구조체 구조
			- `char *h_name`
				- 역할
					- 공식 이름
				- 특징
					- 회사명이 아닌 경우가 많다.
			- `char **h_aliases`
				- 역할
					- 별칭 리스트
				- 특징
					- 다른 이름으로도 등록된 경우를 호환해준다.
			- `int h_addrtype`
				- 역할
					- 호스트 주소 타입
				- 특징
					- 반환된 IP 타입이 v4인지 v6인지 알려준다.
			- `int h_length`
				- 역할
					- 주소 길이
			- `char **h_addr_list`
				- 역할
					- 주소 리스트
				- 특징
					- 실제로는 ==**문자열 배열이 아닌 in_addr 구조체 변수 위치를 가리키는 주소값**==을 가지고 있다.
					- ==**v4와 v6를 둘 다 호환하기 위해 in_addr 구조체 배열이 아닌 char* 배열을 제공하는 것이다.**==
	- API
		- `struct hostent *gethostbyname(const char *hostname);`
			- 역할
				- 문자열 형태의 도메인 이름을 통해 IP 주소 정보 조회
			- 특징
				- 성공 시 hostent 구조체 변수의 주소 값, 실패 시 NULL 포인터 변환
		- `struct hostent *gethostbyname(const char *addr, socklen_t len, int family);`
			- 역할
				- IP 주소 문자열을 이용해 도메인 정보 조회
			- 특징
				- 성공 시 hostent 구조체 변수의 주소 값, 실패 시 null 포인터 반환

## 소켓 옵션

&emsp;&emsp;

- 설명
	- 가장 많이 사용하는 옵션
		- Protocol Level
			- SOL_SOCKET
				- 옵션 종류
					- Get, Set 가능
						- SO_SNDBUF
							- 역할
								- 입력 버퍼 크기와 관련된 옵션
							- 특징
								- 임의로 변경도 가능
						- SO_RCVBUF
							- 역할
								- 입력 버퍼 크기와 관련된 옵션
							- 특징
								- 임의로 변경도 가능
						- SO_REUSEADDR
							- 특징
								- ==**서버측 소켓을 먼저 닫게 되면 다시 실행할 때 3분정도 동일한 번호를 쓸 수 없다.**==
								- 어떤 소켓이든 닫게되면 일정 시간 Time-wait 상태를 유지한다.
									- 이유는 마지막 ACK 메시지를 보낼 때 ==**중간에 소멸되는 경우, 상대방 호스트는 영원히 FIN을 재전송하기 때문에**== 이러한 경우를 방지하고자 존재함
									- 이 문제 때문에 서버를 내렸다가 다시 올릴때 무조건 3분씩 기다린다면 이 3분은 고정된 고장시간이 되고 만다.
									- 이때 ==**SO_REUSEADDR 상태가 0이면 Time-wait라는 의미이며 이걸 다시 1로 변경해주면 재사용 가능한 port**==라는 것을 의미한다.
						- SO_KEEPALIVE
						- SO_BROADCAST
						- SO_DONTROUTE
						- SO_OOBINLINE
					- Get 가능, Set 불가능
						- SO_ERROR
						- SO_TYPE
			- IPPROTO_IP
				- 옵션 종류
					- Get, Set 가능
						- IP_TOS
						- IP_TTL
						- IP_MULTICAST_TTL
						- IP_MULTICAST_LOOP
						- IP_MULTICAST_IF
			- IPPROTO_TCP
				- 옵션 종류
					- Get, Set 가능
						- TCP_KEEPALIVE
						- TCP_NODELAY
							- 특징
								- TCP는 기본적으로 ==**Nagle 알고리즘**==을 사용한다.
									- ==**작은 패킷을 보내보고 응답이 있으면 최대한 큰 패킷으로 전송하는 방식이다(응답을 매번 확인)**==.
									- 이 ==**알고리즘 옵션을 끄면 ACK의 수신의 상관없이 출력버퍼가 채워지면 바로 전송한다.**==
									- Nagle 알고리즘은 파일 전송에는 좋지 않다.
										- 옵션을 끄면 ACK를 기다리지 않고 지속적으로 파일 퍼버가 채워지기만 하면 바로바로 전송하기 때문
								- TCP_NODELAY에 1을 넣으면 Nagle 알고리즘이 꺼진다. (노 딜레이로 전송하는 옵션을 켜는 것이니 Nagle 알고리즘을 끄는 것과 마찬가지)
						- TCP_MAXSEG
	- API
		- `int getsocket(int sock, int level, int optname, void *optval, socklen_t *optlen);`
			- 역할
				- 소켓의 옵션 설정 상태를 조회
			- 특징
				- 성공 시 0, 실패 시 -1 반환
		- `int setsockopt(int sock, int optname, const void *optval, socklen_t optlen);`
			- 역할
				- 소켓의 옵션 설정을 세팅
			- 특징
				- 성공 시 0, 실패 시 -1 반환

## 프로세스의 이해

&emsp;&emsp;

- 설명
	- 다중 접속 서버의 종류
		- 멀티 프로세스 서버
			- 특징
				- 다수의 프로세스를 생성하는 방식으로 서비스를 제공
		- 멀티 플렉싱 서버
			- 특징
				- 입출력 대상을 묶어서 관리하는 방식으로 서비스 제공
		- 멀티 스레딩 서버
			- 특징
				- 클라이언트의 수만큼 스레드를 생성하는 방식으로 서비스 제공
	- 프로세스 정의
		- 메모리 공간을 차지한 상태에서 실행중인 프로그램
	- 프로세스 특징
		- 프로그램 흐름의 기본 단위
		- 멀티 프로세스 서버는 다른 프로세스를 실행하는 것이 아닌 본인 프로세스에서 자식 프로세스를 생성하는 방식으로 진행함
		- 코어의 개수를 넘어서는 프로세스를 실행하면 프로세스 별로 코어에 할당되는 시간이 나뉜다.
		- 생성시 OS로부터 ID를 할당 받으며 이를 프로세스 ID (이하 PID)라고 부른다.
		- ps 명령으로 실행중인 프로세스를 볼 수 있다.
	- 좀비 프로세스 생성 방법
		- fork 함수로 생성된 자식 프로세스가 종료되는 상황 두 가지
			- 인자를 전달하면서 exit 호출
			- main 함수에서 return 문을 실행하면서 값을 반환
		- OS는 ==**exit함수로 전달되는 인자 값 + main 함수에서 return 문으로 반환되는 값**== 모두 운영체제로 전달된다.
			- 이 ==**값은 자식 프로세스를 생성한 부모 프로세스에게 전달될 때까지 자식 프로세스를 소멸시키지 않는다**==.
			- 이 상황에 놓인 프로세스가 좀비 프로세스이다.
			- ==**OS가 자식 프로세스를 좀비 프로세스로 만드는 주체**==이다.
	- 부모 프로세스에게 값을 전달하는 방법
		- ==**부모가 직접 자식에게 적극적으로 값을 요청**==해야한다.
			- ==**요청하지 않으면 자식 프로세스는 좀비로 남게 된다.**==
	- Background Processing
		- 콘솔 창에서 명령어의 실행을 뒤에서 진행하도록 하는 방식
			- `./<프로그램 명> &`
			- & 연산자가 프로그램 실행을 백그라운드에서 진행되도록 한다.
	- API
		- `pid_t fork(void);`
			- 역할
				- 프로세스 생성
			- 특징
				- 성공 시 PID 반환, 실패 시 -1 반환
				- 호출한 프로세스의 복사본을 생성한다.
				- 완전히 동일한 프로세스로 ==**메모리 영역까지 동일하게 복사**==한다.
				- ==**fork 함수 실행 이후로부터 자식 프로세스의 실행흐름이 진행된다.**==
					- 프로그래밍 시 fork() 함수 반환값을 기준으로 흐름을 분기해야한다.
						- 부모 프로세스의 fork 함수 반환 값은 자식 프로세스의 pid
						- 자식 프로세스의 fork 함수 반환 값은 0
		- `pid_t wait(int *statloc);`
			- 역할
				- 이미 종료된 자식 프로세스가 존재한다면 자식 프로세스가 종료되면서 전달한 값이 매개변수로 전달된 주소의 변수에 저장된다.
			- 특징
				- 즉, 부모 프로세스가 본인의 자식 프로세스의 종료여부를 모르기때문에 직접 호출하는 함수이다.
				- ==**종료된 자식 프로세스가 없다면 자식 프로세스가 종료될 때까지 블로킹 상태에 놓이게 된다.**==
		- `pid_t waitpid(pid_t pid, int *statloc, int options);`
			- 역할
				- 특정 종료된 자식 프로세스가 전달한 값이 매개변수로 전달된 주소의 변수에 저장된다.
			- 특징
				- 첫번째 파라미터로 자식의 프로세스 ID를 넘겨 결과 값을 받을 수 있고 -1을 넘기면 wait() 처럼 임의의 자식 프로세스가 종료되어 결과 값을 받을 때까지 기다린다.
				- 두번째 파라미터로 매개변수 값을 받아올 참조 변수값을 넣어준다.
				- ==**sys/wait.h에 선언된 상수 WNOHANG을 인자로 전달하면 종료된 자식 프로세스가 존재하지 않아도 블로킹 상태에 있지 않고, 0을 인자로 반환하면서 함수를 빠져나온다.**==
		- WIFEXITED
			- 역할
				- 자식 프로세스가 정상 종료한 경우 참을 반환하는 매크로 함수
			- 특징
				- wait() 함수의 반환값을 파라미터로 받는다.
				- wait() 함수 결과 값의 분리 과정에 필요한 매크로 함수
		- WEXITSTATUS
			- 역할
				- 자식 프로세스의 전달 값을 반환한다.
			- 특징
				- wait() 함수의 반환값을 파라미터로 받는다.
				- wait() 함수 결과 값의 분리 과정에 필요한 매크로 함수

## 시그널 핸들링

&emsp;&emsp;

- 설명
	- 시그널 정의
		- 운영체제가 프로세스에게 특정 상황이 발생했음을 전달하는 메시지
	- 시그널 등록 정의
		- 프로세스가 운영체제에게 프로세스 본인의 자식 프로세스가 종료되면, 특정 함수를 호출하도록 등록하는 것
	- 시그널 특징
		- ==**시그널 발생시 sleep 함수의 호출로 블로킹에 있던 프로세스가 깨어난다.**==
			- 시그널 발생시 시그널에 해당하는 ==**시그널 핸들러 호출을 위해서**== sleep 함수의 호출로 블로킹 상태에 있던 프로세스는 깨어난다.
	- API
		- `void (*signal(int signo, void (*func)(int)))(int);`
			- 역할
				- 시그널 발생시 호출되도록 이전에 등록된 함수의 포인터를 반환한다.
			- 특징
				- 함수 이름
					- signal
				- 매개변수 선언
					- `int signo`
						- 정수형 파라미터를 넘겨 특정 상황에 대한 정보를 넘긴다.
					- `void (*func)(int))`
						- 특정 상황에서 호출될 함수의 포인터 값을 넘긴다.
				- 반환형
					- 매개 변수 형이 int이고 반환형이 void인 함수 포인터
				- 첫번째 매개변수에 있는 특정 상황이 발생하면, 두번째 파라미터로 넘긴 함수가 실행된다.
		- `unsigned int alarm(unsigned int seconds);`
			- 역할
				- 인자로 보낸 시간 초 이후 SIGALRM이 발생하도록 예약
			- 특징
				- 0 또는 SIGALRM 시그널이 발생하기 까지 남아있는 시간을 초 단위로 반환
				- 양의 정수를 인자로 보내면, 양의 정수 만큼의 초 시간이 지나면, SIGALRM이 발생
				- 0을 인자로 전달하면, 이전에 설정된 SIGALRM 예약이 취소
				- 시그널 발생시 호출될 함수를 지정해주어야 프로세스가 그냥 종료되지 않는다.
		- `int sigaction(int signo, const struct sigaction *act, struct sigaction *oldact);`
			- 역할
				- signal과 같은 역할을 하는 함수
			- 특징
				- 마지막 인자로 이전에 등록했던 함수 포인터를 얻는데 사용되는 인자, 필요 없으면 0을 전달하면 된다.
	- sigaction 구조체
		- `void (*sa_handler)(int)`
			- 역할
				- 시그널 핸들러의 함수 포인터 값
		- `sigset_t sa_mask;`
			- 역할
				- 시그널 관련 옵션 및 특성의 지정에 사용되는 변수
			- 특징
				- 모든 비트를 0으로 초기화한다.
				- sigemptyset() 함수를 사용해 0으로 초기화한다.
		- `int sa_flags;`
			- 역할
				- 시그널 관련 옵션 및 특성의 지정에 사용되는 변수
			- 특징
				- 0으로 초기화한다.
	- 시그널 상수
		- SIGALRM
			- alarm 함수 호출을 통해 등록된 시간이 된 상황
		- SIGINT
			- CTRL + C가 입력된 상황
		- SIGCHLD
			- 자식 프로세스가 종료된 상황

		- 에코 서버는 accept 함수 호출을 통해 연결 요청을 수락하고 이때 얻은 fd를 자식 프로세스를 생성해서 넘겨준다.

		- fork()를 호출하면 하나의 소켓에 두개의 fd가 할당된다.
			- 따라서 ==**자식 프로세스에서는 서버의 fd을 닫아야하고 부모 프로세스에서는 클라이언트의 fd을 닫아야한다.**==

## 프로세스간 통신 기본 개념

&emsp;&emsp;

- 설명
	- 특징
		- ==**프로세스간 그 어떠한 메모리 공간을 공유하지 않기 때문**==에 ==**특정한 메모리 공간을 마련해 이를 통해 통신**==을 해야한다.
	- 프로세스간 통신
		- 변수를 통해 자신의 상태를 전달하면 상대 프로세스는 그 변수를 통해 상태를 전달 받는다.
		- IPC 공유 메모리 영역을 새로 만들면 접근이 가능하지만, PIPE라는 전통적인 방법도 존재한다.
		- 다만 일반 PIPE는 부모-자식 관계의 프로세스만 가능하다, 그러나 지명 PIPE를 사용하면 또 그렇지도 않는다.

### 파이프 기반 프로세스간 통신


&emsp;&emsp;

- 설명
	- API
		- `int pipe(int filedes[2]);`
			- 역할
				- 파이프를 생성한다.
			- 특징
				- 성공시 0, 실패시 -1을 반환한다.
				- 0번 인덱스는 출구(읽기), 1번 인덱스는 입구(쓰기)이다.
				- 길이가 2인 배열을 인자로 넘겨야한다.
	- 특징
		- 부모 프로세스 혼자서 파이프에 데이터를 넣고 빼는 것도 가능은 하다.
		- 부모 프로세스의 목적은 자식 프로세스와의 데이터 송수신이므로 ==**입구 또는 출구에 해당하는 FD**==를 전달해야하는데 ==**부모의 모든 영역을 복사해 자식 프로세스에 상속하는 fork()의 특성을 이용**==하면된다.
	- 양방향 파이프 구현시 매우 주의해야하는 점
		- 데이터 전달시 먼저 가져가는 프로세스에게 전달된다.
			- 들어간 데이터는 소유권이 없어 read()를 먼저 호출한 프로세스가 가져간다.
			- 따라서 파이프는 2개를 만들어야 이런일이 없다.
			- 이것이 바로 생산자-소비자 문제라고 부른다.

## IO 멀티 플렉싱 기반의 서버

&emsp;&emsp;

- 설명
	- 멀티프로세스 서버의 단점과 대안
		- 프로세스를 매번 생성하면 많은 메모리 공간을 부모 프로세스 만큼 할당해줘야한다.
			- 부모 프로세스의 크기가 크면 클수록 클라이언트와의 연결이 많을수록 엄청나게 많은 메모리를 할당하게 되는 구조이다.
	- 멀티 플렉싱
		- 서버 프로세스가 손을 든 (데이터를 수신한) 소켓이 있는지 확인하고 손을 든 소켓을 통해 전송된 데이터를 수신하는 구조
	- FD 설정
		- 수신, 전송, 예외를 구분해서 이벤트를 모아야한다.
			- 이것을 묶어서 사용하는 것이 fd_set 타입이다.
				- 0과 1로 표현되는, 비트단위로 이뤄진 배열이다.
		- fd_set에서 관찰 대상 fd는 비트가 1이다.
	- 검사 범위 지정
		- select의 첫번째 파라미터로 fd의 개수를 받으며 가장 큰 fd 값의 + 1을 해주면 된다.
			- fd는 0부터 시작해 +1을 하기 때문이다.
		- 마지막 매개변수는 timeval 구조체이다.

## select 함수의 이해와 서버의 구현

&emsp;&emsp;

- 설명
	- API
		-  `int select(int maxfd, fd_set *readset, fd_set *writeset, fd_set *exceptset, const struct timeval* timeout);`
			- 역할
				- 여러 소켓을 모아놓고 동시에 이들을 관찰한다.
			- 특징
				- 관찰항목은 다음과 같다.
					- 수신된 데이터를 지닌 소켓
					- 블로킹되지 않고 데이터의 전송이 가능한 소켓
					- 예외 상황이 발생한 소켓
				- 관찰항목은 각각 이벤트라고 부른다.
					- 해당 상태의 소켓이 등장하면 이것을 이벤트라고 칭한다.
				- 오류 발생시 -1, 타임아웃시 0, 관심대상으로 등록된 fd의 해당 관심에 관련된 변화가 생기면 0보다 큰 값이 반환됨
					- 변화가 발생된 fd의 총 수가 반환된다.
			- 각 매개변수 역할
				- maxfd
					- 검사 대상이 되는 fd의 수
				- readset
					- 이 변수에 수신된 데이터의 존재 여부에 관심이 있는 fd 정보를 모두 등록해 그 변수의 주소값을 전달
				- writeset
					- 이 변수에 블로킹 없는 데이터 전송의 가능 여부에 관심 있는 fd 정보를 모두 등록해서 그 변수의 주소 값을 전달
				- exceptset
					- 이 변수에 예외상황 발생 여부에 관심이 있는 fd 정보를 모두 등록해서 그 변수의 주소 값을 전달
				- timeout
					- select 호출 이후 무한 블로킹 상태에 빠지지 않도록 설정하기 위한 변수
			- 호출 방법
				1. 초기화
					1. FD 설정
					2. 검사 범위 지정
					3. 타임 아웃 설정
				2. select invoke
				3. 호출 결과 확인
		- `FD_ZERO(fd_set *fdset)`
			- 역할
				- 매크로 함수
				- 인자로 전달된 주소의 fd_set 타입 변수의 모든 비트를 0으로 초기화
		- `FD_SET(int fd, fd_set *fdset)`
			- 역할
				- 매크로 함수
				- 매개변수 fdset으로 전달된 주소의 매개변수에 매개변수 fd로 전달된 fd 정보를 등록
		- `FD_CLR(int fd, fd_set *fdset)`
			- 역할
				- 매크로 함수
				- 매개변수 fdset으로 전달된 주소의 매개변수에 매개변수 fd로 전달된 fd 정보를 삭제
		- `FD_ISSET(int fd, fd_set *fdset)`
			- 역할
				- 매크로 함수
				- 매개변수 fdset으로 전달된 주소의 매개변수에 매개변수 fd로 전달된 fd 정보가 있으면 양수, 없으면 음수 반환
			- 특징
				- select() 호출 결과를 확인하는 용도로 사용된다.
	- timeval 구조
		- `long tv_sec`
			- 역할
				- 초단위 정보
			- 특징
				- seconds
		- `long tv_usec`
			- 역할
				- 마이크로초 단위 정보
			- 특징
				- microseconds

### send & recv 입출력 함수 - 리눅스

&emsp;&emsp;

- 설명
	- API
		- `ssize_t send(int sockfd, const void *buf, size_t nbytes, int flags);`
			- 역할
				- 윈도우 send()와 동일
		- `ssize_t recv(int sockfd, const void *buf, size_t nbytes, int flags);`
			- 역할
				- 윈도우 recv()와 동일
	- flags 옵션 종류
		- MSG_OOB
			- 역할
				- 긴급 데이터(Out-of-band data)의 전송
			- 특징
				- send, recv 사용가능
				- 수신측에서 이 메시지를 긴급 데이터로써 받아 들일때 SIGURG 시그널을 통해서 받는다.
					- 따라서 sigaction()을 통해 후킹 함수를 등록해야 정상적으로 긴급 메시지를 따로 처리하게 만들 수 있다.
				- 예제에서는 fcntl() 함수를 통해 fd의 컨트롤을 시도한다.
					- 소켓의 소유자를 getpid() 함수가 반환하는 id의 프로세스로 변경하겠다라고 함수 파라미터를 넘긴다.
					- 이 함수는 fork()와 같은 함수를 호출해 fd가 다른 프로세스에도 공유해서 소유하게 되는 경우 어느 함수를 호출할지 구분시켜주기 위해 세팅하는 것이다.
						- 이 경우에는 현재 프로세스의 시그널 처리 함수를 사용하기 위해 세팅하는 것이다.
				- ==**1바이트만 제공된다.**==
					- OOB의 의미는 전혀 다른 통신 경로로 전송되는 데이터를 의미해 이 통신 경로를 확보해 다른 데이터를 전송받아야한다.
					- URG 패킷을 보게되면 오프셋을 지정하는 길이의 + 1을 한 곳이 Urgent Pointer이다.
						- 상대에게는 Urgent Pointer 앞에 있는 메시지가 긴급 메시지임을 표현하는 것
					- 물론 이렇게 되면 실제 긴급한 데이터는 1바이트 내에서 무엇인지 알 수는 없지만, 패킷자체를 긴급 패킷으로 보내며 메시지 처리를 재촉하는 것에 의미가 있다.
		- MSG_PEEK
			- 역할
				- 입력 버퍼에 수신된 데이터의 존재 유무를 확인
			- 특징
				- recv만 사용가능
				- MSG_DONTWAIT과 함께 설정되 입력 버퍼에 수신된 데이터가 존재하는지 확인하는 용도로 사용된다.
					- ==**같이 설정하는 이유는 데이터가 없어도 블로킹 상테에 두지 않기 위함이다.**==
		- MSG_DONTROUTE
			- 역할
				- 데이터 전송과정에서 Rounting 테이블을 참조하지 않을 것을 요구, 즉 로컬 네트워크에서만 목적지를 찾는 옵션
			- 특징
				- send만 사용가능
		- MSG_DONTWAIT
			- 역할
				- 입출력 함수 호출 과정에서 블로킹되지 않을 것을 요구, 즉 논블로킹 IO 요구에 사용되는 옵션
			- 특징
				- send, recv 사용가능
		- MSG_WAITALL
			- 역할
				- 요청한 바이트 수에 해당하는 데이터가 전부 수신될 때까지 호출된 함수가 반환되는 것을 막기 위한 옵션
			- 특징
				- recv만 사용가능

### readv, writev 입출력 함수 - 리눅스

&emsp;&emsp;

>[!info] Definition
> readv & writev의 기능은 데이터를 모아서 전송, 수신하는 기능의 함수이다.



&emsp;&emsp;

- 설명
	- API
		- `ssize_t writev(int filedes, const struct iovec *iov, int iovcnt);`
			- 역할
				- 데이터를 여러 버퍼에 나눠 한번에 전송
			- 특징
				- 성공 시 전송된 바이트 수, 실패 시 -1 반환
				- filedes은 데이터의 전송 목적지를 나타내는 fd
					- 파일도 가능
				- iov는 구조체 iovec 배열의 주소값이다.
					- 이 구조체 변수에는 전송할 데이터의 위치 및 크기 정보가 담긴다.
				- iovcnt는 주소 값이 가리키는 배열의 길이 정보
		- `ssize_t readv(int filedes, const struct iovec *iov, int iovcnt);`
	- iovec 구조
		- `void *iov_base;`
			- 역할
				- 버퍼의 주소 정보
		- `size_t iov_len;`
			- 역할
				- 버퍼의 크기 정보
	- 특징
		- 전송할 데이터가 여러 개의 배열에 나뉘어 있는 경우에 유용하다
			- 보통 대부분의 경우에 유용하다.
			- write() syscall, read() syscall 횟수를 획기적으로 줄일 수 있는 방법이다.
		- nagle 알고리즘을 명시적으로 중지시킨 경우에 매우 활용의 가치가 높다.
			- 즉, NO_Delay 옵션을 켜고 추가적인 ack 확인을 하지 않은 경우에는 여러번 전송을 할 수 밖에 없지만, 이때 writev()를 사용하면 단, 한번에 보내는 것이다.

### 멀티캐스트의 데이터 전송 방식

&emsp;&emsp;

- 설명
	- 특징
		- **==UDP 기반==**
		- 멀티캐스트 ==**서버**==는 ==**특정 그룹을 대상**==으로 ==**데이터를 딱 한번 전송**==
		- 특정 그룹에 있는 ==**클라이언트는 모두 데이터를 수신**==한다.
		- 그룹의 수는 IP 주소 범위 내에서 얼마든지 추가가 가능하다.
		- 특정 그룹으로 전송되는 데이터를 수신하려면 해당 그룹에 가입해야한다.
		- 멀티 케스트 그룹은 **==224.0.0.0 ~ 239.255.255.255==**이다.
			- 서브넷팅을 하면 원하는 만큼 잘라서 구획을 나눠 쓸 수 있다.
		- 라우터의 도움으로 가능하다.
		- ==**MBone이라는 가상 네트워크를 기반으로 동작**==
	- TTL 설정 방법
		- `setsockopt(send_sock, IPPROTO_IP, IP_MULTICAST_TTL, (void *) &timeLive, sizeof(timeLive));`
			- 특징
				- 소켓 DRAM 설정 이후 진행한다.
				- ==**그룹 가입과 관련된 프로토콜 레벨은 IPPROTO_IP**==이다.
					- 옵션의 이름은 IP_ADD_MEMBERSHIP
		- `struct ip_mreq joinAddress`
			- 특징
				- `imr_multiaddr.s_addr` 세팅이 필요하다.
					- 멀티캐스트 그룹의 주소정보를 문자열로 넣는다.
				- `imr_interface.s_addr` 세팅이 필요하다.
					- 그룹에 가입할 호스트의 주소정보를 의미한다.
	- Sender, Receiver
		- Sender
			- 역할
				- 파일에 저장된 뉴스 정보를 AAA 그룹으로 방송함
		- Receiver
			- 역할
				- AAA 그룹으로 전송된 뉴스정보를 수신함

### 브로드 캐스트

>[!info] Definition

- 설명
	- 특징
		- UDP 기반
	- 종류

						- 192.168.12.255라는 뜻
				- 특정 지역의 모든 호스트에게 전달하는 방식
		- Local 브로드캐스트
			- 특징
				- 255.255.255.255로 예약된 주소로 보내는 방식
				- 192.168.12.x 네트워크인 경우 (마스크는 255.255.255.0인 경우)
					- 255.255.255.255로 보내면 192.168.12.x 에 있는 모든 호스트에게 알아서 전달되는 방식
	- 전송 방법
		 1. 1로 초기화한 변수 선언 `int bcast = 1;`
		 2. `setsockopt(send_sock, SOL_SOCKET, SO_BROADCAST, (void *) &bcast, sizeof(bcast))`
			 1. sender에게만 필요하고 recevier에게는 필요없다.
