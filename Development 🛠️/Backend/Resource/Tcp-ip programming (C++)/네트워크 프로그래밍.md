## 표준 입출력 함수

&emsp;&emsp;

- 설명
	- 장점
		- 이식성이 좋다.
			- 모든 운영체제에서 지원하도록 ANSI C에서 표준으로 정의했다.
		- 버퍼링을 통한 성능 향샹에 도움이 된다.
			- ==**소켓 생성시 생성되는 입출력 버퍼와는 별개로 입출력 함수 또한 버퍼가 존재한다.**==
				- 따라서 ==**TCP 프로토콜을 통해 입출력 함수를 사용하면 버퍼를 총 2번 이용하는 것이다**==. (==**입출력 함수 버퍼 + 소켓 입출력 버퍼**==)
	- 단점
		- 양방향 통신이 어렵다.
		- 상황에 따라서 fflush 함수의 호출이 빈번히 등장해야한다.
			- 읽기에서 쓰기, 쓰기에서 읽기로 변환할때마다 이 함수가 필요하다.
		- 파일 디스크립터를 FILE 구조체의 포인터로 변환해야한다.
	- 특징
		- 버퍼는 성능 향상으로 둔다.
			- 하지만 소켓 버퍼는 TCP 구현을 위한 목적으로써 제공하는 목적성이 좀 더 강하다.
			- 버퍼를 사용하면 전송하는 ==**데이터 양이 클수록 성능 차이가 매우 크다**==.
			- 따라서 ==**표준 입출력 함수와 시스템 함수의 성능은 상당한 차이를 보인다.**==
		- 리눅스에서 소켓도 파일로 사용하기 때문에 표준 입출력 함수를 사용할 수 있다.
			- 따라서, 표준 입출력 버퍼의 이점을 그대로 이용할 수 있다.
	- API
		- `FILE *fdopen(int fildes, const char *mode)`
			- 역할
				- fd를 파일 구조체로 변환
			- 특징
				- 마지막 인자로 포인터의 모드를 문자열로 전달
		- `FILE *fileno(int fildes, const char *mode)`
			- 역할
				- 파일 구조체를 fd로 변환

### 입출력 스트림 분리


&emsp;&emsp;

- 설명
	- 장점
		- 입력 루틴(코드)과 출력 루틴의 독립을 통한 구현의 편의성 증대
		- 입력에 상관없이 추력이 가능하게 함으로 속도의 향상 기대
	- 단점
		- EOF에 문제가 발생함
			- EOF 전달 방법은 shutdown() 함수호출을 이용해 Half-close 기반의 eof 을 전달한다.
			- 송신용 파일 포인터를 닫으면 될것 같지만 실제로는 파일 포인터가 닫히면 바라보던 FD도 같이 소멸된다.
	- 주요 목적
		- FILE 포인터는 읽기모드와 쓰기 모드 구분이 필요함
		- 읽기모드, 쓰기 모드의 구분을 하면 구현의 편의성이 증대됨
		- 입출력 버퍼의 구분으로 인한 버퍼링 기능 향상
	- Half-Close 구현방법
		1. FD 복사본을 하나 더 만들고 쓰기용 File 포인터가 복사본을 바라보게 한다.
			1. 이는 fork()의 동작과는 다르다. 따라서 ==**dup, dup2와 같은 복사 함수**==를 사용해야한다.
		2. 쓰기용 File 포인터를 소멸시킨다.
			1. 이어서 쓰기용 File 포인터가 바라보던 FD 복사본이 소멸된다.
	- API
		- `int dup(int fildes);`
			- 역할
				- fd 복사
			- 특징
				- 성공시 복사된 FD, 실패시 -1 반환
		- `int dup2(int fildes, int fildes2);`
			- 역할
				- fd 복사
			- 특징
				- 성공시 복사된 FD, 실패시 -1 반환

## Epoll

&emsp;&emsp;

### epoll의 이해

&emsp;&emsp;

- 설명
	- select의 단점
		- select를 사용하기 위해서는 모든 fd를 대상으로 for문을 돌려야한다.
		- select 호출 시 인자로 매번 관찰대상에 대한 fd_set을 넘겨아한다.
			- 이것이 가장 큰 단점이다.
				- 매번 OS에게 큰 데이터를 전달하는 것은 프로그램에 많은 부담을 준다. 
				- 함수 시그니처 구조상의 문제때문에 필연적으로 코드 개선을 통해 해결도 불가능하다.
					- 소켓은 OS 관리 대상이기때문에 어쩔 수 없다.
	- epoll의 특징
		- OS에게 관찰 목표 소켓에 대한 정보를 딱 한번만 알려주고, 대상의 범위나 내용에 변경이 있을 때만 변경사항만 넘긴다.
			- 이것을 리눅스에서는 epoll, 윈도우는 iocp라고 한다.
		- 서버에 접속자 수가 많지 않고 다양한 운영체제에서 운영이 가능해야하면 select를 쓰는 것이 낫다.
			- 하지만 대다수 서버의 환경은 리눅스이므로 epoll을 쓰지 않는 것은 의미가 없긴하다.
		- ==**epoll 방식은 관찰대상인 fd의 저장을 OS가 담당한다.**==
			- FD 저장을 위한 저장소 생성을 OS에 요청하기위해 사용된다.
		- ==**epoll_event를 기반으로 상태 변화가 발생한(이벤트가 발생한) FD가 별도로 묶인다.**==
			- 이 구조체의 배열을 epoll_ctl, epoll_wait 호출시 인자로 넘긴다.
		- 리눅스 커널 버전에 따라 없을 수도 있다.
			- 요즘 대부분 커널 버전은 5버전을 넘어간다.
			- 요즘은 io_uring이라는 좀 더 성능이 좋은 멀티플렉스 API도 새로이 등장했다.
	- epoll의 장점
		- 상태 변화 확인을 위한, 전체 파일 디스크립터를 대상으로 하는 반복문이 필요 없다.
		- select 함수에 대응되는 epoll_wait 함수 호출시, 관찰 대상의 정보를 다시 매번 전달할 필요가 없다.
	- API
		- `int epoll_create(int size);`
			- 역할
				- epoll 파일 디스크립터 저장소 생성
			- 특징
				- 성공시 epoll fd, 실패시 -1 반환
				- FD 저장을 위한 저장소 생성을 OS에 요청하기위해 사용된다.
				- 리눅스 커널 2.6.8부터는 완전히 저 ==**전달인자는 무시**==된다.
					- **커널 내에서 epoll의 인스턴스 크기를 상황에 맞게 적절히 늘리기도 하고 줄이기도 하기 때문**이다.
				- 호출시 생성되는 리소스는 소켓처럼 OS에 의해서 관리된다.
				- 반환하는 epoll fd 값은 epoll 인스턴스 구분용으로 사용된다.
					- **소멸시 다른 fd와 마찬가지로 close 함수 호출을 통한 종료과정이 필요하다.**
		- `int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);`
			- 역할
				- 저장소에 파일 디스크립터 등록 및 삭제
			- 특징
				- 성공시 0, 실패 시 -1 반환
				- epoll FD 저장소에 관찰 대상(fd) 추가, 삭제를 OS에 요청하기 위한 함수
				- 첫번째 파라미터는 epoll fd
				- 두번째 파라미터는 관찰 대상의 추가, 삭제, 또는 변경 여부
					- EPOLL_CTL_ADD, EPOLL_CTL_DEL, EPOLL_CTL_
					  MOD가 있다.
				- 세번째 파라미터는 등록할 관찰 대상의 fd
				- 네번째 파라미터는 관찰대상의 관찰 이벤트 유형
					- 삭제시 null을 넣는다.
		- `int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout);`
			- 역할
				- fd의 변화 대기
			- 특징
				- epoll_event를 기반으로 상태 변화가 발생한(이벤트가 발생한) FD가 별도로 묶인다.
				- 따라서 반복문으로 하나씩 fd_set을 확인하던 select 방식과는 다르게 효율이 좋다.
	- epoll_event 구조
		- `__uint32_t events;`
			- 역할
				- 이벤트의 유형
			- 특징
				- OR 연산자로 복수 등록이 가능하다.
			- 종류
				- EPOLLIN
					- 역할
						- 수신할 데이터가 존재
				- EPOLLOUT
					- 역할
						- 출력버퍼가 비워져 당장 데이터 전송 가능
				- EPOLLPRI
					- 역할
						- OOB 데이터 수신
				- EPOLLRDHUP
					- 역할
						- 연결 종료 또는 Half-close 진행
					- 특징
						- 엣지 트리거 방식에서 매우 유용함
				- EPOLLERR
					- 역할
						- 에러 발생
				- EPOLLET
					- 역할
						- 이벤트 감지 방식을 엣지 트리거 방식으로 동작
				- EPOLLONESHOT
					- 역할
						- 이벤트 1회 감지 이후 더 이상 fd에서 이벤트를 발생시키지 않음
					- 특징
						- epoll_ctl 함수로 다시 이벤트 재설정이 필요함
		- `epoll_data_t data;`
			- 특징
				- 유니온
	- epoll_data_t 구조
		- `void *ptr;`
		- `int fd;`
		- `__uint32_t u32;`
		- `__uint64_t u64;`

### 레벨 트리거 방식과 엣지 트리거 방식

&emsp;&emsp;

- 설명
	- 레벨 트리거 방식 특징
		- ==**입력버퍼에 데이터가 남아있는 동안 계속 이벤트가 등록된다.**==
			- 서버의 입력 버퍼로 50바이트의 데이터가 수신되는 경우
				- OS는 이를 이벤트로 등록한다. (변화가 발생한 FD로 등록)
				- 입력버퍼에서 20바이트만 수신하고 30바이트가 남는 경우에도 이벤트로 등록된다.
				- ==**이후 버퍼에 데이터가 남아있기만 하면 계속 이벤트로 등록한다.**==
		- **참고로 Select 함수는 레벨 트리거 방식이다.**
	- 엣지 트리거 방식 특징
		- 입력버퍼에 데이터가 수신되는 경우에만 딱 한번만 이벤트가 등록된다.
			- ==**버퍼에 데이터가 남아있어도 이벤트로 등록하지 않는다.**==
	- 엣지 트리거 방식의 주의점
		- errno 변수는 리눅스에서 전역으로 선언해둔 오류정보 제공용 변수이다.
			- 이 변수 접근을 위해서는 errno.h 헤더파일을 include 해야한다.
			- 함수별로 오류발생시 변수 errno에 저장되는 값이 다르기 때문에 이 변수에 저장되는 값을  모두 알필요는 없지만 필요할때마다 참조해야한다.
			- ==**단, errno 변수에 상수 EAGAIN이 저장되는 시점은 read 함수에 입력버퍼가 비어서 읽어들일 데이터가 없는 경우이다.**==
		- 소켓을 넌플로킹 IO 소켓으로 바꾸는 방법을 알아야한다.
			- 엣지 트리거 방식은 데이터 수신시 딱 한번 이벤트가 등록되는 특성 때문에 **입력 관련 이벤트 발생시 버퍼에 저장된 데이터를 전부 읽어내야한다.**
				- 따라서 ==**수신된 데이터의 크기에 따라 블로킹으로 동작하는 경우 오랜시간 서버가 멈출 가능성이 높아지게 된다.**==
				- 따라서 ==**논블로킹 방식으로 만들어야**== 이런 가능성이 없는 서버를 만들 수 있다.
	- 엣지 트리거 방식의 장점
		- 데이터의 수신과 데이터의 처리 시점 분리가 가능하다.
			- ==**입력버퍼에 데이터가 수신된 상황에도 이를 읽어들이고 처리하는 시점을 서버가 결정하도록 만드는 것은 서버 구현에 엄청난 유연성을 제공한다.**==
			- 
	- API
		- `int fcntl(int fildes, int cmd, ...);`
			- 역할
				- 파일의 특성을 변경 및 참조한다.
			- 특징
				- 성공시 매개변수 cmd에 따른 값, 실패시 -1 반환
				- 첫번째 파라미터는 fd
				- 두번째 파라미터는 cmd이다.
					- F_GETFL을 전달하면 fd의 정보를 int 형으로 얻을 수 있다.
					- F_SETFL을 전달하면 fd의 특성 정보를 변경할 수 있다.
				- 파일을 넌블로킹 모드로 변경하기 위해서는 다음과 같이 호출한다.
					- `int flag = fcntl(fd, F_GETFL, 0);`
						- 플래그 값을 얻기 위해 먼저 F_GETFL로 호출한다.
					- `fcntl(fd, F_SETFL, flag|O_NONBLOCK);`
						- 첫번째 호출에서 얻은 플래그 값을 사용해 넌블록 특성을 더해 재설정한다.
					- 이로써 read, write 호출시 데이터 유무에 상관없이 ==**블로킹이 되지 않는 소켓**==이 만들어진다.

## 멀티 스레딩
