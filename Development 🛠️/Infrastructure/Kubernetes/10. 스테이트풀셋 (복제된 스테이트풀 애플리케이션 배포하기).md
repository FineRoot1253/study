
## 스테이트풀 파드 복제하기

- 설명
	- RS는 하나의 파드 템플릿에서 여러 개의 파드 레플리카를 생성하고 레플리카는 이름과 IP 주소를 제외하면 서로 똑같다.
		- 파드 템플릿이 특정 PVC를 참조하는 볼륨을 포함하면 RS의 모든 레플리카는 정확히 동일한 PVC를 사용할 것이고 클레임에 바인딩된 동일한 PV를 사용하게 된다.
	- 파드 템플릿에는 클레임에 관한 참조가 있으므로 각 레플리카가 별도의 PVC를 사용하도록 만들 수는 없다.
		- 각 인스턴스가 별도의 스토리지를 필요로 하는 분산 데이터 저장소를 실행하려면 레플리카셋을 사용할 수 없다.
			- 적어도 단일 RS를 사용해서는 불가능하다.

&emsp;&emsp;

### 개별 스토리지를 갖는 레플리카 여러 개 실행하기

- 설명
	- 기본 방법
		- 동일 볼륨을 여러 개의 디렉터리로 사용하기
			- 단일 파드 템플릿으로부터 파드 레플리카의 설정을 다르게 하는 것은 불가능하기 때문에 인스턴스 생성 시점에 데이터 디렉터리를 자동으로(가능하다면 새로 생성)하도록 할 수 있다.

&emsp;&emsp;

### 각 파드에 안정적인 아이덴티티 제공하기

- 설명
	- 안정적 ID의 필요성
		- RS가 파드 교체시 새 파드가 갖는 스토리지 볼륨의 데이터는 종료된 이전의 파드의 것이여도 완전히 새로운 호스트 이름과 IP를 가진다.
			- 이런 새로운 네트워크 ID가 문제를 야기할 수도 있다.
		- 특정 앱이 안정적인 네트워크 ID를 요구하는 이유는 분산 스테이트풀 애에서는 보편적이다.
			- 이런 특정 앱은 관리자가 다른 모든 클러스터 멤버의 리스트와 멤버들의 IP 주소를 각 멥버의 설정 파일에 기재해야 한다.
			- k8s는 매번 파드가 재스케줄링되고 새로운 파드는 새로운 호스트이름과 IP 주소를 할당받으므로 구성 멤버가 재스케줄링될 때마다 모든 앱 클러스터가 재구성돼야 한다.
	- 각 파드 인스턴스 별 전용 서비스 사용하기
		- 각 개별 멤버에게 전용 k8s 서비스를 생성해 클러스터 멤버에 안정적인 네트워크 주소를 제공하는 것
		- 서비스 IP는 고정적이기 때문에 설정에서 각 멤버를 서비스 IP를 통해 가리키는 것이다.
		- 이는 각 멤버에게 개별 스토리지를 제공하려고 RS을 생성한 것과 비슷하다.
		- 그러나 이러면 RS를 서비스 별로 따로 생성해야하고 문제들도 해결을 못한다.

## 스테이트 풀 셋(이하 STS)

&emsp;&emsp;

### STS와 RS 비교

- 설명
	- RS
		- 관리되는 파드를 가축처럼 다룬다.
		- 죽으면 바로 재 생성하고 의도된 수보다 많으면 바로 죽인다.
	- STS
		- 관리되는 파드를 애완동물처럼 다룬다.
		- 의도된 수보다 많으면 ID와 상태를 유지하면서 (비석처럼) 새로 생성할때는 이 ID와 상태를 물려주며 생성한다.
	- 공통점
		- 파드 템플릿을 통해 생성한다.

&emsp;&emsp;

### 안정적인 네트워크 ID 제공하기

- 설명
	- STS 이름 특징
		- 각 파드의 이름과 호스트 이름은 RS와 비교하면 해시값이 아닌 0부터 시작하는 숫자 인덱스를 부여받는다.
	- 거버닝 서비스
		- STS는 거버닝 헤드리스 서비스를 생성해 각 파드에 실제 네트워크 아이덴티티를 제공한다.
			- 각 파드는 이를 통해 자체 DNS 엔트리를 가지며 클러스터 피어 혹은 클러스터의 다른 클라이언트가 호스트 이름을 통해 파드의 주소를 지정할 수 있다.
		- foo라는 거버닝 서비스가 있고 default 네임스페이스에 a-0 파드가 존재한다면
			- FQDN은 a-0.foo.default.svc.clustuer.local로 파드에 접근할 수 있다.
			- foo.default.svc.clustuer.local라는 도메인의 SRV 레코드를 조회해 모든 STS 파드의 이름을 찾는 목적으로 DNS 활용이 가능하다.
	- 스케일링
		- 항상 끝번호 인스턴스가 제거되고 추가된다.
		- 보통 스케일링은 하나 이상의 파드가 비정상 상태인 경우 진행이 불가능하다.
			- 분산 데이터 스토어 구조에서 여러개의 노드가 죽으면 데이터 손실이 일어날 가능성이 존재하기 때문에 혹시나 모를 사고를 대비하기 위해서이다.

&emsp;&emsp;

### 각 STS 인스턴스에 안정적인 전용 스토리지 제공하기

- 설명
	- 방법
		- STS의 파드 템플릿처럼 PVC 템플릿을 생성해야한다.
			- STS 정의에 파드 템플릿과 VC 템플릿을 같이 작성하는 것이다.
	- 생성과 삭제 과정
		- STS의 파드 생성시 두 개 이상의 API 오브젝트(파드 + 파드가 참조하는 1개 이상의 PVC)가 생성된다.
		- STS의 파드 삭제시 파드만 삭제하고 클레임은 남겨둔다.
			- 클레임은 삭제되면 컨텐츠가 손실된다.
	- 동일 파드의 새 인스턴스에 PVC 붙이는 방법
		- 실수로 스케일다운하여 컨텐츠를 잃어버려도 스케일 업으로 돌릴 수 있고 새 파드는 동일한 이름뿐만 아니라 동일한 지속된 상태를 가진다.

&emsp;&emsp;

### 스테이트풀셋 보장 이해하기

- 설명
	- k8s는 동일한 ID를 가지는 인스턴스를 직접 수동으로 올리면 멀쩡하게 동작하는데 STS가 보장하는 것은 이런 일이 발생하지 않도록 하는 것이다.
	- 교체 파드를 생성하기 전에 파드가 더 이상 실행 중이지 않는다는 점을 절대적으로 확신해야한다는 뜻이다.

## STS 사용하기

&emsp;&emsp;

### 앱과 컨테이너 이미지 생성

- 설명
	- 요청 받은 내용을 텍스트 파일에 저장하는 구조의 컨테이너이다.

&emsp;&emsp;

### STS를 통한 앱 배포하기

- 설명
	- 배포전 생성 필수 조건
		- PV
		- 거버닝 서비스
		- STS
	- PV 생성하기
		- `pv.yaml` 로부터 생성한다.
			- 디스크는 미니쿱 기준으로 생성하면 된다.
	- 거버닝 서비스 생성
		- 헤드리스 서비스를 먼저 생성한다.
			- spec.clusterIP를 None으로 잡으면 된다.
	- STS 디스크립션 생성
		- 생성해보면 파드가 하나씩 올라오는데 특정 클러스터된 앱의 경우 레이스 컨디션에 빠질 수 있기 때문에 이렇게 동작한다.


&emsp;&emsp;

### 파드 가지고 놀기

- 설명
	- API 서버를 통해 파드와 통신하기
		- 헤드리스 서비스를 만들었기 때문에 직접 파드와 통신을 해야만 하므로 API 서버있는 기능 중 하나인 개별 파드에 프록시 서버 만들기 기능을 사용해볼 ㅅ ㅜ있다.
	- 파드에 데이터를 전달하고 저장이 되었으면, 삭제해 보고 다시 생성해보자
		- 제대로 연동이 되는 것을 볼 수 있다.
	- STS 스케일링
		- 스케일 다운을 하면 pvc를 수동을 삭제하지 않는한 그대로 살아있다가 스케일 업을 하면 그대로 가져다 재사용된다.

## STS가 노드 실패를 처리하는 과정 이해하기

&emsp;&emsp;

### 노드의 네트워크 연결 해제 시뮬레이션

- 설명
	- 노드 네트워크 어뎁터 셧다운
		- 특정 노드의 네트워크를 일부러 막는다.
	- k8s 마스터에서 본 노드 상태 확인
		- 노드의 네트워크가 내려간 것을 확인할 수 있다.
		- 파드의 상태는 Unknown이 된다.
	- Unknown 파드 이해하기
		1. 노드가 준비되었는지 확인을 해보고 아니면 파드는 자동으로 노드에서 제거된다.
			1. 마스터가 이 동작을 수행한다.
		2. 파드가 deletion으로 표시되면 kubelet은 파드 종료를 시작한다.

&emsp;&emsp;

### 수동으로 파드 삭제하기

- 설명
	-
