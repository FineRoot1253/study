## 도커를 이용한 컨테이너 이미지 생성, 실행, 공유

- 설명
	- 간단한 node.js 앱을 생성하고 앱을 컨테이너 이미지로 패키징한다.
	- 이미지 기반 컨테이너를 실행한다.
	- 누구든 실행할 수 있게 도커 허브에 이미지를 푸시한다.

&emsp;&emsp;

- 내용 분기
	- 나는 내 프로젝트를 위해 여기에서부터는 프로메테우스 이미지를 기준으로 작업을 시작할 예정이다.

&emsp;&emsp;

- 이미지 빌드 프로세스
	- 디렉터리 전체의 컨텐츠가 도커 데몬에 업로드되고 그곳에서 이미지가 빌드된다.
	- 도커 클라이언트와 데몬은 같은 머신에 있을 필요는 없다.
	- 리눅스가 아닌 OS에서 도커를 사용하는 경우 도커 클라이언트는 호스트 OS에 위치하고 데몬은 가상머신 내부에서 실행된다.
	- 빌드 디렉터리의 모든 파일이 데몬에 업로드되어야하기 때문에 데몬이 로컬로 실행 중이지 않은 상황에서 큰 파일이 다수 포함되면 시간이 오래 걸릴 수 있다.
	- 주의할 점
		- 원격 머신에 도커 데몬이 위치한 경우 파일이 빌드 프로세스의 속도 저하를 야기한다.
		- 쓸모 없는 파일을 빌드 디렉터리에 포함시켜서는 안되는 이유이다.
	- 빌드 프로세스 동안 이미지가 사용자 컴퓨터에 없으면 도커허브에서 들고 온다.

&emsp;&emsp;

- 이미지 레이어의 이해
	- 이미지는 하나의 바이너리 덩어리가 아닌 여러 개의 레이어로 구성된다.
	- 서로 다른 이미지가 여러 개의 레이어 공유가 가능하기 때문에 이미지 저장과 전송에 효율적이다.
	- 예를 들면 node의 이미지를 통해 또 다른 이미지를 구성한다고 가정하면 이미 nodejs의 이미지가 존재하기 때문에 직접 만든 앱만 레이어를 따로 구성한다.
	- 각 도커파일이 새로운 레이어를 하나만 생성한다고 생각할 수도 있지만 그렇지 않다.
		- 이미지를 빌드하는 동안 기본 이미지의 모든 레이어를 가져온 다음 도커는 그 위에 새로운레이러를 생성하고 app.js 파일을 그 위에 추가한다.
		- 그리고 마지막 레이어는 이미지 파일의 태그를 지정한다.
	- nodejs 기반 도커 파일 appjs 레이어 예시
		- cmd node app.js
		- add app.js/app.js <- 여기 까지는 node.js 이미지 밖, node.js를 이용하는 모든 이미지는 이 아래까지의 레이어를 공유하는 것이다. 따라서 한번 이미지를 다운 받았다면 이 아래의 동작들은 반복할 이유는 없는 레이어이다.
		- cmd node
		- run curl ...
		- ...
		- run apt-get ...
	- 빌드 프로세스가 완료되면 새로운 이미지가 로컬에 저장된다.
	- 기존 이미지에서 컨테이너 실행 후 최종 상태를 새로운 이미지로 커밋할 수 있다.
	- 요즘은 build.io와 같은 이미지 자동화 툴이 대단히 많다.
		- 책에서는 도커파일이 훨씬 좋다고 설명하지만 요즘은 누구도 도커파일을 잘 작성하진 않는다.
		- 로컬 머신에서 앱을 개발한 다음 앱 위에 동작할 os와 환경설정을 한 다음 이를 기반으로 이미지 패키징을 자동화하는 툴이 요즘 대단히 많이 개발된 상황이다.
		- 이미지를 패키징하는 이 일련의 과정도 원격 빌드 머신에서 스크립트로 진행하면 된다.

## 컨테이너 이미지 실행

- 설명
	- `docker run --name '컨테이너명' -p 로컬머신포트:컨테이너포트 -d '이미지명'`
		- 이 cli 명령은 docker-compose 파일로 세팅하면 다 의미가 없긴하다.
		- 그러나 지금은 명령에만 좀 집중해서 보자

&emsp;&emsp;

- 도커 프로세스 확인하기
	- `docker ps`
		- 올라간 도커 프로세스들을 간략하게 보여준다.
	- `docker inspect`
		- 올라간 도커 특정 컨테이너의 좀 더 자세한 정보를 출력한다.


<br><br>


## 실행 중인 컨테이너 내부 탐색하기

- 설명
	- `docker exec -it '컨테이너 명' bash`
		- 컨테이너 내부에 bash를 실행한다.
		- i
			- 표준 입력을 연다.
			- 셀에 입력하기 위해 스트림을 여는 것이다.
		- t
			- 의사 터미널 tty를 할당한다.
	- it 옵션은 일반적인 셸 이용을 원한다면 필요한 옵션이다.

&emsp;&emsp;

- 실행 중인 컨테이너 내부에서 프로세스 탐색
	- 설명
		- `ps aux`

&emsp;&emsp;

- 호스트 OS에서 실행 중인 컨테이너에서 실행되는 프로세스 이해해보기
	- 설명
		- 다른 터미널을 열고 호스트 OS에서 프로세스를 조회하면 컨테이너에서 실행중인 프로세스를 볼 수 있다.
			- 이때 특징은 수행된 커맨드는 같지만 PID가 서로 다르다.
			- 여기에서 이 컨테이너 자체 리눅스 PID 네임스페이스르 사용하여 고유의 시퀀스 번호를 가지고 완전히 분리된 프로세스 트리를 가지고 있음을 의미한다.

&emsp;&emsp;

- 격리된 컨테이너 파일 시스템
	- 설명
		- 컨테이너는 격리된 프로세스를 가진 것과 매한가지로 격리된 파일 시스템을 가지고 있다.
		- 컨테이너 루트 디렉터리의 내용을 보면 컨테이너 안의 파일만 보여준다.
			- 이 파일들은 이미지에 있는 모든 파일과 컨테이너가 실행하는 동안 생성한 파일등을 포함한다.
			- app.js 파일과 사용한 node:7 기본 이미지의 일부인 시스템 디렉터리를 포함한다.
		- 컨테이너에서 나오기 위해 exit 명령으로 셸을 빠져나오면 호스트 머신으로 돌아오게 된다.

## 컨테이너 중지와 삭제

- 설명
	- `docker stop '컨테이너명'`
		- 도커에게 컨테이너를 중지하도록 명령한다.
	- `docker rm '컨테이너명'`
		- 도커에게 컨테이너 삭제를 명령한다.
		- `docker ps -a`를 하면 사실 중지만 해서는 컨테이너가 중지만하지 살아는 있다는 것을 알게 된다.
		- 이 명령을 사용하면 재시작은 불가능하고 다시 이미지로부터 다시 컨테이너를 생성해야한다.

## 이미지 레지스트리에 이미지 푸시


- 추가 태그로 이미지에 태그 지정하기
	- 설명
		- 도커 허브의 규칙으로 태그 지정이 필수이다.
		- `docker tag '이미지명' '도커허브ID/태그명'`
			- 태그는 '도커 허브 ID/태그명' 이런식으로 작성해야 푸시가 가능하다.
			- 같은 이미지에 여러번 다른 태그를 작성해도 태그만 들어나고 이미지는 그대로다.

&emsp;&emsp;

- 도커 허브에 이미지 푸시하기
	- 설명
		- `docker push '도커허브ID/태그명'`
			- 참고로 도커허브의 ID로 로그인을 해야 푸시가 가능하다.

&emsp;&emsp;

- 다른 머신에서 이미지 실행하기
	- 설명
		- `docker run -p 로컬머신포트:컨테이너포트 -d '도커허브ID/태그명'`
			- 도커가 설치된 머신이면 이 명령어 한줄로 어디서든 환경을 배포할 수 있게 되었다.

## k8s 클러스터 설치

- 설명
	- 쿠버네티스 클러스터가 설치되어있다면 도커에서 실행하지 않고 쿠버네티스 클러스터 배포할 수 있다.
	- 완전 다중 노드 k8s 클러스터의 설치는 리눅스와 네트워크 관리에 능숙하지 않다면 단순한 작업이 아니라서 튜토리얼에서도 로컬머신에서의 k8s 경험은 단일 노드에서 진행하도록 알려준다.
	- 클러스터 설치 방법
		- 구글 k8s 머신(이하, GKM)인 관리형 k8s 클러스터를 사용하는 방법
			- GKM이 완전한 다중 노드 k8s 클러스터이다.
		- 단일 노드 k8s 클러스터를 설치하는 방법
		- kubedm 도구를 사용해서 클러스터를 설치하는 방법
			- 지금 설명해봐야 좀 의미가 없다.
			- aws에서 제공하는 kops 도구가 kubedm을 래핑하여 만든 도구이다.
			- aws위의 운영 수준의 고가용성 k8s 클러스터를 배포하는데 도움을 주며 다른 엔진도 지원되도록 준비중이다.


<br><br>

## minikube로 단일 노드 로컬 k8s 클러스터 사용해보기

- 설명
	- 완전하게 동작하는 k8s 클러스터를 접하는 가장 빠르고 간단한 방법이다.
		- 모든 기능을 볼 수는 없지만 대다수의 기능을 살펴볼 수 있다.
	- 우선 도커 런타임을 실행시키고 minikube을 실행시키야한다.
		- `minikube start`로 수행한다.
	- 이걸 우선 설치해야 kubectl (CLI 클라이언트)을 설치했을 때 정상 동작을 한다. 
	- 튜토리얼에서는 kubectl을 먼저 설치하게 하는데 k8s와 같은 클러스터가 없으면 에러를 띄운다.
		- 우선 k8s api 서버가 올라왔는지 확인하기 때문이며 이와 같은 이유로 minkube과 같은 클러스터를 우선 설치해야 kubectl이 동작한다.

&emsp;&emsp;

- 클러스터 작동 확인
	- `kubectl cluster-info`

&emsp;&emsp;

- GKM을 활용한 관리형 k8s 클러스터 사용하기
	1. 구글에 가입하여 GCP 콘솔을 연다.
	2. 콘솔에서 프로젝트를 하나 생성하고 k8s 엔진 api를 활성화 한다.
	3. GC SDK를 다운로드하고 설치한다.
	4. `gcloud components install kubectl` 이 명령으로 kubectl 명령행 도구를 설치한다.
	5. `gcloud container clusters create '이미지명' --num-nodes 3`
		- 이 명령이면 워커 노드가 3개인 쿠버네티스 클러스터가 구성된다.
		- 이를 바탕으로 다중 노드를 활용하는 기능을 제대로 시연도 가능하다.

&emsp;&emsp;

- 클러스터 개념 이해하기
	- 설명
		1. 각 노드는 도커(혹은 컨테이너 런타임), kubelet, kube-proxy를 실행한다.
		2. kubectl 클라이언트 명령어는 마스터 노드에서 실행 중인 k8s API 서버로 REST 요청을 보내 클러스터와 상호작용한다.
		3. `kubectl get nodes`를 입력하면 클러스터의 모든 노드를 조회할 수 있다.
			- kubectl 명령으로 모든 종류의 쿠버네티스 오브젝트 조회가 가능하다.
				- 이 get 명령이 가장 자주 사용되는 명령이다.
		4. `kubetctl describe node '노드명'`
			- `describes`는 오브젝트의 상세 정보를 보는 명령이다.
			- 출력 결과
				- cpu
				- 메모리
				- 시스템 정보
				- 노드에 실행 중인 컨테이너 등을 포함한 노드 상태
			- 노드명을 생략하면 모든 노드에 대한 상세정보를 전달해준다.

## kubectl alias와 명령줄 자동 완성 설정하기

- 설명
	- 보통 kubectl 명령조차 입력하기 힘들어서 alias를 붙여두는 편이다.
		- zshrc같은 파일에 저장해두자.
	- `alias k=kubectl`
	- 별칭을 이용해도 입력할 명령은 썩어 넘치기 때문에 zshrc-completion 패키지를 설치하고 명령을 실행해야한다.
		- 윈도우는 잘 모르겠넹
		- `source <(kubectl completion zsh | sed s/kubectl/k/g)`
		- `sed s/kubectl/k/g`를 넣는 이유는 별칭으로 등록한 k와 함께 정상 동작하게 만들기 위함이다.

## 앱 구동하기

- 설명
	- `k run '이미지명' --image='도커허브ID/태그명' --port=앱이 수신해야할 포트 --generator=run/v1`
		- `--generator`는 보통 사용안하지만 k8s에서 디플로이먼트 대신 레플리케이션 컨트롤러를 사용하기 위해 붙였다.
			- 이 옵션은 1.18 이후 사라진 옵션이다.
			- 정리하는 지금은 1.32버전으로 따로 레플리케이션 컨트롤러를 yaml 기반으로 만들고 이를 기반으로 생성해둬야한다. 그래서 이젠 귀찮게 yaml로 전부 선언해놓고 써야한다.
		- 디플로이먼트는 후반에 설명할 예정이고 레플리케이션 컨트롤러부터 바로 살펴보자
		- 요즘은 `create`로 파드를 생성하며 간단하게 deployment와 한번에 생성되는 파드를 보고 싶을 때 `run`을 사용한다.

- 파드란
	- 설명
		- k8s는 개별 컨테이너들을 직접 다루지 않고 파드라는 이름의 컨테이너를 그룹으로 묶은 개념을 사용한다.
		- 각 파드는 자체 IP, 호스트 이름, 프로세스등이 있는 논리적으로 분리된 머신이다.
		- 앱은 단일 컨테이너로 실행되는 단일 프로세스 일수도 있고, 개별 컨테이너에서 실행되는 주 애플리케이션 프로세스와 부가적으로 도와주는 프로세스로 이뤄질 수 있다.
		- 다른 파드에서 실행 중인 컨테이너는 동일한 논리적인 머신에서 실행하는 것처럼 보이는 반면, 다른 파드에서 실행 중인 컨테이너는 같은 워커 노드에서 실행 중이라 할지라도 다른 머신에서 실행 중인 것으로 나타난다.
		- 결론적으로 하나의 노드에는 여러 파드가 올라갈 수 있고, 파드에는 여러 컨테이너가 올라갈 수 있다.
			- 집합의 개념으로 보자면, 노드 ) 파드 ) 컨테이너 개념이다.
	- 정의
		- 하나 이상의 밀접하게 연관된 컨테이너의 그룹으로 같은 워커 노드에서 같은 리눅스 네임스페이스로 함께 실행된다.

&emsp;&emsp;

- 파드 조회하는 법
	- 설명
		- 컨테이너는 독립적인 k8s 오브젝트가 아니기 때문에 개별 컨테이너 조회는 불가능하지만 파드는 가능하다.
	- `k get pods`
		- 노드에 올라간 파드들을 볼 수 있다.
		- 상태 종류
			- pending 
				- 파드 내부 컨테이너를 준비시키는 중
			- Running
				- 파드 내부의 컨테이너가 생성되고 동작 중
	- `k describe pod`
		- 파드의 세부 정보를 볼 수 있다.

&emsp;&emsp;

- 백그라운드에 일어난 앱 구동시 동작 이해하기
	1. k8s 명령 수행시 k8s REST API 서버를 통해 요청을 전달하고 클러스터에 새로운 레플리케이션 컨트롤러 오브젝트를 생성한다.
	2. 레플리케이션 컨트롤러는 새 파드를 생성하고 스케줄러에 의해 워커 노드 중 하나에 스케줄링 된다.
	3. 해당 워커 노드의 kubelet은 파드가 스케줄링됐다는 것을 보고 이미지가 로컬에 없으므로 도커 허브에서 다운 받아 컨테이너를 생성하고 실행한다.

## 앱에 접근하기

- 설명
	- 각 파드는 IP 주소를 가지고 있지만 클러스터 내부에만 존재하며 외부에서 접근이 불가하다.
	- 외부에서 가능하게 하려면 서비스 오브젝트를 통해 노출시켜야한다.
	- 서비스(cluster ip 서비스)를 생성하면 클러스터 내부에서만 접근이 가능하므로 Load Balancer 유형의 특별한 서비스를 생성해야한다.
		- 외부 로드 밸런서가 생성되어 로드 벨런서의 퍼블릭 IP를 통해 파드에 연결할 수 있다.
	- 

&emsp;&emsp;

- 서비스 오브젝트 생성하기
	- 설명
		- `k expose rc '파드명' --type=LoadBalancer --name '서비스명'`
			- rc는 래플리케이션 컨트롤러의 약어이다.
			- 사실 대부분의 리소스 유형은 약어를 더 많이 사용한다.
				- pod = po
				- service = svc
			- 현재 rc는 생성되진 않으므로 pod을 넣으면 된다
			- 그리고 loadBalancer 이 타입은 GKM이나 AWS와 같은 환경에서만 지원이 된다. 즉, 단일 노드 클러스터에서는 지원되진 않아 NodePort를 써줘야한다.

&emsp;&emsp;

- 서비스 조회하기
	- 설명
		- `k get svc`
			- 출력에 Kubernetes와 같이 내가 생성한 서비스가 나와야 정상이다.
			- 클라우드의 경우 EXTERNAL-IP 컬럼에 외부 IP가 좀 기다리면 생성된다.
			- minikube의 경우 `minikube service 서비스명`을 입력하면 서비스에 접근 가능한 IP를 반환해준다.
				- 참고로 저걸 입력하면 터널이 연결된다.

&emsp;&emsp;

### 시스템 논리적 구성 살펴보기

- 레플리케이션 컨트롤러, 파드, 서비스가 서로 동작하는 방식
	1. run 수행시 레플리케이션 컨트롤러를 생성하고 레플리케이션 컨트롤러가 파드를 생성한다.
		1. 현재 버전에서는 바로 파드를 생성해주게끔 수정되었다. 따로 레플리케이션 컨트롤러를 명령줄에서 생성되는 것을 방지한 것이다.
		2. 현재 파드를 생성하는 것은 deployment이나 여기에 속한 ReplicaSet이 담당한다.
	2. 클러스터 외부에서 파드에 접근케 하기 위해 k8s에게 모든 파드를 단일 서비스로 노출하도록 명령한다.

&emsp;&emsp;

- 파드와 컨테이너의 이해
	- 설명
		- 시스템의 가장 중요한 구성 요소는 파드이다.
		- 예제는 파드가 하나의 컨테이너를 지니고 있긴하지만 보통 파드가 원하는 만큼의 컨테이너를 넣어줄 수 있다.
		- 컨테이너 내부에는 앱 프로세스가 있고 포트에 바인딩 되어 요청을 대기한다.(listen)
		- 파드는 자체의 고유한 사설 IP 주소와 호스트 이름을 갖는다.

&emsp;&emsp;

- ReplicaSet의 이해
	- 설명
		- 항상 정확히 하나의 파드 인스턴스를 실행하도록 지정한다.
		- 파드를 복제하고 항상 실행 상태로 만든다.
		- 지금 예제에서는 파드의 레플리카를 지정한 것은 아니기 때문에 하나만 생성된 것이다.
		- 어떤 이유로 파드가 사라진다면 레플리카셋은 세로운 파드를 생성하게 된다.

&emsp;&emsp;

- 서비스가 필요한 이유
	- 설명
		- ==**파드는 일시적이기 때문에 필요하다.**==
		- 파드는 동적으로 사라질 수도 있고 삭제될 수도 있고 레플리카셋이 알아서 생성해서 대체한다.
		- 새로운 파드는 다른 IP 주소를 늘 할당 받으며 이로인해 서비스가 늘 필요하게 된다.
		- 서비스는 파드의 존재 위치나 상태를 상관없이 파드 중 하나로 연결해 요청을 처리하게 한다.
		- 서비스는 동일한 서비스를 제공하는 하나 이상의 파드 그룹의 정적 위치를 나타낸다.
		- 서비스의 IP와 포트로 유입된 요청은 그 순간 서비스에 속해있는 파드 중 하나에게 전달된다.

&emsp;&emsp;

### 애플리케이션 수평 확장

- 설명
	- 책에서는 rc가 존재하기 때문에 간단하게 파드를 생성하지만 현재 버전에서는 run명령 만으로는 독립파드에 서비스가 붙은 상태이기 때문에 파드가 삭제되거나 중단되거나 하면 그것으로 끝난다.
	- 따라서 디플로이먼트이나 이 하위의 replicaset을 생성해줘야한다.
		- `k create deployment 디플로이먼트 이름 --image=도커허브ID/태그명` 이 명령으로 생성해주자.
			- 참고로 저 이미지를 지정하면 레플리케이션을 시도할 이미지를 지정해주는 것이다.
	- `k scale deploy 디플로이먼트 이름 --replicas=3`
		- 이 명령을 넣으면 래플리카 개수가 증가한다.
		- 실제로 `k get deploy`를 입력하면 늘어난 레플리카 개수를 확인 할 수 있다.
	- k8s에게 항상 유지해야할 복사본 개수를 입력해준 것이다.
		- 단순히 복사본 개수를 늘려준 것이 아니라 k8s에게 이를 유지할 수 있도록 모든 노력을 다하라는 명령을 내린 것이다.
		- 이것이 바로 k8s의 핵심 이론으로 정확히 어떤 액션을 수행해야하는지 지정하는 것이 아닌 시스템이 의도하는 상태를 선언적으로 지정하면 k8s가 실제 현재 상태를 검사해 의도한 상태로 **==재조정한다.==**
			- 그래서 이름이 늘 항로를 재 조정하는 쿠버네티스(항해사)라는 뜻이다.
		- 이제 `k get po`를 해보면 늘어난 pod를 확인할 수 있다.

&emsp;&emsp;

- 정리
	- 앱 스케일링은 앱이 프로덕션에서 실행 중이라면 언제든 확장이 필요할 때 새로운 복제본을 수동으로 설치하고 실행하지 않더라도 명령어 하나로 인스턴스를 추가할 수 있다.

&emsp;&emsp;

- 주의점
	- 앱 자체가 수평확장에 열린 구조로 만들어져야한다.
		- 예를 들면 앱이 세션을 처리해야하는 스테이트 풀 파드라고 가정하면 따로 이 세션 상태 DB를 분리하지 않는 한 애플리케이션은 세션에 열린 구조가 아니게 된다.
	- 참고로 직접 호출을 해보면 각각 다른 파드를 리턴한다.
		- 서비스 타입을 로드벨런서로 끼워두면 이런식이다.
		- NodePort로 뒀다면 이 서비스는 항상 하나의 파드를 바라보는 구조이다.
	- 근데 지금 세팅에서는 불가능하다. 애초에 run은 이제 단일 파드 하나만을 생성하는 명령이기 때문이다.
		- 서비스를 지우고 다시 올려야한다.
			- 우선 미니쿱에서 로드벨런싱이 가능하게 하려면 minikube tunnel로 로드벨런싱 타입 서비스를 외부 IP에 할당해준다.
				- 다만 이러면 프로세스가 새로 올라가기 때문에 터미널의 표준입력이 막히게되서 백그라운드로 돌려줘야한다.
				- `Start-Process minikube -ArgumentList "tunnel" -WindowStyle Hidden`윈도우 명령은 이렇고, 맥이나 리눅스는 nohup을 사용해주자.
			- `k expose deploy prom-depl --port=9090 --target-port=9090 --type=LoadBalancer --name prom-svc`로 새로 올려주자.
		- 그럼 이제 로드벨런싱으로 파드들이 돌려가며 리턴하는 모습을 볼 수있다.

### 앱이 실행 중인 노드 검사하기

- 설명
	- 파드 조회시 파드 IP와 실행 중인 노드 표시하기
		- `k get po -o wide` 이렇게 하면 추가 컬럼을 요청할 수있고 여기에 노드 정보도 적혀있다.
		- `k describe po 파드명` 이렇게 하면 파드 세부 정보를 볼 수 있고 여기에도 노드 정보가 있다.
