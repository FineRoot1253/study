## 볼륨

- 설명
	- 파드는 내부에 프로세스가 실행되고 CPU, RAM, 네트워크 인터페이스 등의 리소스를 공유하는 노리적 호스트와 유사하다.
	- 파드 내부의 각 컨테이너는 고유하게 분리된 파일시스템을 가지며 파일시스템은 컨테이너 이미지가 제공하기 때문에 프로세스가 디스크 또한 공유할 것 같지만 사실은 그렇지 않다고 앞에서 설명하였다.
	- 새로운 컨테이너 시작마다 컨테이너 이미지를 빌드할 때 추가한 파일들을 갖는 컨테이너를 시작한다.
		- ==**새로 시작한 컨테이너는 이전 컨테이너에 쓰여진 파일 시스템의 어떤 것도 알수가 없다.**==
	- k8s에서는 도커와 비슷하게도 볼륨을 정의하여 마운트를 할 수 있다.
		- k8s는 스토리지 볼륨을 정의하여 이를 제공하며 파드와 같은 최상위 리소스는 아니지만 파드의 일부분으로 정의되며 파드와 동일한 라이플사이클을 가진다.
		- 파드가 시작되면 볼륨이 생성되고 파드가 삭제되면 볼륨도 삭제된다.
		- 파드 내부의 컨테이너를 다시 날려도 파드는 계속 살아있기 때문에 계속 데이터가 남아있는 구조이다.
		- 파드 내부의 여러 컨테이너가 모든 컨테이너가 볼륨을 공유할 수도 있다.

&emsp;&emsp;

## 볼륨

- 설명
	- 정의
		- 파드의 구성 요소, 컨테이너와 동일하게 파드 스펙에서 정의되며 독립적인 k8s 오브젝트가 아니므로 자체적으로 생성, 삭제될 수 없다.
	- 특징
		- 볼륨은 파드의 모든 컨테이너에서 접근 가능하지만 컨테이너에서 각각 마운트가 되어야한다.
			- 각 컨테이너에서 파일 시스템의 어느 경로에서나 볼륨에 마운트할 수 있다.

&emsp;&emsp;

### 예제의 볼륨 설명

- 설명
	- /var/htdocs
		- 첫 컨테이너는 이 디렉터리에서 HTML 페이지를 서비스한다. (컨텐츠 웹 서빙)
	- /var/logs
		- 첫 컨테이너는 이 디렉터리에 HTML 페이지를 서비스한 뒤 로그를 기록한다.
		- 세번째 컨테이너는 이곳에서 로그를 압축, 분석등의 처리한다. (로그 정리)
	- /var/html
		- 두번째 컨테이너는 이 디렉터리에서 파일을 생성한다. (컨텐츠 생성기)
	- 예제의 특징
		- 별도의 분리된 컨테이너에서는 각각 /var 내부의 특정 디렉토리들에서 작업을 쳐도 상호작용이 일어지 않는다. 공유된 볼륨을 마운트한 상황이 아니기 때문
		- publicHTML 볼륨과 logVol 볼륨을 각각 마운트한다.
			- 이러면 각각 컨테이너마다 필요한 디렉토리에 접근하여 각자 하는 역할의 업무들이 다른 컨테이너들에 영향을 주게 된다.

&emsp;&emsp;

	- 클라우드 제공자 전용 스토리지 볼륨

## 볼륨을 이용한 컨테이너 간 데이터 공유

&emsp;&emsp;

### emptyDir 볼륨 사용

- 설명
	- 특징
		- 동일 파드내에서 실행 중인 컨테이너간 파일 공유에 매우 유용하다.
		- 단일 컨테이너 내에서 가용한 메모리에 넣기 큰 데이터 세트의 정렬 작업을 수행하는 것과 같이 임시데이터를 디스크에 쓰기 목적으로 사용 가능하다.
		- 데이터는 컨테이너 자체 파일 시스템에도 쓸 수 있지만 컨테이너 파일시스템은 쓰기가 불가할 수도 있고 마운트된 볼륨에 쓰는 것이 유일한 옵션일 수도 있다.
		- 가장 단순하며 이 볼륨을 기반으로 만들어진 볼륨들이 많다.
	- 파드에서 emptyDir 볼륨 사용
		- 예제에서는 fortune 인용구를 index.html에 저장하는 컨테이너를 추가한다.
	- emptyDir을 사용하기 위한 매체 지정하기
		- tmpfs 파일시스템을 변환도 가능하다.
			- `volumes.emptyDir.medium: Memory`를 넣으면 된다

&emsp;&emsp;

### 깃 레퍼지토리를 볼륨으로 사용하기

- 설명
	- gitRepo 볼륨 특징
		- emptyDir 볼륨이며 파드가 시작되면 깃 레포지터리를 복사하고 특정 리비전을 체크아웃해 데이터로 채운다.
		- gitRepo 또한 이미 deprecated된 리소스이며 emptyDir을 사용해 initContainers로 대체하는 것으로 바뀌었다.

``` yaml
spec:
  containers:
  - image: imageName
    name: containerName
    volumeMounts:
    - name: html
      mountPath: /usr/share/nginx/html
      readyOnly: true
    ports:
    - containerPort: 80
      protocol: TCP
  volumes:
  - name: html
    gitRepo:
      repository: repositoryUrl
      revision: release
      diretory: .
```

위가 옛날 버전 아래가 요즘 버전이다.

``` yaml
apiVersion: v1 
kind: Pod 
metadata: 
  name: example-pod 
spec: 
  volumes: 
  - name: git-volume 
    emptyDir: {} 
  initContainers: 
  - name: clone-repo 
    image: alpine/git 
    command: 
    - sh 
    - -c 
    - git clone https://repositoryUrl.git /gitrepo 
    volumeMounts: 
    - name: git-volume 
      mountPath: /gitrepo 
    containers: 
    - name: main-container 
      image: nginx 
      volumeMounts: 
      - name: git-volume 
        mountPath: /usr/share/nginx/html 
      ports: 
      - containerPort: 80
```

## 워커 노드 파일시스템의 파일 접근

- 설명
	- 대부분의 파드는 호스트 노드를 인식하지 못하므로 노드의 파일시스템에 있는 어떤 파일도 접근해서는 안된다.
	- 특전 시스템 레벨의 파드(데몬셋으로 관리되는 것들)는 노드의 파일을 읽거나 파일시스템을 통해 노드 디바이스를 접근하기 위해 노드의 파일시스템을 사용해야한다.

&emsp;&emsp;

### hostPath 볼륨

- 설명
	- 정의
		- 노드 파일시스템의 특정 파일이나 디렉터리를 가리키는 볼륨
	- 특징
		- 퍼시스턴스 스토리지 중 하나이다.
			- 파드가 종료되도 사라지지 않는다.
		- 파드 삭제시 다음 파드가 호스트의 동일 경로를 가리키는 hostPath 볼륨을 사용하고 이전 파드와 동일한 노드에 스케쥴링이 된다는 조건하에 새로운 파드는 이전 파드가 남긴 모든 항목을 볼 수 있다.
		- DB 데이터 디렉터리를 저장할 위치로 사용할 생각이라면 다시 고려해봐야한다.
			- 볼륨의 컨텐츠는 특정 노드의 파일시스템에 저장되므로 DB 파드가 다른 노드로 다시 스케줄링되면 더 이상 이전 데이터를 볼 수 없게된다.
			- hostPath 볼륨은 파드가 어떤 노드에 스케줄링 되느냐에 따라 민감하기 때문에 일반적인 파드에서 사용하는 것은 좋지않다.
			- 보통 DB 파드의 경우 스테이트풀 셋을 사용해야한다.
			- 스케일링, 환경에 따라 파드가 뒤바뀜 등등에 따라 어떤 조건으로 뒤바뀔지 아무도 예상하진 못하기 때문에 DB 파드의 경우에는 스테이트풀셋이라는 리소스로 세팅해야한다.
	- 결론
		- 노드의 시스템 파일에 읽기 쓰기를 하기위해서 사용해야한다.
			- 여러 파드에 걸쳐 데이터를 유지하기 위해 사용해서는 절대 안된다.

&emsp;&emsp;

### hostPath 볼륨을 사용하는 시스템 파드 검사하기

- 설명
	- `k get pod s --nampspace kube-system`
		- 시스템 전역 파드를 확인할 수 있다.
		- 첫번째 파드를 보면 hostPath를 사용중인 것을 알 수 있다.

## 기반 스토리지 기술과 파드 분리

- 설명
	- NFS 기반의 볼륨을 생성하기 위해서는 개발자는 NFS 익스포트가 위치하는 실제 서버를 알아야한다.
		- 이는 인프라의 세부사항에 대한 걱정을 없애고, 앱과 앱 개발자로부터 인프라를 몰라도 되게끔 한다는 k8s 기본 아이디어에 반하는 것이다.
		- 이상적으로는 k8s에 앱을 배포하는 개발자는 기저에 어떤 종류의 스토리지 기술이 사용되는지 알 필요가 없어야 하고 동일한 방식의 파드를 실행하기 위해 어떤 유형의 물리 서버가 사용되는지 알 필요가 없어야 한다.
			- 인프라 관련 처리는 클러스터 관리자(k8s)의 영역이여만 한다.
	- 개발자가 앱을 위해 일정량의 퍼시스턴트 스토리지를 필요로하면 k8s에 요청가능해야하며 동일한 방식으로 파드 생성시 cpu, 메모리와 다른 리소스를 요청할 수 있어야 한다.
		- 시스템 관리자는 클러스터를 구성해 앱이 요구한 것을 제공할 수 있어야 한다.

&emsp;&emsp;

### 퍼시스턴트 볼륨과 퍼시스턴트 볼륨 클레임

- 설명
	- 퍼시스턴트 볼륨(이하 PV)
		- 파드가 기술적인 세부 사항을 기재한 볼륨을 추가하는 대신 클러스터 관리자가 기반 스토리지를 설정하고 k8s API 서버에 PV 리소스를 생성해 k8s에 등록한다.
		- 퍼시스턴트 볼륨이 생성되면 관리자는 크기와 지원 가능한 접근 모드를 지정한다.
	- 퍼시스턴트 볼륨 클레임 (이하 PVC)
		- 클러스터 사용자가 파드에 퍼시스턴트 스토리지를 사용해야한다고 하면 먼저 최소 크기와 필요한 접근 모드를 명시한 PVC 매니페스트를 생성한다.
			- 이후 사용자는 PVC 매니페스트를  k8s API에 전송하고 k8s는 적절한 PV를 찾아 클레임에 볼륨을 바인딩한다.
			- 이후 PVC는 파드 내부의 볼륨 중 하나로 사용될 수 있다.
			- PVC의 바인딩을 삭제해 릴리스될 때까지 다른 사용자는 동일한 PV를 사용할 수 없다.

&emsp;&emsp;

### 퍼시스턴트 볼륨 생성

``` yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: prom-pv
spec:
  capacity:
    storage: 10Gi
  accessModes:
  - ReadWriteOnce
    persistentVolumeReclaimPolicy: Retain
  hostPath:
    path: "C:\\Users\\gjhon\\prom" 
    type: DirectoryOrCreate
```


- 설명
	- `k apply -f prom-pv.yaml`으로 생성해주자
		- create보다 apply를 권장하는 이유는 업데이트가 필요한 경우 업데이트를 같이 진행하기 때문이다.
	- DirectoryOrCreate
		- 이걸 넣어줘야 없는 디렉토리는 생성해서 작업을 진행한다.
	- persistentVolumeReclaimPolicy
		- 이것을 Retain으로 지정해야 관련 클레임이 해제되어도 볼륨을 유지한다.
	- ReadWriteOnce
		- 단일 클라이언트의 읽기/쓰기용이라는 뜻이다.
		- 보통 ReadOnlyMany와 같이 쓰이곤 한다. 이는 여러 노드(클라이언트)에서 동시에 읽는 것을 허용한다.
			- 참고로 파드의 수와는 관계가 없고 노드의 수와 관계가 있다.
	- `k get pv`
		- 이것을 입력하면 사용가능한(available)한 볼륨을 볼 수 있다.
			- 클레임 할당 대기 중이라는 뜻이다.

&emsp;&emsp;

### PVC 생성을 통한 PV 요청

``` yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: prom-pvc
spec:
  resources:
	requests:
	    storage: 10Gi
  accessModes:
  - ReadWriteOnce
  storageClassName: ""
```

- 설명
	- 파드가 재 스케줄링(삭제되거나 줄어들거나 등등)되더라도 동일한 PVC이 사용 가능한 상태로 유지되기를 원하므로 ==**PVC는 파드를 생성하는 것과 별개이다.**==
	- 접근 모드의 다수의 클라이언트는 파드가 아닌 노드와 연관이 있다.
	- `k get pvc`
		- 생성된 pvc를 나열한다.
		- Bound 상태가 된것은 pv가 제대로 할당이 된 것을 의미한다.
	- 접근 모드 별 사용되는 약어
		- RWO
			- 단일 노드만이 읽기/쓰기용으로 볼륨을 마운트할 수 있음
		- ROX
			- 다수 노드가 읽기용으로 볼륨을 마운트할 수 있음
		- RWX
			- 다수 노드가 읽기/쓰기 용으로 볼륨을 마운트할 수 있음

&emsp;&emsp;

### 파드에서 PVC 사용하기

``` yaml
apiVersion: v1
kind: Pod
metadata:
  name: prom-with-pvc
spec:
  containers:
  - image: prom/prometheus:latest
    name: prom-with-pvc
    args: 
    - "--config.file=/etc/prometheus/prometheus.yml" 
    - "--storage.tsdb.path=/prometheus"
    ports: 
    - containerPort: 9090
    volumeMounts:
    - name: prom-pv 
      mountPath: /prometheus
  volumes: 
  - name: prom-pv 
    persistentVolumeClaim: 
      claimName: prom-pvc
```


&emsp;&emsp;

### PV와 PVC의 장점

- 설명
	- 앱 개발자가 기저에 사용된 실제 스토리지 기술을 알 필요가 없다.
	- 돌일한 파드와 PVC 매니페스트는 인프라와 관련된 어떤 것도 참조하지 않으므로 다른 k8s 클러스터에도 써먹을 수 있다.

&emsp;&emsp;

### PV 재사용하기

- 설명
	- 파드와 pvc를 삭제하고 다시 pvc를 생성하면 바로 바인딩 되지 않는다.
		- 이전에 사용한 파드가 설치한 데이터가 볼륨에 존재하기 때문에 비워내지 않으면 진행되지 않는 것이다.
	- 수동으로 PV 다시 클레임하기
		- PV 리소스를 삭제하고 다시 생성한다.
			- 다만 백업은 해야한다.
	- 자동으로 PV 다시 클레임하기
		- Recyle 정책
			- 볼륨의 컨텐츠를 삭제하고 볼륨이 다시 클레임될 수 있도록 볼륨을 만든다.
			- PV가 여러번 다른 PVC와 다른 파드에서 재사용할 수 있다.
			- GCE에서는 사용 불가능하다.
		- Delete 정책
			- 볼륨의 컨텐츠를 아예 삭제시켜버린다.


<br><br>

## PV 동적 프로비저닝

- 설명
	- 프로비저닝
		- 실제 스토리지를 사용가능한 상태로 확보하는 것
	- 동적 프로비저닝
		- 클레임이 삭제되어도 자동으로 실제 스토리지를 사용가능한 상태로 확보하는 것
	- PV 프로비저너
		- 클러스터 관리자가 PV를 생성하는 대신를 사용해 사용자(개발자)가 선택 가능한 PV의 타입을 하나 이상의 스토리지 클래스 오브젝트(이하 SC)로 정의할 수 있다.
		- 사용자가 PVC에서 SC를 참조하면 프로비저너가 퍼시스턴트가 스토리지를 프로비저닝할 때 이를 처리한다.
	- SC
		- PV와 비슷하게 동작하는 리소스
		- PV와 마찬가지로 네임스페이스에 속하지 않는다.
		- 관리자가 많은 퍼시스턴트 볼륨을 미리 프로비저닝하는 대신 하나 혹은 그 이상의 SC를 정의하면 시스템은 누군가 PVC를 통해 요청시 새로운 PV를 생성한다.
			- 이의 장점은 항상 사용 가능한 PV가 존재한다는 것이고 단점은 스토리지 용량이 금새 부족해진다.
	- 주의점
		- k8s는 대부분의 인기 있는 클라우드 프로바이더(이하 CP)의 프로비저너를 포함하므로 관리자가 항상 프로비저너를 배포할 필요가 없지만 온프레미스 환경에서는 사용자 정의 프로비저너가 배포되어야 한다.

&emsp;&emsp;

### SC 리소스를 통해 사용 가능한 스토리지 타입 정의하기

``` yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast
provisioner: k8s.io/minikube-hostpath
parameters:
  type: pd-ssd
```

- 설명
	- 내 경우는 미니쿱에서 제공하는 hostpath 기반 프로비저너를 사용한다.
	- 만약 GCE를 사용한다면 zone까지 입력해둬야한다.

&emsp;&emsp;

### PVC에서 SC 요청하기

``` yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: prom-pvc
spec:
  resources:
	requests:
	    storage: 10Gi
  accessModes:
  - ReadWriteOnce
  storageClassName: fast
```

- 설명
	- `k get pvc prom-pvc`
		- 바로 바운드된 모습을 볼 수 있다.
	- `k get pv`
		- 새로 생성된 pv를 볼 수 있다.
		- 기존에 생성된 pv는 pvc가 날라갔으므로 Released 상태에 있다. 비워야 사용 가능 상태가 된다
	- SC 사용법
		- 클러스터 관리자는 성능이나 기타 특성이 다른 여러 SC를 생성할 수 있고 개발자는 생성할 클레임에 맞는 SC를 골라서 쓰기만 하면된다.
		- SC는 클레임 이름으로 이를 참조하는 것이 장점이다.
			- 다른 클러스터간 SC의 이름을 똑같이 사용한다면 PVC 정의를 다른 클러스터로 이식이 가능하다.
				- GKE를 지금까지 사용했다면 AWS나 Minikube에도 PVC 정의를 그대로 가져다 쓸 수 있다는 것
				- 클러스터관리자는 다른 SC를 생성해두긴 해야한다.

&emsp;&emsp;

- 주의점
	- 동적 프로비저닝은 항상 Delete가 기본 동작방식이다.
		- PVC를 지울 때에는 항상 PV를 백업해야한다.
		- CP 환경에서는 EBS를 사용하여 스냅샷을 뜨면 되지만 미니쿱에서는 따로 Velero 같은 스냅샷 도구를 사용해야한다.

&emsp;&emsp;

### SC 미지정 동적 프로비저닝

- 설명
	- 이미 따로 만들어두지 않아도 GKE, 미니쿱에서 사용 가능한 sc가 존재한다.
		- `k get sc standard -o yaml`로 확인해보자

&emsp;&emsp;

``` yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: prom-pvc
spec:
  resources:
	requests:
	    storage: 10Gi
  accessModes:
  - ReadWriteOnce
```

- 스토리지 클래스를 지정하지 않고 PVC 생성하기
	- storageClassName 필드를 통째로 날려도 정상 동작하는 것을 볼 수 있다.
		- 이 필드를  "" 빈문자열로 두면 미리 확보된 PV로 바인딩된다.
			- 반드시 큰 따옴표여야 한다.


&emsp;&emsp;

- PV 동적 프로비저닝 전체 그림 이해하기
	- PVC(storageClassName를 지정한)와 파드만 생성하는 것이 퍼시스턴트 스토리지를 연결하는 최적의 방법이다.
