## 파드에서 실행 중인 앱 업데이트

- 설명
	- 모든 파드를 업데이트 하는 방법
		- 기존 파드를 모두 삭제한 다음 새  파드 시작
			- 장점
				- 비교적 빠르게 모든 파드 업데이트 가능
			- 단점
				- 짧은 시간 동안 앱 사용 불가능
		- 새로운 파드 시작 후 기동 시 기존 파드를 삭제
			- 새 파드를 모두 추가한 다음 한꺼번에 기존 파드를 삭제하거나 순차적으로 새 파드를 추가하고 기존 파드를 점진적으로 제거한다.
			- 장점
				- 앱 사용 불가능 시간이 없음
			- 단점
				- 사용자는 한순간동안 버전이 다른 앱을 이용해야한다.
					- 새 버전이 이전 버전을 손상 시킬 수 있는 데이터 스키마나 데이터의 수정을 해서는 안된다.

&emsp;&emsp;

### 오래된 파드 삭제 후 새 파드 교체 방식

- 설명
	- RS의 파드 템플릿은 언제든지 업데이트가 가능하다.
		- 새 인스턴스 생성시(파드 생성시) 업데이트된 파드 템플릿을 사용한다.
		- v1을 관리하는 파드 템플릿을 지닌 RS가 있다면 v2로 파드 템플릿을 업데이트한 후 이전 파드 인스턴스를 삭제한다.
	- 짧은 시간 발생하는 다운 타임을 허용하는 경우에만 사용된다.

&emsp;&emsp;

### 새 파드 기동과 이전 파드 삭제 방식

- 설명
	- 한 번에 이전 버전에서 새 버전 전환
		- 블루 - 그린 디플로이먼트
		- 새 버전을 실행하는 파드를 불러오는 동안 서비스는 파드의 이전 버전에 연결되는 구조
		- 모든 파드가 새버전이 되면 서비스의 레이블 셀렉터를 수정하여 서비스를 새 파드로 전환한다.
		- 전환 후 새 버전에 문제가 없으면 이전 RS를 삭제해 이전 파드를 삭제한다.
	- 롤링 업데이트 수행
		- 이전 버전의 RS를 천천히 스케일 다운하고 새 파드를 스케일 업해 이를 수행한다.
		- 서비스의 파드 셀렉터에 이전 파드와 새 파드를 모두 포함시켜 요청을 두 파드 세트로 보낼 수 있다.
		- 수동으로 롤링 업데이트는 어렵고 오류가 발생하기 쉽다.
			- 레플리카 수에 따라 업데이트 프로세스를 수행하려면 올바른 순서로 수많은 명령어 수행이 필요하다.
				- k8s에서는 간단하다.

## RS로 자동 롤링 업데이트 수행

&emsp;&emsp;

### 앱의 초기 버전 실행

``` yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: my-replicaset-v1
  namespace: my-namespace
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-container
        image: my-image
---
apiVersion: v1
kind: Service
metadata:
  name: my-service
  namespace: my-namespace
spec:
  selector:
    app: my-app
  ports:
    - protocol: TCP
      port: 80
      targetPort: 8080
```

- 설명
	- 레플리카셋 yaml에 `---`으로 구분 지어 서비스를 추가할 수 있다.
	- 주의점
		- 동일한 이미지로 태그를 달아 다른 버전의 앱을 배포하는 경우 이미지를 동일하게 가져와 파드를 생성한다.
			- 변경한 내용을 같은 이미지 태그로 푸시하면 변경이 되지 않는다.
			- 파드가 이미 올라간 노드에서는 늘 예전 버전을 사용하게되고, 파드를 새로 올리게 될 노드에서는 새 버전을 들고오는 문제가 발생한다.
		- imagePullPolicy는 기본이 ifNotPresent이지만 always를 달아주면 항상 새로 이미지를 받아와서 노드에 배포를 한다.
			- 시간은 좀 걸리지만 이 방식을 사용하면 항상 새 파드를 올리게 된다.
		- 가장 좋은 방법은 항상 새 버전에는 새로이 태그를 달아 이미지를 푸시하는 것이다.
	- `k rolling-update my-replicaset-v1 my-replicaset-v2 --image=newImage`
		- 이 명령으로 롤링 업데이트를 간단하게 수행한다.
		- 수행과정
			1. 기존 파드의 레이블들을 수정한다.
			2. kubectl은 기존 RS의 레이블들도 파드의 레이블과 동일하게 수정한다.
				1. 서비스는 파드의 레이블을 보고 요청을 전달하기 때문에 기존 파드의 레이블과 RS의 레이블을 동시에 수정하는 것이다.
			3. kubectl은 기존 RS를 복사하고 해당 파드 템플릿에서 이미지를 변경해 새 RS를 만든다.
				1. 이때 레플리카 수는 0이다.
			4. 첫번째 v2 파드를 만들고 이전 RS에서 하나씩 스케일 다운한다.
			5. 이전 RS의 래플리카 수가 0이 되면 이전 RS는 삭제돼 이제 V2 파드만 지원한다.

&emsp;&emsp;

### k rolling-update를 더 이상 사용하지 않는 이유

- 설명
	- 롤링 업데이트의 모든 수행의 주체는 kubectl 클라이언트이다.
		- kubectl 클라이언트가 k8s API 서버에 요청을 보내 k8s 마스터 대신 스케일링을 수행한다.
		- 클라이언트가 주체이기 때문에 서버가 맛탱이가 간다면, 롤링 업데이트 수행 절차 자체가 망가지게 된다.
		- 생각보다 k8s API 서버는 잘 고장이 난다.
	- 클라이언트가 명령하는 방식이기 때문에 사용되지 않는다.
		- k8s는 자연스럽게 의도하는 시스템의 상태를 선언하고 k8s가 알아서 수행하는 것이 궁극적인 목표이지 직접 명령을 받아서 행동 하나하나를 수행하는 방식이 아니다.

## 앱을 선언적으로 업데이트하기 위한 디플로이먼트 사용하기

- 설명
	- 디플로이먼트는 RS보다 더 높은 수준의 리소스이다.
	- 디플로이먼트 생성시 RS 리소스가 그 아래에 생성된다.

&emsp;&emsp;

### 디플로이먼트 생성

``` yaml
apiVersion: apps/v1beta1
kind: Deployment
metadata:
  name: kubia
spec:
  replicas: 3
  template:
    metadata:
      name: kubia
      labels:
        app: kubia
    spec:
      containers:
      - iamge: imageName
        name: appName
```

- 설명
	- 디플로이먼트 리소스 생성
		- `k create -f deploymentFile.yaml --record`
			- `--record`
				- 개정 이력에 명령어를 기록해 나중에 유용히 사용이 가능하므로 반드시 넣어줘야한다.
		- `k apply -f deploymentFile.yaml --record`
			- apply
				- 이게 더 많이 사용된다. 업데이트가 필요한 경우 업데이트를 같이 진행하기 때문
	- 디플로이먼트 롤아웃 상태 출력
		- `k rollout status deployment deploymentName`
	- 디플로이먼트가 RS를 생성하는 방식과 RS가 파드를 생성하는 방식
		- 파드 이름 중간에 파드 템플릿 해시값이 하나 들어가는데 RS가 이러한 파드를 관리함을 뜻한다.
			- 정확하게는 파드템플릿 해시값이 맨 끝에 달리고 RS 해시값이 중앙에 달린다.
			- `k get rs`를 하면 RS 이름 뒤에 똑같은 해시값이 들어가있는 것을 볼 수 있다.
		- ==**디플로이먼트는 파드 템플릿의 각 버전마다 하나씩 여러 개의 RS를 만든다.**==
	- 서비스로 파드 엑세스하기

&emsp;&emsp;

### 디플로이먼트 업데이트

- 설명
	- 기본 동작
		- 원하는 의도를 디플로이먼트 리소스 파드 템플릿을 수정하여 반영하면 k8s가 리소스에 정의된 상태로 만드는 데 필요한 모든 단계를 수행한다.
	- 사용 가능한 디플로이먼트 전략
		- 기본 전략
			- RollingUpdate
				- 이전 파드를 하나씩 제거하고 동시에 새 파드를 추가해 전체 프로세스에서 앱을 계속 사용할 수 있도록한다. 
				- 앱에서 이전 버전과 새 버전을 동시에 지원해도 클라이언트에 지장이 없는 경우에만 사용해야한다.
				- 서비스 다운 타임이 없다.
		- 대안 전략
			- Recreate
				- 새 파드 생성 전 모든 파드를 삭제하므로 새 버전 실행 이전에 이전 버전의 완전한 중지가 필요한 경우에만 사용된다.
				- 서비스 다운타임이 무조건 발생한다.
		- 특징
			- 의도하는 래플리카 수를 많게하거나 적게하여 상한과 하한을 설정할 수 있다.
	- 데모 목적으로 롤링 업데이트 속도 느리게 하기
		- `k patch deployment deploymentName -p '{"spec":{"minReadySeconds": 10}}'`
			- minReadySeconds필드 속성을 부여해 의도적으로 약간 느리게 업데이트를 수행할 수 있다.
			- 이것으로 롤링 업데이트의 수행 과정을 천천히 지켜 볼 수도 있다.
		- patch 명령은 리소스 속성 한두개를 변경하는데 자주 사용된다.
			- 파드 템플릿을 변경한 것은 아니여서 파드가 업데이트 되지는 않는다.
				- 의도하는 레플리카 수, 디플로이먼트 정책과 같은 배포 속성도 롤아웃을 야기하진 않는다.
				- 롤아웃은 기존 파드에 영향을 주지 않기 때문이다.
	- 롤링 업데이트 시작
		- 수행 과정
			1. 파드 컨테이너에 사용된 이미지 버전을 변경한다.
				1. `k set image deployment deploymentName containerName=iamgeName`
				2. `k set image`명령을 이용하면 yaml 수정 없이 진행 가능하다.
	- 디플로이먼트의 장점
		- 파드 템플릿 변경 하나만으로 애플리케이션을 최신 버전으로 업데이트 했다.
			- 파드에 직접 하나씩 올리고 내리고를 똥 고생을 해보면 이게 말도 안되는 일이라는 것을 알 수 있다.

&emsp;&emsp;

### 디플로이먼트 롤백

- 설명
	- `k rollout undo deployment deploymentName`
		- 오류가 존재하는 앱이 올라간 경우 이전 버전으로 되돌린다.
		- 이 명령은 롤아웃이 진행되는 도중에 적용해도 이전 버전 파드는 냅두고 롤아웃을 이미 진행한 파드를 다시 되돌리는 작업을 진행한다.
			- 따라서 롤아웃이 느리게 진행된다고 가정하더라도 도중에라도 명령 호출이 문제되지 않고 잘 적용이 된다.
	- 롤아웃 이력 표시 (revision history)
		- 특징
			- 이력은 기본 RS에 저장된다.
			- 롤아웃 완료시 이전 RS는 삭제되지 않으므로 모든 버전으로 롤백이 가능하다.
				- 이전 롤링 업데이트에서는 삭제했지만 롤아웃 명령의 경우에는 revisionHistoryLimit만큼 남게된다.
				- 10개가 디폴트이다.
		- 명령
			- `k rollout history deployment deploymentName`
	- 특정 디플로이먼트 개정으로 롤백
		- `k rollout undo deployment deploymentName --to-revision=revisionNumber`
			- undo 명령시 `--to-revision`으로 버전을 지정할 수 있다.

&emsp;&emsp;

### 롤아웃 속도 제어

- 설명
	- 롤링 업데이트 전략
		- maxSurge
			- 디플로이먼트가 의도하는 레플리카 수보다 얼마나 많은 파드 인스턴스 수를 허용할 수 있는지 결정한다.
			- 기본적으로 25%로 설정되며 의도한 개수보다 25%보다 많은 파드 인스턴스 수를 벗어날 수 없다는 것을 의미한다.
			- 의도하는 레플리카수가 4인 경우, 업데이트 도중 동시에 5개 이상 파드 인스턴스가 실행되지 않는다.
				- 추가로 생성 가능한 레플리카의 수는 25프로인 1개를 의미한다.
			- 백분율을 절대 숫자로 변환하면 숫자가 반올림된다.
				- 26인 경우 2개로 간주한다.
			- 백분율 대신 절댓값 지정이 가능하다.
		- maxUnavailable
			- 업데이트 중에 의도하는 레플리카 수를 기준으로 사용할 수 없는 인스턴스 수를 결정한다.
			- 25%가 기본이며 사용가능한 파드의 인스턴스 수는 의도하는 레플리카 수의 75%이하로 떨어지지 않아야한다.
			- 레플리카 수가 4이고 25%인 경우 하나의 파드만 사용할 수 없어야한다.
				- 항상 처리 가능한 파드는 3개이상 존재해야만 한다는 것을 의미한다.
			- 백분율을 절대 숫자로 변환하면 숫자가 반올림된다.
			- 백분율 대신 절댓값 지정이 가능하다.
		- 특징
			- maxSurge, maxUnavailable 둘 다 0이면 롤링 업데이트 자체가 불가능해진다.
			- maxSurge가 1, maxUnavailable가 0이면, 파드를 하나를 생성하고 사용 가능해지면 하나를 삭제하는 방식으로 진행한다.
				- 생성 즉시 가용가능한 상태가 아니지만, 그럼에도 불구하고 생성이 가능한 이유는 ==**설정해둔 래플리카 수의 기준으로 동작하기 때문이다.**==
			- maxSurge가 0, maxUnavailable가 1이면, 파드를 하나를 삭제하고 다시 하나를 추가하는 방식으로 진행한다.

&emsp;&emsp;

### 롤아웃 프로세스 일시 중지

- 설명
	- 롤아웃 일시 정지
		- 특징
			- 일시적으로 두가지 버전을 존재하게 하여 문제가 없는지 확실하게 확인하기 위해 존재하는 기능
				- 이 경우를 카나리 릴리즈라고 부르는데 이 효과를 볼 수 있는 방법이다.
				- 잘못된 버전의 앱이 롤아웃되 모든 사용자에게 영향을 주는 위험을 최소화하는 기법이다.
				- 새 버전의 파드를 적은 수만 만들고 이전 버전을 유지하는 것
		- 명령
			- `k set image deployment deploymentName containerName=imageName`
			- `k rollout pause deployment deploymentName`
	- 롤아웃 재개
		- 명령
			- `k rollout resume deployment deploymentName`
	- 롤아웃 방지를 위한 일시 중지 기능
		- 특징
			- 일단 디플로이먼트 롤아웃 업데이트를 막은 다음 충분히 속성을 변경한 다음 적용하게 할 수 있다

&emsp;&emsp;

### 잘못된 버전의 롤아웃 방지

- 설명
	- `minReadySeconds`
		- 특징
			- 파드가 사용 가능하다는 판단이 내려지기 전에 새로 만든 파드를 준비할 시간을 지정한다.
			- 모든 파드의 레디니스 프로브가 성공해야 파드가 준비된다.
			- ==**만약 이 시간이 지나기 전에 레디니스 프로브가 실패하면 새 버전의 롤아웃이 효과적으로 차단된다.**==
			- 롤아웃을 계속하기 전에 파드가 준비된 후 k8s가 10초 동안 대기하도록해 롤아웃 프로세스 속도를 늦추는 것이다.
			- ==**보통 실제 트래픽을 수신하기 시작한 후 파드가 준비 상태를 보고할 수 있도록 충분히 여유있게 시간을 세팅하는 편이다.**==
	- 오류 버전을 완전 배포하는 것을 방지하는 레디니스 프로브 정의 방법
		- minReadySeconds = 10, maxSurge = 1, maxUnavailble = 0, periodSeconds = 1로 설정하는 경우
			- 오류 버전이 하나 생성되어 올라가지만 ready 상태가 되진 않는다.
	- 롤아웃 데드라인 설정
		- 롤아웃은 10분동안 진행되지 않으면 실패로 간주한다.
			- `k describe deployment`를 입력하면 `ProgressDeadlineExceeded` 조건이 명시된다.
		- `progressDeadlineSeconds` 속성으로 설정할 수 있다.
