## 안정적으로 파드 유지 하기

- 설명
	- k8s의 주요 장점은 k8s에 컨테이너 목록을 제공하면 해당 컨테이너를 클러스터 어딘가 계속 실행하고 있도록 만들 수 있다는 점이다.
		- 파드 리소스를 생성하고 k8s가 이 파드를 실행할 워커 노드를 지정하며 해당 노드에서 파드의 컨테이너가 실행되도록 함으로써 이 작업을 수행한다.
	- 파드는 노드에 스케줄링 되는 목표 노드의 kubelet은 파드의 컨테이너를 실행하고 파드가 존재하는 한 컨테이너가 계속 실행되게끔한다.
		- 만약 컨테이너의 주 프로세스에 크래시가 나면 kubelet이 컨테이너를 다시 시작한다.
		- k8s의 자동 리커버리 기능은 자동으로 주어진다는 뜻이다.
	- 만약 스프링 앱에서 OOM이 발생하더라도 JVM 프로세스는 계속 수행될 것이다.
		- 이런 경우 앱이 더이상 정상 작동을 하지 않는다는 것을 k8s에 보내 k8s가 앱을 재시동하도록 만드는 것이 좋다.
		- 크래시가 발생한 컨테이너는 자동으로 다시 시작하긴 하지만 모든 크래시에 대해서 해결하지는 못한다.
			- 특히 이런 경우는 외부의 존재에게 의해 오류를 감시하다가 k8s에게 알려주도록 하는 것이 좋다.

### 라이브니스 프로브

- 설명
	- 정의
		- k8s가 컨테이너 외부에서 컨테이너의 생존 여부를 확인하는 오브젝트
	- 특징
		- 파드의 spec에 각 컨테이너의 라이브니스 프로브를 지정할 수 있다.
		- k8s는 주기적으로 프로브를 실행하여 프로브가 실패하는 경우 컨테이너를 재시동한다.
	- k8s가 기용하는 컨테이너와 프로브의 매커니즘
		- HTTP GET 프로브
			- 지정한 IP, 포트, 경로에 HTTP GET 요청을 보낸다.
			- 프로브가 응답 수신 후 응답 코드가 오류를 나타내지 않는 경우 성공으로 간주한다.
			- 서버가 오류 응답 코드를 반환하거나 응답이 없는 경우 실패로 간주하여 컨테이너를 재시동한다.
		- TCP 소켓 프로브
			- 컨테이너의 지정된 포트에 TCP 연결을 시도한다.
			- 연결에 성공하는 경우 프로브는 성공으로 간주한다.
			- 연결에 실패하는 경우 프로브는 실패로 간주한다.
		- Exec 프로브
			- 컨테이너 내의 임의의 명령을 실행하고 명령의 종료 상태 코드를 확인한다.
			- 상태 코드가 0이면 프로브는 성공으로 간주한다.
			- 상태 코드가 0이외의 모든 숫자인 경우 실패로 간주한다.

&emsp;&emsp;

### HTTP 기반 라이브니스 프로브 생성

``` yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: prometheus-liveness  
spec:  
  containers:  
    - name: prometheus-manual  
      image: bitnami/prometheus:latest  
      imagePullPolicy: IfNotPresent
      livenessProbe:
	  httpGet:
	    path: /
		port: 9090
      ports:  
        - containerPort: 9090  
          protocol: TCP
```

- 설명
	- 주기적으로 / 경로에 9090포트로 HTTP GET을 요청해서 컨테이너가 정상 동작하는지 확인하도록 httpGet 라이브니스 프로브를 정의한다.
		- 이 요청은 컨테이너 실행 즉시 시작된다.

### 동작 중인 라이브니스 프로브 확인

- 설명
	- 예시에서는 문제가 있는 앱을 감싼 이미지를 통해 컨테이너를 생성하기 때문에 매번 재시작을 하는 것을 볼 수 있긴하다.
		- RESTARTS 컬럼을 보게 되면 이것이 재시동한 횟수이다.
	-`k logs podName --previous`
		- 명령은 이전 파드의 로그 기록을 보기 위한 명령이다.
		- 왜 재시동을 했는지 파악하기 위해 필요한 명령옵션이다.
	- 예시에서는 137 시그널을 보내며 종료한 로그가 남아있는데 이는 128 + x이며 x는 프로세스에 전송된 시그널 번호이며 이 시그널로 인해 컨테이너가 종료된 것이다.
		- x는 SIGKILL, 시그널 번호 9번이다.
	- ==**재시동하는 컨테이너는 처음부터 컨테이너를 재생성하는 것이지 정지했다가 다시 시작하는 것이 아니다.**==

### 라이브니스 프로브의 추가 속성 설정

- 설명
	- `k describe`는 라이브니스 프로브에 대한 추가정보도 표시해준다.
		- 명시적으로 지정한 라이브니스 프로브 옵션 외에도 지연, 제한 시간, 기간 등과 같은 추가 속성을 볼 수도 있다.
		- delay
			- 지연 시간
			- 0인 경우 프로브가 컨테이너가 시작된 후 바로 프로브가 시작된다는 것을 나타낸다.
		- timeout
			- 제한 시간
			- 컨테이너는 이 시간 안에 응답을 못하면 실패로 간주하여 컨테이너를 재시작한다.
		- period
			- 프로브 반복 수행시간
			- 10초인 경우 10초에 한번 씩 요청을 보내 프로브 역할을 수행한다. 
		- success
			- 성공 제한 횟수
		- failure
			- 실패 제한 횟수
			- 이 횟수 이상 실패하는 경우 컨테이너를 재시작한다.
	- initialDelaySeconds
		- 초기 지연 시간
		- 라이브니스 프로브는 이 시간 이후 요청을 보내 프로브 역할을 수행하기 시작한다.
		- 대부분 컨테이너는 제대로 초기화하는데 시간이 걸리므로 초기 지연시간을 충분히 주지 않으면 안된다.
	- 컨테이너 종료 코드
		- 137 또는 143은 외부에서 파드를 종료 했음을 알 수 있다.
		- 137
			- 128 + 9 (SIGKILL)
		- 143
			- 128 + 15 (SIGTERM)

### 효과적인 라이브니스 프로브 생성 전략

- 설명
	- 라이브니스 프로브가 확인해야할 사항
		- 대부분 액추에이터와 같은 라이브러리와 같은 방식으로 health 상태를 표기하는 방식을 사용하는 것이 일반적이며 이것만 해둬도 대다수의 문제 상황을 해결해주곤 한다.
			- 대신 이러한 경로에는 권한을 둬서는 안된다. 어디에서든 접속이 가능하게끔 표현해야한다.
	- 프로브는 앱의 상황만 체크하고 외부 상황은 체크하도록 해서는 안된다.
		- 만약 FE 파드가 DB 접속에 실패하였다고 한들 이건 BE 파드의 문제이기 때문에 FE 라이브니스 프로브가 실패로 간주해서는 안된다.
	- 프로브는 가볍게 두기
		- 너무 많은 연산 리소스를 사용하게끔 하면 안된다.
			- 가볍고 자주 수행되어야 하며 무조건 1초 이내에 확인이 완료되어야한다.
			- 너무 무거운 프로브는 연산에 부하만 주는 나쁜 리소스 돼지가 될 수도 있다.
		- JVM은 기동 절차가 상당히 무거운 편에 속한다.
			- 소스 컴파일, 클래스 파일 재 로딩 등 상당히 무거운 편이긴하다.
			- 이런 경우에는 Exec보단 HTTP Get 방식이 대다수 현명한 선택인 편이다.
				- Exec를 사용하면 jvm 자체를 새로 기동해야하기 때문이다.
	- 프로브는 재시도 루프를 둬선 안된다.
		- 어차피 k8s가 컨테이너가 실패했다고 간주하기 전에 여러 번 요청을 보내 재시도를 하므로 굳이 재시도 루프를 구현할 필요가 없다. 헛수고이다.
	- 라이브니스 프로브 요약
		- 라이브니스 프로브는 외부에서 컨테이너의 생존 여부를 확인하는 활동이다.
		- 노드의 kubelet에서 수행되며 마스터에서 실행 중인 k8s 컨트롤플레인 구성 요소는 이에 관여하지 않는다.
			- 크래시 발생시 중단된 모든 파드의 대체 파드 생성에만 관여한다.
			- ==**직접 생성한 파드의 경우 kubelet에서 관리되기 때문에 이런 파드는 노드가 고장나면 아무것도 할 수 없다.**==
			- 애플리케이션이 다른 노드에서 다시 시작되도록하려면 레플리케이션 컨트롤러 또는 유사한 매커니즘으로 파드를 관리해야한다.

## 레플리케이션 컨트롤러

- 설명
	- 정의
		- k8s의 리소스로 파드가 항상 실행되도록 보장한다.
	- 특징
		- k8s 클러스터에서 노드가 사라지거나 노드에서 파드가 제거되는 경우 사라진 파드를 감지해 교체 파드를 생성한다.
		- 레플리케이션 컨트롤러가 관리하지 않는 노드의 파드는 사라져도 관여를 하지 않는다.
		- 지금은 디플로이먼트와 레플리카셋이 이 위치를 대체하고 있다. 옛날에는 이런게 있었구나 하고 보고 넘어가자

&emsp;&emsp;

### 레플리케이션 컨트롤러의 동작

- 설명
	- 역할
		- 실행 중인 파드 목록을 지속적으로 모니터링
		- 특정 타입의 실제 파드 수가 의도하는 수와 일치하는지 항상 확인
		- 파드가 너무 적게 실행 중인 경우 파드 템플릿에서 새 복제본을 만든다.
		- 너무 많은 파드가 실행 중인 경우 초과 복제본을 제거한다.
	- 파드 수가 너무 많아지는 경우
		- 누군가 같은 타입의 파드를 수동으로 생성
		- 누군가 기존 파드의 타입을 변경
		- 누군가 의도하는 파드의 수를 줄이기
	- 주의점
		- 파드 타입은 존재하지 않는다.
			- 앞에서 본 레이블 셀렉터에 의해 동작하는 것이며 이것을 이해하기 쉽게 그냥 타입이라고 줄여 부른 것이다.
	- 컨트롤러 조정 루프 과정
		1. 레이블 셀렉터와 매치되는 파드를 찾는다.
		2. 매치되는 파드 수와 의도하는 파드 수를 비교한다.
			1. 너무 많은 경우 초과하는 파드를 삭제하여 동일하게 만든다.
			2. 너무 적은 경우 현재 탬플릿에서 파드를 추가 생성하여 동일하게 만든다.
		3. 2번 동작 수행 결과 동일하거나 동일해졌다면 1번 동작부터 다시 수행한다.
	- 레플리케이션 컨트롤러의 세 가지 요소
		- 레이블 셀렉터
			- 레플리케이션 컨트롤러의 범위에 있는 파드를 결정
		- 레플리카 카운트
			- 실행할 파드의 의도하는 수를 지정
		- 파드 템플릿
			- 새로운 파드 레플리카를 만들 때 사용
	- 컨트롤러의 레이블 셀렉터 또는 파드 템플릿 변경의 영향 이해
		- 레이블 셀렉터, 파드 템플릿의 변경은 기존 파드에 영향을 주지 않는다.
		- 레이블 셀렉터 변경시 기존 파드가 레플리케이션 컨트롤러의 범위를 벗어난다.
			- 이때 컨트롤러가 해당 파드에 대한 관리를 중지한다.
		- 레플리케이션 컨트롤러는 파드 생성 이후 파드의 실제 컨텐츠 (컨테이너 이미지, 환경 변수 및 기타 사항)에 신경을 쓰지 않는다.
			- 그러므로 템플릿은 레플리케이션컨트롤러(이하 rc)로 새 파드를 생성할 때만 영향을 미친다.
			- 템플릿을 새 파드를 만들기 위한 쿠키 커터로 생각할 수 있다.
	- rc 사용시 이점
		- 기존 파드가 사라지면 새 파드를 시작해 파드가 항상 실행되도록 한다.
		- 클러스터 노드에 장애 발생시 장애가 발생한 노드에서 실행 중인 모든 파드에 관한 교체 복사본이 생성된다.
		- 수동 또는 자동으로 파드를 쉽게 수평으로 확장할 수 있게 한다.
	- 주의점
		- 파드 인스턴스는 다른 노드로 재배치 되지 않는다.
			- RC는 교체되는 인스턴스와 관련이 없는 완전히 새로운 파드 인스턴스를 만든다.

### rc 작동 확인

- 설명
	- `k get po`를 이용해 실제 rc 정의에 따라 파드가 생성되었는지 확인 가능하다.
	- 삭제된 파드에 관한 rc 반응 확인
		- `k delete po 파드명`
			- 하나를 수동으로 삭제하면 파드 수를 다시 맞추는 지 확인 할 수 있다.
	- rc 정보 얻기
		- `k get rc`
			- 의도하는 파드 수
			- 실제 파드 수
			- 준비된 파드 수
		- `k describe rc rc명`
			- 실제 대 의도하는 파드 인스턴스 수
			- 파드의 상태별 파드 인스턴스 수
			- rc 관련된 이벤트
	- rc가 새로 파드를 생성한 이유를 파악하기
		1. rc가 새 파드를 만들어 파드 삭제에 대응한다. (정확히는 정의한 레플리카 수에 대응하는 것)
		2. rc는 삭제되는 파드에 대해 즉시 통지를 받는데 api 서버는 클라이언트가 리소스 및 리소스 목록의 변경을 감시할 수 있도록 허용하기 때문이다.
			1. 이 통지 자체가 대체 파드를 생성하게 하는 것이 아니라 실제 파드 수를 확인하고 적절히 조치하기 위한 트리거로써의 역할을 수행한다.
	- 노드 장애 대응
		- 노드에 장애가 발생하는 경우 k8s는 즉시 새 파드를 올린다.

&emsp;&emsp;

### RC의 범위 안팎으로 파드 이동하기

- 설명
	- rc가 생성하느 파드는 어떤 식으로든 rc와 묶이지 않는다.
		- rc는 레이블 셀렉터와 일치하는 파드만을 관리한다.
		- 파드의 레이블을 변경하면 rc 범위에서 제거되거나 또는 추가될 수도 있다.
			- 그래서 한 rc에서 다른 rc로 이동할 수도 있다.
	- 파드는 metadata.ownerReferences 필드에서 rc를 참조한다.
		- 파드의 레이블을 변경해 더 이상 rc의 레이블 셀렉터와 일치하지 않게 만들면 해당 파드는 수동으로 만든 다른 파드처럼 된다.
			- 더 이상 아무도 이 파드를 관리하지 않는다.
			- 파드를 실행하는 노드에 장애 발생시 파드는 다시 스케줄링되지 않는다.
				- 단, 파드의 레이블을 변경하면 파드가 하나 사라진 것을 rc가 감지하고 사라진 파드를 대체하기 위해 새로운 파드를 기동한다.
	- rc는 파드에 추가 레이블이 있는지를 상관하지 않기 때문에 별도의 레이블을 추가하는 것은 별 효과가 없다.
		- 오직 파드가 레이블 셀렉터에서 참조하는 모든 레이블을 갖고 있는지만 고려한다.
	- rc가 관리하는 파드에 레이블 추가
		- 관리되는 파드에 레이블을 추가해도 rc는 상관안하는 것을 볼 수 있다.
	- rc가 관리하는 파드의 레이블 변경
		- 관리되는 파드의 레이블을 더 이상 rc에 정의된 레이블 셀렉터와 다르게 변경하면 rc는 해당 파드를 찾지 못해 파드가 줄어들었다고 판단하여 새로운 복사본을 만들어낸다.
	- rc에서 파드를 제거하는 실제 사례
		- ==**특정 파드에서 어떤 작업을 하려는 경우, 해당 파드의 레이블 셀렉터를 변경해 RC 범위 부터 변경해야한다.**==
			- 예시
				- 일정 시간이 지난 후 특정 이벤트가 발생한 후 파드가 제대로 동작하지 않는 버그가 있다고 가정해보자, 파드가 오작동한다는 것을 안다면 파드를 RC 밖으로 빼 RC가 새 파드로 교체하게 끔 한 다음, 오작동하는 파드를 디버깅하거나 문제를 재현해볼 수 있다.
					- 완료되면 파드는 삭제한다.
	  - rc의 레이블 셀렉터 변경
		  - 이 경우 기존의 파드를 두고 새로운 파드를 레플리카 수에 맞춰서 생성한다.
		  - k8s는 rc의 레이블 셀렉터를 변경하도록 허용하지만 이 챕터 후반부에 나오는 리소스들은 그렇지 않다.

&emsp;&emsp;

### 파드 템플릿 변경

- 설명
	- rc의 파드 템플릿은 언제든지 수정이 가능하다.
		- ==**기존 파드를 수정하려면 해당 파드를 삭제하고 RC가 새 템플릿 기반으로 새 파드를 교체해야한다.**==
	- `k edit rc rc명`
		- 이 명령을 사용시 yaml 정의 편집기가 열린다.
		- 변경사항 수정 후 편집기를 종료하고 파드를 조회해보면 별 변경사항이 없는 것을 알 수 있다.
			- 기존 파드 삭제 후 새로 생성하는 경우 변경되는 것을 볼 수 있다.
		- 참고로 9장에서 더 나은 수정 방법이 존재한다.
	- `zshrc`파일 등에 `export KUBE_EDITOR="/usr/bin/vim` 등을 넣어 편집기를 선택할 수 있다.
		- 기본 정의된 편집기를 사용하지만 이걸 넣어주면 이 방식대로 동작한다.

&emsp;&emsp;

### 수평 파드 스케일링

- 설명
	- ctl 사용법
		- `k scale rc rc명 --replicas=10`
	- 정의 편집 사용법
		- `k edit rc rc명`
			- spec.replicas 필드 숫자 변경 후 저장시 바로 파드가 10개로 늘어난다.
	- 스케일 다운하는 법
		- ctl 사용법
			- `k scale rc rc명 --replicas=3`
	- 스케일링의 선언적 접근 방법
		- 파드 수평 확장은 x개의 인스턴스 수행을 원한다는 이 바램을 전달하는 것이다.
			- 의도하는 상태를 지정하는 것이다.
		- 이 책 후반부에 나오는 오토 스케일링이 k8s의 꽃이라고 할 수 있다.

&emsp;&emsp;

### rc 삭제

- 설명
	- `k delete`을 통해 삭제시 파드는 그대로 살아있다.
		- `--cascade` 옵션을 추가하면 해당 파드를 같이 삭제시킬 수 있다.

## RC 대신 레플리카셋(이하 RS) 사용하기

- 설명
	- 최신 도입 기능로 이 책에서는 여기서부터 RS 만을 사용한다. 
		- 진작 이랬어야하지 않나 싶다 이젠 아예 deprecated 되었다.
		- 디플로이먼트 리소스를 생성하면 RS가 자동으로 생성된다.
	- RC와 RS의 차이
		- RS는 RC보다 풍성한 표현식을 사용하는 파드 셀렉터를 지니고 있다.
			- 레이블 셀렉터와는 달리 레이블이 없는 파드, 레이블의 값과 상관없이 특정 레이블이 키를 갖는 파드를 매칭시킬 수 있다.
			- RC는 레이블이 서로 다른 값을 지닌 경우에 같은 파드를 선택할 수 없는 단점이 존재한다.
				- RS는 레이블이 서로 다른 값을 지녀도 한 그룹에 묶을 수 있는 표현식을 지니고 있다.
			- RS는 레이블 키의 존재만으로 매칭 시킬 수 있다.

&emsp;&emsp;

### RS 정의하기

``` yaml
apiVersion: apps/v1beta2
kind:ReplicaSet
metadata:
  name: rsName
spec:
  replicas: 3
  selector:
    matchLabels:
		app: padName
template:
  metadata:
  labels:
    app: rsName
 spec:
   containers:
	- name: containerName
	  image: userName/imageName:tag

```

- 설명
	- 주의점
		- RS는 기본적으로 v1 API의 일부가 아니다.
			- 리소스 생성시 적절한 apiVersion을 선택해야만 한다.
		- 파드가 가져야하는 레이블은 selector.matchLables 필드로 내려야한다.
			- RS에서 레이블을 지정하는 가장 단순한 방식이다.
		- apiVersion 속성은 두 가지를 지정하는데, Api 그룹과 Api 버전이다.
			- Api 그룹은 이 경우 apps이며, Api 버전은 v1beta2이다.
		- 이미 셀렉터와 매칭되는 파드는 여전히 실행중이기 때문에 새 파드가 생성되지 않을 것이다.
			- 이런 경우 RS는 기존의 3개의 파드를 자신의 관리하에 두게 된다.

&emsp;&emsp;

### RS 생성 및 검사

- 설명
	- RS 조회
		- `k get rs`
	- RS 상세 조회
		- `k describe rs`

&emsp;&emsp;

### RS의 더 세밀한 표현의 레이블 셀렉터 사용하기

``` yaml
selector:
  macthExpressions:
    - key: app
      operator: In
      values:
        - labelName
```

- 설명
	- 연산자 종류
		- In
			- 레이블이 지정된 값 중 하나와 일치해야한다.
		- NotIn
			- 레이블의 값이 지정된 값과 일치하지 않아야 한다.
		- Exists
			- 파드는 지정된 키를 가진 레이블이 포함되어야 한다.
			- 값 필드 지정 금지
		- DoesNotExist
			- 파드에 지정된 키를 가진 레이블이 포함되어 있지 않아야한다.
			- 값 필드 지정 금지
	- 여러 표현식을 지정하는 경우 모든 표현식이 true 이여야 한다.
		- matchLabels와 matchExpressions를 모두 지정하면 셀렉터가 파드 지정을 위해 모든 레이블이 일치하고 모든 표현식이 true로 평가되어야한다.

&emsp;&emsp;

### RS 정리

- 설명
	- RS 삭제 명령
		- `k delete rs rsName`

## 데몬셋(이하 DS)을 사용해 각 노드에서 정확히 한 개의 파드 실행하기

- 설명
	- 클러스터의 모든 노드에 노드 당 하나의 파드만 실행되길 바라는 경우 사용된다.
	- 예시
		- 시스템 수준의 작업을 수행하는 인프라 관련 파드
		- 모든 노드에서 로그 수집기와 리소스 모니터를 실행하려는 경우
		- kube-proxy
			- 서비스 작동을 위해 모든 노드에서 실행되어야한다.
	- k8s가 아닌 환경에서는 보통 노드 부팅 중 시스템 초기화 스크립트(init script) 또는 systemd 데몬을 통해 시작된다.
	- k8s 노드에서도 systemd를 사용해 시스템 프로세스를 수행할 수 있지만 k8s의 기능을 제대로 활용하는 방식은 아니라서 권장되지는 않는다.


&emsp;&emsp;

### DS로 모든 노드에 파드 실행하기

- 설명
	- 모든 클러스터 노드마다 파드를 하나만 실행하려면 DS 오브젝트를 생성해야한다.
	- DS에 의해 생성되는 파드가 존재할 타깃 노드가 이미 지정되어 있고 k8s 스케줄러를 건너뛰는 것을 제외하면 이 오브젝트는 RC나 RS와 매우 비슷하다.
	- DS는 파드 셀렉터와 일치하는 파드 하나가 각 노드에서 실행 중인지 확인 하는 것이 DS의 역할이다.
	- 노드 다운시 DS는 다른 곳에서 파드를 생성하지 않는다.
		- 새 노드가 클러스터에 생성되는 경우에는 즉시 새 파드 인스턴스를 새 노드에 배포한다.
		- 노드에 누가 실수중 DS의 파드를 날리는 경우에도 포함된다.
	- DS도 RS와 매한가지로 그 안에 구성된 파드 템플릿을 기반으로 파드를 생성한다.

&emsp;&emsp;

### DS을 사용해 특정 노드에서만 파드 실행하기

- 설명
	- node-Selector를 사용하면 가능하다.
	- 주의점
		- DS가 관리하는 파드는 스케줄러와 무관하다.
			- 스케줄링 되지 않는 노드에도 시스템 서비스를 실행할 수 있어야 하므로 DS를 사용하는 것이 마땅하다.

&emsp;&emsp;

### 예제와 함께 보는 DS

- 설명
	- SSD를 갖는 모든 노드에서 실행되어야하는 ssd-monitor라는 데몬을 넣는다고 가정한다.
		- ssd를 갖고 있다고 표현된 모든 노드에서 이 데몬을 실행하는 DS를 만든다.
		- 클러스터 관리자가 이런 모든 노드에 disk=ssd label을 사용했으므로 레이블이 있는 노드만 선택하는 node-selector를 사용해 ds를 작성한다.

&emsp;&emsp;

``` yaml
apiVersion: apps/v1beta2
kind:DaemonSet
metadata:
  name: ssd-monitor
spec:
  selector:
    matchLabels:
		app: ssd-monitor
template:
  metadata:
  labels:
    app: ssd-monitor
 spec:
   nodeSelector:
	  disk: ssd
   containers:
	- name: main
	  image: userName/imageName:tag
```

&emsp;&emsp;

- 설명
	- 데몬셋 생성
		- `k create -f ssd-monitor-ds.yaml`
	- 데몬셋 조회
		- `k get ds`
	- 데몬셋 상세조회
		- `k describe ds ssd-monitor`
	- 필요한 레이블 추가하기
		- `k get node`
			- 레이블을 추가할 노드 확인하기
		- `k label node minikube disk=ssd`
			- 레이블 추가
		- `k get po`
			- 레이블 추가 후 파드 생성 확인
	- 레이블 변경하기
		- `k get node`
		- `k label node minikube disk=hdd --overwrite`
		- `k get po`
			- 레이블 변경 후 파드 제거 확인

## 완료 가능한 단일 작업을 수행하는 파드 실행

- 설명
	- completable task는 프로세스 종료 후 다시 생성되지 않는 파드를 의미한다.

&emsp;&emsp;

### 잡 리소스

- 설명
	- 정의
		- 완료 가능한 테스크를 수행 후 다시 시작하지 않는 파드
	- 특징
		- 어느 작업을 완료 후 다시 실행하지 않는 파드를 실행할 수 있다.
		- 노드에 장애가 발생한 경우 노드에 있던 잡이 관리하는 파드는 RS 파드와 같은 방식으로 다른 노드로 다시 스케줄링된다.
		- 프로세스 자체에 장애가 발생한 경우 잡에서 컨테이너를 다시 시작할 것인지 설정도 가능하다.
		- 장애 발생시 노드에서 제거되는 경우 수동으로 다시 생성해야한다.
		- 이 또한 잡이 관리하는 파드가 존재한다.
		- ==**파드 완료시 삭제가 되진 않는 이유는 로그 검사를 위함이다.**==
	- 예시
		- 임시 작업
		- 데이터를 어딘가에 저장하고 있고 이 데이터를 변환해 어딘가로 전송해야하는 경우
			- 배치작업이 대표적이다.
			- 배치식으로 알림을 보내는 경우도 있을 것이고 배치는 아직도 여러 군데에서 사용되는 유용한 해결방식이다.

&emsp;&emsp;

### 잡 리소스 정의

``` yaml
apiVersion: batch/v1beta2
kind:Job
metadata:
  name: batch-job
spec:
  template:
    metadata:
    labels:
      app: ssd-monitor
   spec:
     restartPolicy: OnFailure
     containers:
	  - name: main
	    image: userName/imageName:tag
```

&emsp;&emsp;

### 파드를 실행한 잡 조회

- 설명
	- `k get jobs`
	- `k get po`
	- `k get po -a`
		- 프로세스 수행 후 얼마 뒤 완료된 파드를 확인 하는 명령이다.
		- completed로 상태가 표기되어있다.

&emsp;&emsp;

### 잡을 여러 파드 인스턴스로 수행하기

``` yaml
apiVersion: batch/v1beta2
kind:Job
metadata:

```

- 설명
	- 이 잡은 차례로 5개의 파드를 실행한다.
		- 하나의 파드가 완료되면 다음 파드를 만들어서 수행한다.

#### 병렬 잡 파드 수행

``` yaml
apiVersion: batch/v1beta2
kind:Job
metadata:
  name: batch-job
spec:
  completions: 5
  parallelism: 2
  template:
    metadata:
    labels:
      app: ssd-monitor
   spec:
     restartPolicy: OnFailure
     containers:
	  - name: main
	    image: userName/imageName:tag
```


- 설명
	- 파드를 2개를 동시에 생성해서 병렬로 수행하고 완료되면, 다시 다음 파드를 2개를 생성해서 동시에 병렬로 수행한다.
	- `k scale job multi-completion-batch-job --replicas 3`
		- 스케일링 명령을 통해 동시에 수행될 파드의 수를 조절할 수 있는데 replica 명령으로 조절이 가능하다.

&emsp;&emsp;

### 잡 파드가 완료되는 데 걸리는 시간 제한하기

- 설명
	- `activeDeadlineSeconds` 속성을 설정해 파드의 시행 시간을 제한할 수 있다.
		- 보통 잡에는 이 옵션을 충분히 주는 경우가 많다.
	- `spec.backoffLimit` 필드를 조절하면 재시도 횟수를 제한할 수 있다.
		- 기본 값은 6이다.

## 잡을 주기적으로 또는 한 번 실행되도록 스케줄링하기

- 설명
	- 잡 리소스는 생성 즉시 파드를 생성해 수행한다.
	- 배치 잡은 주로 지정 시간 또는 지정된 간격으로 반복 수행되므로, 리눅스에서는 크론 이라는 이름으로 부르고 있다.
	- k8s에서는 CronJob(이하 CJ)이라는 리소스를 만들어 구성한다.
	- 잡 실행을 위해 위한 스케줄은 잘 알려진 크론 형식으로 지정한다.
	- k8s는 설정된 시간에 잡 리소스를 크론잡 오브젝트에 설정한 잡 템플릿에 따라 생성한다.
		- 잡 리소스가 생성되면 앞에서 배운 것처럼 하나 이상의 파드 복제본이 잡의 파드 템플릿에 따라 생성되고 시작된다.

&emsp;&emsp;

### CJ 리소스 생성하기

``` yaml
apiVersion: batch/v1beta2
kind:CronJob
metadata:
  name: batch-job-every-fifteen-minutes
spec:
  schedule: "0, 15, 30, 45 * * * *" // 매일, 매 시간 0, 15, 30, 45분에 수행된다.
  jobTemplate:
    metadata:
    labels:
      app: ssd-monitor
   spec:
     restartPolicy: OnFailure
     containers:
	  - name: main
	    image: userName/imageName:tag
```

- 설명
	- 크론 스케줄 타입
		- 왼쪽부터 오른쪽으로 다섯개의 항목을 가진다.
			- 분
			- 시
			- 일
			- 월
			- 요일
	- 예시
		- 매일, 새벽 1시에 수행되어야한다.
			- "0 1 * * *"
		- 매 달, 첫째 날에 30분마다 수행되어야한다.
			- "0, 30 * 1 * *"
	- 잡 템플릿의 특성에 따라 잡 리소스를 생성한다.

&emsp;&emsp;

### 스케줄된 잡의 실행 방법 이해

- 설명
	- 잡 리소스는 대략 예정된 시간에 크론잡 리소스에서 생성된다.
	- 잡이나 파드가 상대적으로 늦게 생성되고 실행될 수 있다.
		- `startingDeadlineSeconds` 필드를 이용해 데드라인을 설정할 수 있다.
		- 현재 정의한 yaml에 15로 지정하는 경우 매 0분 15초, 15분 15초, 30분 15초, 45분 15초 까지 수행되지 않으면, 실패로 표시한다.
		- 크론잡은 설정한 각 실행에 항상 하나의 잡을 생성하지만, 두 개의 잡이 동시에 생성되거나 전혀 생성하지 않을 수도 있다.
			- 두 개의 잡이 동시에 생성되는 경우 해결법
				- 항상 여러번 수행되도 문제 없도록 멱등성을 지니게 해야한다.
			- 전혀 생성되지 않는 경우 해결법
				- 다음 번 잡 실행이 이전의 실행해서 완료했어야하는 작업을 수행하는지 확인해야한다.
					- 잡을 한번에 다 수행을 이어서 할 수 있는지 검증되어야한다.
