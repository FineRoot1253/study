# 서비스: 클라이언트가 파드를 검색하고 통신을 가능케 함

- 설명
	- 대다수 앱들은 외부의 요청을 응답하기 위한 처리를 위해 사용되며 이것이 k8s를 사용하는 대부분의 이유이다.
	- MSA의 경우 파드는 대게 클러스터 내부의 다른 파드나 클러스터의 외부의 클라이언트에서 오는 HTTP 요청에 응답한다.
	- 파드가 다른 파드에게 제공하는 서비스를 이용하기 위해서는 다른 파드를 찾는 방법이 필요하다.
		- k8s가 아닌 세계에서는 시스템 관리자가 클라이언트 구성파일에 서비스를 제공하는 서버의 정확한 IP 주소나 호스트 이름을 지정해 각 클라이언트 애플리케이션을 구성하는 것과 달리 k8s에서는 제대로 동작하지 않는다.
	- 다른 환경의 방식이 k8s에서 제대로 동작하지 않는 이유
		- 파드는 일시적이라 노드에서 제거되거나, 누군가 파드 수를 줄이거나, 클러스터 노드 장애로 다른 노드로 이사갈 수도 있다.
		- k8s는 노드를 파드에 스케줄링한 후 파드가 시작되기 바로 전에 IP를 할당하므로 미리 IP를 지정할 수가 없다. 미리 알 수 없기 때문
		- 수평 스케일링은 여러 파드가 동일한 서비스를 제공할 수 있음을 의미한다.
			- 각 파드는 고유한 IP 주소가 있고 클라이언트는 서비스가 지원하는 파드의 수와 IP에 상관하지 않아야한다.
				- 즉, 클라이언트 측에서 파드의 개별 IP 목록을 몰라야 제대로 된 서비스 수행이 가능하다는 것이다. 그래서 단일 Ip 주소로 서비스를 제공가능해야 한다.


<br><br>

## 서비스 리소스

- 설명
	- 정의
		- 동일한 서비스를 제공하는 파드 그룹에 지속적인 단일 접점을 제공하기 위해 존재하는 리소스
	- 특징
		- 각 서비스는 서비스가 존재하는 동안 절대 바뀌지 않는 IP 주소와 포트가 있다.
		- 클라이언트는 해당 IP와 포트로 접속한 다음 해당 서비스를 지원하는 파드 중 하나로 연결된다.
		- 서비스의 클라이언트는 서비스를 제공하는 개별 파드의 위치를 알 필요 없으므로 이 파드는 언제든지 클러스터 안에서 이동할 수 있다.
	- 예제를 통한 설명
		- FE 웹서버와 BE 데이터베이스 서버가 있는 예제를 보자 FE 역할을 하는 파드는 여러 개 있을 수 있지만 BE 파드는 하나만 있을 것이다.
			- 시스템 기동시 해결이 필요한 문제
				- 웹 서버가 수 백 수 천 개가 있어도 외부 클라이언트는 FE 파드에 연결을 할 수 있어야 한다.
				- FE 파드는 BE에 연결해야한다.
					- DB는 파드내에서 실행되므로 시간이 지남에 따라 클러스터 주위를 이동해 IP 주소가 변경될 가능성이 있어 BE DB가 이동할때마다 FE 파드를 재설정해야할 필요성을 제거해야한다.
			- 해결 방안
				- 서비스는 총 두 개를 만들어야한다.
					- FE 외부 연결용 서비스
					- BE DB 내부 연결용 서비스

&emsp;&emsp;

### 서비스 생성

- 설명
	- 서비스는 레이블 셀렉터를 이용해 생성한다.

&emsp;&emsp;

``` yaml
apiVersion: v1
kind: Service
metaData:
  name: svcName
spec:
  ports:
  - port: 80
    targetPort: 9090
  selector:
    app: podAppLableName
```

- 설명
	- 새 서비스를 80포트에 연결하고 app=podAppLableName과 일치하는 레이블 셀렉터와 매칭하여 파드의 8080포트로 라우팅한다.
	- `k expose`를 이용하는 것이 가장 단순하지만 yaml을 API에 던져주는 것이 k8s에서 권장하는 방식이다.

&emsp;&emsp;

- 설명
	- 새 서비스 조회
		- `k get svc`
	- 클러스터 내에서 서비스 테스트 방법
		- 서버의 클러스터 IP에 요청을 보내고 응답으로 로그로 남기는 파드를 만드는 것
			- 가장 확실하고 가장 자주 사용하는 방식이다.
			- 파드의 로그만 확인하면 서비스의 응답확인이 쉽고 빠르기 때문이다.
		- k8s 노드로 ssh 접속하고 curl을 날린다.
		- k exec 명령으로 기존 파드에서 curl 날린다.
	- 실행 중인 컨테이너에 원격으로 명령어 실행
		- `k exec` 명령 사용시 기존 파드 컨테이너 내에서 원격으로 임의의 명령어 수행이 가능하다.
			- 컨테이너 내용, 상태, 환경 검사시 유용하다.
		- `k get po`로 파드 조회 후 `k get svc` 로 IP를 찾은 뒤 `k exec`를 IP를 집어넣어 수행한다.
			- 예시
				- `k exec podName -- curl -s http://10.111.249.153`
					- 동작 순서
						1. curl 요청으로 서비스  IP에 요청을 전달한다.
							- 이 ==**요청 자체는 웹 서버가 있는 파드 컨테이너 내부에서 보내는 것**==이다.
							- ==**컨테이너 내부에서 수행한 요청이 서비스 프록시로 전달**==되는 것이다.
						2. k8s 서비스 프록시가 연결을 인터셉트해 3개의 파드 중 임의의 파드로 요청을 전달한다.
						3. 해당 파드 내에서 실행 중인 웹서버는 요청을 처리하여 해당 파드의 이름을 포함하는 http 응답을 반환한다.
						4. curl은 표준 출력으로 응답을 출력하고 kubectl이 있는 로컬 시스템의 표준 출력으로 전달한다.
		- 주의점
			- `--` 더블 대시 사용 이유
				- kubectl 명령줄 옵션의 끝을 의미한다. 이 뒤는 파드 내에서 실행되어야 하는 명령이다.
				- 명령줄에 대시로 시작하는 인수가 없는 경우 더블 대시를 사용할 필요가 없다.
					- 인수가 없는 경우 사용할 이유는 없다.
				- 이전 예시에서는 더블 대시를 사용하지 않으면 오류가 발생한다.
					- `k exec podName curl -s http://10.111.249.153`
						- 이건 서비스가 연결을 거부하는 것과는 아무런 연관이 없고 kubectl이 10.111.249.153과 연결을 할 수 없기 때문이다.
						- `k expose -s` 옵션은 kubectl에게 기본값과 다른 api 서버에 연결하도록 지시하는 옵션이다.
	- 세션 어피니티 구성 방법
		- 특정 클라이언트의 요청을 매번 같은 파드로 하고 싶다면(리디렉션을 하고 싶다면) 서비스의 세션 어피니티 (spec.sessionAffinity)속성을 None이 아닌 ClientIP로 설정한다.
		- 주의점
			- 쿠키 기반 세션 어피니가 없는 이유는 HTTP 수준에서 작동하지 않는데, TCP와 UDP 패킷을 처리하고 그들이 가진 페이로드는 신경쓰지 않는다.(서비스 리소스는 L4 전송계층만 관여한다는 의미이다.)
				- 쿠키는 HTTP 프로토콜의 구성이므로(L7 응용 계층이므로) 서비스는 쿠키를 알지  못하기 때문에 쿠키를 기반으로 동작하는게 불가능하다.
	- 동일한 서비스에서 여러 개의 포트 노출
		- 서비스는 단일 포트만 노출하지만 여러 포트에 지원 가능하다.
			- 파드가 두 개의 포트를 수신한다면 하나의 서비스를 이용해 파드의 포트를 전달한다.
			- 굳이 두 개의 서비스를 만들 필요는 없다는 뜻이다.
			- spec.ports는 여러개의 포트가 매핑이 가능하다.
	- 이름이 지정된 포트 사용
		- 파드 정의에 spec.containers.ports에서 name과 containerPort를 각각 여러개를 등록한다고 가정하면 이를 서비스에서 사용할 수 있다.
			- 서비스에서 사용할 때는 spec.ports.targetPort를 사용한다.
		- ==**이는 현업에서 꽤 자주 사용되는데, 이렇게 사용하면 파드의 포트를 변경할 때 서비스까지 변경할 일이 없어진다.**==

&emsp;&emsp;

### 서비스 검색

- 설명
	- 환경변수를 통한 서비스 검색
		- 만약 파드 생성 전 서비스를 올린다면 서비스는 환경변수에서 파드를 검색하여 파드에 대한 정보를 얻을 수 있다.
		- 그러나 파드 생성 후 서비스를 올린다면, 서비스에 대한 환경 변수를 설정할 수 없다.
			- 따라서 모든 파드를 삭제하고 RS에서 새로 파드를 생성해야만 한다.
			- `k delete po -all`
			- 이후 `k exec` 명령어의 대상으로 파드를 선택가능하게 된다.
			- 대상 파드를 선택하면 `k exec podName env`이 `env` 명령어를 실행해 환경변수를 조회할 수 있게 된다.
		- 즉, ==**결론적으로 서비스가 파드를 바라보는 구조가 아니라 파드가 서비스를 바라보는 구조이기 때문에 환경변수 설정이 안되는 것이다.**==
			- 파드가 서비스를 바라보아야 파드가 재생성이 되어도 서비스가 찾아가기때문에 서비스는 파드보다 우선적으로 생성되어야 환경변수 세팅에 문제가 없다.
		- 참고
			- 서비스 이름의 -는 \_로 변환된다.
	- DNS를 통한 서비스 검색
		- kube-system 네임스페이스에 존재하는 파드인 kube-dns를 이용하는 방법이다.
		- 클러스터에서 실행 중인 다른 모든 파드는 자동으로 이를 사용하도록한다.
			- `/etc/resolv.conf`파일에서 관련 세팅을 수정할 수 있다.
		- 파드에서 실행 중인 프로세스에서 수행된 모든 DNS 쿼리는 시스템에서 실행중이 모든 서비스를 알고 있는 k8s의 자체 DNS서버로 처리된다.
			- 참고로 파드가 내부 DNS 서버를 이용할지 안할지는 각 파드 spec.dnsPolicy 속성으로 구성할 수 있다.
	- FQDN을 이용한 서비스 연결
		- DNS에서 검색이 가능하기 때문에 FQDN을 사용할 수 있다.
			- `serviceName.default.svc.cluster.local`
				- serviceName
					- 서비스이름
				- default
					- 포함된 네임 스페이스 이름
				- svc.cluster.local
					- 모든 클러스터의 로컬 서비스 이름에 사용되는 클러스터 도메인 접미사
		- 참고로 같은 네임스페이스에 있는 파드의 서비스에 연동하는 것은 더욱 간단하다.
			- 접미사와 네임스페이스를 날릴 수 있기 때문인데 이러면 더 간단하게 서비스 이름만 사용하면 된다.
		- 따라서 대다수는 IP가 아닌 FQDN을 이용해서 서비스에 엑세스하는 것이 정석이다.
			- `k exec -it podName bash`
			- `curl http://serviceName`
				- 이렇게 서비스 이름만으로 접속이 가능해진다.
			- 컨테이너 내부의 `/etc/resolv.conf`를 보면 다음과 같을 것이다.
				- `search default.svc.cluster.local svc.cluster.local cluster.local ...`
					- k8s 내부 dns resolver가 구성되어 생략이 가능한것이다. 쿼리를 날린 기록이 남아있다
	- 주의할 점
		- 서비스 클러스터 IP는 가상 IP이므로 서비스 포트와 결합된 경우에만 의미가 있기 때문에 ping은 동작하지 않는다.
		- 따라서 포트 연동 이후 서비스에 핑을 날려서 왜 안돼지를 말해봐야 의미가 없다.

## 클러스터 외부에 있는 타 서비스 연결

- 설명
	- 위의 설명은 내부의 서비스와 파드간의 연결이며 외부 서비스를 노출 하는 방법은 여러가지가 존재한다.
	- 외부 IP와 포트로 연결을 전달하는 것인데, 서비스 로드벨런싱과 서비스 검색 모두 활용 가능하다.

&emsp;&emsp;

### 서비스 엔드포인트 소개

- 설명
	- ==**서비스는 파드에 직접 연결되는 구조가 아니며 그 사이에 엔드포인트 리소스가 존재한다**==.
	- `k get endpoints svcName`
		- 서비스로 노출되는 파드의 IP 주소와 포트 목록 반환
	- 파드 셀렉터는 서비스 스펙에 정의돼 있지만 들어오는 연결을 전달할 때 직접 사용하지는 않는다.
		- 대신 셀렉터는 IP와 포트 목록을 작성하는 데 사용되고 엔드 포인트 리소스에 저장된다.
	- 클라이언트가 서비스에 연결하면 서비스 프록시는 이들 중 하나의 IP와 포트 쌍을 선택하고 들어온 연결을 대상 파드의 수신 대기 서버로 전달한다.
	- ==**실제 서비스는 클라이언트와 파드와의 연결을 매칭시켜 줄 때 파드의 실제 IP 정보가 있는 엔드포인트 리소스를 살펴보고 이를 통해 연결하는 것이다.**==

&emsp;&emsp;

### 서비스 엔드포인트 수동 구성

- 설명
	- 서비스의 엔드포인트와 서비스를 분리하면 엔드포인트를 수동으로 구성하고 업데이트 가능해진다.
	- 파드 셀렉터 없이 서비스를 만들면 k8s는 엔드포인트 리소스를 만들지 못한다. (파드 셀렉터가 없으니 서비스에 포함된 파드를 알지 못해서 만들지 못하는 것)
	- 수동으로 관리되는 엔드포인트를 사용해 서비스를 만들려면 서비스와 엔드포인트 리소스를 모두 만들어야한다.

&emsp;&emsp;

``` yaml
apiVersion: v1
kind: Service
metadata: 
  name: extarnal-svc
spec:
  ports:
  - port: 80
```


``` yaml
apiVersion: v1
kind: Endpoints
metadata: 
  name: extarnal-svc
subsets:
  - addresses:
    - ip: 11.11.11.11
    - ip: 22.22.22.22
    ports:
    - port: 80
```

- 설명
	- 서비스
		- 서비스는 파드 셀렉터를 제외하고 구현하였다.
	- 엔드포인트
		- 엔드포인트의 이름은 서비스의 이름과 일치해야한다.
		- 서비스를 제공하는 대상 IP 주소와 포트 목록을 가져야한다. (파드 IP 목록이다.)
	- 서비스와 엔드포인트 리소스가 모두 서버에 게시되면 파드 셀렉터가 있는 일반 서비스처럼 서비스를 사용할 수 있다.
	- 서비스가 만들어진 뒤 파드컨테이너는 서비스의 환경변수가 포함되며 IP:포트 쌍에 대한 모든 연결은 서비스 엔드포인트 간에 로드벨런싱한다.

&emsp;&emsp;

- 결론
	- 외부서버 11.11.11.11:80과 22.22.22.22:80에 대한 정보를 서비스를 알게 되어 파드가 사용가능해진 것이다.

&emsp;&emsp;

### 외부 서비스를 위한 별칭 생성

``` yaml
apiVersion: v1
kind: Service
metadata: extarnal-svc
spec:
  type: ExternalName
  externalName: someApi.someCompany.com
  ports:
  - port: 80
```

- 설명
	- 파드는 서비스의 FQDN을 사용하는 대신 external-svc.default.svc.cluster.local 도메인 이름 (혹은 external-svc)로 외부 서비스에 연결이 가능해진다.
	- 나중에 externalName 속성을 변경하거나 type을 ClusterIP로 변경하여 서비스 스펙을 추가하여 수정하면 다른 서비스를 가리키게 할 수도 있다.
	- 서비스를 위한 엔드 포인트 오브젝트를 수동 혹은 서비스에 레이블 셀렉터를 지정해 엔드포인트가 자동으로 생성되도록 한다.
	- ExternalName 서비스는 당연하게도 DNS 레벨에서만 구현된다.
		- 서비스에 관한 간단한 CNAME DNS 레코드가 생성된다.
			- CNAME은 domain - domain간 가리키는 레코드이다. (FQDN-FQDN)
		- 서비스에 연결하는 클라이언트는 서비스 프록시를 완전히 무시하고 외부 서비스에 직접 연결된다.
		- 즉 이러한 이유로 ClusterIP를 얻지 못한다.

## 외부 클라이언트에 서비스 노출

- 설명
	- 방법
		- 노드포트로 서비스 타입 설정
			- 각 클러스터 노드는 노드 자체에서 포트를 열어 해당 포트로 트래픽을 서비스로 전달한다.
				- 이 서비스는 내부 클러스터 IP와 포트로 액세스할 수 있을 뿐만 아니라 모든 노드의 전용 포트로도 액세스할 수 있다.
		- 노드포트 확장 타입인 로드벨런서 서비스 타입 설정
			- k8s가 실행 중인 클라우드 인프라에서 프로비저닝된 전용 로드벨런서로 서비스에 액세스한다.
			- 로드밸런서는 트래픽을 모든 노드의 노드포트로 전달한다.
				- 클라이언트는 로드밸런서의 IP로 서비스에 액세스한다.
		- 단일 IP 주소로 여러 서비스를 노출하는 인그레스 리소스 설정
			-  HTTP 레벨에서 작동하므로 4계층 서비스보다 더 많은 기능을 제공할 수 있다.

&emsp;&emsp;

### 노드포트 서비스 사용

- 설명
	- k8s는 노드포트 서비스가 만들어지면 모든 노드에 특정 포트를 할당하고 서비스를 구성하는 파드로 들어오는 역할을 한다.
		- 일반 서비스인 ClusterIP와 유사하지만 내부 클러스터 IP 뿐 아니라 모든 노드의 IP와 할당된 노드포트로 서비스에 액세스할 수 있다.

``` yaml
apiVersion: v1
kind: Service
metadata: 
  name: svc-nodeport
spec:
  type: NodePort
  ports:
  - port: 80
    targetPort: 8080
    nodePort: 30123
  selector:
    app: svc
```

- 설명
	- 노드포트 서비스 yaml
	-  모든 클러스터 노드에 바인딩되어야하는 노드 포트를 지정한다.
		- 80은 서비스 내부 클러스터 IP의 포트이다.
		- 8080은 서비스 대상 파드의 포트이다.
		- 30123은 각 클러스터 노드의 포트 30123으로 서비스에 액세스 할 수 있다.
	- 아웃바운드 요청 처리시 진행 과정
		- 외부 -> 노드 IP:30123 -> 서비스 -> ClusterIP(가상IP):80 -> 파드 (가상IP:80->localhost:8080)
	- 노드 포트는 반드시 지정할 필요는 없으며 생략을 해도 k8s가 알아서 선택할 것이다.

&emsp;&emsp;

- 노트포트 서비스 확인하기
	- `k get svc svc-nodeport`
		- EXTERNAL-IP
			- `<nodes>`로 표시돼고 클러스터 노드 IP 주소로 서비스에 접근이 가능함을 의미한다.
		- PORT(S)
			- 클러스터 IP의 내부 포트 : 노트 포트 로 표시된다.
		- 서비스 접근 방법
			- 클러스터의 IP:80
			- 노드 IP:30123
	- 결론
		- 외부 클라이언트는 노드나 클러스터의 IP를 직접 알아낸 다음 클러스터 IP와 80포트 혹은 노드 IP와 30123 포트를 붙여서 접속한다.
		- 서비스는 이중 각 파드에 패킷을 파드에 보내는 역할을 진행한다.

&emsp;&emsp;

- 주의점
	- 방화벽은 해제할 것
		- GKE같은 경우 `gcloud compute firewall-rules create svc-rule --allow=tcp:30123`으로 해제하자
		- 방화벽을 해제하면 인터넷 어디서든 포트 30123으로 액세스 할 수 있다.
	- 클라이언트가 첫번째 노드만 알고 그 노드만 사용하다 고장나면 클라이언트는 서비스에 접근할 수 없다.
		- 일반적으로 로드벨런서나 인그레스를 배치한다.

&emsp;&emsp;

### 외부 로드밸런서로 서비스 노출

- 설명
	- 클라우드 공급자에서 실행되는 k8s 클러스터는 일반적으로 클라우드 인프라에서 로드밸런서를 자동으로 프로비저닝하는 기능을 제공한다.
	- 생성 방법은 노드포트와 동일한데 type만 LoadBalancer로 바꾸면된다.
	-  주의점
		- GKE같은 환경에서 사용해야한다. 개인 PC에서 minukube tunnel 열고 해봤는데 안된다.
			- NodePort로 만들거나 포트포워딩으로 0.0.0.0 주소를 바인딩을 해야한다.
				- 루프백 주소만을 바인딩하고 있어 외부요청은 들어오질 않는다.

&emsp;&emsp;

### 외부 연결의 특성 이해

- 설명
	- 불필요한 네트워크 홉의 이해와 예방
		- 외부 클라이언트가 노트포트로 서비스에 접속하는 경우 (또는 로드벨런서를 먼저 통과하는 경우) 임의로 선택된 파드가 연결을 수신한 동일한 노드에서 실행 중일 수도 있고, 그렇지 않을 수도 있다.
			- 파드에 도달하려면 추가적인 네트워크 홉이 필요할 수 있으며 이것이 항상 바람직한 것이 아니다.
			- ==**외부의 연결을 수신한 노드에서 실행 중인 파드로만 외부 트래픽을 전달하도록 서비스를 구성**==하여 추가 홉을 방지할 수 있다.
				- `spec.externalTrafficPolicy: Local`
			- 서비스 정의에 이 설정이 포함돼 있고 서비스의 노드포트로 외부 연결이 열린 경우 서비스 프록시는 로컬에 실행 중인 파드를 선택하며 로컬 파드가 존재하지 않으면 연결이 중단된다.
			- 일반적으로 연결은 모든 파드에 균등하게 분산되지만 이 어노테이션을 사용할 때는 더 이상 적용되지 않는다.
				- 즉, ==**각각 노드에 생성된 파드들이 부담하는 트래픽 양이 고르게 분산되지 않는다는 것**==이다.
	- 클라이언트 IP가 보존되지 않음 인식
		- 클러스터 내부 클라이언트가 서비스로 연결 시도시 서비스의 파드는 클라이언트의 IP 주소를 얻을 수 있다.
			- 노드포트로 연결을 수신하면 패킷에서 소스 네트워크 주소 변환(SNAT)이 일어나므로 패킷 시작 IP 지점이 변경된다.
			- 파드는 실제 클라이언트 IP를 볼 수 없다는 것이다.
			- 이는 클라이언트 IP를 알아야 하는 일부앱에서 문제가 될 수 있다.
			- 로컬 외부 트래픽 정책은 연결을 수신하는 노드와 대상 파드를 호스팅하는 노드 사이에 추가 홉이 없기 때문에 클라이언트 Ip 보존에 영향을 미친다.(SNAT 안 일어남)

### TLS 트래픽을 처리하도록 인그레스 수정

- 설명
	- 인그레스를 위한 인증서 생성
		- `openssl genras -out tls.key 2048`
		- `openssl req -new -x509 -key tls.key -out tls.cert -days 360 -subj CN=prom.littok.com`
		- `k create secret tls tls-secret --cert=tls.cert --key=tls.key`
			- 시크릿을 생성하는 과정이다.
			- 참고로 certbot 파드를 크론잡으로 등록해두면 편하다.
	- 팁
		- CertificateSigningRequest 리소스로 인증서 서명
			- 인증서를 직접 서명하는 대신 CSR 리소스를 만들어 인증서에 서명할 수 있다.
			- 사용자 또는 해당 앱이 일반 인증서 요청을 할 수 있고 CSR에 넣으면 그 다음 운영자나 자동화된 프로세스가 다음과 같이 요청을 승인할 수 있다.
				- `k certificate approve csrName`
			- 이후 CSR의 status.certificate 필드에서 서명된 인증서를 검색할 수 있다.
			- 인증서 서명자 구성 요소가 클러스터에서 실행 중이여야 한다.
				- CSR의 생성 과 승인 또는 거부가 전혀 작동하지 않는다.

``` yaml
tls: 
- hosts: 
  - prom.littok.com 
    secretName: prom-tls
```

- 설명
	- 위의 문장을 spec 바로 밑에 추가하면 된다.

## 파드가 연결을 수락할 준비가 됐을 때 신호 보내기

- 설명
	- 파드의 레이블이 서비스의 파드 셀렉터와 일치할 경우 파드가 서비스의 엔드포인트로 포함된다는 것을 이미 배웠다.
		- 적절한 레이블을 가진 새 파드가 만들어지자마자 서비스의 일부가 요청이 파드로 전달 되기 시작하는데 만약 그 파드가 즉시 요청처리를 못하는 경우 문제가 발생한다.

&emsp;&emsp;

### 레디니스 프로브

- 설명
	- 정의
		- 특정 파드가 클라이언트 요청을 수신할 수 있는지 여부를 결정 짓는 프로브, 주기적으로 호출된다.
	- 특징
		- 컨테이너의 레디니스 프로브 호출이 성공하면 컨테이너가 요청을 수신할 수 있다고 간주한다.
		- being ready 표시는 각 컨테이너 마다 다르다.
		- k8s는 컨테이너에서 실행되는 앱이 간단한 Get 요청에 응답하는지 또는 특정 URL 호출할 수 있는지 확인하거나 필요에 따라 앱이 준비됐는지 확인하기 위해 전체적인 항목을 검사한다.
		- 앱의 특성에 따라 상세한 레디니스 프로브를 작성하는 것은 앱 개발자의 몫이다.
	- 레디니스 프로브 타입
		- Exec 프로브
		- Http Get 프로브
		- TCP 소켓 프로브
	- 레디니스 프로브의 동작
		- 컨테이너 시작시 k8s는 첫 레디니스 점검 수행전 구성 대기 시간을 경과할 때까지 대기하도록 구성 가능하다.
		- 주기적으로 프로브를 호출하고 레디니스 프로브의 결과에 따라 작동한다.
		- ==**파드가 준비되지 않았다고 하면 서비스(엔드포인트)에서 제거된다.**==
		- 파드가 다시 준비되면 서비스에 추가된다.
			- 사실 서비스가 아닌 엔드포인트에 추가된다.
		- ==**라이브니스 프로브와 달리 컨테이너가 준비 상태 점검에 실패해도 컨테이너가 다시 시작하거나 종료하지 않는다.**==
		- 레디니스 프로브는 요청을 처리할 준비가 된 파드의 컨테이너만 요청을 수신하도록 한다.
		- 컨테이너 시작시 주로 필요하지만 시작 이후에도 종종 유용하다.
	- 레디니스 프로브의 필요성
		- 파드 그룹이 다른 파드에 제공하는 서비스에 의존한다고 가정
			- FE 파드 중 하나가 연결 문제가 발생해 DB 연결이 불가능한 경우 해당 시점에 파드가 해당 요청을 처리할 준비가 되지 않았다는 신호를 레디니스 프로브가 k8s에 알리는 것이 현명하다.
			- 다른 파드 인스턴스에 동일한 유형의 연결문제가 발생하지 않는다면 정상적인 요청을 처리할 수 있다.
		- ==**레디니스 프로브를 사용하면 클라이언트가 정상 상태인 파드만 사용중이고 시스템이 문제가 있는지 알수가 없다.**==

&emsp;&emsp;

### 파드에 레디니스 프로브 추가

- 설명
	- 파드 템플릿에 레디니스 프로브 추가
		- `k edit rs rsName`
		- containers.readinessProbe.exec.command에 ls와 /var/ready를 추가한다.
			- Http Get 방식으로 둘 수도 있고 그렇다.
			- ls 명령어는 파일이 존재하면 0, 아니면 0이 아닌 값을 반환한다.
		- 파드를 날리고 다시 파드를 조회해 확인해보면 READY열에 준비 여부를 알 수 있다.
		- 예제에서는 목표 파드에 /var/ready 파일을 생성하고 변경되었는지 확인한다.
			- 10초마다 한번 조회하니 대기해야한다.

&emsp;&emsp;

### 실제 환경에서 레디니스 프로브가 수행해야하는 기능

- 설명
	- 앱이 수신 가능한지 아닌지 여부에 따라 성공이나 실패를 반환해야한다.
	- 서비스에서 파드를 수동으로 제거하기 위해 수동으로 프로브의 스위치를 전환하는 것이 아닌 파드를 삭제하거나 파드 레이블을 변경해야한다.
	- 레디니스 프로브를 항상 정의해야한다.
		- 파드에 레디니스 프로브를 정의하지 않으면, 파드가 시작하는 즉시 서비스 엔드포인트가 되며 앱이 수신 연결을 하는데 너무 오래 걸리는 경우 클라이언트는 계속해서 Connection refused 유형의 에러를 보게 된다.
	- 레디니스 프로브를 파드의 종료 코드를 포함하지 말라.
		- 파드의 라이프사이클 마지막 단계에서 고려해야하며 연결 오류가 발생한 클라이언트와 관련된 내용이다.
		- 실행되는 앱은 종료 신호를 받자마자 연결 수락을 중단하며 종료 절차가 시작되는 즉시 레디니스 프로브가 실행하도록 만들어 파드가 모든 서비스에서 제거되도록 해야한다고 생각할 수도 있다.
		- ==**k8s는 파드를 삭제하자마자 모든 서비스에서 파드를 제거하기 때문에 이럴 필요가 전혀없다.**==

&emsp;&emsp;

- 결론
	- 항상 레디니스 프로브를 정의해야한다.
		- Http Get 요청을 기본 URL에 보내더라도 항상 정의해야한다.

## 헤드리스 서비스로 개별 파드 찾기

- 설명
	- 만약 클라이언트가 모든 파드 IP를 알아야만 하는 경우 서비스에 클러스터 IP가 필요하지 않다고 가정하면 DNS 서버는 하나의 서비스 IP 대신 파드 IP 여러개를 반환한다.
		- 이를 이용하여 DNS A 레코드를 여러개 반환받고 이를 이용해 연결하면 된다.

&emsp;&emsp;

### 헤드리스 서비스 생성

- 설명
	- spec.clusterIP를 None으로 설정하여 헤드리스 서비스로 구성하는 것이다.


&emsp;&emsp;

### DNS로 파드찾기

- 설명
	- 파드 준비완료시 DNS 조회로 실제 파드 IP lookup 가능하다.
	- 파드 내부에서 조회해야하며 nslookup 바이너리가 있는 이미지를 사용해야한다.

&emsp;&emsp;

- yaml 없이 nslookup이 있는 파드 생성하기
	- `k run dnsutils --image=tutum/dnsutils`

&emsp;&emsp;

- 헤드리스 서비스를 위해 반환된 DNS A 레코드
	- `k exec dnsutils nslookup headlessSvcName`
		- 준비가 되었다고 보고된 파드의 IP 두 개를 반환한다.
	- `k exec dnsutils nslookup svcName`
		- 서비스 클러스터의 IP를 한 개 반환한다.

&emsp;&emsp;

### 준비되지 않은 파드를 포함한 모든 파드 검색

- 설명
	- DNS 조회 메커니즘을 활용해 준비되지 않은 파드 찾기
		- 다음 필드를 추가한다.
			- `publishNotReadyAddresses: "true"`

## 서비스 문제 해결

- 설명
	- 서비스가 파드에 연결 못하는 경우
		- 외부가 아닌 클러스터 내에서 서비스의 클러스터 IP에 접속이 가능한지 확인한다.
		- 서비스에 액세스할 수 있는지 확인하려고 서비스 IP로 핑을 해봐야 가상 IP이므로 가능하지 않다.
		- 레디니스 프로브를 정의했다면 성공했는지 확인하지 않는다면 파드는 서비스 엔드포인트로 추가되지 않는다.
		- 파드가 서비스 일부인지 확인하려면 k get endpoints를 사용해 엔드포인트 오브젝트를 확인하자.
		- FQDN이나 그 일부로 서비스에 액세스 되지 않는 다면 클러스터 IP를 사용해보자
		- 대상 포트가아닌 서비스로 노출된 포트에 연결하고 있는지 확인한다.
		- 파드 IP에 직접 연결해 파드가 올바른 포트에 연결돼 있는지 확인한다.
		- 파드 IP로 애플리케이션에 액세스할 수 없는 경우 앱이 로컬호스트에만 바인딩하고 있는지 확인한다.
			- 내가 직접 로드벨런서 타입 서비스를 사용할 때 항상 문제를 겪던 부분이 이거다.
		-
