## 도입 필요성 요약

- 설명
	-  최근 개발 흐름은 MSA로 진행되고 있다. (물론 돈이 많은 회사의 경우)
	- 사견
		- 심지어 전정부 또한 MSA를 밀어붙이고 있지만 도입 자체가 쉽지 않다고 볼 수 있다. (특히 인력과 돈이 많이 들어가는 구조이다. 확장에는 돈이 적게 들지만, 유지에는 돈이 많이 들어가는 방식이 클라우드와 연동된 MSA라고 할 수 있겠다.)

## 쿠버네티스 (이하, k8s)

- 특징
	- 개발자가 운영 팀의 도움 없이도 자신의 애플리케이션을 원하는 만큼 자주 배포가능케 해준다.
	- 하드웨어 장애 발생시 해당 애플리케이션을 모니터링하고 스케줄링을 조정해 운영팀을 돕는다.
		- 시스템 관리자의 관심은 개발 애플리케이션을 감독하는 것에서 k8s에서 나머지 인프라를 감독하고 관리하는 것으로 바뀌고 애플리케이션은 k8s에서 관리해준다.
	- 하드웨어 인프라를 추상화하고 IDC 전체를 하나의 컴퓨팅 리소스로 제공해준다.
		- 세세한 서버 정보를 알 필요 없이 애플리케이션 구성 요소를 배포하고 실행할 수 있다.
	- 쿠버네티스로 여러 애플리케이션 구성 요소를 배포할 때 각 구성 요소 서버를 선택하고 배포하며 애플리케이션의 다른 구성 요소를 쉽게 찾고 통신하게 한다.
	- 사내 IDC 뿐만 아니라 클라우드 프로바이더가 구축 운영하는 대규모 IDC에도 진가를 발휘한다.
		- 개발자가 모든 유형의 애플리케이션을 배포하고 실행할 수 있는 간단한 플랫폼부터 클라우드 프로바이더의 시스템 관리자가 자신들의 하드웨어에서 실행되는 수만 개의 애플리케이션을 알 필요 없게 한다.
	- 점점 더 많은 대기업에 k8s 모델을 최고의 애플리케이션 운영 방법으로 받아들이며 클라우드 뿐만 아니라 사내 자체 인프라를 k8s로 실행하는 표준이 되고 있다.

## k8s의 필요성

- 이유 1
	- 모놀리스에서 마이크로서비스로 전환이 쉬워짐
		- 모놀리스의 단점
			- 소수의 강력한 컴퓨팅 파워를 요구한다.
			- 시스템의 증가하는 부하를 처리하려면 수직확장 또는 복사본을 만들어 전체 시스템을 수평확장해야한다.
			- 수직확장은 비용이 많이 들고, 수평 확장은 저렴하지만 애플리케이션 코드를 크게 변경해야할필요가 존재한다. (세션 유지, 트랜잭션 분산화 등)
			- 결론적으로 확장이 매우 불편하다.
		- 모놀리스의 장점
			- 초기 시스템을 빌드업하는데 비용이 매우 적게 들고 빠르게 시작하기 좋다.
			- 시스템을 빠르게 개발 착수를 할 수 있는 점 덕분에 초기 프로덕트 배포 시작 전에 많은 장점이 있다.
			- 시스템 확장이 힘든거지 기능 빌드업, 시스템 유지가 힘든 편은 아니다.
		- 사견
			- 시작은 모놀리스로, 그리고 빠르고 급진적으로 확장을 할 필요성이 대두될때 MSA를 도입하는게 가장 좋다.
			- 애초에 사람이 많이 필요한 구조임으로 많은 돈이 밑바탕이 되어야 유지할 수 있다.
				- 요즘 시국에 이걸 모르는 VC는 없다고 봐야한다. 
				- MSA를 밀고 나가는 대표는 잘 모르는 캐시카우 물기에 혈안이 되어있는 사람이라고 보면 편하다.
				- 남의 돈 빼먹기는 그럼에도 불구하고 쉽진 않다. k8s 자체는 알아두면 추후에 도움이 되는 것은 분명하다.

&emsp;&emsp;

- 이유 2
	- 애플리케이션에 일관된 환경 제공하기 쉬움
		- 보편적으로 개발 스테이징 환경 관리도 까다로운 편인데, 팀간의 환경 자체를 이해하고 적절히 제공하는 것이 쉽지 않다. 
			- 이 차이때문에 많은 개발회사에서 아직도 고통받고 있는 것이 사실이긴하다.
		- 프로덕션 환경에서 나타나는 환경 문제를 줄이기 위해 레이어를 구분하고 정해진 api로 통신이 가능하게끔하고 zipkin으로 api 호출을 트레이스하여 디버깅하는 것이 요즘의 트랜드라고 한다.

&emsp;&emsp;

- 이유 3
	- 지속적인 배포로 전환: 데브옵스와 노옵스
		- 데브옵스의 가장 핵심은 개발팀과 운영팀이 한데 어울려 프로세스에서 협업함을 의미한다.
		- 옛날 부터 그래왔지만 애초에 개발자가 하드웨어 인프라를 몰라도 되게끔 운영 팀을 거치지 않고 개발자가 직접 배포하게 하는 것이 가장 이상적이며 이를 노옵스라고 부른다.
			- 하드웨어 인프라를 관리할 누군가는 필요하지만 이상적으로 그 위에 실행되는 각 애플리케이션 특성을 다루지 않아도 된다
			- k8s를 쓰면 된다 이런 이야기

## 컨테이너 기술


- 설명
	- 각 애플리케이션의 실행환경을 격리 시키는 기술
	- 최근 개발자들은 리눅스 기반 컨테이닝 기술인 LXC에 눈독들이고 있다.
		- 대부분 저걸 기반으로 동작하게끔 만든다.
		- 다른 가상환경과 달리 같은 파일 시스템을 공유하며 LXC 기술로 환경만 분리된다.
		- 오버헤드가 극히 적은 것이 특징이라고 할 수 있겠다.
		- 하이퍼-v든 뭐든 syscall을 전환해주는 레이어가 존재하는 것 자체가 오버헤드이다.
			- 동일한 커널에서 동일한 syscall을 호출할 수 있게하는 것이 매우 매력적인 선택지라고 할 수 있겠다.
	- 컨테이너는 부팅이 필요가 없다.
		- 시스템 OS 기반이기 때문에 즉시 프로세스가 올라간다.

&emsp;&emsp;

- 종류
	- 도커
	- OCI 표준 하의 여러 기술


&emsp;&emsp;

- 매커니즘 소개
	- 리눅스 네임스페이스
		- 각 프로세스가 시스템에 대한 독립된 뷰만 볼 수 있게하는 기능
		- 종류
			- mnt
				- 마운트
			- pid
				- 프로세스 ID
			- net
				- 네트워크
			- ipc
				- 프로세스 간 통신
			- uts (Unix Time Sharing)
				- 호스트와 도메인 이름
				- NIS(Network Internet Service) 도메인 이름 관련 네임스페이스이다.
			- user
				- 사용자 ID
		- 추가 설명
			- 리눅스는 초기 구동시 하나의 네임스페이스만 존재한다.
				- 파일시스템, 프로세스 ID, 사용자 ID, 네트워크 인터페이스 등과 같은 모든 시스템 리소스는 하나의 네임스페이스에 속한다.
			- **추가 네임스페이스를 생성하고 리소스 구성이 가능하다.**
			- 프로세스를 실행할 때 해당 네임스페이스 중 하나에서 프로세스를 실행한다.
				- **==프로세스는 동일한 네임스페이스 내에 있는 리소스만 볼 수 있다.==**
				- 여러 종류의 네임스페이스가 있기 때문에 프로세스는 하나의 네임스페이스에만 속하는 것이 아니라 **==여러 네임 스페이스에 속할 수 있다.==**
			- 각 네임 스페이스는 특정 리소스 그룹을 격리시키는데 사용된다.
			- uts의 경우 해당 네임스페이스 내에서 실행중인 프로세스가 사용할 호스트 이름과 도메인 이름을 결정한다.
				- 예를 들면 두 개의 서로 다른 UTS 네임스페이스를 두 개가 한 쌍인 프로세스에 각각 지정하면 서로 다른 로컬 호스트 이름이 보이게끔 만들 수 있다.
			- 네트워크 네임스페이스는 실행 중인 애플리케이션의 프로세스에서 볼 수 있는 네트워크 인터페이스를 결정짓는다.
				- 각 네트워크 인터페이스는 딱 하나의 네임스페이스에 속하지만 한 네임스페이스에서 다른 네임스페이스로 이동할 수 있다.
			- 각 컨테이너는 고유한 네트워크 네임스페이스를 사용하므로 각 컨테이너는 고유한 네트워크 인터페이스 세트를 볼 수 있다.
		- 프로세스의 가용 리소스 제한
			- 컨테이너 격리의 나머지 부분은 컨테이너가 사용할 수 있는 시스템 리소스의 양을 제한하는 것이다.
			- 이는 프로세스의 리소스 사용을 제한하는 리눅스 커널 기능인 cgroups로 이뤄진다.
				- 프로세스는 설정된 양 이상의 CPU, 메모리, 네트워크 대역폭 등을 사용할 수 없다.
				- 프로세스는 다른 프로세스 용으로 예약된 리소스 사용이 불가능하며 이는 각 프로세스가 별도의 시스템에서 실행될 때와 비슷하다.
		- 결론
			- 본래 global 네임스페이스인 부모 네임스페이스(부팅하면 올라오는 네임스페이스)를 따르지 않고 새로운 네임스페이스를 만들고 cgroups로 프로세스의 리소스 사용을 제한하는 리눅스 커널 기능을 사용하고, clone과 같은 syscall을 사용하여 네임스페이스를 분리하여 환경을 분리하는 것이 LXC의 핵심 개념이다.
			- 이러면 네트워크 인터페이스나 UTS등을 부모의 것을 따르지 않고 직접 정의한 뷰를 보게 되는 것이라 환경이 자연스레 분리되는 것이다.
	- 도커 컨테이너 플랫폼
		- 각 환경을 이미지로 패키징하고 실행하는 라이브러리
		- 특징
			- 패키징된 이미지를 실행하면 함께 제공된 파일시스템 내용을 정확하게 볼 수 있다.
				- 도커는 이미지를 만들 때 파일시스템의 계층 하나 하나를 레이어로 만들어 패키징한다.
					- 파일시스템이 복잡할 수록 레이어의 수도 늘어나는 구조이다.
				- 이는 프로덕션 환경 서버가 완전히 다른 리눅스 운영체제를 실행하더라도 개발 머신에서 실행되는지 프로덕션 머신에서 실행되는지 사오간없이 동일한 파일을 볼 수 있게 해준다.
				- 애플리케이션은 실행 중인 서버의 내용을 볼 수 없으므로 서버에 개발 컴퓨터와 다른 설치 라이프러리가 설치되어있는지 중요하지 않다.
					- Centos의 전체 파일과 함께 애플리케이션을 패지킹을 하고 다른 리눅스 배포판 서버에서 실행되도 애플리케이션 자신은 항상 RHEL내에서 실행된다고 생각한다.
					- 가상 머신에 OS를 설치하고 그 안에 애플리케이션을 설치한 다음 가상머신 이미지를 배포하고 실행하는 가상머신 이미지를 만드는 것과 유사하다.
			- 리눅스에서 제공되는 도커는 LXC를 기반으로 제공되기 때문에 가상머신과 비슷하게 동작하지만 오버헤드는 적은 것이 큰 특징이다.
				- 이런 점 때문에 리눅스 환경이 아닌 다른 운영체제에서 도커를 쓰는 것은 성능상 딱히 이득인건 아니다.
			- 가상머신 이미지와 도커 이미지와의 차이점은 여러 이미지에서 공유되고 재사용될 수 있는 레이어로 구성돼 있다는 것이다.
				- 동일한 레이어를 포함하는 다른 컨테이너 이미지를 실행할 때 이미 다른 레이어가 다운되어 있다면 이 이미지의 특정 레이어(예를 들면 애플리케이션 부분)만 다운로드하면 된다.
		- 주요 개념
			- 이미지
				- 애플리케이션과 해당 환경을 패키지화 한 것
				- 애플리케이션에서 사용가능한 파일 시스템, 이미지가 실행될 때 실행되어야 하는 실행파일 경로와 같은 메타데이터가 포함됨
			- 레지스트리
				- 도커 이미지를 저장하고 다른 사람이나 컴퓨터 간에 해당 이미지를 쉽게 공유할 수 있는 저장소
				- 도커허브라는 공개된 레지스트리가 존재하며, 사내서버에서 비공개 레지스트리를 구축할 수도 있다.
					- 이 비공개 레지스트리는 이미지를 땡겨올려면 시크릿이 필요하다.
			- 컨테이너
				- 도커 기반 컨테이너 이미지에서 생성된 일반적인 리눅스 컨테이너
				- 실행 중인 컨테이너는 도커를 실행하는 호스트에서 실행되는 프로세스
					- 호스트와 호스트에서 실행 중인 다른 프로세스와 완전 격리되어있다.
				- cgroups로 리소스 사용량이 제한되어 이 이상으로 사용은 불가능하다.
		- 레이어의 이해
			- 도커 이미지는 레이어로 구성되어있다.
				- 모든 도커 이미지는 다른 이미지 위에 빌드되고 두 개의 다른 이미지는 기본 이미지로 동일한 부모 이미지를 사용할 수 있기 때문에 다른 이미지에는 정확히 동일한 레이어가 포함될 수 있다.
					- 이 점이 네트워크로 이미지를 배포할 때 캐싱이 되는 장점이라서 배포시 빠른 속도를 자랑하는 이유이기도 하다.
			- 레이어는 배포만 효율적으로 할 뿐아니라 이미지의 스토리지 공간을 줄이는데 도움된다.
				- 각 레이어는 동일 호스트에서 한 번만 저장된다.
				- **동일한 기본 레이어를 기반으로 생성된 두 개의 이미지에서 생성한 각 두 개의 컨테이너는 동일한 파일을 읽을 수 있지만 그중 하나가 해당 파일을 덮어 쓰면 다른 컨테이너는 해당 변경 사항을 관찰 할 수 없는 구조이다.**
				- 파일을 공유하더라도 여전히 서로 격리되어 있는데 이것은 컨테이너 이미지 레이어가 일기 전용 공간이기 때문이다.
				- ==**컨테이너가 실행될 때 이미지 레이어 위에 새로운 쓰기 가능한 레이어가 만들어진다.**==
				- 컨테이너 프로세스가 기본 레이어 중 하나의 파일에 쓰면 전체 파일의 복사본의 최상위 레어에 만들어지고 프로세스는 복사본을 쓰게 된다.
					- 도커는 AUFS, OverlayFS와 같은 union filesystem을 사용한다.
						- 이는 읽기 전용 레이어, 쓰기 전용 레이어로 구성된다고 한다.
		- 컨테이너 이미지의 제한적인 이식성
			- 컨테이너화된 애플리케이션이 특정 커널 버전을 필요로 한다면 모든 시스템에서 동작할 수 없는 이미지이다.
			- 머신이 다른 버전의 리눅스 커널로 실행되거나 동일한 커널 모듈을 사용할 수 없는 경우 애플리케이션이 실행될 수 없다.
			- 컨테이너는 가상머신에 비해 월등히 가볍지만 컨테이너 내부에서 실행되는 애플리케이션은 제약이 존재한다.
			- ==**예를 들면 amd64에서 빌드된 이미지는 arm64에서 동작하지 않는다.**==
				- 따라서 크로스빌드를 해야한다.

&emsp;&emsp;

- 도커의 대안 rkt
	- OCI
		- 컨테이너 표준으로 도커의 성공 이후 설립된 업계 표준이다.
		- 이 표준을 따르는 여러 컨테이너 라이브러리가 존재하며 rkt, containerd 등이 대표적이다.
		- 쿠버네티스 또한 이를 지원하지만 k8s 설치 및 학습 가이드라인은 우선 도커를 기반으로 학습하는 것을 권장하고 추후 변경하는 것을 권장한다.

## 쿠버네티스

- 기원
	- 구글에서 보그라는 내부 시스템을 개발해 애플리케이션 개발자와 시스템 관리자가 수천 개의 애플리케이션과 서비스를 관리하는 데 도움을 줬다.
		- 이 과정에 go 언어가 탄생했다.
		- 개발 당시 신규 인력과 고인물 인력들의 지식격차에 큰 고통을 겪었었는데 고인물 인력들이 레퍼런스를 찾아 제공하려던 찰나, 너무 복잡해진 C++ 레퍼런스에 스트레스를 크게 받아 바닥부터 새로 만들어 사내 공통언어로 제공하는 것이 처음이다.
			- go 언어가 지금도 인프라쪽에서 상당히 유명세를 얻게 된 이유이기도 하다.
			- 'be simple'은 모든 실리콘벨리 IT기업에서 강조하는 핵심 개념이지만 격이 다르게 시작이 다른 심플함을 가지려는 그들의 태도에 감탄이 절로 나오게 된다.
	- 2014년 보그, 오메가, 기타 내부 구글 시스템으로 얻은 경험을 기반으로 하는 오픈 소스 시스템인 k8s가 출시됬다.

&emsp;&emsp;

- 쿠버네티스 이해
	- 쿠버네티스란
		- 컨테이너화된 소프트웨어를 쉽게 배포 관리해주는 소프트웨어 시스템
		- 이기종 애플리케이션을 호스트에 수동배포하지 않고 실행시켜준다.
			- 동일한 하드웨어에서 완전히 다른 조직의 애플리케이션을 실행할 때 매우 중요하다.
		- 호스팅된 애플리케이션을 완전히 격리하면서 하드웨어를 최대한 활용하려고 노력하는 클라우드 프로바이더에게 매우 중요하다.
	- 특징
		- 모든 노드가 하나의 거대한 컴퓨터인 것 처럼 수천 대의 컴퓨터 노드에서 소프트웨어 애플리케이션을 실행할 수 있다.
		- 기본 인프라를 추상화하고 개발과 운영 팀 모드의 개발, 배포, 관리를 단순화한다.
		- 클러스터에 노드가 몇 개에서 수 천개가 있든, k8s에 애플리케이션을 배포하는 것은 항상 동일하다
	- 핵심 이해
		- 시스템 구성
			- 마스터 노드
			- 워커 노드
		- 배포
			- 애플리케이션 매니페스트를 마스터에 게시하면 k8s가 해당 앱을 워커 노드 클러스터에 배포한다.
				- 구성 요소가 어떤 노드에 배포되든지 개발자나 시스템 관리자에게 중요하지 않음
		- 특징
			- 특정 앱이 함께 실행되도록 지정할 수도 있고 k8s는 여러 애플리케이션을 동일한 워커 노드에 배포한다.
			- 다른 앱은 클러스터에 걸처 분산되지만 배포된 위치에 상관없이 동일한 방식으로 서로 통신한다.
	- 개발자를 위한 특성
		- k8s는 클러스터의 운영체제로 생각할 수도 있다.
		- 앱 개발자가 특정 인프라 관련 서비스를 앱에 구현하지 않아도 된다.
			- 대신 k8s에 의존해 여러 서비스를 제공한다.
				- 서비스 디스커버리
				- 스케일링
				- 로드벨런싱
				- 셀프 리커버리
				- 리더 선출
		- 앱 개발자는 앱의 실제 기능을 구현하는데 집중할 수 있고 애플리케이션을 인프라와 통합하는 방법을 찾는데 시간 낭비 할 이유가 없어짐
	- 운영팀을 위한 특성
		- k8s는 클러스터 어딘가에 컨테이너화된 앱을 실행하고 구성 요소간 서로를 찾는 방법에 관한 정보를 제공하고 모든 앱을 계속 실행하게 함
		- 앱은 어떤 노드에서 실행되든 상관이 없기에 k8s는 언제든지 앱을 재배치하고 앱을 조합하여 리소스를 수동 스케줄링보다 훨씬 더 잘 활용 가능해진다.

&emsp;&emsp;

- k8s 클러스터 아키텍처
	- 하드웨어 수준에서 나눈 유형
		- 마스터 노드
			- 전체 k8s 시스템을 관리하는 k8s 컨트롤 플레인을 실행
		- 워커 노드
			- 실제 배포되는 컨테이너 앱을 실행
	- 컨트롤 플레인
		- 설명
			- 클러스터를 제어하고 작동시킨다.
			- 하나의 마스터 노드에서 실행하거나 여러 노드로 분할되고 복제돼 고가용성을 보장할 수 있는 여러 구성요소로 구성된다.
			- 구성 요소들은 클러스터의 상태를 유지하고 제어하지만 앱을 실행시키지는 않는다.
				- 이는 노드가 수행한다.
		- 구성 요소
			- 쿠버네티스 API 서버
				- 사용자, 컨트롤 플레인 구성 요소와 통신한다.
			- 스케줄러
				- 앱의 배포를 담당한다.
			- 컨트롤러 매니저
				- 구성 요소 복제본, 워커 노드 추적, 노드 장애 처리 등과 같은 클러스터 단의 기능을 수행
			- Etcd
				- 클러스터 구성을 지속적으로 저장하는 신뢰할 수 있는 분산 데이터 저장소
	- 노드
		- 설명
			- 컨테이너화된 앱을 실행하는 시스템
			- 앱을 실행, 모니터링, 앱을 서비스에 제공하는 작업은 구성요소에 의해 수행된다.
		- 구성 요소
			- 컨테이너 런타임
				- 도커, rkt, kind 등이 있다.
				- 컨테이너를 실행하는 런타임이다.
			- kubelet
				- API 서버와 통신하고 노드의 컨테이너를 관리한다.
			- k8s 서비스 프록시
				- 애플리케이션 구성 요소 간에 네트워크 트래픽을 로드벨런싱한다.

&emsp;&emsp;

- k8s에서 애플리케이션 실행
	- 실행 과정
		1. 앱을 하나 이상의 컨테이너 이미지로 패키징한다.
		2. 앱의 이미지를 이미지 레지스트리로 푸시한다.
		3. k8s API 서버에 앱 디스크립션을 게시한다.
	- 앱 디스크립션
		- 구성 요소
			- 컨테이너 이미지
			- 앱 구성 요소가 포함된 이미지
			- 해당 구성 요소가 서로 통신하는 방법
			- 동일 서버에 함께 배치되어야하는 구성 요소
			- 실행할 각 구성 요소의 복제본 수
			- 내부 또는 외부 클라이언트에 서비스를 제공하는 구성 요소
			- 하나의 IP 주소로 노출해 다른 구성 요소에서 검색 가능하게 해야하는 구성 요소
	- 디스크립션으로 컨테이너 실행시키는 방법
		1. API 서버가 앱 디스크립션을 받는다. 
		2. 스케줄러는 각 컨테이너에 필요한 리소스를 계산 
		3. 스케줄러가 리소스를 계산하는 시점에 각 노드에 할당되지 않은 리소스를 기반으로 사용 가능한 워커 노드에 지정된 컨테이너를 할당한다.
		4. 해당 노드의 kubelet은 컨테이너 런타임에 필요한 컨테이너 이미지를 가져와 컨테이너를 실행하도록 지시
		5.
	- 실행된 컨테이너 유지
		- 설명
			1. 앱이 실행되면 k8s는 앱의 배포상태가 사용자가 제공한 디스크립션과 일치하는지 지속적으로 확인한다.
			2. 프로세스가 중단되거나 응답이 중지될 때와 같이 인스턴스가 제대로 동작하지 않으면 k8s가 자동으로 다시 시작 시킨다.
			3. 워커 노드가 중료되거나 엑세스 불가능하면 k8s는 이 노드에서 실행중인 모든 컨테이너 노드를 새로 스케줄링하고 새로 선택한 노드에서 실행한다.
	- 복제본 수 스케일링
		- 설명
			- 앱이 실행되는 동안 복제본 수를 늘릴지 줄일지 결정할 수 있고 k8s는 추가 복제본을 기동하거나 초과 복제본을 정지 시킬것이다.
			- 최적의 복제본 수를 결정하는 작업을 k8s 자율에 맡길 수도 있다.
			- cpu 부하, 메모리 사용량, 초당 요청 수, 앱이 노출하는 다른 매트릭과 같은 실시간 메트릭을 기반으로 복제본 수를 자동으로 조정 가능하다.
	- 이동한 앱에 접근하기
		- 설명
			- k8s는 컨테이너를 클러스터 안에서 이동시킬 수 있다.
				- 실행 중인 노드가 정지되거나 다른 컨테인를 위한 공간을 만들기 위해 노드에서 제거되거나하면 발생
				- 컨테이너가 외부 클라이언트나 클러스터에서 실행 중인 다른 컨테이너에 서비스를 제공하는 경우 이 컨테이너가 클러스터에 지속적으로 이동을 하거나 컨테이너가 복제돼 전체 클러스터에 분산되어도 클라이언트가 서비스를 제공하는 컨테이너에 접근하는 방법은 클라이언트가 특정 서비스를 제공하는 컨테이너를 쉽게 찾을 수 있도록 k8s에게 동일한 서비스를 제공하는 컨테이너를 알려주면 k8s는 하나의 고정 ip 주소로 모든 컨테이너를 노출하고 해당 주소를 클러스터에 실행 중인 모든 앱에 노출한다.
				- 이는 환경 변수로 제공되지만 클라이언트는 오래전부터 사용된 dns로 서비스 IP를 조회할 수도 있다.
				- 이를 **==서비스 라고 부르는 오브젝트의 역할==**이다.
			- kube-proxy는 서비스를 제공하는 모든 컨테이너에서 서비스 연결이 로드벨런싱되도록 한다.
				- 서비스 ip주소는 일정하게 유지되므로 클라이언트는 컨테이너가 클러스터 내에서 이동하더라도 컨테이너에 항상 연결할 수 있다.
				- kube-proxy는 이 서비스 오브젝트들을 감시하고 연결된 pod들의 엔드 포인트 정보를 지속적으로 감시하여 변경이 존재하는 경우 지속적으로 네트워크 규칙을 업데이트 한다.
				- 이로써 클라이언트는 고정된 서비스 IP만 알아도 올바른 컨테이너(pod)으로 트래픽을 주고 받을 수 있다.
				- 서비스 IP가 노출된 종합 IP이고, 이 서비스와 컨테이너 사이를 관리하는 것이 kube-proxy이다.

&emsp;&emsp;

- k8s 사용의 장점
	- 대표적 장점
		- 모든 서버에 k8s를 설치한 경우 운영 팀이 더이상 앱 배포를 직접 처리하지 않아도 된다.
		- 컨테이너화된 앱은 이미 실행에 필요한 모든 것이 포함되어있으므로 시스템 관리자는 앱을 실행하고 배포하기 위해 무언가를 설치하는 것에 노력할 필요가 없다.
		- k8s가 배포된 모든 노드에서는 시스템 관리자 도움 없이도 즉시 앱 실행이 가능하다.
	- 애플리케이션 배포의 단순화
		- k8s는 모든 워커 노드를 하나의 배포 플랫폼에 제공하기 때문에 앱 개발자는 자체적으로 앱 배포가 가능하여 클러스터를 구성하는 서버를 알 필요가 없다.
		- 모든 노드는 앱이 해당 노드를 사용하기를 기다리는 하나의 컴퓨팅 리소스로 바라보게 된다.
			- 서버는 앱이 적절한 시스템 리소스를 제공할 수 있는 한 앱이 어느 서버에서 실행 중인지 신경 쓰지 않는다.
		- 개발자 앱을 특정 종류의 하드웨어에서 실행해야만 하는 경우가 있다.
			- 노드가 이기종인 경우 특정 기능이 있는 노등에서 앱을 실행하고 다른 앱은 다른 노드에서 실행하는 경우이다.
			- k8s에서는 앱을 실행할 특정 노드를 선택하는 것이 아닌 이 특정 기능이 있는 노드중 하나만 선택하도록 지시할 수 있다.
	- 하드웨어 활용도 대폭 상승
		- 인프라와 앱이 논리적으로 분리되어 k8s에 앱을 실행하도록 지시하면 앱의 리소스 요구 사항에 대한 디스크립션과 각 노드에서 사용 가능한 리소스에 따라 앱을 실행할 가장 적합한 노드 선택이 가능하다.
		- 컨테이너를 사용하고 앱은 클러스터의 특정 노드로 지정하지 않으면 언제든지 앱이 클러스터 사이에 자유로운 이동이 가능하므로 클러스터에서 실행되는 다른 앱의 구성요소를 혼합하여 클러스터 노드에 배치가 가능하다.
			- 이것이 노드의 하드웨어 리소스를 최대한 활용하는 방법이다.
		- k8s는 클러스터간 앱 이동이 언제든 가능하므로 수동으로 수행하는 것보다 훨씬 인프라 활용이 잘 수행된다.
			- 클러스터간 앱의 조합을 잘 찾아내는 것은 컴퓨터가 훨씬 사람보다 낫다
	- 상태 확인과 셀프 리커버리
		- 서버 장애 발생시 언제든 클러스터간 앱을 이동할 수 있으므로 클러스터 크기 증가시 컴퓨터 구성 요소의 고장을 더 자주 처리하게 된다.
		- k8s는 앱 구성요소와 이 앱이 구동 중인 노드를 모니터링 하다가 노드 장애 발생시 자동으로 앱을 다른 노드로 스케줄링한다.
			- 운영팀은 수동으로 앱 구성요소를 마이그레이션할 필요가 없고, 앱을 재배치하는 대신 즉시 노드 자체를 수정해 사용 가능한 하드웨어 리소스 풀에 반환하는 데에만 집중하면 된다.
		- 인프라에 장애가 발생한 노드가 없어도 정상적인 시스템 작동이 가능하도록 충분한 예비 자원이 있는 경우 운영 팀은 새벽 3시에 일어난 장애에 즉시 대응할 필요가 없다.
		- 규칙적인 근무 시간 동안 잠을 잘 수 있고 정규 근무 시간에 장애가 발생한 노드를 처리할 수 있다.
			- 개인적으로 이렇게 하는 팀을 아직 본적 없다.
	- 오토 스케일링
		- k8s를 사용해 배포된 앱을 관리하는 것은 급격한 부하 급증에 대응하기 위해 개별 앱의 부하를 운영팀이 지속적으로 모니터링할 필요가 없음을 의미한다.
		- k8s는 각 앱에서 사용하는 리소스를 모니터링하고 각 앱의 실행중인 인스턴스 수를 계속 조정하도록 지시 가능하다.
		- 클라우드 인프라에서 k8s가 실행 중인 경우 클라우드 제공업체의 API로 쉽게 노드를 추가하면 배포된 앱의 부하에 따라 전체 클러스터 크기를 자동으로 확장하거나 축소할 수 있다.
	- 앱 개발의 단순화
		- 앱이 개발과 프로덕션 환경에서 동일한 환경에서 실행되기 때문에 버그를 쉽게 발견할 수 있다.
		- 앱에서 서비스나 피어를 검색하게끔 만들 필요가 없다.
			- 앱은 특정 환경변수를 조회하거나 DNS 조회를 하도록 수행하면 된다.
			- 더 정보가 필요하다면 k8s API 서버를 쿼리하면 해당 정보나 다른 정보를 얻을 수 있다.
			- 이때 k8s API 서버를 쿼리하면 개발자가 리더 선정 같은 복잡한 매커니즘을 구현하지 않아도 된다.
				- 리더 선정이란 클러스터의 여러 앱의 인스턴스중 k8s API 서버를 이용할 수 있는 권한, 일종의 리소스 락이다.
				- 동작 구조는 모든 인스턴스가 리소스락을 획득하려고 API를 호출하면 성공한 인스턴스가 리더가 되는 구조이다
		- 새 버전 앱 출시 시 새 버전이 뭔가 잘못되었다면 자동으로 감지하고 즉시 롤아웃을 중지하므로 개발자들의 신뢰성 증가로 이어진다.
			- 이는 CD를 가속화하여 조직 전체에 도움이 된다.
