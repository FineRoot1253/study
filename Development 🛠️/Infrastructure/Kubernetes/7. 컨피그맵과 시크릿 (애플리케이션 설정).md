## 컨테이너화된 앱 설정

- 설명
	- k8s에서 앱 설정 방법
		- 앱에 명령줄 인수로 앱에 필요한 설정을 넘긴다.
		- 옵션 목록이 크다면 설정을 파일에 저장하고 사용할 수 있다.
		- 환경 변수를 이용해 지정한다.
	- 도커 컨테이너 내부에 있는 설정 파일을 사용하는 것이 애매한 이유
		- 설정 파일이 있는 볼륨을 컨테이너에 마운트해야하기 때문
			- 그래도 설정 파일을 그대로 이미지 안에 넣고 빌드하는 것은 하드코딩이나 다름없긴하다
			- 누구나 비밀정보를 들여다 볼 수도 있기 때문에 이렇게 지정해서는 안된다.
	- 앱의 설정 지정 방법
		- 컨테이너에 명령줄 인수 전달
		- 각 컨테이너를 위한 사용자 정의 환경변수 지정
		- 특수한 유형의 볼륨을 통해 설정 파일을 컨테이너에 마운트

## 컨테이너에 명령줄 인수 전달

&emsp;&emsp;

### 도커에서 명령어와 인자 정의

- 설명
	- ENTRYPOINT
		- 정의
			- 컨테이너가 시작될 때 호출될 명령어
	- CMD
		- 정의
			- ENTRYPOINT에 전달되는 인자
		- shell과 exec 형식 간의 차이
			- shell
				- `ENTRYPOINT node app.js`
				- 메인 프로세스는 이 방식으로 실행된다.
			- exec
				- `ENTRYPOINT ["node", "app.js"]`
				- ENTRYPOINT 명령은 일반적으로 이 방식을 이용한다.
				- shell 프로세스는 불필요하다.
				- `CMD ["10"]` 이런식으로 인수를 넘기면된다.
	- 주의점
		- 이 설명을 자세히 안하는 이유는 앵간하면 직접 도커파일을 작성할 일이 오늘날에는 별로 없어서 그렇다.
		- 이젠 수많은 도커파일 자동작성 자동화 툴이 많다. 

&emsp;&emsp;

### 쿠버네티스에서 명령과 인자 재정의

``` yaml
kind: Pod
spec:
  containers:
  - image: some/iamge
    command: ["/bin/command"]
    arg: ["1","2","3"]
```

- 설명
	- command
		- 컨테이너에서 실행되는 실행파일
	- args
		- 실행파일에 전달되는 인자

	- command와 args 속성도 활용 가능하다.

## 컨피그 맵으로 설정 분리

- 설명
	- 앱의 구성의 요점은 환경에 따라 다르거나 자주 변경되는 설정 옵션을 앱 소스코드와 별도로 유지하는 것이다.
	- 파드 정의를 앱의 소스코드로 생각한다면(MSA에서는 그렇게 여기긴 한다) 설정을 파드 정의에서 밖으로이동시켜야 한다는 것은 명확하다.

&emsp;&emsp;

### 컨피그 맵 (이하 CM)

- 설명
	- 정의
		- 짧은 문자열에서 전체 설정 파일에 이르는 값을 가지는 키/값 쌍으로 구성된 맵
	- 특징
		- 앱은 CM을 직접 읽거나 CM을 몰라도 된다.
		- 맵의 내용은 컨테이너의 환경변수 또는 볼륨 파일로 전달된다.
		- 환경 변수는 `$(변수)` 구문을 통해 명령줄 인수에서도 참조가 가능하기 때문에 CM 항목으로 프로세스의 명령줄 인자로 전달 가능하다.
		- 앱은 필요한 경우 k8s API 엔드포인트를 통해 CM 내부 내용을 직접 읽을 수도 있긴 하지만 앵간하면 앱이 모르게 해야한다.
		- 앱이 CM을 사용하는 방법에 관계 없이 별도의 독립적인 오브젝트에 설정을 포함하면 각기 다른 환경에 관해 동일한 이름으로 CM에 관한 여러 매니페스트 유지가 가능하다.
		- 파드는 CM을 이름으로 참조하기 때문에 모든 환경에서 동일한 파드 정의를 사용해 각 환경에서 서로 다른 설정을 사용할 수 있다.

&emsp;&emsp;

### CM 생성


```  yaml
apiVersion: v1
data:
  sleep-interval: "25"
kind: ConfigMap
metadata:
  creationTimeStamp: 2025-01-01T20:20:20Z
  name: prom-config
  namespace: default
  resourceVersion: "123456"
  selfLink: /api/v1/namespaces/default/configmaps/prom-config
  uid: uuids
```

- 설명
	- `k create cm fortune-config --from-literal=sleep-interval=25`
		- 이 명령을 수행한 다음의 yaml을 출력하면 위의 yaml이 나오게 된다.
		- 물론 파일을 직접 생성해서 만들 수도 있다.
	- resourceVersion:
		- 리소스의 버전
		- 오브젝트 변경때마다 변경된다.
		- 주로 동시성 제어와 캐싱에 사용된다.
	- selfLink
		- 이 리소스에 접근할 수 있는 API 경로(URL)(Kubernetes의 최신 버전에서는 사용이 중단되었을 수 있음)
	- uuid
		- 클러스터 내에서 해당 리소스를 유일하게 식별할 수 있는 고유 ID

kind: ConfigMap
metadata:

  namespace: default

- 설명
	- 다음과 같은 파일을 prom-local-config.yaml으로 저장한 다음 이 파일을 기준으로 CM 구성이 가능하다.
		- `k create cm prom-config-local --from-file=prom-local-config.yaml`
		- 파일을 여러개 붙일 수도 있다.
		- `k create cm prom-config-local --from-file=prom-local-config.yaml --from-file=grafana-config-local.yaml`
		- `--from-file`인수는 여러개를 붙여 쓸 수 있는 인수값이다.
	- CM을 여러개를 구성해도 되고 CM 안에 파일을 여러개를 넣어도 된다.
		- 네임스페이스로 환경을 분리하는게 더 중요하다.

### CM 항목을 환경변수로 컨테이너에 전달

``` yaml
spec:
  containers:
  - image: luksa/ fortune:env
    env:
    - name: INTERVAL
      valueFrom:
        configMapKeyRef:
          name: fortune-config
          key: sleep-interval
```

- 설명
	- envFrom 또는 env + valueFrom을 사용해서 구현한다.
	- 참고로 파드에 없는 CM을 참조하면 컨테이너 시작에 실패한다.

          name: prom-cm

### CM 볼륨을 이용해 CM 항목을 파일로 노출

``` yaml
apiVersion: apps/v1 
kind: Deployment 
metadata: 
  name: prometheus 
  namespace: monitoring 
spec: 
  replicas: 1 
  selector:
    matchLabels: 
      app: prometheus 
  template: 
    metadata: 
      labels: 
        app: prometheus 
    spec: 
      containers: 
      - name: prom 
        image: prom/prometheus:latest 
        args: 
        - '--config.file=/etc/prometheus/prometheus.yml' 
        - '--storage.tsdb.path=/prometheus' 
        ports: 
        - containerPort: 9090 
        volumeMounts: 
        - name: config 
          mountPath: /etc/prometheus 
          readOnly: true 
        volumes: 
        - name: config 
          configMap: 
            name: prom-config
```

- 설명
	- `kubectl create configmap prom-config --from-file=prometheus.yaml --namespace=monitoring`
		- 이 명령어를 이미 입력해서 진행했을 경우를 가정하고 진행한 Deployment를 구성한 것이다.

&emsp;&emsp;

### 볼륨의 특정한 CM 항목 노출

``` yaml
apiVersion: apps/v1 
kind: Deployment 
metadata: 
  name: prometheus 
  namespace: monitoring 
spec: 
  replicas: 1 
  selector:
    matchLabels: 
      app: prometheus 
  template: 
    metadata: 
      labels: 
        app: prometheus 
    spec: 
      containers: 
      - name: prom 
        image: prom/prometheus:latest 
        args: 
        - '--config.file=/etc/prometheus/prometheus.yml' 
        - '--storage.tsdb.path=/prometheus' 
        ports: 
        - containerPort: 9090 
        volumeMounts: 
        - name: config 
          mountPath: /etc/prometheus 
          readOnly: true 
        volumes: 
        - name: config 
          configMap: 
            name: prom-config
            items:
            - key: prometheus.yaml
              path: prometheus.yml
```

- 설명
	- `items[0].path`는 목표하는 경로의 특정 파일만을 타깃하여 설정한 것이다.
	- 이 방식이 가장 깔끔하게 적용하는 방법이다.

&emsp;&emsp;

### 디렉터리 안에 다른 파일을 숨기지 않고 개별 CM 항목을 파일로 마운트

- 설명
	- 만약 `/etc/prometheus/`내부에 `prometheus.yml`을 제외한 다른 중요 파일이 사라진다면, 컨테이너가 망가질 가능성이 크다.
	- volumeMount시 mountPath를  디렉터리 대신 파일을 지정하고, 대신 subPath 속성으로 파일이나 디렉터리 하나만 볼륨에 마운트할 수 있다.
	- CM 볼륨 모든 파일 권한은 644이다.
		- `configMap.defaultMode`이 필드로 지정 가능하다.

&emsp;&emsp;

### 앱 재시작 없이 동적 설정 업데이트

- 설명
	- 설정관련 세팅은 가장 중요한 것은 앱의 정지 없이 바로 반영이 가능한가 여부이다.
	- CM을 이용해 볼륨을 노출하면 파드를 다시 만들거나 컨테이너 재시작 필요없이 설정 업데이트가 가능하다.
		- 다만 파일이 업데이트될 때까지 엄청 오래걸린다.


&emsp;&emsp;

#### CM 편집

- 설명
	- `k edit cm cmName`
		- 볼륨의 실제 파일도 곧 업데이트 된다.
		- exec 명령으로 출력하면 볼 수 있다.
		- 실제 앱에는 반영되진 않는다.
	- 실제 앱에 반영하는 법
		- `k exec cmVolume -c containerName -- nginx -s reload`
		- 이런식으로 직접 명령어를 전달해 다시 리로드해야한다.

&emsp;&emsp;

#### 파일 한꺼번에 업데이트 하는 방법

- 설명
	- 마운트된 CM 볼륨 안의 파일은 ..data 디렉터리의 파일을 가리지는 심볼릭 링크로 CM 업데이트시 k8s는 새 디렉터리를 생성해 모든 파일을 디렉터리에 작성하고 ..data가 새 디렉터리로 가리키게한다.

&emsp;&emsp;

#### 이미 존재하는 디렉터리에 파일만 마운트했을 때 업데이트가 되지 않는 것 이해하기

- 설명
	- subPath를 사용하면 파일만 마운트하는 경우 업데이트가 되지 않는다.
	- 해결 방법
		- 전체 볼륨을 다른 디렉터리에 마운트한 다음 해당 파일을 가리키는 심볼릭 링크를 생성하는 것
			- 컨테이너 이미지에서 심볼릭링크를 만들거나 컨테이너 시작시 만들 수 있다.

&emsp;&emsp;

#### CM 업데이트의 결과 이해하기

- 설명
	- 동일한 이미지에서 생성된 여러 실행 컨테이너 간에 차이가 없는지 확인할 수 있으므로 컨테이너를 실행하는 데 사용되는 CM을 수정하는 것은 불변성을 우회하는 것이다.
		- 앱이 설정을 다시 읽는 기능을 지원하지 않는 경우에는 문제를 야기한다.
			- 특히 서로 다른 설정을 지닌 인스턴스가 올라가는 문제가 발생한다.
		- CM을 변경한 이후 생성된 파드는 새로운 설정을 예전 파드는 예전 설정을 사용한다.
			- 이것은 새로운 파드에만 국한되는 것이 아니고 파드 컨테이너가 어떠한 이유로든 새로 시작되면 새로운 프로세스는 새로운 설정을 바라보게 된다.
	- 앱은 reloading이 지원되면 CM 수정이 야기하는 불변성 파괴는 큰 문제가 되지 않는다.
		- 다만 모든 인스턴스에 동기적으로 업데이트되지 않기 때문에 개별 파드 파일이 최대 1분 동안 동기화되지 않은 상태로 있을 수 있다.

## 시크릿 리소스

&emsp;&emsp;

### 시크릿

- 설명
	- 정의
		- 보안에 민감한 정보를 보관하고 배포하기 위한 오브젝트
	- 특징
		- CM과 매우 비슷한 키-값 쌍으로 구성
		- k8s는 시크릿에 접근해야하는 파드가 실행되고 있는 노드에만 개별 시크릿을 배포해 시크릿을 안전하게 유지한다.
		- 노드 자체적으로 시크릿을 항상 메모리에만 저장되게하고 물리 저장소에 저장되지 않게 한다.
			- 물리 저장소는 시크릿을 삭제해도 wipe 과정이 필요하기 때문이다.
		- 마스터 노드(etcd)에는 시크릿을 암호화되지 않는 형식으로 저장하므로 마스터 노드를 보호해야만 한다.
			- 권한 없는 사용자가 API를 사용하지 못하게 하는 것과 etcd 저장소를 안전하게 하는 것이 중요하다.
			- 파드를 만들 수 있는 사람은 누구나 시크릿을 파드에 마운트하고 민감한 데이터에 접근하는 것이 가능하기 때문이다.
		- 최신 k8s에서는 etcd가 시크릿을 암호화된 형태로 저장해 시스템을 좀 더 안전하게 만든다.
		- 언제부터 시크릿을 사용할 것인지 CM을 사용할지 올바르게 선택해야한다.
	- 주로 사용하는 상황
		- 환경 변수로 시크릿을 컨테이너에 전달
		- 시크릿 항목을 볼륨 파일로 노출

		- 허나 1.24 버전부터는 임시로 bound service account token이 마운트된다.
		- kubectl로 조회가 가능한 형태가 아니다.
		- projected volume 형태로 Pod에 주입된다.
	- 기본적으로 붙는 시크릿 볼륨의 주소는 `/var/run/secrets/kubernetes.io/serviceaccount`이다.
		- `k describe po podName`을 하게 되면 마운트 영역에서 볼 수 있다.
	- `k exec prometheus-rsg9q -- ls /var/run/secrets/kubernetes.io/serviceaccount`
		- 위의 명령을 진행하면 시크릿의 각 구성요소를 볼 수 있다.
		- 책에서는 그냥 이어 붙이지만 `--`를 붙여야한다.

#### 시크릿의 구성 요소

- 설명
	- ca.crt
	- namespace
	- token

&emsp;&emsp;

### 시크릿 생성

- 설명
	- 생성 절차
		1. `openssl genrsa -out https.key 2048`
		2. `openssl req -new -x509 -key https.key -out https.cert -days 3650 -subj /CN=www.littok.com`
		3. `k create secret generic secretName --from-file=https.key --from-file=https.cert --from-file=foo`
			1. 시크릿에는 3가지 타입이 있다.
				- docker-registry
					- 도커 레지스트리를 사용하기 위한 시크릿이다.
				- tls
					- TLS 통신을 위한 시크릿이다.
				- generic

&emsp;&emsp;

### CM과 시크릿 비교

- 설명
	- 결론
		- 시크릿의 항목은 base64 인코딩 문자열이며 CM은 그렇지 않다.
	- 바이너리 데이터 시크릿 사용
		- base64를 사용하는 이유는 일반 텍스트 뿐 아니라 바이너리 값도 담을 수 있기 때문이다.
		- 민감하지 않은 데이터도 시크릿 사용이 가능하나 1MB까지이다.
	- stringData 필드
		- 쓰기 전용 필드이며 값을 설정할 때만 사용이 가능하다.
	- 파드에서 시크릿 항목 읽기
		- secret 볼륨을 통해 시크릿을 컨테이너에 노출하면 시크릿 항목의 값이 모두 디코딩돼 파일에 기록된다.

&emsp;&emsp;

### 파드에서 시크릿 사용하기

- 설명
	- volumeMounts 필드에 하나씩 추가한다.
		- 마운트 과정은 다른 볼륨들과 동일하다.
	- 시크릿 볼륨을 메모리에 저장하는 이유
		- ==**secret 볼륨은 tmpfs 메모리 파일 시스템을 사용하는데 민감한 데이터는 디스크에 일부러 저장하지 않는 것이다.**==
	- 환경 변수로 시크릿 항목 노출하기
		- configMapKeyRef 처럼 secretKeyRef를 사용하면 노출이 가능하다.
		- 물론 이는 좋은 방법은 딱히 아니다.

&emsp;&emsp;

### 이미지 가져올 때 사용하는 시크릿

- 설명
	- docker private repository는 자격증명이 필요하다.
		- 도커 레지스트리 자격증명을 가진 시크릿을 생성하고 파드 매니페스트 안에 imagePullSecrets 필드에 참조시키면 된다.
	- 생성 방법
		- `k create secret docker-registry dockerhubSecret --docker-username=username --docker-password=password --docker-email=username@email.com`
	- 파드 정의에서 사용하기
		- imagePullSecrets에서 만든 시크릿을 참조시킨다.
	- 모든 파드에서 이미지를 가져올 때 사용할 시크릿을 모두 지정할 필요는 없다.
		- 이미지를 가져올 때 사용할 시크릿을 서비스 어카운트에 추가해 모든 파드에 자동으로 추가될 수 있게 할 수 있다.
