## 애플리케이션에서 파드 메타데이터와 그 외 리소스에 접근하기

&emsp;&emsp;

## Downward API로 메타데이터 전달하기

- 설명
	- 메타데이터가 필요한 경우
		- 파드가 생성되는 시점의 데이터가 필요한 경우
		- 파드의 레이블이나 어노테이션과 같이 어딘가에 이미 설정된 데이터가 필요한 경우
	- Downward API
		- 특징
			- 환경 변수 또는 파일로 파드와 해당 환경의 메타데이터를 전달할 수 있다.
			- 환경변수 또는 파일에 파드의 스펙 또는 상태값을 채우기 위해 사용된다.
				- kubectl api와는 다르다.

&emsp;&emsp;

### 사용가능한 메타데이터

 - 설명
	 - 파드의 이름
	 - 파드의 IP 주소
	 - 파드가 속한 네임스페이스
	 - 파드가 실행 중인 노드의 이름
	 - 파드가 실행 중인 서비스 어카운트 이름
		 - 서비스 어카운트란 파드가 API 서버와 통신할 때 인증하는 계정이다.
	 - 각 컨테이너의 CPU와 메모리 요청
	 - 각 컨테이너의 CPU와 메모리 제한
	 - 파드의 레이블
		 - 볼륨으로만 노출 가능함
	 - 파드의 어노테이션
		 - 볼륨으로만 노출 가능함

&emsp;&emsp;

### 환경 변수로 메타데이터 노출하기


``` yaml
apiVersion: V1
kind: Pod
metadata:
  name: downward
spec:
  containers:
  - name: maln
    image: busybox
    command : [ "sleep", "9999999"]
    resources:
      requests:
        cpu: 15m
        memory: 100Ki
      limits:
        cpu: 100m
        memory: 4Mi
env:
- name: POD_NAME
  valueFrom:
    fieldRef:
      fieldPath: metadata.name
- name: POD NAMESPACE
  valueFrom:
    fieldRef :
      fieldPath: metadata.namespace
- name: POD_IP
  valueFrom:
    fieldRef:
      fieldPath: status.podIP
- name : NODE_NAME
  valueFrom:
    fieldRef:
      fieldPath: spec.nodeName
- name: SERVICE_ACCOUNT
  valueFrom:
    fieldRef:
      fieldPath: spec.serviceAccountName
- name: CONTAINER_CPU_REQUEST_MILLICORES
  valueFrom:
    resourceFieldRef:
      resource: requests.cpu
      divisor: 1m
- name: CONTAINER_MEMORY_LIMIT_KIBIBYTES
  valueFrom:
    resourceFieldRef:
      resource: limits.memory
      divisor: 1Ki
```

 - 설명
	 - 각 환경 변수가 downward API를 통해 조회되는 모습이다.
	 - CPU 제한과 요청의 제수는 1 또는 1m으로 설정한다.
	 - 메모리 제한/요청의 제수는 1, 1k, 1ki, 1M 또는 1Mi 등이 될 수 있다.
	 - 파드를 만든 후에는 `k exec`를 사용해 컨테이너에 있는 모든 환경 변수를 볼 수 있다.

&emsp;&emsp;

### downwardAPI 볼륨에 파이로 메타데이터 전달하기

``` yaml
apiVersion: V1
kind: Pod
metadata:
  name: downward
  labels:
    foo: bar
  annotaions:
    key1: value1
    key2: |
      multi
      line
      value
spec:
  containers:
  - name: maln
    image: busybox
    command : [ "sleep", "9999999"]
    resources:
      requests:
        cpu: 15m
        memory: 100Ki
      limits:
        cpu: 100m
        memory: 4Mi
    volumeMounts:
    - name: downward
      mountPath: /etc/downward
volumes:
- name: downward
  downwardAPI:
    items:
    - path: "podName"
      fieldRef:
        fieldPath: metadata.name
    - path: "podNamespace"
      fieldRef:  
        fieldPath: metadata.namespace
    - path: "labels"
      fieldRef:  
        fieldPath: metadata.labels
    - path: "annotations"
      fieldRef:  
        fieldPath: metadata.annotations
    - path: "containerCpuRequestMilliCores"
      resourceFieldRef:  
        containerName: main
        resource: requests.cpu
        divisor: 1m
    - path: "containerMemoryLimitBytes"
      resourceFieldRef:  
        containerName: main
        resource: requests.memory
        divisor: 1
```

- 설명
	- downward 볼륨은 `/etc/downward` 아래에 마운트한다.
	- downwardAPI 볼륨을 downward라는 이름으로 정의한다.
	- 파드의 이름은 podName 파일에 기록된다.
		- 나머지도 다 downward 볼륨에 기록되는 세팅한 것이다.
	- 환경변수로 메타데이터를 전달하는 대신 볼륨을 정의하고 볼륨을 `/etc/downward`에  마운트한 것이다.
		- `k exec downward -- ls -lL /etc/downward`를 하면 실제로 파일로 저장된 볼륨의 구조를 볼 수 있다.
		- 각 파일은 볼륨 정의의 항목에 해당된다.
			- 복수 지정된 값은 \n으로 표시되 한줄로 기록된다.

&emsp;&emsp;

#### 레이블과 어노테이션  업데이트

 - 설명
	- 레이블이나 어노테이션 변경시 k8s가 이 값을 가지고 있는 파일을 업데이트하여 파드가 항상 최신 데이터를 보게 끔 한다.
		- 이것이 레이블과 어노테이션이 환경변수로 노출될 수 없는 이유이다.
			- 환경변수는 나중에 업데이트를 할 수 있는 부분이 아니기 때문에 파드의 레이블 또는 어노테이션이 환경변수일지라도 나중에 변경이 발생하면 새로운 값을 노출할 방법 자체가 존재하지 않는다.

&emsp;&emsp;

#### 볼륨 스펙에서 컨테이너 수준 메타데이터 참조

- 설명
	- 컨테이너의 리소스 제한 또는 요청과 같은 ==**컨테이너 수준 메타데이터를 노출하는 경우 리소스 필드를 참조하는 컨테이너의 이름을 지정해야 한다.**==
		- resourceFieldRef가 대표적이다.
	- 볼륨은 파드 수준에서 스펙이 지정되어 모든 파드 내 컨테이너들이 공유해서 사용되지만 컨테이너 수준에서의 메타데이터는 다르다. 
		- 그렇기 때문에 컨테이너 수준의 메타데이터를 노출하는 경우는 명확히 컨테이너 명을 지정해야하는 것이다.
	- 볼륨 스펙 내에서 컨테이너의 리소스 필드를 참조할 때는 참조하는 컨테이너의 이름을 명시적으로 지정해야 한다.
		- 컨테이너가 하나인 파드에서도 매한가지이다.
	- 볼륨을 사용해 컨테이너의 리소스 요청이나 제한을 노출하는 것은 환경변수를 사용하는 것보다 약간 더 복잡하지만 필요한 경우 컨테이너의 리소스 필드를 다른 컨테이너에 전달할 수 있는 장점이 있다.

&emsp;&emsp;

#### Downward API 사용 시기 이해하기

- 설명
	- DownwardAPI는 사용이 간편하며 앱을 k8s에 독립적으로 만들어준다.
	- 환경변수의 특정 데이터를 활용하는 앱은 DownwardAPI는 특히 유용하다.
		- 환경변수를 파싱하는 셸 스크립트를 짤 필요가 없어진다.
	- 사용가능한 데이터는 이정도가 다라 나머지 정보는 k8s API와 통신해야한다.

### k8s REST API 살펴보기

&emsp;&emsp;

#### kubectl 프록시로 API 서버를 액세스하기
 
 - 설명
	 - 기본적으로 k8s API 서버는 HTTPS를 사용하기 때문에 프록시 서버의 도움이 필요하다.
	 - 사용법
		 - `k proxy`
			 - 이것을 호출하는 순간 즉시 프록시 터널이 형성되어 localhost로 통신하면 된다.
	 - `curl http://localhost:8001`
		 - 다양한 호출 가능한 경로 정보를 알려준다.
			 - hateos와 비슷하다.
		 - 보면 알겠지만 `apiVersion: 값`이 값에 사용됬던 다양한 버전들이 존재한다.
	 - `curl http://localhost:8001/apis/batch`
		 - 엔드 포인트를 호출하면 사용가능한 버전과 클라이언트가 사용해야 하는 선호 버전에 관한 정보와 관련 API 그룹에 관한 설명이 표시된다.
	 - `curl http://localhost:8001/apis/batch/v1`
		 - 관련 API 그룹에서 하나를 집어 호출하면 리소스 리스트를 반환한다.
			 - 각 리소스 안에 들어있는 속성
				 - 이름
				 - 네임스페이스
				 - kind
				 - verbs
					 - 이 api에서 사용 가능한 행동 API들이다.

&emsp;&emsp;

#### 클러스터에 모든 잡 인스턴스 나열하기

- 설명
	- `curl http://localhost:8001/apis/batch/v1/jobs`
		- 이전에 `curl http://localhost:S0Ø1/apis/batch/v1`의 각 리소스의 name 중 하나를 붙여서 호출하면 이렇게 나온다.
		- 참고로 모든 인스턴스가 등장하는데 없으면 items가 비어있는게 정상이다.
	- `curl http://localhost:8001/apis/batch/v1/namespaces/default/jobs/jobName`
		- 이렇게 하면 특정 이름에 대한 리소스에 대해서 볼 수 있다.
		- `k get job jobName` 과 같은 결과이다.

### 파드 내에서 API 서버와 통신

- 설명
	- 사전 조건
		- API 서버의 위치를 알아야한다.
		- API 서버와 통신하고 있는지 확인해야한다.
		- API 서버로 인증해야한다.
	- API 서버 주소 찾기
		- 서비스에 관해 환경 변수가 구성되어있으며 API 서버의 IP 주소와 포트를 컨테이너 내부 환경변수에서 모두 구할 수 있다.
			- ==**그렇기 때문에 서비스는 반드시 파드보다 먼저 구성되어야하는 것 기억하자.**==
		- 단순히 내부에서 `https://kubernetes`만 사용해도 접속이 가능하다.
			- 다만 컨테이너에 자동으로 연결되는 시크릿 볼륨의 인증서가 필요하다.
	- 서버 아이덴티티 검증
		- `cur1 --cacert /var/run/secrets/kubernetes.io/serviceaccount/ca.crt https://kubernetes`
			- 결과는 Unauthorized가 나와야 정상이다.
		- CURL_CA_BUNDLE 환경 변수에 저 파일을 지정하면 편하게 API 서버에 접속이 가능하다.
	- API 서버 인증
		- 사전 조건
			- 환경변수 TOKEN을 /var/run/secrets/kubernetes.io/serviceaccount/token에서 가져와 저장한다.
			- CURL_CA_BUNDLE 환경 변수에 파일 위치 지정
		- `curl -H "Authorization: Bearer $TOKEN" https://kubernetes`
			- 정상적으로 통신이 가능해진다.
	- 파드에서 실행 중인 namespace 얻기
		- `/var/run/secrets/kubernetes.io/serviceaccount/namespace`
			- 여기에 있는 내용을 NS 환경변수에 넣는다.
				- 그리고 위에서 단일 리소스 조회 curl 요청을 보내면 된다.
	- 팁
		- RBAC 비활성화
			- RBAC이 활성화된 k8s 클러스터를 사용하는 경우 서비스 어카운트가 API 서버에 접속할 권한이 없을 수 있다.
			- `kubectl create clusterrolebinding permissive-binding --clusterrole=c luster-admin --group=system:serviceaccounts`
				- 이것을 사용하면 모든 서비스 어카운트를 마음대로 할 수 있지만 프로덕션 클러스터에서 금기사항이다.

&emsp;&emsp;

### 앰버서더 컨테이너를 이용한 API 서버 통신 간소화

- 설명
	- 앰버서더 컨테이너 패턴
		- 앰버서더 컨테이너
			- k8s API 서버와 통신하는 용도로만 사용되는 컨테이너
		- 메인 컨테이너
			- 앰버서더 컨테이너와 HTTP만으로 통신하는 컨테이너
	- 특징
		- 시크릿 볼륨에 있는 default-token 파일을 사용해 이를 수행한다.
			- 위에서 설명했지만 볼륨은 파드 수준에서 접근이 가능하기 때문에 모든 컨테이너가 접근이 가능하다. 따라서 default-token 파일에도 접근이 가능하다.
		- 보안흐름을 깨뜨리지 않고 모든 파드에서 재사용이 가능한 괜찮은 패턴이다.
	- 추가적인 앰배서더 컨테이너를 사용한 curl 파드 실행
		- curl은 HTTP 일반 요청을 앰배서더 컨테이너 내에서 실행 중인 프록시로 전송한 다음 프록시는 HTTPS 요청을 API 서버로 전송하여 토큰을 전송해 클라이언트 인증을 처리하고 서버의 인증서를 검증해 서버의 신원을 확인한다.
