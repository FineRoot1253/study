## 파드

- 설명
	- 정의
		- 하나 이상의 컨테이너를 지닌 함께 배치되는 컨테이너들의 집합
	- 특징
		- 쿠버네티스의 기본 빌딩 블록
		- k8s는 컨테이너를 운영하는 것이 아닌 컨테이너를 가진 파드를 배포하고 운영한다.
		- 파드에 여러 컨테이너가 포함된 경우 모든 컨테이너는 항상 하나의 워커 노드에서 실행되며 ==**다른 노드에 걸쳐 존재하지 않는다.**==
	- 필요성
		- 여러 프로세스를 실행하는 단일 컨테이너보다 다중 컨테이너가 괜찮은 이유
			- IPC 또는 하나의 파일을 기준으로 프로세스간 통신하는 구조로 실행되는 애플리케이션의 경우를 고려해보면, k8s는 항상 컨테이너에서 실행시키고, 이 각 컨테이너는 격리된 머신과 비슷하기 때문에 여러 프로세스를 단일 컨테이너에서 실행시키는 것이 효율적일 것 같지만 전혀 그렇지 않다.
			- 컨테이너는 단일 프로세스를 실행하는 목적으로 설계되었고, 단일 컨테이너에서 관련 없는 다른 프로세스를 실행한다고 가정하면 해당 모든 프로세스를 실행하고 로그를 관리하는 것은 모드 사용자의 책임이다.
				- 즉, 개별 프로세스가 실패하는 경우 자동으로 재시작하는 매커니즘을 알아서 구현해야한다.
				- 모든 므로세스는 동일한 표준 출력을 사용하여 로그로 기록하기 때문에 어떤 프로세스가 남긴 로그인지 파악하는 것 또한 매우 어렵다.
				- 즉, 각 프로세스를 자체의 개발 컨테이너로 실행해야하는 결정적인 두 가지 이유이다.
	- 파드의 이해
		- 여러 프로세스를 단일 컨테이너로 묶지 않기 때문에 컨테이너를 묶어서 관리할 논리적 단위가 필요하여 등장함
		- 컨테이너 모음을 사용해 밀접하게 관련된 프로세스를 함께 실행하고 격리된 상태를 유지한다.
		- 같은 파드에서 컨테이너 간 부분 격리도 가능하다.
			- k8s는 파드 안에 있는 모든 컨테이너가 파드별 별도의 네임스페이스를 지니게 하지 않고 동일한 리눅스 네임스페이스를 공유하도록 도커를 세팅한다.
			- 파드의 모든 컨테이너는 동일한 네트워크, UTS 네임스페이스 안에서 실행되기 때문에 모든 컨테이너는 같은 호스트 이름과 네트워크 인터페이스를 공유한다.
			- 모든 컨테이너는 네트워크, UTS 네임스페이스처럼 동일한 IPC 네임스페이스를 공유하기 때문에 IPC를 통해 컨테이너 간 상호통신이 가능하다.
				- 다만 최신 k8s와 도커는 비활성화되어 있다.
				- 파드 안에 있는 컨테이너가 별도의 PID 네임스페이스를 사용할 경우에는 ps aux ㅁ여령을 실행하면 해당 컨테이너에 있는 실행중인 프로세스를 볼 수 있다.
		- 파일 시스템의 경우
			- 대부분의 컨테이너의 파일 시스템은 이미지에서 나오므로 기본적인 파일 시스템은 다른 컨테이너와 완전히 분리된다.
			- 물론 k8s의 볼륨을 이용해 컨테이너간 파일 디렉터리를 공유하는 것은 당연히 가능하다.
		- 컨테이너가 동일한 IP와 포트 공간을 공유하는 방법
			- 파드 안의 컨테이너는 네트워크 네임스페이스 안에서 실행되므로 동일한 IP 주소와 포트 공간을 공유한다.
				- 따라서 동일한 파드 안의 컨테이너들 끼리 실행 중인 프로세스가 같은 포트 번호를 사용하지 않도록 주의해야한다.
				- 다른 파드에 있는 컨테이너는 서로 다른 포트 공간을 가지므로 이런 걱정은 안해도 된다.
				- 파드 안에 있는 모든 컨테이너는 동일한 루프백 네트워크 인터페이스를 가지므로 대부분 서로 상호 통신을 할 때는 localhost를 사용하는 것이 국룰이다.
		- 파드 간 플랫 네트워크
			- k8s 클러스터의 모든 파드는 하나의 플랫한 공유 네트워크 주소 공간에 상주하므로 모든 파드는 다른 파드의 IP 주소를 이용해 접근하는 것이 가능하다.
				- 둘 사이에 NAT 또한 존재하지 않는다.
				- 두 파드가 서로 네트워크 패킷을 보내면 상대방의 IP 주소를 패킷 안에 있는 출발지 IP 주소에서 찾을 수 있다.
			- 각 파드는 라우팅 가능한 IP 주소를 알고 있고 (같은 네트워크 네임 스페이스 이므로) 모든 다른 파드는 해당 IP 주소를 통해 파드들을 서로 바라 볼 수 있는 구조이다.
			- 각 파드들은 동일하거나 다른 워커 노드에 있는지는 딱히 중요한 요소가 아니며 두 경우 모두 ==**파드 안에 있는 컨테이너는 플랫 네트워크를 통해 서로 통신이 가능하다.**==
			- 파드는 논리적인 호스트로서 컨테이너가 아닌 환경에서 물리적 호스트 혹은 VM과 매우 유사하게 동작한다.
				- 동일한 파드에서 실행한 프로세스는 각 프로세스가 컨테이너로 캡슐화되었다는 점을 제외하면 같은 물리적 혹은 가상 머신에서 동작하는 것과 동일하다.


### 파드에서 컨테이너의 적절한 구성

- 설명
	- 파드는 특정한 앱만을 호스팅하기
		- 한 호스트에 모든 타입의 앱을 때려넣었던 옛과는 달리 파드는 그렇게 사용하지 않고 상대적으로 가벼운 특징을 가졌기 때문에 오버헤드없이 필요한 만큼만 파드를 얻을 수 있다.
		- ==**모든 것을 파드 하나에 넣는 것이 아닌 앱을 여러 파드로 구성하고 각 파드에는 밀접하게 관련 있는 구성 요소나 프로세스만을 포함해야한다.**==
	- 다계층 앱을 여러 파드로 분할하기
		- 프론트엔드와 백엔드는 항상 다른 파드로 구성되어야한다.
			- 그래야만 다른 노드에서 실행이 되며 두 노드를 가진 k8s 클러스터가 있고, 파드는 하나로 구성되어있다면 하나의 노드는 항상 활용하지 않고 버리는 구조인 것이다.
				- 아님 파드를 두 개로 분리하면 k8s가 fe를 한 노드 be는 다른 노드에 스케줄링해 인프라 활용능력을 끌어 올릴 수 있다.
			- 예를 들면 FrontEnd 서버와 DB를 각각 컨테이너로 구성하는 것은 적절치 않다.
	- 개별 확장이 가능하도록 여러 파드로 분할하기
		- 두 컨테이너를 하나의 파드에 넣지 말아야 하는 이유는 스케일링 때문이다.
		- 파드는 스케일링의 기본 단위이기 때문에 하나에 파드에 컨테이너를 몰빵하면, 요청이 몰리는 구간을 분할하여 확장하는 것이 불가능하게 된다.
		- stateful 파드인 be는 특히나 확장할 때 고려해야할 요소가 많기 때문에 어렵다.
	- 파드에서 여러 컨테이너를 사용하는 경우 고려하기
		- 보통 하나의 파드에 컨테이너를 여러 개를 구성하는 경우는 다른 프로세스를 맞물려 돌려야하는 경우에 의해 그렇게 돌아간다.
			- 예를 들면 꾸준하게 돌아가야하는 데몬 프로세스를 둬야한다거나, 파일 저장을 전담하는 프로세스를 둬 비동기로 동작하게 하여 비동기로 동작하게 만들던가 이런 경우이다.
			- 사이드카 컨테이너의 또다른 예제는 로그 로테이터와 수집기, 데이터 프로세서, 통신 어뎁터 등이 존재한다.
	- 파드에서 여러 컨테이너를 사용하는 경우를 고려하기
		- 컨테이너를 파드로 묶어 그룹으로 둘 때 고려할 사항
			- 컨테이너는 함께 실행해야만 하는지, 아님 다른 호스트에서 실행 가능한지
			- 여러 컨테이너가 꼭 그룹이 되어야 하나의 구성 요소로 완성이 되는지, 아님 개별적으로 분리가 가능한지
			- 컨테이너가 함께 스케일링이 가능한지, 아니면 개별적으로 스케일링 되어야하는지
		- 보통 일단 파드를 분리하는 것이 대다수 괜찮은 선택이다.
			- 보통 아키텍처 관점에서는 의존성과 역할, 부하등을 고려해서 분리하는데 이것만 해도 괜찮게 분리가 된다.

## YAML, JSON 디스크립터로 파드 생성해보기

- 설명
	- k8s REST API에 json이나 yaml을 보내 파드 생성이 가능하다.
		- 보통 run은 파드 하나를 구성하는데 간단하게 테스트하는 용도의 명령어이고 이 방식이 가장 보편적으로 생성하는 선언적 생성 방식이다.
		- k8s는 선언형 방식을 최대한 권장하고 있다.
	- 각 오브젝트의 옵션은 API 레퍼런스를 읽어보고 만들어보도록 하자.

### yaml 정의 직접 작성해보기

``` yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: prometheus-manual  
spec:  
  containers:  
    - name: prometheus-manual  
      image: bitnami/prometheus:latest  
      imagePullPolicy: IfNotPresent  
      ports:  
        - containerPort: 9090  
          protocol: TCP
```

- 설명
	- 컨테이너 포트 지정
		- 포트는 굳이 생략을 해도 다른 클라언트에서 포트를 통해 파드를 연결 할 수 있는 여부에 영향을 미치지 않는다.
		- 파드 스펙에 명시적으로 나열하지 않아도 0.0.0.0 주소에 열어둔 포트를 통한 접속을 열어두면 항상 해당 파드를 다른 파드에서 접속할 수 있다.
		- 다만 명시적으로 적는 이유는 다른 사람들이 파드에서 노출된 포트를 빠르게 볼 수 있는 장점이 있다.
			- ==**포트에 이름을 지정하여 편리하게 사용하기 위함이라고 보면 된다.**==
	- 매니페스트 준비 시 꿀팁
		- k explain 명령을 이용해 개별 api 오브젝트에 지원되는 속성을 확인해볼 수 있다.
			- `k explain pods`
			- `k explain pods.specs`
				- 이런식으로 .속성명 을 추가 하면 각 속성을 자세히 알아 볼 수 있다.
				- api 서버가 다 제공해준다.

&emsp;&emsp;

### k create 명령으로 파드 만들기

- 설명
	- `k create -f prometheus-manual.yaml` 이 명령으로 생성 가능
		- 실제로 파드 생성에 가장 많이 사용되는 명령은 create이다.
	- 실행 중인 파드의 전체 정의 조회하기
		- `k get po prometheus-manual -o yaml`
		- `k get po prometheus-manual -o json`
	- 실행 중인 파드 로그 보기
		- `k logs prometheus-manual`
	- 컨테이너 이름름을 지정해 다중 컨테이너 파드에서 로그 꺼내오기는 방법
		- `k logs prometheus-manual -c prometheus-container`


&emsp;&emsp;

### 파드에 요청 보내기

- 설명
	- 로컬 네트워크 포트를 파드의 포트로 포워딩하기
		- `k port-forward prometheus-manual 39090:9090`
	-  포트 포워딩을 통해 파드 연결
		- 다른 터미널에서 이제 39090 포트로 요청을 보낼 수 있다.

## 레이블을 이용한 파드 구성

- 설명
	- 파드는 부분 집합에 따라 구분 지을 필요가 있으며 이것을 레이블이 돕는다.

&emsp;&emsp;

### 레이블 이해

- 설명
	- 정의
		- 파드와 모든 k8s 리소스를 조직화할 수 있는 단순하고 강력한 k8s의 기능
		- 리소스에 첨부하는 키:값 쌍 조합
	- 특징
		- 레이블 셀렉터를 통해 리소스 선택시 활용된다.
		- 레이블 키가 해당 리소스 내에서 고유하다면 하나 이상의 원하는 만큼 레이블을 가질 수 있다.
	- 예시
		- app
			- 파드가 속한 도메인을 지정한다.
		- branch
			- 파드의 버전을 지정한다.
		- rel
			- 파드에서 실행 중인 앱의 배포 버전을 지정한다.
		- 이렇게 하면, 파드가 속한 도메인, 파드의 버전, 파드의 배포 버전등을 통해 보기 좋게 정리할 수 있다.

&emsp;&emsp;

### 파드를 생성할 때 레이블 지정

``` yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: prometheus-manual-v2
  labels:
    creation_method: manual
	env: prod
spec:  
  containers:  
    - name: prometheus-manual  
      image: bitnami/prometheus:latest  
      imagePullPolicy: IfNotPresent  
      ports:  
        - containerPort: 9090  
          protocol: TCP
```

- 설명
	- `k get po --show-labels`
		- 이렇게 하면 생성된 라벨을 볼 수 있다.
	- `k get po -L creation_method,env`
		- 이렇게 하면 특정 레이블에만 집중해 이 레이블을 결과 테이블의 열에 추가할 수 있다.

&emsp;&emsp;

### 기존 파드 레이블 수정

- 설명
	- `k label po prometheus-manual creation_method=manual`
		- label 명령을 사용하면 레이블을 추가할 수 있다.
	- `k label po prometheus-manual-v2 env=debug --overwrite`
		- 변경시 --overwrite 옵션을 마지막에 붙이면 된다.

&emsp;&emsp;

## 레이블 셀렉터를 이용한 파드 부분 집합 나열하기

- 설명
	- 레이블 셀렉터의 특징
		- 특정 레이블로 태그된 파드의 부분 집합을 선택해 원하는 작업을 수행한다.
	- 레이블 셀렉터의 선택 기준
		- 특정한 키를 포함하거나 포함하지 않는 레이블
		- 특정한 키와 값을 가진 레이블
		- 특정한 키를 가지긴 했지만, 다른 값을 지닌 레이블

&emsp;&emsp;

### 레이블 셀렉터를 사용해 파드 나열

- 설명
	- `k get po -l creation_method=manual`
		- 수동으로 생성한 모든 파드 나열
	- `k get po -l env`
		- env 레이블 키는 지녔지만 값이 무슨 값이 들었든 상관없이 해당 되는 파드를 나열
	- `k get po -l '!env'`
		- env 레이블이 없는 파드 나열
	- `k get po -l 'creation_method!=manual'`
		- 수동 생성이 아는 모든 파드 나열
	- `k get po -l 'env in (prod,devel)'`
		- prod 또는 devel로 환경인 모든 파드 나열
	- `k get po -l 'env notin (prod,devel)'`
		- prod 또는 devel 환경이 아닌 모든 파드 나열


<br><br>

## 레이블과 셀렉터를 이용한 파드 스케줄링 제한

- 설명
	- k8s는 모든 노드를 하나의 대규모 배포 플랫폼으로 노출하기 때문에 파드가 어느 노드에 스케줄링되었는지는 중요하지 않다.
		- 각 파드는 요청한 만큼의 정확한 컴퓨팅 리소스를 할당 받고 다른 파드에서 해당 파드로 접근 하는 것은 파드가 스케줄링된 노드에 아무런 영향을 받지 않는다.
		- 따라서 k8s에서 파드를 어디에 스케줄링할지 알려줄 필요가 없다.
	- 그러나 파드를 스케줄링할 위치를 결정할 때 약간이라도 영향을 미쳐야하는 상황이 존재한다.
		- 하드웨어 인프라가 동일하지 않는 경우가 대표적이다.
			- 예시
				- SDD를 쓰는 컨테이너와 HDD를 쓰는 컨테이너
				- AI를 위해 GPU 가속을 쓰는 컨테이너와 GPU 자원 자체가 필요가 없는 컨테이너
	- ==**k8s 아키텍처 핵심 개념 자체는 개발자는 인프라 자체를 모르게 하는 것**==이지만 ==**필요한 노드의 요구사항 정도는 k8s가 적절히 선택하게끔 하는 것은 필요**==하다.
		- 이때 **레이블**과 **레이블 셀렉터**가 필요하다.

&emsp;&emsp;

### 워커 노드 분류에 레이블 사용

- 설명
	- `k label node 노드명 gpu=true`
		- gpu 컴퓨팅을 사용할 노드로써 노드에 라벨을 추가한다.
		- 노드 조회시 이 레이블을 사용해 나열할 수 있다.
	- `k get nodes -l gpu=true`

&emsp;&emsp;

### 특정 노드에 파드 스케줄링

``` yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: prometheus-manual-gpu
spec:
	nodeSelector:
	  gpu: "true"
  containers:  
    - name: prometheus-manual  
      image: bitnami/prometheus:latest  
      imagePullPolicy: IfNotPresent  
      ports:  
        - containerPort: 9090  
          protocol: TCP
```

- 설명
	- nodeSelector 필드를 추가해 스케줄러가 gpu=true를 가진 레이블을 노드를 찾아 선택을 한다.
	- 단, 이 경우 단일 노드만 존재한다.

&emsp;&emsp;

### 하나의 특정 노드로 스케줄링

- 설명
	- nodeSelector에 실제 호스트이름을 지정하면 해당 노드가 오프라인인 경우 제대로 스케줄링이 되지 않는 문제가 존재한다.
	- 레이블 셀렉터를 통해 지정한 특정 기준을 만족하는 노드의 논리적인 그룹을 생각해야 제대로 적용할 수 있다.

## 파드에 어노테이션 추가하기

- 설명
	- 어노테이션 정의
		- 개발시 팀원간 소통을 편하게 하기 위한 설명 추가 기능으로 모든 파드 및 다른 오브젝트가 지닐 수 있는 기능이다.
	- 특징
		- 어노테이션은 레이블과 달리 파드나 오브젝트를 묶을 수는 없는 기능이다.
		- 특정 어노테이션은 k8s에 의해 자동으로 오브젝트에 추가되며 나머지 어노테이션은 사용자에 의해 추가된다.
		- k8s에서 새로운 기능을 추가할 때 흔히 사용되는 기능이다.
			- api 오브젝트에 바로 새로운 필드를 추가하는 것이 아닌 어노테이션을 사용하고 api 변경이 명료해지면 새로운 필드가 도입되는 식이다.
	- 어노테이션이 유용하게 사용되는 경우
		- 파드나 다른 api 오브젝트에 설명을 추가해 주는 것
			- 다른 사람들이 개별 오브젝트에 대한 설명을 빠르게 찾아 볼 수 있게 된다.

&emsp;&emsp;

### 오브젝트의 어노테이션 조회

- 설명
	- 초기에 yaml로 뽑은 디스크립션을 보게 되면 k8s가 추가한 어노테이션을 볼 수 있다.
	- 어노테이션은 총 256kb까지 넣을 수 있어 상대적으로 큰 표현이 가능하여 다양한 표현을 집어넣을 수 있다.
		- 보통 json을 넣는다.

&emsp;&emsp;

### 어노테이션 추가 및 수정

- 설명
	- `k annotate pod prometheus-manual geekos.com/someannotaion="foo bar"`
		- geekos.com/someannotaion이라는 어노테이션을 foo bar라는 값과 함께 추가하는 명령이다.
		- 이런 방식을 사용하는 이유는 키 충돌 방지를 위해 이런 형식을 사용한다.

## 네임스페이스를 이용한 리소스 그룹화

- 설명
	- 네임 스페이스 특징
		- 네임스페이스는 k8s의 리소스이다.
		- 오브젝트를 겹쳐지지 않는 그룹으로 분할하고자 할 때는 k8s의 네임 스페이스를 활용해볼 수 있다.
		- k8s의 네임스페이스 리눅스의 네임스페이스와는 달리 프로세스를 격리하기 위한 용도로 사용하는 것이 아니다.
		- k8s의 네임스페이스는 오브젝트 이름의 범위를 제공한다.
			- **모든 리소스를 하나의 단일 네임스페이스에 두는 대신 여러 네임스페이스로 분할할 수 있다.**
			- **분리된 네임스페이스는 같은 리소스 이름을 다른 네임스페이스에 걸쳐 여러 번 사용할 수 있게 해준다.**
	- 주의점
		- 여러 사용자 또는 그룹이 동일한 k8s를 사용하고 각자 알아서 리소스를 사용한다고 가정하면, 각각 개인 또는 그룹들의 고유한 네임스페이스를 사용해야한다.
		- 특정 사용자가 지정된 리소스에 접근 할 수 있도록 허용하고 개별 사용자가 사용할 수 있는 컴퓨팅 리소스를 제한하는 데에도 사용된다.

&emsp;&emsp;

### 네임스페이스의 필요성

- 설명
	- 멀티테넌트 환경처럼 리소스를 분리하고 싶은 경우 사용하기 좋다.
		- 리소스를 프로덕션, 개발, QA 환경으로 나누던지 부서별로 나누던지 원하는 방법으로 나눠 사용이 가능해진다.
	- 리소스 이름은 네임스페이스 안에서만 고유하면 된다.
		- 서로 다른 네임스페이스 내부에서 같은 이름의 리소스를 가질 수 있다.
	- 대부분 리소스 타입은 네임스페이스 안에 속하지만 노드와 같은 일부 리소스는 글로벌 리소스이다.


&emsp;&emsp;

### 다른 네임스페이스와 파드 살펴보기

- 설명
	- `k get ns`
		- 클러스터에 존재하는 모든 네임스페이스를 나열한다.
	- `k get po --namespace kube-system`
		- 여기에 나오는 파드들은 k8s 시스템 파드들로 후반부에 자연스레 배울 예정이다.

&emsp;&emsp;

### 네임스페이스 생성

``` yaml
apiVersion: v1
kind: NameSpace
metadata:
  name: custom-namespace
```

- 설명
	- `k create -f custom-namespace.yaml`
		- `k create namespace` 명령으로도 생성이 가능하다.

&emsp;&emsp;

### 다른 네임스페이스의 오브젝트 관리

- 설명
	- 특징
		- 지정 방법은 총 두 가지가 존재한다.
	- 네임스페이스 변경 방법
		- `k config set-context 네임스페이스명 --namespace`
			- 이것을 alias에 지정해서 사용하는 편이다.
	- 지정 방법
		- metadata 섹션에 namespace 필드를 추가해 네임스페이스 이름을 지정
		- `k create -f 오브젝트 파일명 -n 네임스페이스 명`

&emsp;&emsp;

### 네임스페이스가 제공하는 격리

- 설명
	- 특정 네임스페이스에서 다른 네임스페이스간 네트워크 통신이 불가능한지 등등에 대한 격리정도는 k8s와 함께 배포하는 네트워킹 솔루션에 따라 다르지 네임스페이스를 통해서 격리가 되는 것은 아니다.
	- k8s 네트워크 솔루션에서 네임스페이스간 네트워크 격리를 딱히 세팅하지 않았다면 다른 네임스페이스간 ip 주소를 알고 있다면 통신이 충분히 가능하다.

## 파드 중지와 제거

&emsp;&emsp;

### 이름으로 파드 제거하기

- 설명
	- 삭제 명령
		- `k delete po prometheus-gpu`
			- 뒤에 공백을 추가해 여러 파드명을 넣어 복수의 파드를 동시에 제거할 수도 있다.
	- 특징
		- k8s는 파드에 있는 모든 컨테이너에게 SIGTERM 시그널을 프로세스에 보내고 지정된 시간(기본 30초) 동안 대기한다.
		- 시간 내 종료되지 않는 경우 SIGKILL 시그널을 보내 프로세스를 끝낸다.
		- 항상 일반적으로 정상 종료를 시키기 위해서는 SIGTERM으로 끝내는 것이 옳은 방식이다.

&emsp;&emsp;

### 레이블 셀렉터로 파드 제거하기

- 설명
	- `k delete po -l create_method=manual`

&emsp;&emsp;

### 네임스페이스를 제거해 연관된 파드 제거

- 설명
	- `k delete ns custom-namespace`

&emsp;&emsp;

### 네임스페이스를 유지하면서 네임스페이스 안에 있는 모든 파드 삭제

- 설명
	- `k delete po --all`
		- 참고로 레플리카셋 또는 디플로이먼트를 올려둔 상태라면 지워도 새로 생성된다는 것을 볼 수 있다.

&emsp;&emsp;

### 네임스페이스에서 대부분의 리소스 삭제

- 설명
	- `k delete all -all`
		- 현재 네임스페이스에 있는 모든 리소스를 삭제한다.
	- 삭제가 안되는 시크릿 등의 리소스는 직접 명시적으로 삭제해야한다.
