## 목과 스텁 차이점

### 테스트 대역 유형

&emsp;&emsp;

- 설명
	- 테스트 더블 유형
		- 목
			- 목
				- 외부로 나가는 **상호 작용을 모방하고 검사**하는데 도움을 주는 테스트 더블
			- 스파이
				- 객체의 동작을 감시하고 로깅하는 테스트 더블
				- 보통 수동으로 작성하는 편이다. 객체의 어떠한 동작를 감시하느냐에 따라 다르게 작성되기 때문이다.
				- 직접 작성한 목이라고도 부른다.
		- 스텁
			- 스텁
				- 내부로 들어오는 상호 작용을 모방하는 데 도움을 주는 테스트 더블
			- 더미
				- 널 값이나 가짜 문자열과 같이 단순하고 하드코딩된 값으로 구성된 테스트 더블
				- 테스트 적용 대상의 메소드 시그니처를 만족시키기 위해 사용하고 최종 결과를 만드는데에 영향을 주지는 않는다. (당연히 줘서도 안된다.)
			- 페이크
				- 아직 존재하지 않는 의존성을 대체하고자 사용되는 테스트 더블
	- 목의 특징
		- 외부로 나가는 상호 작용을 모방하고 검사하는데 도움을 줌
			- 여기서 상호 작용은 테스트 적용 대상의 상태를 변경하기 위한 의존성을 호출하는 것
	- 목의 예시
		- 이메일 전송 시스템이 테스트 적용 대상인 경우
			- 이메일 발송 기능 테스트에 적용된 테스트 더블은 목으로 SMTP 서버의 상태를 전환한다.
			- 즉, 사이트 이펙트를 일으키는 상호 작용이 목의 적용 대상이다.
	- 스텁의 특징
		- 내부로 들어오는 상호 작용을 모방하는 데 도움을 준다.
			- 여기서 상호 작용은 테스트 적용 대상이 입력 데이터를 얻기 위해 의존성을 호출하는 것
	- 스텁의 예시
		- 이메일 전송 시스템이 테스트 적용 대상인 경우
			- 데이터 검색, 이메일 리스트 탐색등의 기능 테스트에 적용된 테스트 더블은 스텁으로 외부 의존성의 상태를 변경 시키지는 않는다.
			- 즉, 사이드 이펙트를 일으키지 않는 상호 작용이 스텁의 적용 대상이다.

### 도구로서의 목과 테스트 대역으로서의 목

### 스텁으로 상호 작용을 검증하지 말 것

&emsp;&emsp;

- 스텁 사용시 주의점
	- 의존 객체와의 상호 작용 검증을 위해 사용하면 안된다. 테스트가 깨지기 쉬운 안티패턴이 된다.
		- 예시
			- 스텁의 이메일을 전송하는 기능을 테스트 하는 경우, 검증을 스텁을 통해 이메일의 개수를 들고 오는 것(상태를 검증), 이는 최종 결과가 아닌 사항을 검증하는 관행이므로 과잉 명세라고 부른다.
			- 상호 작용을 스텁을 가지고 검사하는 경우 굉장히 쉽게 발견되는 결함이다.
