## 단위 테스트의 정의

+ 설명
	+ 단위 테스트에 대한 견해는 고전파, 런던파 접근으로 나뉜다.
	+ 고전파 (Detriot, 단위 테스트에 대한 classicist 접근 등)
		+ 모든 사람들이 단위 테스트와 TDD에 원론적으로 접근하는 방식
	+ 런던파 (mockist)
		+ 런던의 프로그래밍 커뮤니티에서 시작된 방식
		+ 런던파에게 mock이 가장 핵심이다
	+ 단위 테스트 특징
		+ 작은 코드 조각을 검증한다.
		+ 빠르게 수행된다.
		+ 격리된 방식으로 처리하는 자동화된 테스트이다.
			+ 하나의 클래스가 다른 클래스 또는 여러 클래스에 의존하면 이를 테스트 대역 (test double, 테스트 더블이라고도 부름)으로 대체하여 외부 영향과 분리해 테스트 대상 클래스에만 집중해야한다.
			+ 이러면 코드베이스의 어느 부분이 고장났는지 확연히 알 수 있다.
			+ 객체 그래프로도 분할할 수 있다.

&emsp;&emsp;

### 격리 문제에 대한 런던파 접근

+ 설명
	+ 테스트 더블을 사용해 격리하는 방식

&emsp;&emsp;

#### 예제 (고전파 스타일)

``` Java

void purchase_succeeds_when_enough_inventory(){
	// 준비
	Store store = new Store();
	store.addInventory(Product.Shampoo, 10);
	Customer customer = new Customer();
	// 실행
	bool success = customer.purchase(store, Product.Shampoo, 5);
	// 검증
	AssertThat(success).isTrue();
	AssertThat(store.getInventory).isEqual(5);
}

void purchase_fails_when_not_enough_inventory(){
	// 준비
	Store store = new Store();
	store.addInventory(Product.Shampoo, 10);
	Customer customer = new Customer();
	// 실행
	bool success = customer.purchase(store, Product.Shampoo, 15);
	// 검증
	AssertThat(success).isFalse();
	AssertThat(store.getInventory).isEqual(10);
}

```

+ 설명
	+ 준비
		+ 테스트 대상 시스템 (이하, SUT)과 하나의 협력자를 준비한다.
			+ 고객: SUT
			+ 상점: 협력자
		+ 협력자가 필요한 이유
			+ 고객이 store를 요구하므로 컴파일을 위해 필요함
			+ 검증 단계에서 상점 제품 수량이 감소할 가능성이 있기 때문

+ 결론
	+ 현재 격리가 되어있지 않아 협력자에 문제가 있는 경우 단위 테스트가 정상적으로 동작하지 않는다. 따라서 격리가 필요하다

&emsp;&emsp;

#### 예제

``` Java
@Test  
@DisplayName("재고가 상점에 충분한 경우 구매 성공")  
void purchase_succeeds_when_enough_inventory(){  
    //given  
    Store store = mock(Store.class);  
    when(store.hasEnoughInventory(Product.Shampoo, 5))  
            .thenReturn(true);  
    Customer customer = new Customer();  
    //when  
    boolean success =  customer.purchase(store, Product.Shampoo, 5);  
    //then  
    assertTrue(success);  
    verify(store, times(1)).removeInventory(Product.Shampoo, 5);  
}

@Test  
@DisplayName("재고가 상점에 충분하지 않는 경우 구매 실패")  
void purchase_fails_when_not_enough_inventory(){  
    //given  
    Store store = mock(Store.class);  
    when(store.hasEnoughInventory(Product.Shampoo, 5))  
            .thenReturn(false);  
    Customer customer = new Customer();  
    //when  
    boolean success =  customer.purchase(store, Product.Shampoo, 5);  
    //then  
    assertFalse(success);  
    verify(store, times(0)).removeInventory(Product.Shampoo, 5);  
}
```

+ 설명
	+ 자바 17에서는 vm 옵션을 손봐야해 살짝 애먹었긴 했지만 구현을 한다면 이렇게 할수 있다.
	+ 실제 인스턴스를 사용하지 않고 특정 파라미터가 넘어오면 어떤 값이 넘어올지 미리 셋업을 한다. 
		+ 자바에서는 when을 통해 특정 파라미터가 넘어오면 어떤 값이 넘어올지 저렇게 정하는 것이다.
			+ 내부적으로 cglib을 이용해 프록시를 자체 생성하는 구조이다.
	+ 실제 store 재고를 상태를 수정하지 않고 호출에 어떻게 응답하는지 목에 직접 정의한다.
	+ 예제에서는 더이상 Store를 사용하지 않고 IStore 인터페이스를 사용한다.
		+ 실제로 구체 클래스를 mock으로 만드는 것은 안티패턴이다. 그렇지만 나는 예시로 저렇게 구현하였다.

+ 결론
	+ 고전스타일에서는 상점의 상태를 검증했지만 지금은 고객과 상점과의 상호작용을 검사한다.
		+ 즉, 고객이 상점에서 올바른 호출을 했는지까지도 검증하고 횟수까지도 검증이 가능하다.



### 격리 문제에 대한 고전파 접근

+ 설명
	+ 공유 의존성
		+ 테스트 간에 공유되고 서로의 결과에 영향을 미칠 수 있는 수단
			+ 예시
				+ 정적 가변 필드
	+ 비공개 의존성
		+ 공유하지 않는 의존성
	+ 프로세스 외부 의존성
		+ 애플리케이션 실행 프로세스 외부에 있는 의존성
		+ 경우에 따라 공유 의존성, 비공개 의존성으로 나뉜다.
			+ 예시
				+ 데이터베이스
					+ 공유 의존성을 띄는 프로세스 외부 의존성
				+ 도커 컨테이너로 띄운 테스트용 데이터베이스
					+ 비공개 의존성을 띄는 프로세스 외부 의존성
	+ 공유 의존성 특징
		+ 테스트 대상 클래스들 사이에서 공유되는 것이 아니고 단위 테스트 들 사이에서 공유된다.
			+ 예시
				+ 싱글턴 클래스
					+ 새로운 객체로 주입받아 생성이 가능하다면 이는 공유 의존성이 아니다. 이런 경우에만 공유 의존성을 띄지 않는다고 보고 다른 단위 테스트끼리 공유가 가능하다면 공유 의존성을 띄는 것이다.
				+ 설정 클래스
					+ 이 또한 싱글턴 클래스와 마찬가지이다.
	+ 휘발성 의존성
		+ 공유 의존성과 비슷하지만 다른 의존성
		+ 양자 택일로 들어나는 휘발성 의존성의 특징
			+ 개발자 로컬 머신에 설치된 환경 외의 설정 및 구성을 요구한다.
				+ 예시
					+ 데이터베이스
					+ 외부 api
			+ 비결정적 동작을 포함한다.
				+ 예시
					+ 난수를 생성해 사용함
					+ 현재 날짜를 사용함
		+ 공유 의존성과 함께 바라보는 비교 예시
			+ 데이터베이스
				+ 공유 의존성 true, 휘발성 의존성 true
			+ 파일 시스템
				+ 공유 의존성 true, 휘발성 의존성 false
			+ 난수 발생기
				+ 공유 의존성 false, 휘발성 의존성 true
	+ 공유 의존성을 없애야하는 이유
		1. 공유 의존성이 존재하면 테스트 간의 간섭이 발생한다. 이를 없앨려면 공유 의존성을 없애야한다.
		2. 공유 의존성이 존재하면 테스트의 실행 속도가 내려간다.
			+ 공유 의존성에 대한 호출은 보통 실행 프로세스 외부에 존재하기 때문에 늘 실행속도에 시간이 걸린다.
				+ 비공개 보다 시간이 오래 걸리는 핵심적인 이유이다.
			+ 공유 의존성을 가진 경우 단위 테스트 영역이 아닌 통합 테스트의 영역으로 봐야한다.
	+ 주의점
		+ 굳이 한번에 한 클래스만을 테스트하는 것이 격리라고 볼 수는 없다.
			+ 한 테스트에 여러 클래스가 포함되어도 비공개 의존성을 띈다면 괜찮다.
				+ 이럴땐 오히려 순환 참조를 조심해야한다.

## 단위 테스트에서 바라본 런던파와 고전파

+ 설명
	+ 런던파
		+ 테스트 대상 시스템에서 협력자를 격리함
	+ 고전파
		+ 단위 테스트끼리 격리함

&emsp;&emsp;

| 분파  | 격리주체   | 단위의 크기           | 테스트 대역 사용 대상    |
| --- | ------ | ---------------- | --------------- |
| 런던파 | 단위     | 단일 클래스           | 불변 의존성 외 모든 의존성 |
| 고전파 | 단위 테스트 | 단일 클래스 또는 클래스 세트 | 공유 의존성          |

&emsp;&emsp;

### 런던파와 고전파의 의존성 다루는 법

+ 설명
	+ 런던파
		+ 불변 객체를 제외하고 전부 목으로 교체한다.
			+ 위의 예시를 보면 불변 객체인 enum이였던 Product는 mock으로 바꾸지 않은 모습이다.
			+ 내용에 의해서만 식별되는 값 객체도 포함해서 제외한다.
		+ 공유 의존성 뿐만 아니라 변경 가능한 비공개 의존성에 대해서도 목으로 교체한다.
			+ 예시
				+ 상위 예제의 Store 인스턴스
					+ 변경 가능한 비공개 의존성
				+ 상위 예제의 Product 인스턴스
					+ 변경 불가능한 비공개 의존성
	+ 고전파
		+ 공유 의존성에 대해서만 목으로 교체한다.
	+ 협력자 vs 의존성
		+ 협력자
			+ 공유 하거나 변경 가능한 의존성
			+ 예시
				+ DAO
				+ 상위 예제의 Store 인스턴스
		+ 모든 클래스는 협력자와 값, 두 가지 의존성으로 동작한다.
			+ customer.purchase(store, Product.Shampoo, 5);
				+ 협력자
					+ store
				+ 값
					+ Product.Shampoo
					+ 5
	+ 주의점
		+ 프로세스 외부 의존성의 경우 왠만해서 테스트 대역으로 교체하지만 충분히 연결 안정적이고 빠르면 교체 하지 않는다.
			+ 예시
				+ 파일 시스템의 특정 파일을 읽기용으로만 사용함
		+ 대부분 프로젝트에서 프로세스 외부 의존성치고 공유 의존성이 아닌 경우가 없지만 혼동하지는 말자.
			+ 책에서는 서로 섞어서 사용하긴한다. 그 이유는 아래 항목과 같다.
		+ 공유 의존성이 내부에 있는 경우, 인스턴스만 바꿔서 주입하면 테스트가 편하게 컨트롤 할 수 있기 때문에 해소된다.

## 고전파와 런던파의 비교

+ 설명
	+ 런던파의 장점
		+ 입자성이 좋다.
			+ 세밀한 테스트를 만들어 한 번에 한 클래스만 확인이 가능함
		+ 서로 연결된 클래스의 그래프가 커져도 테스트가 쉽다.
			+ 모든 협력자는 테스트 대역으로 교체되기 때문에 테스트 작성시 걱정이 필요없다.
		+ 테스트 실패시 어느 기능이 문제인지 확인이 매우 쉽다.
			+ 클래스 협력자가 없는 경우 테스트 대상 클래스만 확인하면 된다.
			+ 의존성이 제거된 상태이기 때문에 어느 부분이 문제인지 한 눈에 보기 좋다.

&emsp;&emsp;

### 입자성이 좋다.

+ 설명
	+ 클래스를 단위로 테스트하므로 자연스럽게 클래스를 테스트에서 검증할 원자 단위로 취급한다.
	+ 문제점
		+ 입자성이 좋기 때문에 비즈니스 담당자가 알아보기 매우 힘들다.
			+ 테스트는 코드 단위가 아닌 동작의 단위로 검증을 행해야하지만 입자성이 좋으면 이것이 너무 세밀해져 동작을 알아보기 매우 힘들어진다.
	+ 예시
		+ 응집도가 좋은 경우
			+ 우리집 강아지를 부르면, 나에게 온다.
		+ 응집도가 낮은 경우
			+ 우리집 강아지를 부르면, 왼쪽 앞다리를 먼저 움직이고, 오른쪽 앞다리를 움직인뒤 머리를 돌린다. 그리고 꼬리를 흔들기 시작한다. 그 다음은 ~~ ...

&emsp;&emsp;

### 서로 연결된 클래스의 그래프가 커져도 단위 테스트 하기 쉽다.

+ 설명
	+ SUT에 의존성이 있고 이 의존성이 꼬리물기식으로 의존성이 있는 경우 테스트 하기 쉽다.
	+ 문제점
		+ 애초에 이런 SUT의 설계 구조 자체가 잘못된 것이기 때문에 테스트를 하기 전에 설계 수정이 필요한 상황이다.
			+ 런던파는 입자성이 좋기 때문에 코드 조각을 단위 테스트 집중하려는 의도가 강하다. 즉, 비교적 높은 정확도로 저품질의 코드를 예측성을 가지는 것이다.
			+ 목을 사용하면 이 문제를 감추기만 할 뿐 해결이 되지 않는다.

&emsp;&emsp;

### 테스트 실패시 어느 기능이 문제인지 확인이 매우 쉽다.

+ 설명
	+ SUT에서 버그만 포함된 테스트만을 찾아서 수정하기 쉽다.
	+ 고전파의 경우 하나의 버그가 전체 시스템 테스트의 실패를 야기하기 쉽다.
	+ 문제점
		+ 고전파의 경우 수정 전 버전과 비교하면 어디가 문제인지 한번에 파악하기 쉽다.
			+ 그리고 전체 시스템에 어디에 실패 여파를 주는지 영향 범위 파악에도 도움이 된다. 즉, 이런 경우 보통 고장난 코드 조각이 큰 가치를 증명해낸다.

&emsp;&emsp;

### 고전파와 런던파 사이의 다른 차이점

+ 설명
	1. 테스트 주도 개발을 통한 시스템 설계 방식
		+ 런던파의 경우
			+ 행위에 대한 검증을 하므로 상호작용이 중요하다.
				+ 즉, 인터페이스의 상호작용을 중점으로 만들기 때문에 상위 레벨 테스트 부터 시작한다.
				+ 따라서 하향식 TDD를 진행한다.
			+ 테스트시 SUT의 모든 협력자를 차단해 해당 협력자의 구현을 나중으로 미룰 수 있다.
				+ stub의 경우에는 실제 구체클래스를 활용하므로 다르다. mocking을 의미한다.
		+ 고전파의 경우
			+ 상태에 대한 검증을 하므로 실제 구현된 객체가 중요하다.
				+ 즉, 실제 데이터를 기준으로 만들기 때문에 하위 레벨 도메인부터 다져서 올라간다.
				+ 따라서 상향식 TDD를 진행한다.
	2. 과도한 명세 문제 (테스트 코드 - 프로덕션 코드간 구현 결합도)
		+ 런던파의 경우
			+ 실제 구현에 대한 검증으로써 테스트 내부의 Mock을 기준으로 구현하므로 구현 결합도가 높은 편이다.
			+ 수정된 요구사항이 주어지면, 해당 요구사항에 맞는 응답을 전부 셋업해야하므로 수정에 많은 코스트가 사용된다.
		+ 고전파
			+ 실제 구현에 대한 검증을 구체 클래스를 활용해 진행하므로 구현 결합도가 구체 클래스에 좀 더 비중이 실려있는 편이므로 구현 결합도가 낮은 편이다.
			+ 수정된 요구사항이 주어지면, 해당 요구사항에 맞도록 구체 클래스를 수정하여 테스트를 진행하면 그만이라 런던파보다 적은 코스트가 사용된다.

## 두 분파의 통합 테스트

+ 설명
	+ 런던파의 경우 
		+ 실제 협력자 객체를 하나라도 사용한다면 그건 통합테스트로 간주한다.
	+ 고전파의 경우
		+ 공유 의존성에 접근하는 테스트는 다른 테스트와 분리해 실행할 수 없으므로 이때 통합테스트로 간주한다.
		+ 둘 이상의 동작 단위를 검증하려고 하면 이는 통합 테스트다.

&emsp;&emsp;

### 통합 테스트의 일부인 E2E 테스트

+ 설명
	+ 통합테스트의 정의
		+ 공유 의존성, 프로세스 외부 의존성, 조직 내 다른 팀이 개발한 코드등과 결합해 작동하는지 검증하는 테스트
	+ E2E 테스트
		+ 통합 테스트의 일부로, 프로세스 외부 의존성과 어떻게 동작하는지 검증하는 테스트
		+ 특징
			+ 통합 테스트는 보편적으로 프로세스 외부 의존성을 한 두개만 가지고 동작, E2E 테스트는 프로세스 외부 의존성을 전부 또는 대다수와 함께 작동한다.
				+ 즉, 모든 외부 애플리케이션을 포함해 시스템을 최종 사용자의 관점에서 검증하는 것
				+ 통상 통합 테스트는 DB, 파일시스템까지만 포함한다.
		+ E2E 테스트의 예시
			+ UI 테스트
			+ GUI 테스트
			+ 기능 테스트
		+ 실제 예시
			+ 데이터베이스, 파일 시스템, 결제 게이트웨이로 동작하는 시스템이 있다고 가정하는 경우 
				+ 실제로 결제 게이트웨이의 경우에는 테스트 대역으로 대체해야한다.
					+ 그 이유는 보통 계정 문제, 비용 문제가 가장 크다.
					+ 호출당 비용이 엮이거나 계정이 몇달마다 리프래시를 한다면 통합테스트는 이런 요인까지 관리 코스트로 부담해야한다.
				+ 모든 단위 테스트, 통합 테스트를 통과하면 마지막으로 수행한다. 
					+ 유지 보수 측면에서 너무 많은 코스트가 들기 때문이다.
				+ 모든 외부 프로세스 의존성을 적용하지 못할 수도 있다.
					+ 결제 게이트웨이가 테스트에 불러오지 못한다거나 하는 경우가 발생한다면 통합 테스트에 사용한 테스트 대역을 다시 사용할 수 밖에 없다.
