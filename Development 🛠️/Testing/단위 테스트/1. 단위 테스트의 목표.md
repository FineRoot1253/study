## 단위 테스트 현황

+ 설명
	+ 기업용 어플리케이션의 경우 프로덕션 코드 1 : 테스트 코드 1 ~ 10까지의 비율로 구성되기도 한다.
		+ 높은 비즈니스 복잡도, 긴 프로젝트 수명, 중간 크기의 데이터, 낮은 수준이나 중간 수준 정도의 성능 요구등의 특성을 지녔다.
	+ 테스트 코드를 아무리 열심히 짠다고 해도 잘 짜지 않으면 소용이 없다.


<br><br>

## 단위 테스트의 목표

+ 설명
	+ 단위 테스트와 코드 설계의 관계
		+ 단위 테스트가 어려우면 설계가 잘못된 경우이다.
			+ 보통 강결합이 이루워져 단위 테스트가 힘든 경우이다.
			+ 강결합은 곧 저품질이다.
		+ 단, 저결합이 고품질을 의미하지 않는다는 것도 알아야한다.
	+ **단위 테스트의 목표는 프로젝트를 지속가능한 성장을 이루도록 하는 것이다.**
	+ 개발속도가 빠르게 감소하는 현상을 소프트웨어 엔트로피라고 한다.
		+ 코드 품질을 떨어뜨리는 무질서도를 의미한다.
		+ 테스트와 리팩터링을 통해 이를 해결해야한다.

&emsp;&emsp;

### 좋은 테스트, 나쁜 테스트

+ 설명
	+ **가장 핵심은 모든 테스트를 작성할 필요는 없다.**
	+ 유지보수시 행동강령
		+ 기반 코드 리팩터링시 관련 테스트도 리팩터링할 것
		+ 각 코드 변경시 테스트를 실행할 것
		+ 테스트가 잘못된 경로를 발생시킬 경우 처리할 것
		+ 기반 코드가 어떻게 동작하는지 이해하려고 할 때는 테스트를 읽는 데 시간을 투자할 것
	+ **테스트 작성시 고품질의 좋은 테스트만 작성하도록 집중해야한다.**

## 커버리지 지표

+ 설명
	+ 코드 커버리지
		+ 테스트로 실행된 라인 수와 제품 코드 베이스의 전체 코드 수로 나눈 비율을 의미
		+ 단점
			+ 약간의 조작으로도 손쉽게 눈속임이 가능하다.
	+ 분기 커버리지
		+ 테스트로 실행된 분기 수와 제품 코드 베이스에 존재하는 전체 분기 수로 나눈 비율을 의미
		+ 단점
			+ 외부 라이브러리의 커버리지 지표와 테스트 대상 시스템의 모든 가능한 결과를 추측하는 것은 불가능하다

&emsp;&emsp;

+ 결론
	+ 커버리지는 어디까지나 상태 확인을 위해 활용하는 목적으로만 사용해야한다.
	+ 적어도 60프로 미만은 되지 않게끔 만들면 왠만하면 통과시키는 것이 국룰이다.
	+ 커버리지를 요구사항 기준으로 만들면 프로젝트가 망한다.

## 성공적인 테스트 스위트 특징

+ 설명
	+ **개발 주기 통합**
		+ 자동화가 필수이다.
	+ **코드베이스에서 가장 중요한 부분만**을 대상으로 한다.
		+ 비즈니스 로직 외의 코드 부분
			+ **인프라**
				+ **여기에서 중요한 알고리즘은 반드시 많이 테스트해야한다.**
				+ 단, 도메인 모델에 더 테스트를 많이 하는 것을 기본으로 두어야한다.
			+ 데이터베이스, 서드파티 시스템과 같은 외부 서비스 및 종속성
			+ 모든 것을 하나로 묶는 코드
	+ **최소한의 유지비로 최대의 가치를** 끌어낸다.
		+ 이를 위한 핵심 능력
			+ 가치 있는 테스트, 가치 없는 테스트 골라내기
			+ 가치 있는 테스트만 작성하기
