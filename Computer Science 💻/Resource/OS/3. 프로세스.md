>[!info] Definition
> 프로세스는 실행 중인 프로그램을 의미하며 현대 컴퓨팅 시스템에서의 작업 단위로 사용된다.

&emsp;&emsp;

# 프로세스 개념

&emsp;&emsp;

- 용어 정리
	- 작업(job)
		- 초창기 컴퓨팅 시스템의 작업 단위
		- 스케쥴링에서 범용적으로 사용된다.
	- 프로세스
		- 현대 컴퓨팅 시스템의 단위

&emsp;&emsp;

### 프로세스

&emsp;&emsp;

- 설명
	- 특징
		- 실행중인 프로그램을 비공식적으로 의미함
		- 프로그램의 현재 활동 상태는 프로그램 카운터(PC)와 레지스터 내용으로 나타낸다.
		- 텍스트 및 데이터 섹션의 크기는 고정되어 런타임동안 변치 않는다.
		- 스택 및 힙 섹션은 런타임중 변한다.
		- 함수 호출시 함수 매개변수, 지역 변수 및 복귀 주소를 포함하는 ==**활성화 레코드**==가 스택에 푸시된다.
			- 함수에서 제어가 돌아오면 스택에서 활성화 레코드가 pop된다.
		- ==**프로그램은 프로세스가 아니다.**==
			- 디스크에 저장된 명령어 리스트를 내용으로 가진 파일이다. (==**실행 파일**==이다.)
			- ==**수동적인 존재**==이다.
		- 프로세스는 다음에 실행할 명령어를 지정하는 프로그램 카운터와 관련 자원의 집합을 가진 **==능동적인 존재==**이다.
		- 프로세스는 하나의 프로그램에서 여러 프로세스를 띄울 수 있다.
			- ==**프로세스간 각각의 메모리 영역은 명백히 다르다.**==
	- 프로세스 메모리 배치
		- 힙
		- 스택
			- 함수를 호출할 때 임시 데이터 저장장소
			- 함수 매개 변수, 복귀 주소 및 지역 변수 등이 예시로 들 수 있다.
			- 프로그램 실행 중 동적으로 할당되는 메모리
		- 데이터
			- 전역 변수
		- 텍스트
			- 실행 코드
	- C의 메모리 배치
		- 전역 데이터 섹션에는 초기화된 변수와 아닌 변수를 나누어 따로 섹션을 분리한다.
		- main() 함수에 제공된 매개변수는 스택 세션 상위에 따로 섹션을 분리하여 제공한다.

&emsp;&emsp;

### 프로세스 상태

&emsp;&emsp;

- 설명
	- 종류
		- 생성됨
			- 프로세스가 새로 생성중인 상태
			- 승인이 되면 준비 상태로 넘어간다.
		- 실행중
			- 명령어들이 실행 중인 상태
			- 종료하면 종료 상태로 넘어간다.
			- 인터럽트시 준비 상태로 넘어간다.
			- I/O 또는 이벤트 대기시 대기 상태로 넘어간다.
		- 대기
			- 프로세스가 어떤 이벤트(입출력 완료 또는 신호의 수신과 같은)가 일어나기를 기다리는 상태
			- I/O 또는 이벤트가 완료되면 준비상태로 넘어간다.
		- 준비
			- 프로세스가 처리기에 할당되기를 기다리는 상태
			- 스케줄러 디스패치가 호출하면 실행중 상태로 넘어간다.
		- 종료
			- 프로세스의 실행이 종료된 상태

&emsp;&emsp;

### PCB - 프로세스 제어 블록

&emsp;&emsp;

- 설명
	- 정의
		- OS에서 프로세스를 표현하기 위해 사용되는 표현
		- 약간의 회계 데이터와 함께 프로세스를 시작시키거나 다시 시작시키는데 필요한 모든 데이터를 위한 저장소
	- 구조
		- 프로세스 상태
			- 현재 프로세스의 상태
		- PC
			- 현재 프로세스가 다음에 실행할 명령어의 주소
		- CPU 레지스터
			- 컴퓨터의 구조에 따라 다양한 수와 유형을 가진다.
			- 누산기, 인덱스 레지스터, 스택 레지스터, 범용 레지스터들과 상태 코드 정보가 포함됨
			- ==**PC와 함께**== 이 정보는 ==**나중에 프로세스가 다시 스케줄 될 때 계속 올바르게 실행되도록**== 하기 위해 ==**인터럽트 발생시 저장되어야한다.**==
		- CPU-스케줄링
			- 프로세스 우선순위, 스케쥴 큐에 대한 포인터와 다른 스케쥴 매개변수를 포함한다.
		- 메모리 관리 정보
			- OS에 의해 사용되는 메모리 시스템에 따라 기준 레지스터와 한계 레지스터의 값, OS가 사용하는 메모리 시스템에 따라 테이지 테이블 또는 세그먼트 테이블과 같은 정보
		- 회계 정보
			- CPU 사용시간, 경과된 실시간, 시간 제한,  계정 정보, 잡 또는 PID
		- 입출력 상태 정보
			- 현재 프로세스에 할당된 입출력 장치들과 열린 파일들의 목록

&emsp;&emsp;

### 스레드

&emsp;&emsp;

- 설명
	- 특징
		- 모든 프로세스는 단일 제어 스레드를 지니고 있으며 이 스레드를 통해 프로그램이 수행된다.
		- 한번에 한가지 일만 하도록 허용한다.
		- 프로세스는 한번에 여러 스레드를 가질 수 있어 한번에 여러 일을 할 수 있다.
		- 멀티 스레드를 지원하는 시스템의 PCB는 각 스레드에 관한 정보를 포함하도록 확장된다.

# 프로세스 스케쥴링

&emsp;&emsp;

- 용어 정리
	- 프로세스 스케줄러
		- 코어에서 실행 가능한 여러 프로세스 중에서 하나의 프로세스를 선택한다.
		- 각 CPU 코어는 한 번에 하나의 프로세스를 실행할 수 있다.
		- 다중 코어 시스템의 경우 한 번에 여러 프로세스를 실행 시킬 수 있다.
	- 다중 프로그래밍 정도
		- 현재 메모리의 있는 프로세스의 수
		- 코어 보다 많은 프로세스가 있는 경우 초과 프로세스는 코어의 사용이 가능해지고 다시 스케줄될 때까지 기다려야 한다.
	- I/O 바운드 프로세스
		- 계산보다 I/O에 더 많은 시간을 소비하는 프로세스
	- CPU 바운드 프로세스
		- 계산에 많은 시간을 사용하여 I/O를 자주 요청하지 않는 프로세스

&emsp;&emsp;

### 스케줄링 큐

&emsp;&emsp;

- 설명
	- 스케줄링 구조
		1. 프로세스가 시스템에 들어가면 준비 큐에 들어가서 준비 상태가 되어 CPU 코어에서 실행되길 기다린다.
			1. 큐는 보통 연결리스트로 준비 큐 헤더에는 리스트의 첫 번째 PCB에 대한 포인터가 저장된다. 
			2. 각 PCB에는 준비 큐의 다음 PCB를 가리키는 포인터 필드가 포함된다.
		2. 시스템에는 다른 큐도 존재해 프로세스에 CPU 코어가 할당되면 프로세스는 잠시 실행되어 결국 종료되거나 도중에 인터럽트되거나 I/O 요청의 완료와 같은 특정 이벤트가 발생할 때까지 기다린다.
			1. ==**프로세스보다 디스크 처리속도가 훨씬 느리기 때문에 이때 무조건 디스크가 사용 가능할 때까지 프로세스가 대기해야한다.**==
			2. ==**I/O와 같은 이벤트를 기다리는 프로세스는 대기 큐에 삽입된다.**==
		3. 큐는 각각 대기 큐와 준비 큐가 존재하며 ==**무조건 새 프로세스는 준비큐에 먼저 들어가게 된다**==.
			1. 이 큐에서 프로세스는 디스패치될 때까지 기다린다.
			2. 코어가 할당되고 실행 상태가 되면 여러 이벤트 중 하나가 발생할 수 있다.
	- 실행 상태에서 발생 가능한 이벤트
		- I/O 요청
			- 이벤트 이후 프로세스는 ==**I/O 대기 큐**==에 놓이게 된다.
			- 처리 후 ==**준비 큐**==로 들어간다.
		- 타입슬라이스 종료
			- 이벤트 이후 프로세스가 코어에서 강제로 제거되어 ==**준비 큐**==로 들어간다.
		- 자식 프로세스 생성
			- 이벤트 이후 다음 자식의 종료를 기다리는 동안 ==**자식 종료 대기 큐**==에 놓이게 된다.
			- 처리 후 ==**준비 큐**==로 들어간다.
		- 인터럽트를 기다림
			- 이벤트 이후 프로세스가 코어에서 강제로 제거되어 ==**인터럽트 준비 큐**==로 들어간다.
			- 처리 후 ==**준비 큐**==로 들어간다.
		- 특징
			- 공통적으로 프로세스가 작업을 마치면 다시 코어에 할당될 때까지 대기하기 위해 준비 큐에 모이게 된다.
				- ==**모든 프로세스는 준비 큐를 통한다.**==

&emsp;&emsp;

### CPU 스케줄링

&emsp;&emsp;

- 설명
	- CPU 스케줄러
		- 역할
			- 준비 큐에 있는 프로세스 중 선택된 하나의 프로세스에 CPU 코어 할당
		- 특징
			- 자주 새 프로세스를 선택해야 처리연산 속도가 늘어나는 셈이다.
			- I/O 바운드 프로세스는 I/O 요청을 대기하기 전에 몇 밀리초 동안만 실행 가능하다.
			- CPU 바운드 프로세스는 오랜 시간동안 CPU 코어를 필요로 하지만 스케줄러는 프로세스에게 코어를 장기간 부여할 가능성이 전혀없다.
			- 보통 저 더 자주 선택을 하지만 못해도 100밀리초마다 한 번씩 프로세스 선택을 수행한다.
	- 스와핑
		- 역할
			- 메모리에서 프로세스를 제거해 다중 프로그래밍 정도를 감소 시켜 처리속도를 유리하게 만듬
		- 특징
			- 일부 OS에 존재하는 스케줄링 형태
			- 프로세스를 나중에 메모리에 다시 적재할 수 있고 중단시켰다가 다시 수행할 수도 있다.
			- 프로세스를 메모리에서 디스크로 빼는 것은 스왑 아웃이라고 한다.
			- 프로세스를 디스크에서 메모리로 다시 적재해 복원하는 것을 스왑 인 이라고 한다.
			- 이 방식은 보통 메모리가 모자를때 사용하는 방법이다.
			- 일반적인 스케줄링보단 매우매우 느리지만 개발환경을 강제로 꾸려야하는 특정 환경이라면 하는 방식이다.
				- 배포환경에서는 스와핑은 매우 극단적인 상황을 제외하고서 시도하진 않는게 좋다.
			- 현재 최신 OS들에서는 메모리가 여유가 있어도 스와핑을 시도하는데 이유는 성능 최적화 때문이다.
				- 미리 데이터를 로드하고 다시 스와핑을 하면 덜 사용하는 데이터이지만 나중에 로드하게 된다면 이때 빠르게 프리매칭시킬 수 있기 때문

### 문맥 교환 - Context Switching

&emsp;&emsp;

- 설명
	- 문맥
		- 역할
			- 현재 실행 상태
		- 특징
			- 인터럽트 처리 후 이를 복구하기 위해 인터럽트 발생시 저장된다.
			- 프로세스의 PCB에 표현된다.
			- 커널 모드이건 사용자 모드 이건 CPU의 현재 상태를 저장하는 작업을 수행하고, 연산을 다시 수행하기 위해 상태 복구 작업을 수행한다.
		- 구성
			- CPU 레지스터의 값
			- 프로세스 상태
			- 메모리 관리 정보
	- 문맥 교환
		- 역할
			- 다른 프로세스로 CPU 코어를 교환하기 위해 이전 프로세스의 상태를 보관하고 새로운 프로세스의 보관돤 상태를 북구하는 작업
		- 특징
			- 커널은 과거 프로세스의 문맥을 PCB에 저장하고 실행이 스케줄된 새로운 프로세스를 저장된 문맥을 복구한다.
			- 문맥 교환중 일어나는 작업의 시간은 순전히 오버헤드로 계산된다.
			- 교환 속도는 메모리의 속도, 복사에 필요한 레지스터의 수, 특수 명령어의 존재에 의해 좌우된다.
				- 컴퓨터 마다 다르다는 뜻
				- 보통 마이크로초까지 분포되어있다.
				- 하드웨어 지원에 크게 좌우되므로 일부 처리기는 여러 개의 레지스터 집합을 제공한다.
			- 문맥 교환중에는 현행 레지스터 집합의 포인터를 변경도 한다.
				- 레지스터 집합보다도 활성 프로세스들이 더 많다면 시스템은 전처럼 레지스터 자료를 메모리로 또는 메모리에서 더 많이 복사해야한다.
			- OS가 복잡할수록, 교환시 처리할 작업의 양도 늘어난다.
		- 동작
			1. 프로세스 A에서 인터럽트 또는 시스템 콜을 호출받는다. (실행)
				1. PCB_a에 프로세스 A의 상태를 저장한다. (유휴상태)
				2. PCB_b에서 프로세스 B의 상태를 복원한다. (유휴상태)
			2. 프로세스 B를 수행한다. (실행)
			3. 프로세스 B에서 인터럽트 또는 시스템 콜을 호출 받는다. (실행)
				1. PCB_b에 프로세스 B의 상태를 저장한다. (유휴상태)
				2. PCB_a에서 프로세스 A의 상태를 복원한다. (유휴상태)

### 프로세스 생성

&emsp;&emsp;

- 설명
	- 특징
		- 실행되는 동안 프로세스는 자식을 생성 할 수 있고 자식 또한 자식을 만들 수 있다.
			- 그 결과 프로세스 트리가 형성된다.
		- PID를 이용해 프로세스들을 구분한다.
			- 정수형
			- 프로세스의 속성에 접근하기 위한 인덱스로 사용된다.
		- ps 명령어를 사용하면 현재 시스템에 활성화된 모든 프로세스 정보를 나열해준다.
		- 프로세스가 자식 프로세스를 생성할 때 자원이 필요하다.
			- 자식 프로세스는 이 자원을 OS에서 직접 얻거나 부모 프로세스가 자원을 분할하여 나누어주거나, 메모리나 파일과 같은 몇몇 자원들은 자식 프로세스들이 같이 사용하게 할 수 있다.(공유 메모리 통신)
			- 자식 프로세스는 부모에게서 자원과 특권, 스케줄링 속성까지 상속받는다.
		- 부모 프로세스 자원을 일부분만 사용하도록 자식 프로세스가 쓸 수 있게 제한할 수 있다.
			- 이는 자식 프로세스를 많이 생성해 시스템을 과부하하는 것을 방지한다.
		- 초기화시 부모는 자식에게 데이터를 넘겨줄 수 있다.
			- 두 프로세스 사이에 열린 파일을 두고 데이터를 주고 받게끔도 할 수 있다.
		- 자식 프로세스 생성 후 수행시 부모의 이후 동작
			- 자식과 병행 동작
			- 일부 또는 모든 자식이 끝날 때까지 대기
				- wait() 시스템 콜을 사용해 임의의 수행이 종료된 자식을 삭제한다.
				- exec()를 호출하면 프로세스의 주소 곡간을 새 프로그램으로 덮어쓰기 때문에  exec() 시스템 콜은 오류가 발생하지 않은 한 제어를 반환하지 않는다.
		- 자식 프로세스의 주소 공간
			- 부모 프로세스의 복사본
				- 특징
					- 쉽게 부모와 자식간 통신을 할 수 있다.
					- fork() 시스템 콜의 반환 값이 다르다.
						- 부모는 자식의 pid, 자식은 0을 반환한다.
			- 자식 프로세스가 자신에게 적재될 프로그램을 가지고 있음
				- 특징
					- 자식 실행 중 exec() 시스템 콜을 호출해 메모리 공간을 새로운 프로그램으로 교체하는 방식
					- exec() 시스템 콜은 이진 파일을 메모리로 적재하고 실행한다.
						- 이때 exec() 시스템 콜을 포함하는 원래 프로그램의 메모리 이미지는 파과한다.
		- 윈도우의 경우 자식 프로세스를 새로 생성하면 주소 공간에 명시된 프로그램을 적재한다.
			- 10개 이상의 매개변수까지 요구한다.
	- sytemd 프로세스
		- 역할
			- 모든 사용자 프로세스의 부모 프로세스 
		- 특징

			- 시스템이 부트될 때 생성되는 첫 번짜 사용자 프로세스이다.
			- 시스템이 부팅되면 다양한 사용자 프로세스를 생성한다.

		- logind 프로세스

# 프로세스 간 통신

&emsp;&emsp;

- 설명
	- 프로세스 특징
		- 대다수 프로세스들은 독립적이다.
			- 메모리 공간이 전부 나뉘어져 있기 때문
		- 프로세스끼리 통신시켜 협력하게 만든 프로세스도 존재한다.
	- 프로세스 협력을 제공하는 이유
		- 정보 공유
			- 여러 응용 프로그램들이 동일한 정보에 흥미를 느낄 수 있다면, 그 정보를 병행적으로 접근 가능하게 만들어야한다.
		- 계산 가속화
			- 만약 특정 테스크는 빠르게 실행해야하고 무거운 일처리는 뒤로 미뤄야한다면 이를 서브 태스크로 나눠 병렬로 실행되게 해야한다.
		- 모듈성
			- 시스템의 기능을 프로세스별로 나눠 또는 스레드 별로 나눠 모듈식 형태로 시스템을 구성한다면 필요하다.
				- 예를 들면 브라우저는 탭이 전부 자식 프로세스이다.
	- IPC
		- 역할
			- 프로세스 간 통신, 협력적 프로세스들간 데이터를 교환할 수 있는 기법
		- 모델 종류
			- 공유 메모리 모델
				- 특징
					- 협력 프로세스들에 의해 공유되는 메모리의 영역이 구축된다.
					- 이 영역에 데이터를 읽고 써 정보를 교환한다.
					- 공유 메모리 영역을 확보하는 데에만 시스템 콜을 호출한다.
						- 메모리 영역 구축 이후 커널의 도움은 필요 없다.
			- 메시지 전달
				- 특징
					- 통신이 협력 프로세스들 사이에 교환되는 메시지를 통하여 이루어진다.
					- 충돌을 회피할 이유가 없어 적은 양의 데이터를 교환하는데 적절하다.
					- 시스템 콜을 이용해 구현 되므로 커널 간섭등의 부가적인 시간 소비가 필요해 소요시간은 공유 메모리 모델보다 느리다.

# 공유 메모리 시스템에서의 프로세스 간 통신

&emsp;&emsp;

>[!info] Definition
> 공유 메모리 IPC는 프로세스간 공유 메모리 영역을 구축해 통신하는 방식이다.

&emsp;&emsp;

- 설명
	- 특징
		- 공유 메모리 영역은 메모리 세그먼트를 생성하는 프로세스의 주소공간에 위치한다.
			- 공유 메모리 세그먼트를 이용하여 ==**통신하고자 하는 다른 프로세스들은 이 세그먼트를 자신의 주소 공간에 추가**==하여야 한다.
		- 데이터의 형식은 프로세스들에 의해 결정된다.
		- ==**프로세스들은 동시에 동일한 위치에 쓰지 않도록 책임져야한다.**==
	- 예시
		- 생산자 - 소비자 프로세스
			- 해결책
				- 생산자가 정보를 채우고, 소비자가 소모할 수 있는 항목들의 버퍼가 반드시 사용 가능해야한다.
					- 생산자가 한 항목을 생산하고, 소비자는 다른 항목을 소비하고 있어야 병행 수행이 가능하다.
					- 생산자와 소비자가 동기화되어야 생산되지도 않는 항목들을 소비자가 소비하려고 시도하지 않는다.
	- 두가지 유형의 버퍼
		- 무한 버퍼
			- 생산자 소비자 문제에서는 버퍼의 크기가 실질적인 한계가 없다.
			- 소비자는 새 항목을 기다릴 때도 있지만 생산자는 항상 새로운 항목을 생산한다.
		- 유한 버퍼
			- 버퍼의 크기가 고정되어 있는 경우 
				- 버퍼가 ==**비어져 있으면**== ==**소비자는 무조건 대기**==해야한다.
				- 버퍼가 ==**채워져 있으면**== ==**생산자는 무조건 대기**==해야한다.
			- 보통 유한 버퍼를 이용해 구현하는 것이 일반적이며 어떻게 구현하는지는 개발자 마음이다.

# 메시지 전달 시스템에서의 프로세스 간 통신

&emsp;&emsp;

>[!info] Definition
> 메시지 전달 시스템은 프로세스간 메시지를 전달해 데이터를 교환하는 방식

&emsp;&emsp;

- 설명
	- 장점
		- 주소 공간을 따로 마련할 필요가 없다.
		- 분산 환경에서 매우 유리하다.
	- 특징
		- send()와 receive() 두 가지 연산을 제공한다.
		- 메시지는 고정길이이거나 가변 길이이다.
			- 고정 길이는 시스템 수준 구현이 직선적이지만 개발하기 힘들다.
			- 가변 길이가 개발하기 편하다.
		- 메시지 전송에는 통신 연결이 설정되어야 한다.
			- 이 구현은 매우 다양하다.
	- 구현 방법 종류
		- 직접, 간접 통신
		- 동기식, 비동기식 통신
		- 자동, 명시적 버퍼링

&emsp;&emsp;

### 명명 - naming

&emsp;&emsp;

- 설명
	-  직접 통신
		- 특징
			- 송수신자 이름을 명시해야한다.
			- `send(P, message)`, `receive(Q, message)`
				- 통신을 원하는 각 프로세스의 쌍들 사이에 연결이 자동으로 구축됨
					- 상대방의 신원만 알면 구축된다.
				- 연결은 정확히 두 프로세스 사이에만 연관됨
				- 통신하는 프로세스들의 각 쌍 사이에는 정확하게 하나의 연결이 존재해야한다.
				- 대칭성을 보이는 방식이다.
			- `send(P, message)`, `receive(id, message)`
				- 임의의 프로세스로부터 수신을 받는 방식
				- 비대칭적인 방식이다.
			- 하드 코딩 기법으로 모듈성이 떨어지는 기법이다.
	- 간접 통신
		- 특징
			- 메일박스 또는 port로 송신되고 그것으로부터 수신된다.
				- 메일박스는 프로세스들에 의해 메시지들이 넣어지고 제거되는 객체이다.
				- 각 메일박스는 id를 가진다.
					- Posix 메시지 큐는 메일박스 식별을 위해 정수형 데이터를 이용한다.
			- 프로세스는 다수의 상이한 메일박스를 통해 다른 프로세스들과 통신할 수 있다.
			- `send(A, message)`, `receive(A, message)`
				- 한 쌍의 프로세스들 사이의 연결은 이들 프로세스가 공유 메일박스를 가질 때만 구축된다.
				- 연결은 두 개 이상의 프로세스들과 연관될 수 있다.
				- 통신하고있는 각 프로세스 사이에는 다수의 서로 다른 연결이 존재할 수 있다.
					- 각 연결은 하나의 메일박스에 대응된다.
			- 메일 박스는 한 프로세스나 운영체제에 의해 소유될 수 있다.
				- 메일박스가 한 프로세스에 의해 소유된다면 우리는 소유자와 메일박스의 사용자를 구분할 수 있다.
				- 각 메일박스가 고유한 소유자를 가지고 있어 이 메일박스로 보내진 메시지를 어느 프로세스가 수신할지 혼란에 빠지진 않는다.
				- 메일 박스를 소유한 프로세스가 종료되면 메일박스는 사라진다.
					- 그 후 메일박스로 메시지를 송신하는 모든 프로세스는 더는 메일박스가 존재하지 않는다는 알림을 받아야한다.
			- 운영체제가 소유한 메일박스는 자체적으로 존재한다.
				- 이는 독립적인 것으로 어떤 특정한 프로세스에 예속되지 않는다.
			- 메일박스를 생성하는 프로세스는 그 메일박스의 기본 소유자가 된다.
			- 초기에는 소유자만 메일박스를 통해 수신할 수 있지만 소유권과 수신 특권은 적절한 시스템 콜을 통해 다른 프로세스에 전달될 수 있다.
				- 이 규칙으로 인해 메일박스마다 복수의 수신자들을 만들 수 있다.
		- 복수의 프로세스의 수신 여부에 따른 기법
			- 하나의 링크는 최대 두 개의 프로세스와 연관되도록 허용
			- 한순간에 최대로 하나의 프로세스가 receive() 연산을 실행하도록 허용
			- 어느 프로세스가 수신할 것인지 시스템이 임의로 선택
				- 시스템이 송신자에게 수신자를 알려 주는 방식도 있음
		- 운영체제가 프로세스에 제공해야하는 기능
			- 새로운 메일 박스 생성
			- 메일박스를 통해 메시지 송수신
			- 메일박스 삭제

&emsp;&emsp;

### 동기화

&emsp;&emsp;

- 설명
	- 프리미티브를 구현하기 위한 설계 옵션
		- 블로킹 (동기식)
			- 보내기
				- 송신하는 프로세스는 수신 프로세스 또는 메일박스에 의해 수신될 때까지 대기
			- 받기
				- 메시지가 이용 가능할 때까지 수신 프로세스가 대기된다.
		- 논블로킹 (비동기식)
			- 보내기
				- 송신하는 프로세스는 메시지를 보내고 작업을 다시 수행한다.
			- 받기
				- 송신하는 프로세스가 유효한 메시지 또는 null을 받는다.
	- 블로킹 특징
		- 송신자와 수신자 간에 랑데부를 한다.
		- 생산자와 소비자 문제는 사소한 문제이다.
			- 생산자는 메시지를 전달될때까지 대기하고 소비자는 메시지가 수신될때까지 대기한다.

&emsp;&emsp;

### 버퍼링

&emsp;&emsp;

- 설명
	- 특징
		- 통신 프로세스들의 메시지는 임시 큐에 들어가 있다.
	- 임시큐 구현 방법 종류
		- 무용량
			- 큐의 최대길이가 0
			- 링크는 자체 안에서 대기하는 메시지를 가질 수 없음
			- 송신자는 수신자가 메시지를 수신할 때까지 대기한다.
		- 유한 용량
			- 유한한 n의 길이를 가짐
				- n개의 메시지를 가질 수 있음을 의미
			- 큐가 꽉찬 것이 아니면 메시지는 큐에 놓이며, 송신자는 대기하지 않고 실행을 계속함
				- (메시지는 복사되거나 메시지에 대한 포인터가 유지된다.)
			- 링크는 유한한 용량이므로 만원인 경우에는 무용량과 동일하게 송신지는 수신자가 수신할 때까지 대기해야한다.
		- 무한 용량
			- 무한한 길이를 가진다.
			- 송신자는 절대 블럭되지 않는다.

# IPC 시스템의 사례

&emsp;&emsp;

### POSIX 공유 메모리

&emsp;&emsp;

- 설명
	- 특징
		- 메모리-매핑 파일을 이용해 구현한다.
		- 메모리 매핑 파일은 공유 메모리의 특정 영역을 파일과 연관시킨다.
		- `MAP_SHARED` 플래그 값은 공유 메모리 객체에 변경이 발생하면 객체를 공유 하는 모든 프로세스에 최신의 값을 접근하게 된다는 것을 의미한다.
		- 쓰기 작업은 `sprinft()`함수를 호출하고 쓰기 작업 후 바이트 크기만큼 포인터를 증가시켜야 한다.
	- API
		- `int shm_open(name, O_CREAT | O_RDWR, 0666); 
			- 역할
				- 공유 메모리 영역을 파일과 연관시킨다.
			- 특징
				- fd 값을 성공시 반환한다. 실패시 -1을 반환한다.
				- 첫번째 인자에는 공유 메모리 객체 이름을 넣는다.
				- 두번째 인자에는 객체가 존재하지 않으면 생성되고, 읽기쓰기 가능 상태로 열으라는 것을 의미한다.
				- 마지막 인자에는 공유 메모리 객체에 파일-접근 허가권을 부여한다.
		- `ftruncate(fd, 4096);`
			- 역할
				- 객체의 크기를 바이트 단위로 설정한다.
			- 특징
				- 첫번째 인자는 fd, 두번째 인자는 크기를 넣는다.
		- `nmap()`
			- 역할
				- 공유 메모리 객체에 접근할 때 사용할 메모리 매핑 파일의 포인터를 반환
		- `shm_unlink(name)`
			- 역할
				- 공유 메모리를 제거한다.

&emsp;&emsp;

### UNIX 파이프

&emsp;&emsp;

>[!info] Definition
> 파이프는 두 프로세스를 통신할 수 있게 하는 전달자로서 동작하는 유닉스 IPC 기법 중 하나이다.

&emsp;&emsp;

- 설명
	- 파이프 구현시 옵션
		- 단방향 or 양방향
		- 양방향시 반이중인지, 전이중인지
		- 통신하는 두 프로세스 간 부모-자식과 같은 특정 관계가 존재해야만 하는지
		- 파이프는 네트워크를 통하여 통신이 가능한지, 동일한 기계 안에서 두 프로세스끼리만 가능한지

&emsp;&emsp;

#### 일반 파이프

&emsp;&emsp;

- 설명
	- 특징
		- 생산자 - 소비자 형태
		- 두 프로세스 간의 통신을 허용
		- 생산자는 파이프의 한 종단에 쓰고, 소비자는 다른 종단에서 읽는다.
		- 한쪽으로만 데이터를 전송할 수 있는 단방형 통신만을 가능하게 한다.
		- 양방향 통신을 원한다면 파이프 두 개를 사용해야한다.
		- UNIX는 파이프를 파일의 특수한 타입으로 취급한다.
			- read()와 write() 시스템 콜로 접근이 가능하다.
		- 일반 파이프는 파이프 생성 프로세스 외 접근 불가능하다.
			- 통상 파이프 생성 후 자식 프로세스와 소통하기 위해 사용된다.
			- 자식은 메모리 영역을 복사하기 때문에 파이프까지도 복사한다.
				- tcp에서 정리했지만 fork() 이후 fd를 닫았던 이유도 복사되기 때문이였는데 역으로 파이프는 이점을 이용해서 파이프를 연결하는 것이다.
		- 윈도우에서는 익명 파이프가 유닉스의 일반 파이프와 같은 역할을 한다.
	- API
		- `pipe(int fd[]);`
			- 역할
				- 배열을 통해 파이프를 설정한다.
			- 특징
				- 0번 파일은 읽기, 1번파일은 쓰기로 사용된다.

&emsp;&emsp;

#### 지명 파이프

&emsp;&emsp;

- 설명
	- 특징
		- 양방향
			- 단 반이중만 가능하다.
		- 프로세스의 부모-자식 관계 조건이 필요 없음
		- 지명 파이프 구축시 다수의 프로세스가 사용 가능
		- 통신 프로세스를 종료해도 파이프는 존재함
		- UNIX에서는 FIFO라고 한다.
		- 전이중통신이 필요하다면 2개를 쓰면 된다.
		- `|`명령어를 cmd 창에서 사용하면 이것또한 파이프와 동일한 역할을 한다.
			- 주로 앞 명령어의 출력을 뒤 명령어에 입력으로 전달하는데 사용한다.
	- API
		- `mkfifo()`
			- 역할
				- 지명 파이프 생성
			- 특징
				- open(), read(), write(), close()와 같은 시스템 콜 사용 가능
				- 명시적으로 파일시스템에서 삭제될 때까지 존재한다.
				- 반이중 전송만 가능하다.
