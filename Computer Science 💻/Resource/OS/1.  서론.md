# 서론

>[!info]
> 운영체제(Operating System)는 컴퓨터 하드웨어를 관리하는 소프트웨어이다.

+ 설명
	+ 역할
		+ 응용 프로그램을 위한 기반 제공
		+ 컴퓨터 사용자와 컴퓨터 하드웨어 사이의 중재자
	+ 특징
		+ 광범위한 컴퓨팅 환경에서 본인 역할을 매우 다양한 방법으로 수행함
		+ 근본적인 책임은 CPU, 메모리 및 입출력 장치, 저장장치와 같은 자원들을 프로그램에 할당하는 책임이다.
		+ 덩치가 매우 크고 복잡해 부분별로 생성되어야 한다.
			+ 하나의 부분은 전체 시스템 윤곽에 잘 맞아야 하고, 각 부분들의 입력과 출력, 동작은 주의를 기울여 정의해야한다.
	+ 서론의 목표
		+ 컴퓨터 시스템의 일반적인 구성과 인터럽트 역할을 기술
		+ 현대 다중 처리기 컴퓨터 시스템의 구성요소에 관해 기술
		+ 사용자 모드 - 커널 모드로의 전환에 대해 설명
		+ 다양한 컴퓨팅 환경에서 OS의 사용 방법에 대해 논의
		+ 무료 및 공개 소스 운영체제의 예시

## 사용자 관점

+ 설명
	+ 특징
		+ 사용되는 인터페이스에 의존된다.
			+ 랩탑, 모니터, 키보드, 마우스등 PC 앞에서 작업한다.
				+ 보통 사용자가 수행하는 작업을 최대화하는 것에 목표를 두고 설계된 시스템들이다.
		+ 성능에는 약간만 신경을 쓰고 사용의 용이성을 위해 설계된다.
		+ 다양한 하드웨어와 소프트웨어 자원이 어떻게 공유되느냐의 자원의 이용에는 전혀 신경을 쓰지 않는다.
			+ 모바일의 경우 셀룰러와 같은 무선 통신 기술에 연결되고, 사용자는 터치 스크린을 사용해 시스템과 상호작용한다.
				+ 음성인식도 보편적이다.
		+ 사용자 관점이 배제된 시스템도 많다.
			+ 자동차나 가전제품의 내장형 컴퓨터(임베디드)는 상태를 표현하는 아주 단순한 표시등과 같은 인터페이스만을 가지고 있고 이 컴퓨터나 운영체제와 응용 프로그램은 사용자의 개입 없이 작동되도록 설계되어있다.




&emsp;&emsp;

## 시스템 관점

+ 설명
	+ 관점에 따른 두 가지 분류
		+ OS를 자원 할당자(resource controller)로 보는 경우
			+ 특징
				+ OS를 컴퓨터의 관점에서 하드웨어와 가장 밀접한 프로그램으로 본다.
				+ OS는 여러가지 컴퓨터 시스템이 가지는 자원들의 관리자로 동작한다.
					+ 이때 여러 요청들이 상충하게 되는데 공정하게 운영할 수 있도록 요청에 따른 자원 할당 결정들을 도맡고 있다.
		+ OS를 제어 프로그램(control program)으로 보는 경우
			+ 특징
				+ 여러 가지 입출력 장치와 사용자 프로그램을 제어하는 책임 위주로 보는 경우
				+ 제어 프로그램은 컴퓨터의 부적절한 사용을 방지하기 위해 사용자 프로그램의 수행을 제어한다.
					+ 특히, 입출력 장치의 제어와 작동에 깊이 관여한다.


&emsp;&emsp;

## 운영체제의 정의

>[!info] Definition
> 운영체제는 일반적으로 적합한 정의가 없고 운영체제에 포함되는 요소에도 보편적인 정의는 없다.

+ 설명
	+ 오늘날 컴퓨터 시스템은 다양한 환경에 존재하고 각 다양한 환경에서 딱 필요한 만큼 존재하는 다양한 하드웨어에 사용자를 위해 만들어진 응용 프로그램들, 이 둘 사이에 공통적으로 자원을 제어하는 하나의 소프트웨어로 설명할 수 있다.
	+ 가장 일반적인 정의는 운영체제는 컴퓨터에서 항시 실행되는 프로그램(이하 커널)이다.
		+ 커널과 함께 두 가지 다른 프로그램은 시스템 프로그램과 시스템 작동과 관련 없는 응용 프로그램이다.
	+ 오늘날 모바일 기기는 그래픽, 멀티미디어, 데이터베이스를 지원하는 미들웨어를 함께 포함해 핵심 커널이 포함되어 있다.
	+ 정리
		+ **운영체제는 컴퓨터에서 항시 실행되는 프로그램(이하 커널)이다.**
		+ **운영체제를 포함하는 요소에는 커널, 응용 프로그램 개발을 쉽게 하기 위해 기능 제공을 하는 미들웨어 프레임워크 및 시스템 실행 중 시스템을 관리하느 데 도움이 되는 시스템 프로그램으로 구성된다.**

&emsp;&emsp;

>[!info]
> 운영체제의 기본적인 지식들 운영체제를 작성하는 사람들에게는 필수적이고 프로그램을 작성하는 사람들에게도 매우 유용하다.
> 
> 따라서 운영체제를 공부하는 이유는 컴퓨터 과학에 종사하는 사람이라면 배워야하는 것이 정상적인 학습 흐름이다.

# 컴퓨터 시스템의 구성

+ 설명
	+ 현대의 범용 컴퓨터 시스템 구성
		+ CPU
		+ 공유 메모리
		+ 공통 버스
		+ 각 장치의 컨트롤러
			+ 역할
				+ 제어 하는 주변 장치와 로컬 버퍼 저장소 간에 데이터 이동
			+ 장치 드라이버
				+ 역할
					- OS의 장치 컨트롤러의 작동을 돕기 위해 인터페이스를 제공
				- 특징
					- CPU와 장치 컨트롤러는 병렬로 수행된다.
						- 이때 메모리 사이클을 놓고 경쟁한다.
							- 이때 공유 메모리를 질서 있게 엑세스 하기 위해 메모리 컨트롤러가 액세스에 대한 동기화를 제공한다.






&emsp;&emsp;

## 인터럽트

&emsp;&emsp;

- 설명
	- 입출력 작업을 예시
		- 장치 드라이버
			- 데이터 적재시 서순
				1. 장치 컨트롤러의 적절한 레지스터 값을 적재
				2. 레지스터의 내용을 검사해 수행할 작업을 결정
					- 예: 키보드에서 문자 읽기 수행
				3. 장치에서 로컬 버퍼로 데이터 전송
				4. 데이터 전송 후 ==**컨트롤러는 드라이버에게 작업 완료를 알림**==
					+ ==**이 알림을 인터럽트로 알리는 것**==
			- 읽기 요청시 데이터 또는 데이터에 대한 포인터를 반환하여 OS의 다른 부분에 제어를 넘김
	- 주요 목적
		- 비동기 이벤트를 처리하기 위해 사용된다.



&emsp;&emsp;

### 개요

&emsp;&emsp;

- 설명
	- 특징
		- 하드웨어는 어느 순간이든 ==**시스템 버스를 통해 CPU에 신호를 보내 인터럽트 발생**==이 가능하다.
		- 인터럽트는 많은 목적으로 사용되며 ==**OS와 하드웨어의 상호 작용 방식의 핵심**==이다.
		- 대부분 컴퓨터 시스템은 인터럽트 구조를 공통적으로 가지고 있다.
		- 적절한 서비스 루틴으로 제어를 전달한다.
			- 이런 전달을 관리하는 직선적인 방법은 인터럽트 정보를 조사하는 일반적인 루틴을 호출하는 방법이다.
			- 이 루틴은 인터럽트 고유 핸들러를 호출한다.
				- 아주 빠르게 수행되어야한다.
					- 그래서 ==**인터럽트 루틴에 대한 포인터들의 테이블**==을 대신 이용할 수 있다.
					- ==**테이블을 통해 간접 인터럽트 루틴 호출이 가능**==하다.
					- 테이블은 ==**하위 메모리에 저장**==된다. (처음 100개 정도에 위치)
		- 테이블에는 여러 장치에 관한 인터럽트 서비스 루틴 주소가 들어있다.
			- 이 주소의 배열인 ==**인터럽트 벡터**==는 인터럽트 요청과 함께 주어진 ==**고유의 유일한 장치 번호로 인덱스**== 된다.
			- 윈도우, 유닉스도 이 방식을 사용한다.
		- 인터럽트시 프로그램 카운터에 반드시 상태를 저장하여 복귀 전에 상태를 복원해야한다.
	- CPU 인터럽트시 동작
		1. 인터럽트 발생시 하던일을 중단한다.
		2. 고정된 위치에서 인터럽트 수행을 위한 서비스 루틴 시작 주소를 빼고 실행하던 일을 옮긴다. (프로그램 카운터)
		3. 서비스 루틴 시작 주소를 통해 일을 수행한다.
		4. 인터럽트 서비스 루틴 완료시 인터럽트 되었던 연산을 다시 수행한다.

&emsp;&emsp;

### 구현

&emsp;&emsp;

- 설명
	- 기본 인터럽트 매커니즘 동작
		1. 인터럽트 요청 라인 선을 통해 CPU는 명령어 하나를 수행할 때마다 감지를 수행
		2. 컨트롤러에서 보낸 요청을 감지하면, 인터럽트 번호를 읽고 이 번호를 인터럽트 벡터의 인덱스로 사용해 인터럽트 핸들러 루틴으로 점프한다. (`interrupt_vector[interruptNum]` 이런 식이라는 뜻, ==**디스패치**==한다고 부른다.)
		3. 인터럽트 서비스 루틴 주소에서 실행을 시작
			1. 인터럽트 처리기는 작업 중 변경될 상태를 저장, 인터럽트 원인을 확인, 필요한 처리 수행, 상태 복원 수행, `return_from_interrupt` 명령어를 실행해 CPU를 인터럽트 전 상태로 복원한다.
	- 동작 정리
		- 컨트롤러가 인터셉터 요청 라인에 요청을 선언하여 인터럽트를 발생, CPU는 포착, 인터럽트 핸들러로 디스패치, 핸들러는 장치를 서비스하여 인터럽트를 지운다.
	- 최신 운영체제에서 제공하는 필요한 인터럽트 특성
		- 중요한 처리 중 인터럽트 처리를 연기할 수 있어야한다.
		- 장치의 적절한 인터럽트 핸들러로 효율적으로 디스패치할 방법이 필요하다.
		- 운영체제가 우선순위가 높은 인터럽트와 우선순위가 낮은 인터럽트를 구분하고 적절한 긴급도로 대응할 수 있도록 다단계 인터럽트가 필요하다.
	- 인터럽트 요청라인
		- 마스크 불가능 인터럽트
			- 역할
				- 복구 불능한 메모리 오류와 같은 이벤트를 위해 예약된 요청라인
		- 마스크 가능 인터럽트
			- 역할
				- 장치 컨트롤러가 서비스를 요청하기 위해 사용된다.
	- 인터럽트 동작 종류
		- 벡터 방식 인터럽트 기법
			- 장점
				- 서비스가 필요한 장치를 결정하기 위해 하나의 인터럽트 핸들러가 가능한 모든 인터럽트 자원을 검색할 필요가 줄어든다.
			- 단점
				- 인터럽트 백터보다 많은 장치가 연결된 경우 문제가 생긴다.
		- 인터럽트 체인 기법
			- 특징
				- 인터럽트 벡터의 각 원소는 인터럽트 핸들러 리스트의 헤드를 가리켜 연결리스트를 구성해 요청을 처리할 수 있는 핸들러가 발견될 때까지 상응하는 핸들러를 하나씩 호출한다.
			- 장점
				- 인터럽트 백터보다 많은 장치가 연결된 경우 보완을 해준다.
			- 단점
				- 벡터 방식보단 핸들러를 찾는데 시간이 걸린다.
	- 인터럽트 레벨
		- 특징
			- 모든 인터럽트를 마스킹하지 않고 우선순위가 낮은 인터럽트를 연기할 수 있고 우선순위가 높은 인터럽트가 우선순위가 낮은 인터럽트를 선점할 수 있다.
	- Intel 인터럽트 백터
		- 0
			- 나눗셈 버그
		- 1
			- 디버그 예외
		- 2
			- Null 인터럽트
		- 3
			- 중단점(breakpoint;)
		- 4
			- INTO-검출 오버플로
		- 5
			- 제한 범위 예외
		- 6
			- 유효하지 않는 opcode
		- 7
			- 장치를 가용하지 않음
		- 8
